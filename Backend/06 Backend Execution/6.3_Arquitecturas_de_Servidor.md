# 6.3 Arquitecturas de Servidor

## Definición

Cómo se organizan listeners, acceptors, y readers determina el throughput máximo del servidor. Desde un solo thread que hace todo hasta socket sharding con SO_REUSEPORT, cada variación optimiza un aspecto diferente del pipeline de conexiones.

## Mecánica interna

### Single Listener + Single Worker

```
  [Listener Thread]
    accept() → read() → process() → write()

  Todo secuencial. Un cliente a la vez.
```

### Single Listener + Thread Pool

```
  [Listener Thread]
    client = accept()
    pool.submit(handle, client)

  [Worker Thread 1]: read → process → write
  [Worker Thread 2]: read → process → write
  ...

  Bottleneck: accept() es serializado en un solo thread.
  A ~50K accepts/sec, el listener se satura.
```

### Multiple Acceptors (SO_REUSEPORT)

```
  Sin SO_REUSEPORT:
  ┌──────────────┐
  │ Socket :8080  │ ← una accept queue
  │  accept()     │ ← lock contention entre threads
  └──────────────┘

  Con SO_REUSEPORT (Linux 3.9+):
  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
  │ Socket :8080  │  │ Socket :8080  │  │ Socket :8080  │
  │ Thread 1      │  │ Thread 2      │  │ Thread 3      │
  │ accept queue 1│  │ accept queue 2│  │ accept queue 3│
  └──────────────┘  └──────────────┘  └──────────────┘

  Kernel distribuye conexiones entrantes entre las queues.
  Zero lock contention. Cada thread tiene su propia queue.

  Usado por: Nginx, Envoy, HAProxy.
```

### Pre-fork Model (Apache)

```
  Master process
    │
    ├── fork() → Worker process 1 (listen + accept + handle)
    ├── fork() → Worker process 2
    ├── fork() → Worker process 3
    └── fork() → Worker process N

  Cada worker es un proceso independiente con su propia memoria.
  Aislamiento total: si un worker crashea, los demás continúan.
  Costo: más memoria (no comparten heap).

  Nginx variation: master + N workers, cada worker = event loop.
```

### Nginx Architecture

```
  ┌───────────────────────────────┐
  │         Master Process        │
  │  - Lee config                 │
  │  - Maneja signals             │
  │  - Fork workers               │
  └──────────────┬────────────────┘
                 │
     ┌───────────┼───────────┐
     ▼           ▼           ▼
  [Worker 1]  [Worker 2]  [Worker N]    N = CPU cores
  event loop  event loop  event loop
  epoll       epoll       epoll

  Cada worker: single-threaded event loop.
  SO_REUSEPORT: cada worker tiene su accept queue.
  Resultado: N event loops, zero lock contention, máximo throughput.
```

## Modelo mental

- **Single listener**: un recepcionista que también atiende a los clientes.
- **Listener + pool**: un recepcionista que asigna clientes a empleados.
- **SO_REUSEPORT**: múltiples puertas de entrada, cada una con su propia recepción.
- **Pre-fork**: múltiples sucursales independientes de la misma empresa.

## Modos de fallo

| Síntoma | Arquitectura | Causa |
|---------|-------------|-------|
| Accept rate limitado | Single listener | Un thread no puede accept() lo suficientemente rápido |
| Uneven load entre workers | SO_REUSEPORT sin CBPF | Kernel distribución puede ser desigual |
| Thundering herd | Multiple acceptors sin EPOLLEXCLUSIVE | Todos los workers despiertan por una conexión |

## Insights de ingeniería

- **Worker count = CPU cores**: para event loop workers (Nginx, Envoy), usar más workers que cores causa context switching sin beneficio. Para thread pool workers (Tomcat), usar más threads que cores es correcto si son I/O-bound.
- **EPOLLEXCLUSIVE (Linux 4.5+)**: con múltiples threads haciendo epoll_wait en el mismo fd, solo uno despierta. Resuelve thundering herd.
- **SO_REUSEPORT es casi gratuito**: activarlo en un servidor existente puede mejorar throughput 2-3x sin cambio de código significativo.
