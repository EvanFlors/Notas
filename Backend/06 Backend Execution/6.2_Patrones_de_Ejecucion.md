# 6.2 Patrones de Ejecución

## Definición

El patrón de ejecución define cómo tu servidor asigna recursos (threads, procesos, event loops) para manejar conexiones concurrentes. Es la decisión arquitectónica que determina cuántas conexiones simultáneas puede manejar y cómo utiliza CPU y memoria.

## Mecánica interna

### Single Thread Blocking

```
  while (true):
    client = accept()        ← bloquea hasta nueva conexión
    data = read(client)      ← bloquea hasta datos
    response = process(data)
    write(client, response)
    close(client)

  1 cliente a la vez. Mientras procesa request 1, nadie más puede conectar.
  Uso: prototipos, herramientas CLI, nunca en producción.
```

### Thread-per-Connection

```
  while (true):
    client = accept()
    spawn_thread(handle, client)    ← nuevo thread por cada conexión

  function handle(client):
    data = read(client)              ← cada thread bloquea independientemente
    response = process(data)
    write(client, response)
    close(client)

  10K conexiones = 10K threads.
  Cada thread: ~1MB stack (default) = 10GB RAM solo en stacks.
  Context switching entre 10K threads destruye performance.

  Uso: Apache prefork, PHP-FPM (process-per-request), servlets legacy.
```

### Thread Pool

```
  ┌────────────────────────────────┐
  │         Thread Pool (N=100)    │
  │  [T1: busy] [T2: idle] ...    │
  │                                │
  │  Task Queue:                   │
  │  [client5] [client6] [client7] │ ← esperan un thread libre
  └────────────────────────────────┘

  while (true):
    client = accept()
    thread_pool.submit(handle, client)

  Ventajas: limita recursos, evita crear/destruir threads.
  Desventajas: pool exhaustion si las tasks son lentas.

  Uso: Tomcat, Spring Boot (default), base de datos pooling.
```

### Event Loop (Async I/O)

```
  while (true):
    events = epoll_wait(epfd)      ← bloquea hasta algún fd tenga actividad
    for event in events:
      if event == NEW_CONNECTION:
        client = accept()
        register(client, epfd)
      if event == DATA_READY:
        data = read(event.fd)       ← non-blocking (datos ya disponibles)
        response = process(data)    ← DEBE ser non-blocking
        write(event.fd, response)

  Un solo thread maneja miles de conexiones.
  CERO context switching (no hay threads extra).
  PERO: process() DEBE ser rápido. Si tarda 100ms, TODAS las conexiones esperan.

  Uso: Node.js, Nginx, Redis, Envoy.
```

### Reactor Pattern

```
  ┌─────────────────────────────────────┐
  │            Reactor                   │
  │                                     │
  │  Demultiplexer (epoll)              │
  │       │                             │
  │       ▼                             │
  │  Dispatcher                         │
  │    ├── AcceptHandler (new conn)     │
  │    ├── ReadHandler (data ready)     │
  │    └── WriteHandler (writable)     │
  └─────────────────────────────────────┘

  Single-threaded reactor: Node.js
  Multi-threaded reactor: Netty (1 reactor per thread)
```

### Proactor Pattern

```
  App submits async operation → kernel executes → app receives completion event

  1. App: "lee 1KB del socket X" (submit)
  2. Kernel: ejecuta el read en background
  3. Kernel: "terminé, aquí están los datos" (completion)
  4. App: procesa los datos

  Implementaciones: Windows IOCP, Linux io_uring.
  Diferencia con Reactor: en Reactor, el kernel dice "hay datos listos, tú lee".
  En Proactor, el kernel dice "ya leí los datos, aquí están".
```

### Comparación

| Patrón | Conexiones | Memoria | CPU overhead | Complejidad | Ejemplo |
|--------|------------|---------|-------------|-------------|---------|
| Thread-per-conn | ~1K | Alta | Alto (context switch) | Baja | Apache |
| Thread pool | ~10K | Media | Medio | Media | Tomcat |
| Event loop | ~100K+ | Baja | Bajo | Alta | Node.js, Nginx |
| Event loop + thread pool | ~100K+ | Media | Bajo | Alta | Node.js workers |

## Modelo mental

- **Thread-per-connection**: un empleado dedicado por cliente. Personal, pero costoso.
- **Thread pool**: N empleados compartidos entre clientes. Cola de espera cuando todos están ocupados.
- **Event loop**: un empleado con un panel de luces. Atiende solo a quien tiene algo listo. Extremadamente eficiente si cada tarea es rápida.

## Modos de fallo

| Síntoma | Patrón | Causa |
|---------|--------|-------|
| Thread exhaustion, new connections rejected | Thread pool | Todas las tasks bloqueadas en I/O lento |
| Event loop lag, latencia alta uniforme | Event loop | Operación CPU-bound en el event loop |
| OOM con muchas conexiones | Thread-per-conn | 1MB stack × N connections |

## Insights de ingeniería

- **Hybrid es el estándar moderno**: event loop para I/O + thread pool para CPU-bound tasks. Node.js usa libuv (event loop) + worker_threads. Nginx usa event loop + thread pool para disk I/O.
- **Go runtime**: goroutines son green threads con M:N scheduling. Parecen thread-per-connection (simple de programar) pero internamente usan event loop + thread pool (eficiente). Lo mejor de ambos mundos.
- **Thread pool sizing**: para I/O-bound tasks, pool size > CPU cores (threads pasan tiempo esperando I/O). Para CPU-bound tasks, pool size = CPU cores.
