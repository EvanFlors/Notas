# 6.1 Lifecycle de una Request

## Definición

Cada request HTTP que llega a tu servidor pasa por una cadena precisa de operaciones a nivel de sistema operativo: desde la llegada del paquete a la NIC hasta la respuesta escrita al socket. Entender este ciclo completo te permite identificar exactamente dónde está el bottleneck.

## Mecánica interna

```
  1. PAQUETE LLEGA A LA NIC
     NIC recibe frame Ethernet → DMA copia a ring buffer en RAM
     NIC genera interrupción → kernel procesa

  2. KERNEL NETWORK STACK
     Ethernet → IP → TCP → socket buffer (receive queue)
     Si es SYN → SYN queue (half-open)
     Si completa handshake → accept queue

  3. ACCEPT
     App llama accept() → kernel retorna nuevo fd del accept queue
     Ahora el servidor tiene un socket fd para este cliente

  4. READ
     App llama read(client_fd) → kernel copia datos del socket buffer a user space
     Puede ser blocking (espera datos) o non-blocking (retorna EAGAIN)

  5. PARSE
     Bytes crudos → HTTP request object
     Extraer method, URL, headers, body

  6. ROUTE
     URL + method → handler function
     Extraer path params, query params

  7. MIDDLEWARE
     Auth → logging → rate limit → CORS → validation

  8. PROCESS (handler)
     Lógica de negocio: query DB, call service, compute
     Esta es la parte que TÚ escribes

  9. SERIALIZE
     Response object → bytes (JSON, Protobuf)

  10. WRITE
      App llama write(client_fd, response_bytes)
      Kernel copia de user space a socket send buffer
      TCP envía los datos (segmentos, ACKs, congestion control)

  11. CLOSE (o keep-alive)
      close(client_fd) → TCP 4-way teardown
      O mantener abierto para la siguiente request (HTTP keep-alive)
```

### Dónde está el bottleneck

```
  Paso        Quién controla     Bottleneck típico
  ─────────────────────────────────────────────────
  accept()    Kernel + app       Accept queue llena (backlog bajo)
  read()      Kernel             Client slow-send, red lenta
  parse       App                Payload muy grande
  route       App                Raro (O(1) o O(log n) en la mayoría de routers)
  middleware  App                Auth pesado (DB lookup, JWT verify)
  process     App + downstream   Slow query, downstream timeout
  serialize   App                Payload grande, serialización ineficiente
  write()     Kernel + red       Client slow-read, TCP window shrinks
```

## Modelo mental

Una request es como un paquete en una línea de ensamblaje: llega (accept), se abre (read/parse), se procesa (handler), se empaca (serialize), y se envía (write). El throughput del sistema es el del paso más lento.

## Modos de fallo

| Síntoma | Paso | Causa |
|---------|------|-------|
| Connection refused | accept | Accept queue llena |
| Slow TTFB | process | Slow query o downstream lento |
| Slow download | write | Client en red lenta, TCP window shrinks |
| High CPU, baja latencia | parse/serialize | Payload grande con JSON parsing |

## Insights de ingeniería

- **Profile cada paso**: no asumas dónde está el bottleneck. Instrumenta: tiempo en middleware, tiempo en DB, tiempo en serialización. El 80% de la latencia suele estar en el paso de process (DB queries).
- **Streaming para payloads grandes**: no leas todo el body en memoria. Para uploads de archivos, stream directamente a disco o storage (S3). Para responses grandes, usa chunked transfer.
- **Keep-alive reduce accept overhead**: en HTTP/1.1 con keep-alive, el paso 1-3 se ejecuta solo una vez por conexión, no por request.
