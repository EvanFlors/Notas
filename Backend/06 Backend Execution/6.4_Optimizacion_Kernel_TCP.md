# 6.4 Optimización Kernel y TCP

## Definición

Configuraciones a nivel de kernel y TCP que impactan directamente la latencia y throughput de tu servidor. Nagle's algorithm, delayed ACK, zero-copy, y buffer tuning son controles que la mayoría de desarrolladores ignoran pero que pueden marcar diferencias de 10-100x en escenarios específicos.

## Mecánica interna

### Nagle's Algorithm + Delayed ACK

```
  Nagle (sender): "no envíes un segmento pequeño si hay datos
  pendientes de ACK. Espera a tener más datos o el ACK."

  Delayed ACK (receiver): "no envíes ACK inmediatamente.
  Espera 40ms o hasta que tengas datos para enviar junto con el ACK."

  Problema: los dos esperan al otro.

  Server envía response pequeña (200 bytes)
  → Nagle retiene (hay datos pendientes de ACK)
  → Client delayed ACK espera 40ms
  → 40ms de latencia fantasma

  Solución: TCP_NODELAY = desactiva Nagle.
  Todos los HTTP servers modernos activan TCP_NODELAY.
```

### Zero-copy: sendfile()

```
  Envío normal (4 copies):
  Disk → kernel buffer → user buffer → kernel socket buffer → NIC
  2 context switches (user ↔ kernel)

  sendfile() (2 copies):
  Disk → kernel buffer → NIC (con DMA scatter-gather: 0 CPU copies)
  0 context switches extra

  Uso: servir archivos estáticos (Nginx usa sendfile por defecto).
  10x throughput para archivos grandes.
```

### Buffer Tuning

```bash
# TCP send/receive buffers
sysctl net.ipv4.tcp_wmem    # min default max (send buffer per socket)
sysctl net.ipv4.tcp_rmem    # min default max (receive buffer per socket)

# Valores para servidores de alto throughput:
net.ipv4.tcp_wmem = 4096 65536 16777216    # 4KB / 64KB / 16MB
net.ipv4.tcp_rmem = 4096 65536 16777216

# Backlog queues
net.core.somaxconn = 4096              # Accept queue max
net.ipv4.tcp_max_syn_backlog = 8192    # SYN queue max
```

### Backpressure

```
  Producer rápido → Consumer lento

  Sin backpressure:
  [producer] →→→→→→→→→ [buffer crece infinitamente] → [consumer lento]
  → OOM crash

  Con backpressure:
  TCP flow control: receiver reduce window → sender frena
  App level: queue full → reject/drop nuevos mensajes
  HTTP: 503 Service Unavailable + Retry-After
```

## Modelo mental

Nagle + Delayed ACK es como dos personas educadas en una puerta: "después de usted" — "no, después de usted". TCP_NODELAY dice: "pase primero sin esperar". Zero-copy es enviar un paquete sin abrirlo y re-empacarlo en cada oficina postal.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| 40ms de latencia en responses pequeñas | Nagle + Delayed ACK | `TCP_NODELAY` |
| Alto CPU sirviendo archivos estáticos | Copias de datos user↔kernel | `sendfile()` (Nginx: `sendfile on`) |
| Throughput bajo en WAN | Buffers TCP default demasiado pequeños | Aumentar `tcp_wmem`/`tcp_rmem` |
| Connection refused bajo carga | `somaxconn` o `tcp_max_syn_backlog` bajo | Aumentar ambos |

## Insights de ingeniería

- **TCP_NODELAY es default en HTTP servers modernos**: pero si escribes un server TCP custom o usas una librería de bajo nivel, verifica que esté activado.
- **Nginx sendfile + tcp_nopush**: `sendfile on` + `tcp_nopush on` = Nginx acumula headers + datos en un solo segmento TCP, luego envía con sendfile. Combina lo mejor de Nagle (eficiencia) con lo mejor de sendfile (zero-copy).
- **Buffer sizing para BDP**: el tamaño óptimo del buffer TCP = Bandwidth × Delay (BDP). Para un enlace de 1Gbps con 50ms RTT: 1Gbps × 50ms = 6.25MB. Buffers menores limitan throughput.
