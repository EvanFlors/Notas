# 11.4 Profiling

## Definición

Profiling identifica dónde tu código gasta tiempo de CPU y memoria. Flame graphs visualizan call stacks, mostrando las funciones "hot" (que consumen más recursos). Continuous profiling en producción permite detectar regresiones de rendimiento sin reproducir en dev.

## Mecánica interna

### CPU Profiling

```
  Flame Graph:

  main() ─────────────────────────────────────
  ├── handleRequest() ─────────────────────
  │   ├── parseJSON() ────────              ← 20% CPU
  │   ├── queryDB() ───────────────         ← 40% CPU (hot path)
  │   │   └── serialize() ───────
  │   └── writeResponse() ──
  └── healthCheck() ─

  Ancho = tiempo de CPU. Más ancho = más CPU.
  Stack de abajo (main) a arriba (funciones específicas).
  Busca funciones anchas en la parte superior = oportunidad de optimización.
```

### Memory Profiling

```
  Heap snapshot: qué objetos están en memoria y cuánto ocupan.

  Útil para detectar:
  - Memory leaks: objetos que crecen indefinidamente
  - Excessive allocation: crear/destruir demasiados objetos temporales
  - Large objects: buffers o caches más grandes de lo esperado

  Herramientas:
  - Go: pprof (go tool pprof)
  - Node.js: --inspect + Chrome DevTools
  - Python: tracemalloc, memory_profiler
  - Java: JFR (Java Flight Recorder), async-profiler
```

### Continuous Profiling

```
  Profiling en producción, siempre activo, bajo overhead (~2% CPU).

  App → [Profiling agent] → [Pyroscope / Parca / Datadog CP]

  Permite:
  - Comparar flame graphs entre deploys (¿esta versión es más lenta?)
  - Detectar regresiones de rendimiento automáticamente
  - Investigar incidentes pasados ("¿qué consumía CPU hace 3 horas?")
```

## Insights de ingeniería

- **Profile en producción, no en dev**: el comportamiento bajo carga real es diferente. Los datos reales, la concurrencia real, y el traffic pattern real producen hot paths diferentes que en tests.
- **JSON parsing aparece en muchos flame graphs**: si tu servicio procesa mucho JSON, serialización/deserialización puede consumir 10-30% del CPU. Considera Protobuf para inter-service.
- **Memory leak detection**: toma heap snapshots periódicos y compara. Si un tipo de objeto crece monotónicamente, tienes un leak.
