# 11.1 Logs

## Definición

Logs son registros textuales de eventos en tu aplicación. Structured logging (JSON) con campos consistentes permite búsqueda, filtrado, y correlación eficiente. Correlation IDs conectan logs de un mismo request a través de múltiples servicios.

## Mecánica interna

### Structured Logging

```
  Unstructured (no usar):
  2024-01-15 10:30:45 ERROR Failed to process order 42 for user alice

  Structured (JSON):
  {
    "timestamp": "2024-01-15T10:30:45.123Z",
    "level": "error",
    "message": "Failed to process order",
    "service": "orders-api",
    "request_id": "req-abc-123",
    "user_id": "alice",
    "order_id": 42,
    "error": "PaymentDeclined",
    "duration_ms": 234
  }

  Ventajas: filtrable (level=error AND service=orders-api),
  indexable, parseable automáticamente por ELK/Loki.
```

### Log Levels

| Level | Cuándo usar | Ejemplo |
|-------|-------------|---------|
| DEBUG | Desarrollo, troubleshooting fino | "Query executed: SELECT..." |
| INFO | Eventos normales de negocio | "Order created: id=42" |
| WARN | Algo inesperado pero recuperable | "Retry #2 to payment service" |
| ERROR | Fallo que requiere atención | "Payment failed: timeout" |
| FATAL | Sistema no puede continuar | "Database connection lost" |

### Correlation IDs

```
  API Gateway genera: X-Request-Id: req-abc-123

  Service A (log): {"request_id": "req-abc-123", "msg": "received order"}
  Service A → Service B: propaga X-Request-Id header
  Service B (log): {"request_id": "req-abc-123", "msg": "checking inventory"}
  Service B → Service C: propaga X-Request-Id header
  Service C (log): {"request_id": "req-abc-123", "msg": "charging payment"}

  Búsqueda en Kibana: request_id:"req-abc-123" → todos los logs de ese request.
```

### Centralización

```
  Servicios → [Log shipper: Fluentd/Filebeat] → [Storage: Elasticsearch/Loki]
           → [Dashboard: Kibana/Grafana]

  ELK Stack: Elasticsearch (storage+search) + Logstash (processing) + Kibana (UI)
  Loki + Grafana: más ligero, indexa solo labels (no full-text), mejor con K8s.
```

## Insights de ingeniería

- **Log lo necesario, no todo**: DEBUG logs en producción generan terabytes de datos. Usa INFO para eventos de negocio, ERROR para fallos. Habilita DEBUG solo para troubleshooting.
- **No loguees datos sensibles**: passwords, tokens, credit card numbers, PII sin sanitizar. Usa masking: `"email": "a***@example.com"`.
- **Log rotation**: sin rotation, los logs llenan el disco → servicio crashea. Configura logrotate o deja que el log shipper maneje la rotación.
- **Correlation ID desde el edge**: el primer servicio que recibe la request (API Gateway) genera el ID. Todos los downstream lo propagan. Sin esto, debugging en microservicios es imposible.
