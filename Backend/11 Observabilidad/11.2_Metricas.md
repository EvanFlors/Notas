# 11.2 Métricas

## Definición

Métricas son valores numéricos que representan el estado del sistema en el tiempo. Permiten alerting, dashboards, y capacity planning. Los dos frameworks clave son RED (para servicios) y USE (para recursos).

## Mecánica interna

### RED Method (para servicios)

```
  Rate:     requests por segundo
  Errors:   porcentaje de requests fallidas
  Duration: latencia (p50, p95, p99)

  Dashboard de un servicio:
  ┌─────────────────────────────────┐
  │ Rate:     1,200 req/s           │
  │ Errors:   0.5% (5xx)           │
  │ Duration: p50=12ms p99=250ms   │
  └─────────────────────────────────┘
```

### USE Method (para recursos)

```
  Utilization: % del tiempo que el recurso está ocupado
  Saturation:  trabajo en cola (queue depth)
  Errors:      errores del recurso

  Aplicar a: CPU, memoria, disco, red, connection pools.

  CPU:  util=75%, saturation=load_avg>cores, errors=0
  Disk: util=90%, saturation=iowait, errors=disk_errors
  Pool: util=80/100 connections, saturation=5 waiting, errors=timeouts
```

### Tipos de métricas

| Tipo | Descripción | Ejemplo |
|------|-------------|---------|
| Counter | Solo incrementa | Total de requests, total de errores |
| Gauge | Sube y baja | Conexiones activas, temperatura, queue depth |
| Histogram | Distribución de valores | Latencia de requests (buckets: <10ms, <50ms, <100ms) |
| Summary | Percentiles calculados en el cliente | p50, p95, p99 de latencia |

### SLI / SLO / SLA

```
  SLI (Service Level Indicator): métrica que mide calidad.
    Ejemplo: "proporción de requests con latencia < 200ms"

  SLO (Service Level Objective): target para el SLI.
    Ejemplo: "99.9% de requests con latencia < 200ms por mes"

  SLA (Service Level Agreement): contrato con consecuencias.
    Ejemplo: "si SLO se incumple, descuento del 10% en la factura"

  Error budget: 100% - SLO = margen de error.
    99.9% SLO → 0.1% error budget → 43.8 min de downtime/mes permitido.
```

### Prometheus + Grafana

```
  App expone métricas: GET /metrics → texto plano

  # TYPE http_requests_total counter
  http_requests_total{method="GET",status="200"} 1234
  http_requests_total{method="POST",status="500"} 5

  # TYPE http_request_duration_seconds histogram
  http_request_duration_seconds_bucket{le="0.01"} 500
  http_request_duration_seconds_bucket{le="0.1"} 900
  http_request_duration_seconds_bucket{le="1"} 990

  Prometheus scrapes /metrics cada 15s → almacena time series.
  Grafana consulta Prometheus → dashboards.
  Alertmanager evalúa reglas → notifica (Slack, PagerDuty).
```

## Insights de ingeniería

- **p99 > p50**: un p50 de 10ms con p99 de 2s significa que 1% de tus usuarios sufren. El p99 es lo que importa para la experiencia real.
- **Error budgets para velocidad de desarrollo**: si tienes error budget disponible, puedes deployar más rápido (asumiendo riesgo). Si el budget se agotó, priorizas estabilidad sobre features.
- **Four Golden Signals (Google SRE)**: latency, traffic, errors, saturation. Equivalente práctico a RED + USE.
- **Alertar sobre síntomas, no causas**: alerta si "latencia p99 > 500ms" (síntoma), no si "CPU > 80%" (causa). El CPU alto no siempre es un problema.
