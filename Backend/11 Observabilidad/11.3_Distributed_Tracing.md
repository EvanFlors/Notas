# 11.3 Distributed Tracing

## Definición

Distributed tracing sigue un request a través de múltiples servicios, mostrando cada paso (span), su duración, y las dependencias. Es la herramienta para responder "¿por qué esta request tardó 3 segundos?" en un sistema con 10 microservicios.

## Mecánica interna

### Conceptos

```
  Trace: el viaje completo de un request (colección de spans)
  Span: una operación individual (e.g., "query a PostgreSQL")

  Trace ID: abc-123 (único por request, propagado a todos los servicios)

  API Gateway [span: 0-500ms] ─────────────────────
    └── Users Service [span: 10-100ms] ────────
    │     └── PostgreSQL query [span: 20-80ms] ──
    └── Orders Service [span: 100-480ms] ──────
          └── Redis cache [span: 110-115ms] ──
          └── PostgreSQL query [span: 120-450ms] ── ← bottleneck!

  Cada span tiene: trace_id, span_id, parent_span_id, start_time, duration, tags.
```

### Context Propagation

```
  Service A genera trace_id, lo pasa como header:
  traceparent: 00-abc123-span1-01

  Service B recibe el header, crea un child span con el mismo trace_id.
  Service C recibe de B, crea otro child span.

  W3C Trace Context es el estándar (traceparent header).
```

### OpenTelemetry

```
  SDK unificado para traces + metrics + logs.

  App code:
  tracer = opentelemetry.trace.get_tracer("my-service")
  with tracer.start_as_current_span("process-order") as span:
      span.set_attribute("order_id", 42)
      result = db.query(...)
      # child span creado automáticamente por la instrumentación de DB

  Collector:
  App → [OTel Collector] → Jaeger/Tempo/Zipkin (storage + UI)
```

### Sampling

```
  100% de traces = demasiados datos y costo de storage.

  Head-based: decide al inicio (sample 10% de requests)
  + Simple, predecible
  - Puede perder traces de errores/outliers

  Tail-based: decide al final (sample 100% pero solo almacena errores/slow)
  + Captura outliers
  - Más complejo, requiere buffering
```

## Insights de ingeniería

- **Tracing responde "por qué"**: logs dicen "qué pasó", métricas dicen "cuánto", tracing dice "por qué fue lento y dónde".
- **Auto-instrumentation**: OTel SDKs instrumentan automáticamente HTTP clients, DB drivers, y frameworks. Mínimo código manual.
- **Tail-based sampling para producción**: captura 100% de traces de errores y requests lentas (>p99), sample 1% de requests normales. Máximo valor, mínimo costo.
