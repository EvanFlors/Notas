# 10.1 Autenticación y Autorización

## Definición

Autenticación (AuthN) verifica quién eres. Autorización (AuthZ) verifica qué puedes hacer. Son procesos separados: puedes estar autenticado pero no autorizado para un recurso.

## Mecánica interna

### JWT (JSON Web Token)

```
  header.payload.signature

  Header: {"alg": "RS256", "typ": "JWT"}
  Payload: {"sub": "user_42", "role": "admin", "exp": 1704067200}
  Signature: RS256(base64(header) + "." + base64(payload), private_key)

  Verificación: el servidor verifica la firma con la public key.
  No necesita DB lookup → stateless.

  Access token: corto (15 min), no revocable.
  Refresh token: largo (7 días), almacenado en DB, revocable.

  Flow:
  1. Login → server emite access_token + refresh_token
  2. Requests → Authorization: Bearer <access_token>
  3. Access token expira → POST /auth/refresh + refresh_token → nuevo access_token
  4. Logout → invalidar refresh_token en DB
```

### OAuth 2.0 (Authorization Code Flow + PKCE)

```
  Usuario → App → Authorization Server (Google, GitHub)

  1. App redirige a: auth.google.com/authorize?
     response_type=code&client_id=...&redirect_uri=...&code_challenge=...
  2. Usuario se autentica con Google
  3. Google redirige a: app.com/callback?code=ABC123
  4. App intercambia code por token:
     POST auth.google.com/token
     {code: "ABC123", code_verifier: "...", client_id: "...", client_secret: "..."}
  5. Google retorna: {access_token: "...", id_token: "..."}

  PKCE: previene intercepción del authorization code.
  Client credentials flow: para service-to-service (sin usuario).
```

### Session-based Auth

```
  1. POST /login {username, password}
  2. Server crea session en Redis: "sess:abc123" → {user_id: 42}
  3. Response: Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Strict
  4. Requests subsecuentes: Cookie: session_id=abc123
  5. Server verifica en Redis → user_id: 42
  6. Logout: DELETE sess:abc123 de Redis

  HttpOnly: JavaScript no puede leer la cookie (previene XSS theft)
  Secure: solo se envía sobre HTTPS
  SameSite=Strict: previene CSRF
```

### mTLS (service-to-service)

```
  Cada servicio tiene un certificado. Ambos lados verifican.
  No necesitas JWT ni API keys entre servicios internos.
  Service mesh (Istio) automatiza emisión y rotación de certificados.
```

### Comparación

| Mecanismo | Stateless | Revocable | Complejidad | Uso |
|-----------|-----------|-----------|-------------|-----|
| JWT | Sí | No (sin blocklist) | Media | APIs, mobile |
| Sessions | No (Redis) | Sí | Baja | Web apps |
| API Keys | Sí | Sí (DB check) | Baja | Third-party APIs |
| OAuth 2.0 | N/A (delegated) | Sí | Alta | Login con Google/GitHub |
| mTLS | Sí | Sí (revoke cert) | Alta | Service-to-service |

## Insights de ingeniería

- **JWT access + refresh token**: el pattern más usado en APIs modernas. Access token corto (no necesita DB check), refresh token largo (revocable en DB).
- **Never store JWT in localStorage**: vulnerable a XSS. Usa httpOnly cookies para web apps. Para mobile/SPA, httpOnly cookie con SameSite=Strict.
- **Authorization check en CADA endpoint**: no asumas que porque el usuario está autenticado, puede acceder a cualquier recurso. `GET /users/42/orders` → ¿user_id del token == 42?
