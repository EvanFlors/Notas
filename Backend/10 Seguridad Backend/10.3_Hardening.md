# 10.3 Hardening

## Definición

Hardening es el proceso de reducir la superficie de ataque de tu sistema: configurar TLS correctamente, gestionar secretos de forma segura, aplicar security headers, escanear dependencias, y aplicar el principio de menor privilegio.

## Mecánica interna

### Security Headers

```
  # Configurar en Nginx o en la aplicación:

  Strict-Transport-Security: max-age=31536000; includeSubDomains
  → Fuerza HTTPS por 1 año. Browser rechaza HTTP.

  Content-Security-Policy: default-src 'self'; script-src 'self'
  → Solo ejecuta scripts del mismo dominio. Bloquea XSS inline.

  X-Content-Type-Options: nosniff
  → Previene MIME-type sniffing.

  X-Frame-Options: DENY
  → Previene clickjacking (tu página en un iframe).

  Referrer-Policy: strict-origin-when-cross-origin
  → Controla qué info se envía en el Referer header.

  Permissions-Policy: camera=(), microphone=()
  → Bloquea APIs del browser no necesarias.
```

### Secrets Management

```
  NUNCA:
  - Secrets en código fuente (git history es permanente)
  - Secrets en variables de entorno sin cifrar en CI/CD
  - Secrets compartidos por Slack/email

  SIEMPRE:
  - HashiCorp Vault, AWS Secrets Manager, GCP Secret Manager
  - Rotación automática de secrets
  - Acceso auditado (quién accedió a qué secret, cuándo)
  - Diferentes secrets por environment (dev ≠ staging ≠ prod)
```

### Principle of Least Privilege

```
  - DB user de la app: solo SELECT, INSERT, UPDATE en sus tablas.
    No SUPERUSER, no DROP, no acceso a otras DBs.
  - Container: non-root user, read-only filesystem, no capabilities extra.
  - IAM roles: solo los permisos necesarios. No AdministratorAccess.
  - Network: security groups que solo permiten puertos necesarios.
```

### Dependency Scanning

```bash
  # npm
  npm audit

  # Python
  pip-audit
  safety check

  # Go
  govulncheck ./...

  # CI/CD: Dependabot (GitHub), Snyk, Trivy
  # Escanea dependencias por CVEs conocidos automáticamente.
```

## Insights de ingeniería

- **Security headers toman 5 minutos y previenen clases enteras de ataques**: HSTS previene downgrade, CSP previene XSS, X-Frame-Options previene clickjacking. No hay excusa para no tenerlos.
- **Git history es permanente**: si commiteaste un secret, rotarlo inmediatamente. `git filter-branch` o BFG Repo-Cleaner pueden eliminarlo del history, pero asume que ya fue comprometido.
- **Automated scanning en CI/CD**: ejecuta `npm audit`, `trivy` (containers), y SAST (static analysis) en cada PR. Los CVEs se descubren constantemente en dependencias.
- **Input validation en CADA capa**: no confíes en la validación del frontend. Valida en el API gateway, en el handler, y en la base de datos (constraints). Defense in depth.
