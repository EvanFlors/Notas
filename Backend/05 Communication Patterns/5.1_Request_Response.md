# 5.1 Request/Response

## Definición

El patrón request/response es el modelo fundamental de comunicación en backend: el cliente envía un request con toda la información necesaria, el servidor procesa y retorna un response. Es síncrono desde la perspectiva del cliente (espera la respuesta) aunque el servidor puede procesar asincrónicamente.

## Mecánica interna

### Flujo completo dentro del servidor

```
  bytes de red
       │
       ▼
  1. Parsing del request
     - Leer method, URL, headers, body
     - Validar formato HTTP
       │
       ▼
  2. Routing
     - Matchear URL path + method contra rutas registradas
     - Extraer path params (/users/:id → id=42)
       │
       ▼
  3. Middleware chain
     - Auth: verificar JWT/session
     - Logging: registrar request
     - Rate limiting: verificar cuota
     - CORS: validar origen
       │
       ▼
  4. Handler / Controller
     - Validar input (body, query params)
     - Lógica de negocio
     - Queries a DB, calls a otros servicios
       │
       ▼
  5. Serialización del response
     - Status code + headers + body (JSON/Protobuf)
       │
       ▼
  bytes de red → cliente
```

### Content Negotiation

```
  Request:
  Accept: application/json, application/xml;q=0.9, */*;q=0.1

  El servidor elige el formato con mayor prioridad que soporta.
  q=1.0 (default) > q=0.9 > q=0.1

  Response:
  Content-Type: application/json
  Vary: Accept              ← Indica que el response depende del Accept header
```

### Transfer Encoding

```
  Content-Length (tamaño conocido):
  Content-Length: 1234
  [1234 bytes de datos]

  Chunked (tamaño desconocido, streaming):
  Transfer-Encoding: chunked

  1a\r\n                    ← 26 bytes en hex
  [26 bytes de datos]\r\n
  0f\r\n                    ← 15 bytes
  [15 bytes de datos]\r\n
  0\r\n                     ← chunk final (tamaño 0)
  \r\n

  Útil para: streaming de responses grandes, server-side rendering
  progresivo, responses donde no conoces el tamaño total al inicio.
```

## Modelo mental

Request/response es como pedir en un restaurante: das tu orden completa (request), esperas, y recibes el plato (response). El mesero no te trae platos que no pediste, y si quieres algo más, debes hacer otro pedido.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Timeout del cliente | Servidor tarda demasiado (slow query, downstream lento) | Timeouts en cada capa, async processing para tareas pesadas |
| Response truncada | Conexión cortada durante chunked transfer | Retry, verificar Content-Length vs bytes recibidos |
| 502/504 en proxy | Backend no responde al reverse proxy | Health checks, circuit breaker |

## Impacto en sistemas distribuidos

- **Request tracing**: cada request debe llevar un correlation ID (X-Request-Id) que se propaga a todos los servicios downstream. Esto permite reconstruir el flujo completo en distributed tracing.
- **Timeout budgets**: si el cliente tiene timeout de 5s y el request pasa por gateway (100ms) → service A (200ms) → service B (?), el service B tiene máximo 5s - 300ms = 4.7s. gRPC propaga deadlines automáticamente.

## Insights de ingeniería

- **Chunked transfer para TTFB**: en server-side rendering, enviar el `<head>` inmediatamente (chunked) mientras calculas el body reduce el tiempo percibido por el usuario.
- **No mezcles sync y async en el mismo endpoint**: si el endpoint hace algo que tarda >1s, considera retornar 202 Accepted con un task ID y que el cliente poll el resultado.
- **Content negotiation real**: pocas APIs lo implementan bien. La mayoría hardcodean JSON. Pero es útil para APIs públicas que soportan múltiples formatos o versiones.
