# 5.3 Comunicación en Tiempo Real

## Definición

Patrones para enviar datos del servidor al cliente sin que el cliente los pida explícitamente. Desde el ineficiente short polling hasta WebSockets full-duplex, cada patrón tiene trade-offs de latencia, complejidad, y escalabilidad.

## Mecánica interna

### Short Polling

```
  Cliente                          Servidor
     │── GET /notifications ────►  │ → []        (nada nuevo)
     │   (espera 5 segundos)       │
     │── GET /notifications ────►  │ → []        (nada nuevo)
     │   (espera 5 segundos)       │
     │── GET /notifications ────►  │ → [{msg}]   (hay algo)

  Problema: N clientes × requests/min = carga constante,
  mayoría de responses son vacías.
  Latencia: hasta polling_interval de retraso.
```

### Long Polling

```
  Cliente                          Servidor
     │── GET /notifications ────►  │
     │       (conexión abierta)    │  (espera hasta tener datos)
     │       ...                    │  ...
     │◄── 200 [{msg}] ────────   │  (datos disponibles, responde)
     │                              │
     │── GET /notifications ────►  │  (inmediatamente reconecta)
     │       (conexión abierta)    │

  Ventajas: latencia mínima, menos requests vacías.
  Desventajas: una conexión HTTP abierta por cliente,
  complejidad de timeout management.
```

### Server-Sent Events (SSE)

```
  GET /events HTTP/1.1
  Accept: text/event-stream

  HTTP/1.1 200 OK
  Content-Type: text/event-stream
  Cache-Control: no-cache

  id: 1
  data: {"type": "notification", "msg": "hello"}

  id: 2
  event: price-update
  data: {"symbol": "AAPL", "price": 195.50}

  : keep-alive comment (cada 15s para evitar timeout)

  Features:
  - Reconexión automática del browser (Last-Event-ID)
  - Event types para categorizar
  - Unidireccional: server → client
  - Funciona sobre HTTP estándar (proxies, CDN friendly)
```

### WebSocket (repaso de uso para real-time)

```
  Bidireccional, full-duplex, baja latencia.

  Use cases:
  - Chat: ambos lados envían mensajes
  - Trading: server push de precios + client send de órdenes
  - Gaming: estado del juego sincronizado bidireccionalmente
  - Collaborative editing: cada keystroke se envía y recibe
```

### Comparación

| Aspecto | Short Polling | Long Polling | SSE | WebSocket |
|---------|--------------|-------------|-----|-----------|
| Latencia | Alta (polling interval) | Baja | Baja | Muy baja |
| Dirección | Client → Server | Client → Server | Server → Client | Bidireccional |
| Complejidad | Mínima | Media | Baja | Alta |
| Connections | Nuevas cada poll | 1 (re-opened) | 1 persistent | 1 persistent |
| Reconexión | Built-in (es un loop) | Manual | Automática (browser) | Manual |
| Proxy/CDN friendly | Sí | Parcial | Sí | No (requiere upgrade) |
| Scaling concern | Requests/s | Open connections | Open connections | Open connections + state |
| Ideal para | Dashboard refresh | Chat simple | Notifications, feeds | Chat, gaming, trading |

### Push vs Pull

```
  Pull (polling): el cliente pide periódicamente.
  + Simple
  + Cacheable
  - Latencia de hasta polling_interval
  - Carga constante incluso sin cambios

  Push (SSE, WebSocket): el servidor envía cuando hay datos.
  + Latencia mínima
  + Solo envía cuando hay cambios
  - Conexión persistente por cliente
  - Scaling más complejo (state per connection)
```

## Modelo mental

- **Short polling**: preguntar "¿hay correo?" cada 5 minutos en la oficina de correo.
- **Long polling**: ir a la oficina de correo y decir "espero aquí hasta que llegue algo".
- **SSE**: suscribirte a un servicio de correo que trae las cartas a tu puerta cuando llegan.
- **WebSocket**: tener una línea telefónica directa permanente.

## Modos de fallo

| Patrón | Síntoma | Causa | Solución |
|--------|---------|-------|----------|
| Long Polling | Reconnect storm | Servidor responde inmediatamente sin datos | Esperar en el servidor, no retornar 204 inmediatamente |
| SSE | Se desconecta cada 30s | Proxy timeout | Keep-alive comments, ajustar proxy idle timeout |
| WebSocket | Mensaje perdido | No ACK a nivel de aplicación | Implementar acknowledgments propios |
| Todos | Server OOM con muchos clientes | 1 connection/thread model | Event loop, connection limits |

## Impacto en sistemas distribuidos

- **WebSocket + horizontal scaling**: si el usuario conecta al server A, pero el evento se genera en server B, necesitas un pub/sub backend (Redis Pub/Sub, Kafka) para que B notifique a A.
- **SSE es suficiente para el 80% de los casos**: notifications, price feeds, log streaming, progress bars. Solo necesitas WebSocket si el cliente también envía datos frecuentemente.

## Insights de ingeniería

- **Heartbeats**: para cualquier conexión persistente (SSE, WebSocket), implementa heartbeats. Sin ellos, no detectas conexiones muertas (half-open) hasta que intentas escribir.
- **Connection limits**: un servidor con 16GB RAM puede manejar ~100K WebSocket connections (cada una consume ~100KB de buffers). Planea capacity accordingly.
- **Fallback chain**: WebSocket → SSE → Long Polling. Si WebSocket no funciona (firewall, proxy), caer a SSE. Si SSE no funciona, long polling. Socket.IO implementa esto automáticamente.
