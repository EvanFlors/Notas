# 5.5 Estado y Sesiones

## Definición

Stateful backends mantienen estado de la sesión entre requests. Stateless backends no — toda la información necesaria viaja en cada request. La idempotencia garantiza que ejecutar una operación múltiples veces produce el mismo resultado, crítica para retries seguros en sistemas distribuidos.

## Mecánica interna

### Stateless Backend

```
  Request 1: GET /api/users
             Authorization: Bearer eyJhb...
             → Server A procesa (cualquier instancia)

  Request 2: POST /api/orders
             Authorization: Bearer eyJhb...
             → Server B procesa (cualquier instancia)

  No hay estado en el servidor. El token contiene toda la identidad.
  Load balancer puede enviar a cualquier instancia.
```

### Stateful Backend

```
  Request 1: POST /login → Set-Cookie: session_id=abc123
             Servidor almacena: abc123 → {user: "Alice", cart: [...]}

  Request 2: GET /cart
             Cookie: session_id=abc123
             → DEBE ir al mismo servidor (o a un store compartido)
```

### Gestión de sesiones

| Mecanismo | Stateful? | Scalable? | Descripción |
|-----------|-----------|-----------|-------------|
| Server-side session | Sí (memoria) | No | Estado en RAM del proceso |
| Session en Redis | Sí (externo) | Sí | Estado en store compartido |
| JWT (access token) | No | Sí | Estado en el token, no en el server |
| Cookie + DB | Sí (externo) | Sí | Session ID en cookie, datos en DB |

### Idempotencia

```
  Operaciones naturalmente idempotentes:
  GET /users/42        → siempre retorna el mismo recurso
  PUT /users/42 {...}  → siempre deja el recurso en el mismo estado
  DELETE /users/42     → primera vez borra, segunda retorna 404 (pero no falla)

  Operaciones NO idempotentes:
  POST /orders {...}   → cada llamada crea un nuevo order

  Solución: Idempotency Key
  POST /orders
  Idempotency-Key: uuid-abc-123

  Servidor:
  1. ¿Ya procesé uuid-abc-123? → retornar resultado almacenado
  2. No → procesar, almacenar resultado con key uuid-abc-123, retornar
```

### Delivery Guarantees

```
  At-most-once:   enviar y olvidar. Puede perderse.
                  Uso: métricas, logs no críticos.

  At-least-once:  retry hasta ACK. Puede duplicarse.
                  Uso: tareas con idempotency key.

  Exactly-once:   cada mensaje procesado exactamente una vez.
                  En teoría imposible en redes no confiables.
                  En práctica: at-least-once + idempotent consumers.
```

## Modelo mental

- **Stateless** = máquina expendedora: insertas monedas y seleccionas producto cada vez. No recuerda quién eres.
- **Stateful** = barista que te conoce: sabe tu orden habitual. Pero si cambia el barista, tienes que repetirla.
- **Idempotency** = un interruptor de luz: presionarlo 1 vez o 10 veces deja la luz en el mismo estado (encendida).

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Sesión perdida al cambiar de server | Estado en memoria, no compartido | Externalizar en Redis |
| Orden duplicada | POST retry sin idempotency key | Implementar idempotency keys |
| JWT no revocable | JWT stateless = no puedes invalidarlo antes de expirar | Short-lived access tokens + refresh tokens + blocklist en Redis |
| Sticky session failure | Server muere, estado perdido | Externalizar estado, evitar sticky sessions |

## Impacto en sistemas distribuidos

- **Stateless es el objetivo**: horizontal scaling trivial, failover sin pérdida de estado, deployment sin interrupción (rolling update).
- **Idempotency keys son obligatorias para pagos**: Stripe, PayPal, todos los payment gateways lo requieren. Un retry no debe cobrar dos veces.
- **Exactly-once en Kafka**: Kafka Transactions + idempotent producer + consumer offset commit atómico da semántica de exactly-once dentro del cluster Kafka. Fuera de Kafka, sigue siendo at-least-once.

## Insights de ingeniería

- **JWT vs Sessions**: JWT = stateless, no revocable, auto-contenido (no DB lookup por request). Sessions = stateful, revocables, requieren store. Para la mayoría de apps: JWT access token (15min TTL) + refresh token en DB (revocable).
- **Idempotency key storage**: almacenar en Redis con TTL (24-48h). Key = idempotency key, value = response serializado. Si llega un retry, retornar el response almacenado sin re-procesar.
- **Sticky sessions como último recurso**: si necesitas sticky sessions (WebSocket, upload de archivos multi-part), usa consistent hashing en el load balancer para que el mismo usuario vaya al mismo server.
