# 5.6 Multiplexing y Connection Management

## Definición

Multiplexing permite enviar múltiples streams de datos sobre una sola conexión. Connection pooling reutiliza conexiones establecidas para evitar el overhead de handshakes repetidos. Juntos, son la base del rendimiento de cualquier backend que hace llamadas a otros servicios, bases de datos, y caches.

## Mecánica interna

### HTTP/2 Multiplexing

```
  HTTP/1.1 (sin multiplexing):
  Conn 1: [req1 ──── resp1] [req2 ──── resp2]  ← secuencial
  Conn 2: [req3 ──── resp3] [req4 ──── resp4]  ← paralelo requiere otra conn

  HTTP/2 (multiplexing):
  Conn 1: [req1]  [req2]  [req3]  [req4]
          [resp2] [resp1] [resp3] [resp4]  ← intercalados en frames

  Una sola conexión TCP, múltiples streams simultáneos.
```

### Connection Pooling

```
  Sin pool:
  request → TCP handshake → TLS handshake → query → close
  request → TCP handshake → TLS handshake → query → close  ← overhead repetido

  Con pool:
  ┌─────────────────────────────┐
  │     Connection Pool         │
  │  [conn1: idle]              │
  │  [conn2: in-use]            │
  │  [conn3: idle]              │
  │  [conn4: in-use]            │
  │                             │
  │  min: 2, max: 10            │
  │  idle_timeout: 30s          │
  │  max_lifetime: 5min         │
  └─────────────────────────────┘

  request → borrow conn1 → query → return conn1 to pool
  request → borrow conn3 → query → return conn3 to pool

  Handshake pagado solo 1 vez por conexión.
```

### Kernel-level Demultiplexing

```
  NIC recibe frame → kernel lee IP header → local? → lee TCP header
  → dst_port = 8080 → socket lookup → entrega al proceso correcto

  Un servidor puede tener miles de sockets (connections) en el mismo puerto.
  El kernel identifica cada conexión por el tuple:
  (src_ip, src_port, dst_ip, dst_port, protocol)
```

## Modelo mental

- **Multiplexing** = una autopista con múltiples carriles. Los autos (streams) viajan simultáneamente sin esperar.
- **Connection Pool** = un estacionamiento de taxis. En vez de llamar un nuevo taxi cada vez, tomas uno que ya está esperando.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Connection pool exhausted | Max connections alcanzado, todas in-use | Aumentar pool size, investigar slow queries |
| Stale connections | Conexión idle cortada por firewall/LB | Health check, max_lifetime, validation on borrow |
| Single TCP bottleneck (HTTP/2) | Una conexión TCP no satura el bandwidth | Múltiples conexiones HTTP/2 |

## Insights de ingeniería

- **Pool size = número de threads/goroutines concurrentes**: si tienes 20 goroutines haciendo queries, un pool de 5 es bottleneck (15 esperan). Un pool de 100 desperdicia conexiones. Target: pool size ≈ concurrent requests.
- **PgBouncer para PostgreSQL**: PostgreSQL fork un proceso por conexión (costoso). PgBouncer multiplexa 1000 app connections → 50 DB connections. Esencial en producción.
- **Connection lifetime**: rota conexiones periódicamente (max_lifetime) para redistribuir carga cuando hay cambios de DNS (ej: failover de base de datos).
