# 5.7 Sidecar Pattern

## Definición

El sidecar pattern coloca un proxy junto a cada instancia de servicio (mismo pod en K8s, mismo host). Este proxy intercepta todo el tráfico de entrada y salida, agregando funcionalidad (mTLS, retries, observabilidad, rate limiting) sin modificar el código de la aplicación. Es la base de los service meshes.

## Mecánica interna

```
  Sin sidecar:
  ┌─────────┐          ┌─────────┐
  │ Service A│ ──HTTP──► │Service B│
  │ (con TLS,│          │(con TLS,│
  │  retries,│          │ metrics,│
  │  metrics)│          │ retries)│
  └─────────┘          └─────────┘
  Cada servicio implementa cross-cutting concerns.

  Con sidecar:
  ┌────────────────────┐     ┌────────────────────┐
  │ Pod A               │     │ Pod B               │
  │ ┌────────┐┌───────┐│     │┌───────┐┌────────┐ │
  │ │Service A││Envoy  ││─────││Envoy  ││Service B│ │
  │ │(solo    ││proxy  ││mTLS ││proxy  ││(solo    │ │
  │ │business ││       ││     ││       ││business │ │
  │ │logic)   ││       ││     ││       ││logic)   │ │
  │ └────────┘└───────┘│     │└───────┘└────────┘ │
  └────────────────────┘     └────────────────────┘

  Envoy maneja: mTLS, retries, circuit breaking, load balancing,
  metrics, tracing, rate limiting. Transparente para la app.
```

### Service Mesh

```
  ┌──────────────────────────────────────────┐
  │              Control Plane                │
  │         (Istio / Linkerd / Consul)        │
  │  - Certificate management (mTLS)          │
  │  - Traffic policies (routing, retries)    │
  │  - Observability config                    │
  └──────────────────┬───────────────────────┘
                     │ config push
         ┌───────────┼───────────┐
         ▼           ▼           ▼
  ┌──────────┐ ┌──────────┐ ┌──────────┐
  │[App][Envoy]│ │[App][Envoy]│ │[App][Envoy]│  ← Data Plane
  └──────────┘ └──────────┘ └──────────┘

  Control Plane: configura las políticas
  Data Plane: los sidecars que ejecutan las políticas
```

### Qué hace Envoy como sidecar

| Feature | Descripción |
|---------|-------------|
| mTLS | Encripta toda comunicación inter-service automáticamente |
| Retries | Retry transparente con backoff y budget |
| Circuit Breaking | Corta llamadas a servicios degradados |
| Load Balancing | L7 load balancing entre instancias |
| Observability | Métricas (latency, error rate), distributed tracing |
| Rate Limiting | Limita requests por servicio |
| Traffic Splitting | Canary deployments (90% v1, 10% v2) |

## Modelo mental

El sidecar es como un asistente personal para cada servicio. El servicio dice "envía esto a service B", y el asistente se encarga del cifrado, reintentos, monitoreo, y manejo de errores. El servicio solo se preocupa por la lógica de negocio.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Latencia adicional de ~1ms por hop | Overhead del proxy | Aceptable en la mayoría de casos; medir si es significativo |
| Sidecar crashea | Bug en Envoy o OOM | Restart policy, resource limits |
| Config inconsistente entre sidecars | Control plane lento en propagar | Verificar sync status, rolling updates |

## Impacto en sistemas distribuidos

Un service mesh con sidecars resuelve de forma uniforme los problemas transversales de 100 microservicios: seguridad (mTLS sin cambio de código), resiliencia (retries, circuit breaking), y observabilidad (métricas y tracing uniformes). El costo es complejidad operacional del mesh.

## Insights de ingeniería

- **No necesitas service mesh con 3 microservicios**: el overhead operacional no se justifica. Implementa retries y health checks en tu código. Service mesh brilla con 20+ servicios.
- **Envoy es el estándar**: Istio, AWS App Mesh, y Consul Connect usan Envoy como data plane. Aprende Envoy y entiendes todos los service meshes.
- **Sidecar overhead**: ~10-20MB RAM por sidecar, ~1ms de latencia por hop. Con 100 servicios = 2GB de RAM solo en sidecars. Evalúa si el beneficio justifica el costo.
