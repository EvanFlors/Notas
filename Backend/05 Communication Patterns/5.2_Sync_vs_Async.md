# 5.2 Synchronous vs Asynchronous

## Definición

Sync: el caller bloquea hasta recibir la respuesta. Async: el caller continúa ejecutando y recibe notificación cuando la respuesta está lista. Blocking/non-blocking es ortogonal: describe si la llamada de I/O retorna inmediatamente o espera. Las 4 combinaciones existen.

## Mecánica interna

### Las 4 combinaciones

```
                    Blocking                   Non-blocking

  Sync        read() espera hasta datos      read() retorna EAGAIN si no hay
              Caller espera respuesta        Caller hace polling
              (thread-per-request)           (busy wait)

  Async       N/A conceptualmente           Callback/future cuando hay datos
                                             Caller sigue trabajando
                                             (event loop, epoll + callbacks)
```

### Sync Request-Response

```
  Service A                    Service B
     │                            │
     │── HTTP GET /users ────►   │
     │       (esperando...)       │ procesa...
     │       (esperando...)       │ query DB...
     │◄── 200 OK ────────────   │
     │                            │
     │  Continúa ejecución       │

  El thread de A está bloqueado durante todo el proceso.
  Si B tarda 2 segundos, A desperdicia 2 segundos de thread.
```

### Async con Message Queue

```
  Service A                Queue              Service B
     │                       │                    │
     │── publish(task) ──►  │                    │
     │◄── ACK ──────────   │                    │
     │                       │── deliver(task) ──►│
     │  (sigue trabajando)   │                    │ procesa...
     │                       │◄── ACK ───────────│
     │                       │                    │

  A no espera a B. A solo sabe que el mensaje fue encolado.
  B procesa cuando puede. Si B está caído, la queue retiene el mensaje.
```

### Cuándo usar cada uno

| Escenario | Patrón | Razón |
|-----------|--------|-------|
| GET /users (respuesta necesaria ahora) | Sync | El cliente necesita los datos para continuar |
| POST /orders (requiere confirmación) | Sync | El usuario necesita saber si se creó |
| Enviar email de confirmación | Async | No bloquear el response del usuario |
| Procesar video subido | Async | Tarda minutos, no puedes hacer esperar al usuario |
| Notificar a 1000 usuarios | Async | Fan-out masivo, no acoplable a un request |
| Sync entre microservicios | Sync con timeout | OK si la latencia es baja y predecible |

## Modelo mental

Sync es llamar por teléfono: esperas hasta que contesten. Async es enviar un email: lo envías, sigues con tu trabajo, y eventualmente recibes respuesta. Blocking es quedarte mirando el teléfono. Non-blocking es revisar el teléfono cada 5 minutos.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Cascading failures | Sync call chain: A→B→C. Si C cae, B cae, A cae | Circuit breaker, timeouts, async donde sea posible |
| Lost messages | Queue no durabled, broker crashea | Persistent queues, acknowledgments |
| Out-of-order processing | Async messages procesados en orden diferente | Sequence numbers, o diseñar para order-independence |
| Fire-and-forget sin retry | Mensaje se pierde, nadie lo sabe | Dead letter queues, monitoring de queue depth |

## Impacto en sistemas distribuidos

- **Sync chains son frágiles**: si tienes A→B→C→D sincrónicamente, la disponibilidad del sistema es el producto de las disponibilidades individuales: 0.99 × 0.99 × 0.99 × 0.99 = 0.96. Con async, A solo depende de la queue.
- **Async = eventual consistency**: cuando usas async, el estado no se actualiza inmediatamente. El usuario puede ver datos stale. Debes diseñar la UX para esto.
- **Back-pressure**: en async, si el producer es más rápido que el consumer, la queue crece indefinidamente. Implementa back-pressure (rate limit al producer cuando la queue pasa cierto umbral).

## Insights de ingeniería

- **202 Accepted pattern**: para operaciones async vía HTTP, retorna 202 con un `Location` header apuntando a un endpoint de status. El cliente poll ese endpoint hasta que el resultado esté listo, o usa WebSocket/SSE para notificación.
- **No hagas async lo que puede ser sync**: si la operación tarda <100ms y el cliente necesita el resultado, sync es más simple. Async agrega complejidad (queues, retries, monitoring, eventual consistency).
- **Timeout en toda llamada sync**: SIEMPRE. Un servicio downstream que no responde puede congelar tu servicio entero si no tienes timeout. Connect timeout (rápido, ~1-3s) + read timeout (depende del SLA del downstream).
