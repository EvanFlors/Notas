# 7.3 Casos Especiales de Proxying

## Definición

WebSockets, gRPC, y TLS tienen requisitos especiales al pasar por proxies. Entender estos casos previene problemas de conectividad y rendimiento que son difíciles de diagnosticar.

## Mecánica interna

### WebSocket Proxying

```
  Cliente ──HTTP Upgrade──► [Proxy] ──HTTP Upgrade──► Backend

  El proxy debe:
  1. Reconocer el header "Upgrade: websocket"
  2. Mantener la conexión abierta (no timeout)
  3. Pasar tráfico bidireccional sin buffering

  Problemas:
  - Idle timeout: proxy corta conexión inactiva (default 60s en Nginx)
    Solución: proxy_read_timeout 3600s + WebSocket ping/pong
  - Sticky sessions: WebSocket es stateful, debe ir al mismo backend
    Solución: IP hash o cookie-based routing
```

```nginx
# Nginx WebSocket config
location /ws {
    proxy_pass http://backend;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_read_timeout 3600s;
}
```

### gRPC Proxying

```
  gRPC usa HTTP/2 con trailers (headers al final del response).

  Requisitos del proxy:
  - Soporte HTTP/2 end-to-end
  - Soporte de HTTP trailers (gRPC status code va en trailers)
  - L7 routing (para routing por gRPC service/method)

  Nginx: soporta gRPC desde v1.13.10 (grpc_pass)
  Envoy: soporte nativo (diseñado para gRPC)
  HAProxy: soporte desde v2.0
```

### TLS Termination vs Passthrough vs Re-encryption

```
  TLS Termination:
  Cliente ──HTTPS──► [Proxy: descifra] ──HTTP──► Backend
  + Proxy maneja certificados (un solo lugar)
  + Backend más simple (no gestiona TLS)
  - Tráfico interno sin cifrar (requiere red segura)

  TLS Passthrough:
  Cliente ──HTTPS──► [Proxy: no toca] ──HTTPS──► Backend
  + End-to-end encryption
  + Proxy no ve el contenido (L4 only)
  - No puede hacer L7 routing (no puede leer headers)
  - Cada backend gestiona su certificado

  TLS Re-encryption:
  Cliente ──HTTPS──► [Proxy: descifra + re-cifra] ──HTTPS──► Backend
  + L7 routing posible
  + Tráfico interno también cifrado
  - Doble overhead de crypto
  - Proxy necesita certificado del backend (o mTLS)
```

## Modos de fallo

| Caso | Síntoma | Causa | Solución |
|------|---------|-------|----------|
| WebSocket | Desconexión periódica | Proxy idle timeout | Ping/pong + timeout alto |
| gRPC | Error sin status code | Proxy no soporta HTTP/2 trailers | Usar proxy compatible (Envoy) |
| TLS | Certificate mismatch | Backend cert CN ≠ hostname que proxy usa | Verificar SNI y CN |

## Insights de ingeniería

- **Envoy es el proxy más completo para casos especiales**: soporte nativo de WebSocket, gRPC, HTTP/2, HTTP/3. Si tu stack usa gRPC, Envoy es la opción más natural.
- **TLS termination en el LB es el patrón más común**: simplifica certificate management. En AWS, el ALB hace TLS termination. Dentro de la VPC, el tráfico HTTP plano es aceptable (o usa mTLS con service mesh para zero-trust).
- **gRPC-web para browsers**: browsers no soportan gRPC nativo (requiere HTTP/2 trailers). Envoy tiene un filtro grpc-web que traduce automáticamente.
