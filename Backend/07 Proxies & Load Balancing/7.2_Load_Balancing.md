# 7.2 Load Balancing

## Definición

Load balancing distribuye tráfico entre múltiples instancias de un servicio para maximizar throughput, minimizar latencia, y garantizar alta disponibilidad. Opera en capa 4 (TCP) o capa 7 (HTTP), con algoritmos que determinan a qué instancia enviar cada request.

## Mecánica interna

### Layer 4 vs Layer 7

```
  L4 (Transport):
  Cliente ──TCP──► [LB] ──TCP──► Backend

  LB lee: IP src/dst, port src/dst
  LB NO lee: HTTP headers, URL, cookies
  Decisión: basada solo en IP:puerto
  Velocidad: muy rápido (no parsea contenido)

  L7 (Application):
  Cliente ──HTTP──► [LB] ──HTTP──► Backend

  LB lee: URL, headers, cookies, body
  Decisión: basada en contenido de la request
  Ejemplos: /api/* → backend A, /static/* → backend B
  Velocidad: más lento (debe parsear HTTP)
```

### Algoritmos

| Algoritmo | Descripción | Cuándo usar |
|-----------|-------------|-------------|
| Round Robin | Rota secuencialmente entre backends | Backends homogéneos |
| Weighted Round Robin | Más requests a backends con más peso | Backends con diferente capacidad |
| Least Connections | Envía al backend con menos conexiones activas | Requests de duración variable |
| IP Hash | Mismo cliente → mismo backend (hash de IP) | Session affinity sin cookies |
| Random | Selección aleatoria | Simple, surprisingly effective |
| Least Response Time | Backend más rápido recibe más tráfico | Heterogeneous backends |

### Health Checks

```
  Active health check:
  LB envía probe cada N segundos:
  GET /health → 200 OK → backend healthy
  GET /health → timeout/500 → backend unhealthy → remove from pool

  Passive health check:
  LB observa responses reales:
  5 errores consecutivos → marcar unhealthy
  3 éxitos consecutivos → marcar healthy

  Ambos se usan juntos en producción.
```

### Session Persistence (Sticky Sessions)

```
  Sin sticky sessions:
  Request 1 → Backend A (session creada en A)
  Request 2 → Backend B (session no existe en B!) ← error

  Con sticky sessions:
  Request 1 → Backend A (session creada, cookie: srv=A)
  Request 2 → Backend A (cookie: srv=A, LB dirige a A)

  Problema: si Backend A muere, la sesión se pierde.
  Mejor solución: externalizar sesión en Redis (stateless backends).
```

## Modelo mental

L4 LB es como un operador telefónico: solo ve el número que marcaste (IP:puerto) y te conecta a una línea libre. L7 LB es como un recepcionista que lee tu solicitud y te dirige al departamento correcto.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Tráfico desigual entre backends | Round robin con requests de duración variable | Least connections |
| Session lost after deploy | Sticky session + rolling update (new IP) | External session store |
| Backend overloaded | Health check no detecta degradación | Passive health checks + connection limits |
| Single point of failure | Un solo LB | LB pair (active-passive) o DNS-based LB |

## Insights de ingeniería

- **L7 es el estándar para microservicios**: necesitas routing por path, header-based routing (canary deployments), y visibilidad en métricas HTTP.
- **Least connections para APIs con latencia variable**: si algunos endpoints tardan 10ms y otros 5s, round robin sobrecarga backends lentos. Least connections distribuye naturalmente.
- **DNS load balancing (GeoDNS)**: para global load balancing, DNS retorna diferentes IPs según la ubicación del cliente. Cloudflare, AWS Route 53, Google Cloud DNS.
