# 7.4 Herramientas de Proxy y Load Balancing

## Definición

Las cuatro herramientas principales de proxy/LB en el ecosistema backend, sus fortalezas, y cuándo usar cada una.

## Mecánica interna

### Nginx

```
  Rol: Reverse proxy + web server + load balancer
  Arquitectura: Master + N workers (event loop, epoll)
  Protocolo: HTTP/1.1, HTTP/2, gRPC, WebSocket, TCP/UDP (stream)
  Config: archivo estático (nginx.conf), reload con SIGHUP

  Fortalezas:
  - Servir archivos estáticos (sendfile, zero-copy)
  - Reverse proxy + TLS termination
  - Amplia adopción, documentación extensa
  - Bajo consumo de recursos

  Limitaciones:
  - Config estático (no auto-discovery)
  - Reload requiere SIGHUP (puede perder conexiones)
```

### HAProxy

```
  Rol: Load balancer especializado (L4/L7)
  Arquitectura: Multi-threaded event loop
  Protocolo: TCP, HTTP/1.1, HTTP/2
  Config: archivo estático, reload con zero-downtime (seamless reload)

  Fortalezas:
  - Rendimiento máximo en load balancing
  - Health checks avanzados
  - Estadísticas detalladas (dashboard built-in)
  - Zero-downtime reloads

  Limitaciones:
  - No sirve archivos estáticos
  - No es web server (solo proxy/LB)
```

### Envoy

```
  Rol: Proxy L7 moderno, base de service meshes
  Arquitectura: Multi-threaded, C++, event loop
  Protocolo: HTTP/1.1, HTTP/2, HTTP/3, gRPC, TCP, UDP
  Config: API-driven (xDS), dynamic (no reload)

  Fortalezas:
  - Config dinámica (sin reload, sin downtime)
  - gRPC nativo, HTTP/3, WebSocket
  - Distributed tracing integrado (Zipkin, Jaeger)
  - Base de Istio, AWS App Mesh, Consul Connect
  - Observabilidad first-class (métricas, access logs)

  Limitaciones:
  - Más complejo de configurar que Nginx
  - Mayor consumo de memoria
```

### Traefik

```
  Rol: Reverse proxy con auto-discovery
  Arquitectura: Go, single binary
  Protocolo: HTTP/1.1, HTTP/2, TCP, UDP
  Config: Auto-discovery (Docker labels, K8s Ingress, Consul)

  Fortalezas:
  - Descubrimiento automático de servicios
  - Let's Encrypt automático
  - Dashboard web built-in
  - Ideal para Docker/K8s

  Limitaciones:
  - Menor rendimiento que Nginx/Envoy
  - Menos opciones de tuning fino
```

### Cuándo usar cada uno

| Escenario | Herramienta | Razón |
|-----------|-------------|-------|
| Web server + reverse proxy | Nginx | Estándar, eficiente, simple |
| Load balancing de alto rendimiento | HAProxy | Especializado, máximo throughput |
| Service mesh / microservicios | Envoy | Config dinámica, gRPC, observabilidad |
| Docker/K8s con auto-discovery | Traefik | Zero-config service discovery |
| API Gateway | Nginx + Lua / Kong / Envoy | Rate limiting, auth, routing |

## Insights de ingeniería

- **Nginx para el 80% de los casos**: si necesitas reverse proxy + TLS + load balancing + static files, Nginx es la opción probada y con menor overhead operacional.
- **Envoy para cloud-native**: si tu stack es K8s + gRPC + service mesh, Envoy es el estándar de facto. Su config API-driven (xDS) permite cambios sin restart.
- **No uses Nginx como load balancer de bases de datos**: para TCP load balancing a PostgreSQL/MySQL, usa PgBouncer/ProxySQL (entienden el protocolo de la DB) o HAProxy (L4 TCP).
