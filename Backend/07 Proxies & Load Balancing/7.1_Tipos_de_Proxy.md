# 7.1 Tipos de Proxy

## Definición

Un proxy es un intermediario entre cliente y servidor. Un forward proxy actúa en nombre del cliente (el cliente lo configura). Un reverse proxy actúa en nombre del servidor (el cliente no sabe que existe). En backend, casi siempre trabajas con reverse proxies.

## Mecánica interna

### Forward Proxy

```
  Cliente ──► [Forward Proxy] ──► Internet

  El CLIENTE configura el proxy.
  El servidor destino no sabe que hay un proxy.

  Usos:
  - VPN corporativa
  - Filtrado de contenido (escuelas, empresas)
  - Anonimización (ocultar IP del cliente)
  - Cache de contenido (Squid)
```

### Reverse Proxy

```
  Internet ──► [Reverse Proxy] ──► Backend servers

  El SERVIDOR configura el proxy.
  El cliente no sabe que hay un proxy (cree que habla directo con el server).

  Usos:
  - Load balancing entre instancias
  - TLS termination (descifra HTTPS, reenvía HTTP al backend)
  - Caching de responses (reduce carga en backend)
  - Compression (gzip/brotli)
  - Rate limiting
  - Security (WAF, DDoS protection)
  - Static file serving
```

### Diferencia fundamental

```
  Forward: cliente → PROXY → servidor
           ↑ controlado por el cliente

  Reverse: cliente → PROXY → servidor
                     ↑ controlado por el servidor

  El cliente de un reverse proxy hace requests a proxy.example.com
  sin saber que internamente hay 10 servidores backend.
```

## Modelo mental

Forward proxy = tu abogado. Actúa en TU nombre ante otros. Los demás hablan con tu abogado, no contigo directamente.

Reverse proxy = la recepcionista de una empresa. Los clientes llaman a un número central, la recepcionista dirige la llamada al departamento correcto. El cliente no conoce los números internos.

## Impacto en sistemas distribuidos

Reverse proxies son el entry point de toda arquitectura de producción. Nginx/Envoy frente a tus microservicios proveen un punto central para TLS termination, routing, observabilidad, y seguridad.

## Insights de ingeniería

- **Siempre hay un reverse proxy en producción**: incluso si tienes un solo backend server, poner Nginx delante te da TLS termination, gzip, rate limiting, y la capacidad de escalar sin cambiar la URL pública.
- **X-Forwarded-For**: cuando el reverse proxy reenvía la request, agrega la IP real del cliente en este header. Tu app debe leer este header (no `request.remote_addr` que será la IP del proxy).
- **Proxy protocol**: alternativa a X-Forwarded-For para L4 proxies. Envía la IP del cliente como parte del protocolo TCP, no como header HTTP.
