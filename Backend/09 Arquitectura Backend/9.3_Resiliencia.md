# 9.3 Resiliencia

## Definición

Patrones que permiten a tu sistema funcionar (posiblemente degradado) cuando partes fallan. En producción, los fallos son inevitables: servicios caen, redes se particionan, bases de datos se saturan. La resiliencia es la capacidad de manejar estos fallos gracefully.

## Mecánica interna

### Circuit Breaker

```
  Estados:
  CLOSED → llamadas pasan normalmente
    │ N fallos consecutivos
  OPEN → todas las llamadas fallan inmediatamente (fast-fail)
    │ después de timeout
  HALF-OPEN → permite 1 llamada de prueba
    │ éxito → CLOSED
    │ fallo → OPEN

  Ejemplo:
  Service A → [Circuit Breaker] → Service B

  B falla 5 veces seguidas → circuit OPEN
  → A recibe error inmediato sin esperar timeout de B
  → A usa fallback (cache, default response)
  → Después de 30s, intenta 1 request a B (HALF-OPEN)
  → Si funciona → CLOSED, tráfico normal
```

### Retry con Exponential Backoff + Jitter

```
  Intento 1: falla → espera 1s
  Intento 2: falla → espera 2s + random(0, 1s)
  Intento 3: falla → espera 4s + random(0, 2s)
  Intento 4: falla → circuit breaker open o return error

  Sin jitter: todos los clientes retry al mismo tiempo → thundering herd.
  Con jitter: retries distribuidos en el tiempo.

  Retry budget: máximo 20% de requests totales pueden ser retries.
  Previene que retries amplifiquen la carga en un servicio degradado.
```

### Timeout

```
  SIEMPRE configurar dos timeouts:

  Connect timeout: 1-3s
  Si no puedes conectar en 3s, el servicio está caído.

  Read timeout: depende del SLA del downstream
  /api/users: 500ms (fast endpoint)
  /api/reports: 30s (heavy computation)

  Sin timeout: un servicio colgado consume threads/goroutines
  de tu servicio indefinidamente → cascading failure.
```

### Bulkhead

```
  Sin bulkhead:
  ┌──────────────────────────────┐
  │  Thread Pool (100 threads)    │
  │  Service A calls: 50 threads  │
  │  Service B calls: 50 threads  │ ← si B es lento, ocupa 100 threads
  │  → A también deja de funcionar│
  └──────────────────────────────┘

  Con bulkhead:
  ┌──────────────────┐  ┌──────────────────┐
  │ Pool A (50)      │  │ Pool B (50)      │
  │ calls to A       │  │ calls to B       │ ← B lento solo agota su pool
  └──────────────────┘  └──────────────────┘
  A sigue funcionando normalmente.
```

### Fallback y Graceful Degradation

```
  request → service B → timeout
         → fallback: retornar datos del cache (posiblemente stale)
         → o: retornar respuesta default
         → o: retornar respuesta parcial (sin datos de B)

  Ejemplo: página de producto
  - Precio: del servicio de pricing (caído) → mostrar "precio no disponible"
  - Reviews: del servicio de reviews (caído) → ocultar sección
  - El producto se sigue mostrando. Degradado pero funcional.
```

## Modelo mental

- **Circuit Breaker** = un fusible eléctrico. Cuando hay sobrecarga, se corta para proteger el sistema. Se resetea periódicamente para probar si la línea ya funciona.
- **Bulkhead** = compartimentos de un barco. Si uno se llena de agua, los demás siguen a flote.
- **Timeout** = poner una alarma. Si no obtienes respuesta antes de que suene, sigues con tu vida.

## Insights de ingeniería

- **Retry solo para errores transitorios**: 503 (overloaded), timeout, connection error → retry. 400 (bad request), 404 (not found) → NO retry (el resultado será el mismo).
- **Combina todos los patrones**: timeout + retry + circuit breaker + fallback. Timeout previene espera infinita, retry maneja errores transitorios, circuit breaker previene cascading failure, fallback mantiene funcionalidad.
- **Chaos engineering para validar**: inyecta fallos intencionalmente (Chaos Monkey, Litmus) para verificar que tus patrones de resiliencia funcionan antes de un incidente real.
