# 9.5 API Design

## Definición

Un API bien diseñada es predecible, consistente, y evolucionable. REST conventions, versionamiento, paginación, y error handling son las decisiones que determinan la experiencia del desarrollador y la mantenibilidad a largo plazo.

## Mecánica interna

### REST Conventions

```
  Recursos como sustantivos, verbos HTTP como acciones:

  GET    /users           → listar usuarios
  GET    /users/42        → obtener usuario 42
  POST   /users           → crear usuario
  PUT    /users/42        → reemplazar usuario 42
  PATCH  /users/42        → modificar usuario 42
  DELETE /users/42        → eliminar usuario 42

  Nested resources:
  GET    /users/42/orders → órdenes del usuario 42
  POST   /users/42/orders → crear orden para usuario 42

  Máximo 2 niveles de nesting. Más → aplanar con query params.
```

### Versioning

```
  URL path (recomendado):
  /v1/users, /v2/users
  + Explícito, fácil de routear

  Header:
  Accept: application/vnd.myapi.v2+json
  + URL limpia
  - Difícil de probar en browser

  Query param:
  /users?version=2
  + Simple
  - Fácil de olvidar
```

### Pagination

```
  Offset-based:
  GET /users?page=3&per_page=20
  → OFFSET 40 LIMIT 20
  + Simple
  - OFFSET alto es lento (DB escanea y descarta filas)
  - Inconsistente si se insertan datos entre páginas

  Cursor-based (recomendado):
  GET /users?after=cursor_abc&limit=20
  → WHERE id > cursor_abc ORDER BY id LIMIT 20
  + Rendimiento constante
  + Consistente
  - No puedes saltar a página N
```

### Error Handling

```json
  // Formato consistente de error:
  {
    "error": {
      "code": "VALIDATION_ERROR",
      "message": "Invalid input",
      "details": [
        {"field": "email", "message": "Must be a valid email"},
        {"field": "age", "message": "Must be positive"}
      ],
      "request_id": "req-abc-123"
    }
  }

  // HTTP status code SIEMPRE refleja el tipo de error:
  400 + error body → input inválido
  401 + error body → no autenticado
  404 + error body → recurso no existe
  500 + error body → error interno (sin detalles sensibles)
```

## Insights de ingeniería

- **Cursor-based pagination para feeds y listas largas**: offset-based se degrada linealmente con el tamaño de la tabla. Cursor-based mantiene rendimiento constante.
- **Consistency matters more than perfection**: elige convenciones y síguelas en TODA la API. Un API consistente pero imperfecta es mejor que una API inconsistente con endpoints "perfectos".
- **Breaking changes requieren nueva versión**: cambiar el tipo de un campo, eliminar un campo, o cambiar el significado de un status code son breaking changes. Agregar campos opcionales NO es breaking.
- **request_id en todos los errores**: permite al usuario reportar un error con un ID que tú puedes buscar en tus logs. Essential para debugging en producción.
