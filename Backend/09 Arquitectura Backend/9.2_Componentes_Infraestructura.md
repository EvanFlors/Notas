# 9.2 Componentes de Infraestructura

## Definición

Los componentes de infraestructura son las piezas que conectan, protegen, y hacen descubribles tus servicios: API Gateway como entry point, service discovery para encontrar instancias, service mesh para comunicación segura, y message brokers para async.

## Mecánica interna

### API Gateway

```
  Internet → [API Gateway] → Microservicios

  Responsabilidades:
  - Routing: /users → users-service, /orders → orders-service
  - Authentication: verificar JWT/API key
  - Rate limiting: 100 req/min por API key
  - Request transformation: agregar headers, modificar paths
  - Response aggregation: combinar responses de múltiples servicios
  - TLS termination

  Herramientas: Kong, AWS API Gateway, Envoy, Nginx + Lua
```

### Service Discovery

```
  Problema: ¿cómo sabe orders-service la IP:puerto de users-service
  cuando hay 10 instancias que cambian dinámicamente?

  Client-side discovery:
  App → [Service Registry] → "users-service: 10.0.1.5:8080, 10.0.1.6:8080"
  App elige una instancia y conecta directamente.

  Server-side discovery (DNS-based, K8s):
  App → DNS: "users-service" → 10.96.0.1 (ClusterIP)
  K8s Service proxy distribuye a los pods.

  Herramientas: Consul, etcd, K8s Services (DNS), AWS Cloud Map
```

### Service Mesh

```
  Control Plane (Istio/Linkerd)
       │ config
  ┌────▼────┐    ┌─────────┐    ┌─────────┐
  │[App][Proxy]│──│[App][Proxy]│──│[App][Proxy]│
  └─────────┘    └─────────┘    └─────────┘

  Cada proxy (Envoy sidecar) automáticamente:
  - mTLS entre servicios
  - Retries con budget
  - Circuit breaking
  - Load balancing
  - Métricas y tracing

  Sin cambiar una línea de código de la aplicación.
```

### Message Broker

```
  Service A ──► [Kafka/RabbitMQ] ──► Service B

  Desacopla servicios temporalmente:
  - Si B está caído, los mensajes esperan en la queue
  - A no necesita conocer la existencia de B
  - Múltiples consumers pueden procesar el mismo evento
```

## Modelo mental

- **API Gateway** = la recepción del edificio. Verifica identidad, dirige a la oficina correcta.
- **Service Discovery** = el directorio telefónico interno que se actualiza automáticamente.
- **Service Mesh** = el sistema de seguridad del edificio que cifra todas las comunicaciones y monitorea todo el tráfico entre oficinas.
- **Message Broker** = el sistema de correo interno con buzones.

## Insights de ingeniería

- **K8s Services como service discovery**: en Kubernetes, los Services proporcionan DNS-based discovery + load balancing. No necesitas Consul/etcd para service discovery si ya estás en K8s.
- **API Gateway vs Service Mesh**: el API Gateway es para tráfico norte-sur (external → internal). El service mesh es para tráfico este-oeste (internal ↔ internal). No son mutuamente excluyentes.
- **No necesitas todas las piezas**: un monolito + Nginx + PostgreSQL + Redis es suficiente para la mayoría de productos. Agrega complejidad solo cuando el problema lo requiere.
