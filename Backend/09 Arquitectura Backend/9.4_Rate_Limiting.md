# 9.4 Rate Limiting

## Definición

Rate limiting controla cuántas requests un cliente puede hacer en un período de tiempo. Protege contra abuso, DDoS, y evita que un cliente monopolice recursos.

## Mecánica interna

### Token Bucket

```
  Bucket: capacidad 10 tokens, refill 1 token/segundo

  t=0:  [10 tokens] request → consume 1 → [9 tokens]  ✅
  t=0:  [9 tokens]  request → consume 1 → [8 tokens]  ✅
  ...burst de 10 requests...
  t=0:  [0 tokens]  request → no tokens  → 429         ❌
  t=1:  [1 token]   request → consume 1  → [0]         ✅

  Permite bursts (hasta bucket capacity) + sustained rate (refill rate).
```

### Leaky Bucket

```
  Requests entran al bucket. Se procesan a tasa constante.
  Si el bucket está lleno → request rechazada.

  ┌─────────┐
  │ ○ ○ ○ ○ │ ← requests en queue
  │ ○ ○ ○   │
  └────┬────┘
       │ leak rate: 1 req/seg
       ▼
  [procesada]

  No permite bursts. Tasa constante siempre.
```

### Fixed Window vs Sliding Window

```
  Fixed window (100 req/min):
  |  min 1: 98  | min 2: 2+98=100 |
  Problema: 98 al final de min 1 + 98 al inicio de min 2 = 196 en 1 min real.

  Sliding window:
  Cuenta requests en los últimos 60 segundos (deslizante).
  Más preciso, más costoso de implementar.

  Sliding window counter (aproximación):
  current_window_count + previous_window_count × overlap_percentage
```

### Rate Limiting Distribuido

```
  Con múltiples instancias de tu API:

  Instance 1 → [Redis counter: "user:42:count" = 45] ← atomic INCR + EXPIRE
  Instance 2 → [Redis counter: "user:42:count" = 46]
  Instance 3 → [Redis counter: "user:42:count" = 47]

  Redis garantiza atomicidad con INCR + verificar < limit.
  Lua script en Redis para hacer check + increment atómico.
```

## Modelo mental

- **Token bucket** = una máquina de chicles con recarga automática. Puedes sacar varios de golpe (burst), pero se recarga a tasa constante.
- **Leaky bucket** = una manguera con flujo constante. No importa cuánta agua le eches, sale a la misma velocidad.

## Insights de ingeniería

- **Token bucket para APIs**: permite bursts razonables (usuario carga una página = 5 requests simultáneas) + límite sostenido.
- **Response headers**: retorna `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` para que los clientes se auto-regulen.
- **Rate limit por dimensiones**: por IP (público), por API key (autenticado), por endpoint (proteger endpoints costosos), por tenant (multi-tenant SaaS).
- **429 + Retry-After**: siempre retorna `Retry-After` header con el número de segundos que el cliente debe esperar.
