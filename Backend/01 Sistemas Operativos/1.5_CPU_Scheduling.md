# 1.5 CPU Scheduling

## Definición

El CPU scheduler del OS decide qué thread/proceso se ejecuta en cada core en cada momento. Para backend, esto determina la latencia de respuesta bajo carga concurrente, el fairness entre requests y el impacto de tareas CPU-bound en el throughput general.

## Mecánica interna

### Conceptos base

- **Time slice (quantum)**: tiempo máximo que un thread puede ejecutar antes de ser preemptado (~1-10 ms)
- **Preemption**: el scheduler interrumpe un thread para dar CPU a otro
- **Runqueue**: cola de threads listos para ejecutar (una por CPU core)
- **Voluntary context switch**: el thread cede CPU voluntariamente (e.g., `read()` que bloquea)
- **Involuntary context switch**: el scheduler fuerza el cambio (quantum expirado)

### Algoritmos

| Algoritmo | Mecánica | Pros | Cons | Usado en |
|-----------|---------|------|------|----------|
| **FIFO** | Primero en llegar, primero en ejecutar | Simple | Starvation, convoy effect | Sistemas embebidos |
| **Round Robin** | Cada thread recibe un quantum fijo | Fair | Ineficiente si los quanta son muy cortos | Base de muchos schedulers |
| **Priority** | Threads con mayor prioridad ejecutan primero | Control preciso | Priority inversion, starvation | RTOS |
| **CFS** | Tiempo virtual equalizado entre threads | Fair, adaptativo | Overhead de red-black tree | Linux (default) |

### CFS (Completely Fair Scheduler) — Linux

CFS mantiene un red-black tree ordenado por `vruntime` (tiempo virtual ejecutado):

```
  Red-Black Tree (ordenado por vruntime)

          Thread C (vruntime=15)
         /                    \
  Thread A (vruntime=10)    Thread D (vruntime=20)
       /
  Thread B (vruntime=5)  ← próximo a ejecutar (menor vruntime)
```

- El thread con menor `vruntime` se ejecuta primero
- Threads I/O-bound acumulan menos vruntime (bloquean antes del quantum) → reciben boost natural
- Threads CPU-bound acumulan más vruntime → son preemptados más frecuentemente

### Priority inversion

```
  Thread de alta prioridad (A) espera un lock
  Thread de baja prioridad (C) tiene el lock pero no ejecuta
  Thread de media prioridad (B) ejecuta y preempta a C

  Resultado: A (alta) está bloqueado por B (media)

  Solución: Priority inheritance — C hereda temporalmente la prioridad de A
```

Ejemplo real: el Mars Pathfinder tuvo priority inversion en 1997 que causaba resets.

## Modelo mental

El scheduler es un director de orquesta que decide quién toca en cada momento. CFS le da a cada músico tiempo proporcional, pero los que tocan notas cortas (I/O-bound) naturalmente dejan espacio para otros, mientras los que quieren tocar solos todo el tiempo (CPU-bound) son interrumpidos para dar fairness.

## Modos de fallo

| Síntoma | Causa probable | Señal |
|---------|---------------|-------|
| Latencia p99 alta con CPU disponible | Thread importante compitiendo con threads de baja prioridad | Involuntary context switches altos |
| Starvation | Thread CPU-bound monopoliza el core | Un thread con 100% CPU, otros esperando |
| Latencia inconsistente entre cores | NUMA: thread migra entre sockets | `numastat` muestra remote memory access |

## Implicaciones de rendimiento

- **CPU pinning** (`taskset`, `isolcpus`): fijar threads críticos a cores específicos elimina migration overhead y mejora cache locality
- **NUMA awareness**: en servidores multi-socket, allocar memoria en el nodo local del core donde ejecuta el thread
- **`nice` y `renice`**: ajustar prioridad relativa de procesos (-20 a +19)
- **cgroups**: en contenedores, `cpu.shares` y `cpu.cfs_quota_us` controlan cuánto CPU puede usar un contenedor

```bash
# CPU pinning
taskset -c 0,1 ./my-server              # Fijar a cores 0 y 1

# Ver scheduling info
chrt -p <PID>                            # Scheduling policy y prioridad
cat /proc/<PID>/sched                    # Estadísticas de scheduling

# Context switches por proceso
pidstat -w -p <PID> 1                    # cswch/s (voluntario) y nvcswch/s (involuntario)
vmstat 1                                 # cs total del sistema

# NUMA
numactl --cpunodebind=0 --membind=0 ./my-server  # Fijar a NUMA node 0
```

## Insights de ingeniería

- **Trade-off**: quantum corto = mejor fairness pero más context switches. Quantum largo = menos overhead pero peor latencia interactiva. Linux CFS adapta el quantum dinámicamente.
- **Pitfall en Kubernetes**: no configurar `resources.requests.cpu` hace que el scheduler de K8s coloque tu pod en cualquier nodo, y CFS le dé poca CPU si hay contención. `requests` define el peso en `cpu.shares`.
- **Regla**: si tu servicio tiene p99 latency spikes en un host con otros servicios: revisa involuntary context switches. Si son altos, necesitas CPU isolation (dedicated cores o mejor resource limits).
