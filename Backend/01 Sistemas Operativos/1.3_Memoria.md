# 1.3 Memoria

## Definición

La gestión de memoria determina cómo un proceso almacena, accede y libera datos en tiempo de ejecución. Para backend, entender stack vs heap, virtual memory y paging es la diferencia entre un servicio estable y uno que crashea bajo carga con OOM kills.

## Mecánica interna

### Layout de memoria de un proceso

```
  Direcciones altas
  ┌──────────────────────┐
  │       Stack          │  ← Crece hacia abajo
  │  (variables locales, │    Tamaño fijo por thread (~1-8 MB)
  │   return addresses)  │    Allocation/deallocation: automático (LIFO)
  ├──────────────────────┤
  │         ↓            │
  │    (espacio libre)   │
  │         ↑            │
  ├──────────────────────┤
  │        Heap          │  ← Crece hacia arriba
  │  (allocation         │    Tamaño dinámico
  │   dinámica: malloc,  │    Allocation: manual o GC
  │   new, make)         │    Fragmentación posible
  ├──────────────────────┤
  │   BSS (uninitialized │    Variables globales sin valor inicial
  │    globals)          │
  ├──────────────────────┤
  │   Data (initialized  │    Variables globales con valor
  │    globals)          │
  ├──────────────────────┤
  │   Text (code)        │    Instrucciones ejecutables (read-only)
  └──────────────────────┘
  Direcciones bajas
```

### Stack vs Heap

| Aspecto | Stack | Heap |
|---------|-------|------|
| Allocation | Push/pop automático (O(1)) | `malloc`/`free` o GC (costoso) |
| Deallocation | Al salir de la función | Manual o GC |
| Tamaño | Fijo por thread (1-8 MB) | Limitado por RAM disponible |
| Fragmentación | No | Sí (externa e interna) |
| Thread-safe | Sí (cada thread tiene el suyo) | No (compartido, requiere sync) |
| Qué almacena | Variables locales, parámetros, return addresses | Objetos, strings dinámicos, buffers, estructuras compartidas |
| Velocidad | Rápido (ya en cache, localidad) | Lento (pointer chasing, posible page fault) |

### Virtual memory

Cada proceso ve un espacio de direcciones virtual continuo, independiente del layout real en RAM:

```
  Proceso A                        RAM física
  ┌─────────────┐                  ┌────────────────┐
  │ Page 0 ─────│──── Page Table ──│► Frame 47      │
  │ Page 1 ─────│──────────────────│► Frame 12      │
  │ Page 2 ─────│──────────────────│► (en disco)    │ ← Page fault
  │ Page 3 ─────│──────────────────│► Frame 83      │
  └─────────────┘                  └────────────────┘

  Proceso B
  ┌─────────────┐
  │ Page 0 ─────│──── Page Table ──│► Frame 5       │
  │ Page 1 ─────│──────────────────│► Frame 47      │  ← Shared (e.g., libc)
  └─────────────┘
```

Beneficios:
- **Aislamiento**: un proceso no puede acceder a memoria de otro
- **Overcommit**: el kernel puede prometer más memoria de la que tiene (Linux lo hace por defecto)
- **Memory-mapped files**: mapear archivos directamente al espacio de direcciones (`mmap`)
- **Copy-on-write**: `fork()` no copia realmente la memoria hasta que alguien escribe

### Paging

- RAM dividida en **frames** (típicamente 4 KB)
- Espacio virtual dividido en **pages** (mismo tamaño)
- **Page table**: mapeo page → frame, gestionado por el kernel
- **Page fault**: acceso a una page que no está en RAM → el kernel la trae del disco (swap) o asigna un nuevo frame

Tipos de page fault:
- **Minor**: la page existe en RAM pero no está mapeada (se resuelve actualizando la page table)
- **Major**: la page está en disco (swap) → I/O de disco → ~5-10 ms de latencia

### TLB (Translation Lookaside Buffer)

Cache hardware de traducciones virtual → físico:

```
  CPU necesita dirección virtual 0x7fff1234
      │
      ├── TLB hit? → Frame 47 directo (~1 ns)
      │
      └── TLB miss? → Recorrer page table en RAM (~10-100 ns)
                       └── Actualizar TLB
```

- TLB tiene ~64-1024 entradas
- **TLB flush** ocurre en context switch entre procesos (no entre threads del mismo proceso) — por eso process switch es más caro

### Garbage Collection

Lenguajes con GC (Java, Go, Python, C#) automatizan la liberación de heap:

| Estrategia | Cómo funciona | Pausa | Usado en |
|-----------|---------------|-------|----------|
| **Mark & Sweep** | Marca objetos alcanzables, barre el resto | Stop-the-world | Java (old gen) |
| **Generational** | Objetos jóvenes se recolectan más frecuentemente | Pausas cortas + ocasionales largas | Java, .NET |
| **Concurrent** | GC corre en paralelo con la app | Mínimas | Go, Java G1/ZGC |
| **Reference counting** | Cada objeto cuenta sus referencias | Sin pausa, pero no detecta ciclos | Python, Swift |

Go GC: concurrent, tri-color mark & sweep. Target: pausas <1 ms. En producción, GC pressure aparece cuando se aloca mucho en heap (strings temporales, encoding/decoding).

## Modelo mental

La memoria virtual es como un mapa de calles de una ciudad virtual. Cada proceso tiene su propio mapa, pero las calles reales (RAM) son compartidas. El TLB es un GPS que cachea las rutas más frecuentes. Un page fault es cuando necesitas ir a una calle que ni siquiera existe todavía — el kernel tiene que construirla.

## Modos de fallo

| Síntoma | Causa probable | Señal |
|---------|---------------|-------|
| OOM Kill | Heap crece sin control (memory leak) | `dmesg | grep -i oom` |
| Latencia p99 spikes periódicos | GC stop-the-world | Logs de GC, métricas de pausa |
| Thrashing (sistema muy lento) | RAM llena → swap constante → page faults masivos | `vmstat`: alto `si`/`so` (swap in/out) |
| Stack overflow | Recursión muy profunda o stack demasiado pequeño | `SIGSEGV`, crash con "stack overflow" |
| Segfault | Acceso a memoria no mapeada o protegida | `SIGSEGV` en logs, core dump |

## Impacto en sistemas distribuidos

- **Memory overcommit** de Linux: el kernel promete más RAM de la que tiene. En Kubernetes, si no pones `resources.limits.memory`, el OOM killer elige tu pod cuando el nodo se queda sin RAM.
- **Heap sizing en JVM**: en contenedores, la JVM debe conocer el límite real de memoria (`-XX:MaxRAMPercentage`). Sin esto, la JVM ve la RAM del host, no del contenedor.
- **GC pauses** causan health check failures → el orquestador reinicia el pod → cascading restarts.

## Implicaciones de rendimiento

- `mmap` para archivos grandes evita copiar datos al user space (zero-copy read)
- Huge Pages (2 MB o 1 GB en vez de 4 KB): reducen TLB misses para bases de datos con grandes working sets (PostgreSQL, Redis)
- Pre-allocar buffers en el heap evita allocation pressure y GC
- Object pooling: reutilizar objetos en vez de allocar/liberar (connection pools, buffer pools)

## Herramientas de diagnóstico

```bash
# Uso de memoria de un proceso
cat /proc/<PID>/status | grep -E 'VmSize|VmRSS|VmSwap'
# VmSize: virtual memory total
# VmRSS: resident (realmente en RAM)
# VmSwap: en swap

# Mapa de memoria detallado
pmap -x <PID>
cat /proc/<PID>/smaps

# Page faults
ps -o min_flt,maj_flt -p <PID>
# min_flt: minor page faults
# maj_flt: major page faults (disco)

# Swap usage
swapon --show
vmstat 1                    # Columnas si/so = swap in/out

# GC logs (Java)
java -Xlog:gc* -jar app.jar

# GC trace (Go)
GODEBUG=gctrace=1 ./my-server

# Memory profiling
valgrind --tool=massif ./my-server    # C/C++
go tool pprof http://localhost:6060/debug/pprof/heap    # Go
```

## Insights de ingeniería

- **Trade-off**: GC elimina memory leaks pero introduce pausas impredecibles. C/Rust dan control total pero requieren disciplina. Go busca el punto medio con GC concurrent de baja pausa.
- **Pitfall**: en Docker/K8s, no configurar `resources.limits.memory` = el OOM killer elige arbitrariamente qué matar cuando el nodo se queda sin RAM. Siempre definir limits.
- **Regla de producción**: RSS (Resident Set Size) es lo que importa, no virtual Size. Monitorea RSS y alerta cuando supera el 80% del limit.
- **Scaling consideration**: si tu servicio es memory-bound (cache en proceso, grandes datasets), vertical scaling puede ser más efectivo que horizontal — pero tiene techo.
