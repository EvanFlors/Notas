# 1.7 CLI y Entorno de Ejecución

## Definición

La CLI (Command Line Interface) es la interfaz principal de operación de servidores en producción. Dominar el shell, la gestión de procesos, y las herramientas de diagnóstico del sistema operativo es fundamental para deployments, debugging, y operaciones de backend.

## Mecánica interna

### Shell: ciclo de ejecución

```
  $ ls -la /var/log | grep error | head -5

  Shell parsea:
  1. Tokeniza: ["ls", "-la", "/var/log"] | ["grep", "error"] | ["head", "-5"]
  2. Crea pipeline: 3 procesos conectados por pipes

  ┌──────┐  stdout → stdin  ┌──────┐  stdout → stdin  ┌──────┐
  │  ls  │ ──── pipe() ──── │ grep │ ──── pipe() ──── │ head │
  └──────┘                  └──────┘                  └──────┘

  Internamente:
  - Shell llama fork() 3 veces
  - Conecta stdout/stdin con pipe() entre cada par
  - Cada hijo llama exec() con su comando
  - Shell espera con waitpid() al último proceso del pipeline
```

### Variables de entorno

```
  ┌─────────────────────────────────────────┐
  │           Proceso (envp[])              │
  │                                         │
  │  PATH=/usr/bin:/usr/local/bin           │
  │  HOME=/home/user                        │
  │  DATABASE_URL=postgres://...            │ ← config de la app
  │  API_KEY=sk-...                         │ ← secretos
  │  NODE_ENV=production                    │ ← modo de ejecución
  │  PORT=8080                              │ ← puerto del servidor
  └─────────────────────────────────────────┘

  Herencia: fork() copia el environment al hijo.
  exec() preserva el environment (a menos que uses execve con nuevo envp).
```

### Gestión de procesos en producción

```bash
# Señales relevantes para backend
kill -SIGTERM <PID>    # Graceful shutdown (tu app debe manejar esta señal)
kill -SIGKILL <PID>    # Kill inmediato, sin cleanup (último recurso)
kill -SIGHUP <PID>     # Reload config sin restart (Nginx, HAProxy)
kill -SIGUSR1 <PID>    # Log rotation (Nginx), heap dump (Node.js)
```

| Señal | Número | Default | Uso en backend |
|-------|--------|---------|---------------|
| SIGTERM | 15 | Terminar | Graceful shutdown, K8s preStop |
| SIGKILL | 9 | Terminar (no interceptable) | Forzar muerte |
| SIGHUP | 1 | Terminar | Reload configuración |
| SIGINT | 2 | Terminar | Ctrl+C |
| SIGUSR1 | 10 | Terminar | Custom (log rotate, debug toggle) |
| SIGCHLD | 17 | Ignorar | Notificación de hijo terminado |

### Herramientas esenciales de diagnóstico

```bash
# === Procesos ===
ps aux --sort=-%mem | head -20       # Top procesos por memoria
top -bn1 | head -20                  # Snapshot de recursos
htop                                 # Interactivo (mejor que top)

# === Archivos y conexiones ===
lsof -p <PID>                       # Todos los fds abiertos por un proceso
lsof -i :8080                       # Quién escucha en puerto 8080
lsof -i TCP -s TCP:ESTABLISHED      # Conexiones establecidas

# === System calls ===
strace -p <PID> -e trace=network    # Syscalls de red de un proceso
strace -c -p <PID>                  # Resumen estadístico de syscalls
strace -e trace=open,read,write     # Solo I/O de archivos

# === Memoria ===
free -h                             # Memoria del sistema
cat /proc/<PID>/status | grep Vm    # Memoria de un proceso específico
pmap -x <PID>                       # Mapa de memoria detallado

# === Disco ===
iostat -x 1                         # I/O por dispositivo, cada segundo
df -h                               # Uso de disco por partición
du -sh /var/log/*                   # Tamaño de cada directorio

# === Red ===
ss -tlnp                            # Puertos TCP en escucha (reemplaza netstat)
ss -s                               # Resumen de sockets
curl -w "@curl-format.txt" URL      # Timing detallado de HTTP request
```

### Process management: systemd

```
  systemd (PID 1)
  │
  ├── myapp.service          ← tu backend
  │   ├── Restart=on-failure
  │   ├── WatchdogSec=30     ← health check cada 30s
  │   └── LimitNOFILE=65535  ← max file descriptors
  │
  └── nginx.service          ← reverse proxy
```

```ini
# /etc/systemd/system/myapp.service
[Unit]
Description=My Backend API
After=network.target postgresql.service

[Service]
Type=notify
User=appuser
WorkingDirectory=/opt/myapp
ExecStart=/opt/myapp/bin/server
ExecReload=/bin/kill -HUP $MAINPID
Restart=on-failure
RestartSec=5
LimitNOFILE=65535
Environment=NODE_ENV=production
EnvironmentFile=/etc/myapp/env

[Install]
WantedBy=multi-user.target
```

```bash
systemctl start myapp        # Iniciar
systemctl status myapp       # Ver estado y logs recientes
journalctl -u myapp -f       # Logs en tiempo real
systemctl daemon-reload      # Después de modificar el .service
```

## Modelo mental

La CLI es tu panel de control del servidor. `ps`/`top` son los medidores, `strace`/`lsof` son las sondas de diagnóstico, las señales son los interruptores, y `systemd` es el supervisor automático que reinicia las cosas cuando fallan.

## Modos de fallo

| Síntoma | Causa probable | Herramienta |
|---------|---------------|-------------|
| Proceso muere sin logs | SIGKILL (OOM killer) | `dmesg | grep -i oom` |
| Puerto ya en uso al restart | Proceso zombie en el puerto | `lsof -i :<port>`, luego kill |
| App no arranca | Variable de entorno faltante | `systemctl status`, logs con `journalctl` |
| Disco lleno, servicio caído | Logs sin rotación | `du -sh /var/log/*`, configurar logrotate |
| Conexiones TIME_WAIT acumuladas | No reusar sockets | `ss -s`, ajustar `tcp_tw_reuse` |

## Impacto en sistemas distribuidos

En producción, rara vez haces SSH a servidores individuales. Pero entender estas herramientas es esencial porque:
- Los contenedores ejecutan estos mismos comandos internamente
- Kubernetes `exec` te da un shell dentro del pod
- Las alertas de monitoreo reportan las mismas métricas (CPU, memoria, fds, sockets)
- El debugging de problemas de red entre microservicios requiere `ss`, `tcpdump`, `strace`

## Herramientas de diagnóstico

```bash
# Debugging completo de un backend que no responde
PID=$(pgrep -f "myapp")
echo "=== Estado del proceso ==="
ps -p $PID -o pid,ppid,stat,vsz,rss,etime,args

echo "=== File descriptors ==="
ls /proc/$PID/fd | wc -l              # Cuántos fds abiertos
cat /proc/$PID/limits | grep "open files"  # Límite

echo "=== Conexiones de red ==="
ss -tnp | grep $PID | awk '{print $4}' | sort | uniq -c | sort -rn

echo "=== Syscalls activas ==="
timeout 5 strace -c -p $PID 2>&1      # Perfil de 5 segundos
```

## Insights de ingeniería

- **12-Factor App**: la configuración va en variables de entorno, no en archivos de config dentro del código. Esto permite cambiar comportamiento entre environments (dev/staging/prod) sin rebuild.
- **Graceful shutdown**: tu app DEBE capturar SIGTERM, dejar de aceptar conexiones nuevas, terminar las requests activas, cerrar conexiones a DB/cache, y luego salir. Kubernetes envía SIGTERM y espera 30s antes de SIGKILL.
- **No uses `kill -9` como primera opción**: siempre intenta SIGTERM primero. SIGKILL no permite cleanup (conexiones a DB quedan abiertas, archivos temporales no se borran, locks no se liberan).
- **File descriptor awareness**: cada conexión TCP, archivo abierto, pipe, y socket consume un fd. El límite default (1024) es insuficiente para servidores de alto tráfico. Ajusta con `ulimit -n` o `LimitNOFILE` en systemd.
