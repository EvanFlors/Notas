# 1.1 Arquitectura de Computadoras

## Definición

La arquitectura de computadoras describe cómo se organizan CPU (Unidad Central de Procesamiento), memoria y dispositivos de I/O (Entrada/Salida) para ejecutar instrucciones. Para backend, lo relevante es entender la jerarquía de latencia entre componentes y cómo el hardware limita o potencia el software.

## Mecánica interna

### CPU

La CPU ejecuta instrucciones en un pipeline de etapas (fetch → decode → execute → writeback). Los componentes relevantes para backend:

- **ALU (Arithmetic Logic Unit)**: ejecuta operaciones aritméticas y lógicas
- **Registros**: almacenamiento ultra rápido dentro de la CPU (~0.3 ns)
- **Pipeline**: permite ejecutar múltiples instrucciones en diferentes etapas simultáneamente
- **Branch predictor**: predice saltos condicionales para mantener el pipeline lleno; una mala predicción cuesta ~15-20 ciclos

### Jerarquía de cache

```
  CPU Core
  ┌──────────────────┐
  │  Registros       │  ~0.3 ns    (bytes)
  │  ┌──────────┐    │
  │  │ L1 Cache │    │  ~1 ns      (32-64 KB por core)
  │  └──────────┘    │
  │  ┌──────────┐    │
  │  │ L2 Cache │    │  ~4 ns      (256 KB - 1 MB por core)
  │  └──────────┘    │
  └──────────────────┘
  ┌──────────────────┐
  │  L3 Cache        │  ~10 ns     (8-64 MB compartido)
  └──────────────────┘
  ┌──────────────────┐
  │  RAM (DRAM)      │  ~100 ns    (GBs)
  └──────────────────┘
  ┌──────────────────┐
  │  SSD (NVMe)      │  ~25 µs     (TBs)
  └──────────────────┘
  ┌──────────────────┐
  │  HDD             │  ~5 ms      (TBs)
  └──────────────────┘
```

Cada nivel es ~10x más lento que el anterior. Esta jerarquía define por qué:
- El buffer pool de PostgreSQL cachea páginas en RAM
- Redis mantiene todo en memoria
- Un cache miss de L3 a RAM cuesta 100 ns (en un servidor procesando millones de requests, esto suma)

### Interrupciones

- **Hardware interrupts**: dispositivos (NIC, disco) notifican a la CPU que hay datos listos
- **Software interrupts (traps)**: system calls, excepciones
- Cada interrupción causa un context switch parcial — en servidores de alta frecuencia (10Gbps NICs), el overhead de interrupciones puede saturar un core

### DMA (Direct Memory Access)

El DMA controller transfiere datos entre dispositivos y RAM sin involucrar la CPU:

```
  Sin DMA:  Disco → CPU → RAM        (CPU ocupada copiando bytes)
  Con DMA:  Disco → DMA → RAM        (CPU libre para procesar)
                     │
                     └── Interrumpe a CPU solo cuando termina
```

Relevante para `sendfile()` (zero-copy): datos del disco al socket sin pasar por user space.

## Modelo mental

La CPU es un chef ultra rápido con una mesa de trabajo pequeña (registros). Los caches son estantes cercanos. La RAM es el refrigerador. El disco es el almacén al fondo del restaurante. Cada vez que necesita algo del almacén, pierde tiempo caminando — por eso organiza los ingredientes más usados cerca.

## Modos de fallo

| Síntoma | Causa probable | Señal |
|---------|---------------|-------|
| Alta latencia en operaciones simples | Cache thrashing (working set > L3) | `perf stat` muestra alto cache-miss ratio |
| CPU al 100% en un core | Busy-wait o spin-lock | `top` muestra un thread consumiendo un core |
| Throughput de disco bajo | Sin DMA o filesystem mal configurado | `iostat` muestra alto `%util` con bajo throughput |

## Impacto en sistemas distribuidos

- **NUMA (Non-Uniform Memory Access)**: en servidores multi-socket, acceder a RAM del otro socket cuesta ~2x más. Relevante para bases de datos y servidores con muchos cores.
- Servidores con 64+ cores requieren NUMA-aware allocation para evitar degradación.

## Implicaciones de rendimiento

- Estructuras de datos cache-friendly (arrays contiguos) rinden hasta 10x más que linked lists (pointer chasing)
- False sharing: dos threads escribiendo en la misma cache line causan invalidaciones constantes
- Para backend: la elección entre JSON (parsing CPU-bound) y Protobuf (binario, cache-friendly) tiene impacto medible en p99

## Herramientas de diagnóstico

```bash
# Performance counters (Linux)
perf stat -e cache-misses,cache-references,instructions ./my-server

# CPU info
lscpu                          # Arquitectura, cores, caches
cat /proc/cpuinfo              # Detalle por core

# NUMA topology
numactl --hardware
numastat

# Interrupciones por segundo
watch -n1 cat /proc/interrupts
```

## Insights de ingeniería

- **Trade-off**: más cores vs más GHz — servidores de DB prefieren frecuencia alta (queries single-threaded); servidores web prefieren más cores (requests paralelos)
- **Pitfall en producción**: no configurar CPU affinity en contenedores puede causar que un proceso salte entre cores constantemente, destruyendo cache locality
- **Escalabilidad**: vertical scaling (más CPU/RAM) tiene techo físico; entender dónde está el bottleneck (CPU-bound vs I/O-bound) dicta si necesitas escalar vertical u horizontalmente
