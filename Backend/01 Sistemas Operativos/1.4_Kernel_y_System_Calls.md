# 1.4 Kernel y System Calls

## Definición

Las **system calls** son la interfaz entre user space (tu aplicación) y kernel space (el OS). Cada operación de red, disco o proceso que tu backend ejecuta pasa por una syscall. Los **file descriptors** son handles enteros que representan cualquier recurso de I/O abierto.

## Mecánica interna

### User space vs Kernel space

```
  ┌──────────────────────────────────────┐
  │           User Space                  │
  │  Tu código, librerías, runtime        │
  │  (no puede acceder a hardware)        │
  │                                       │
  │  read(fd, buf, len)  ← llamada       │
  └──────────┬───────────────────────────┘
             │ trap (interrupción software)
             │ cambio de modo: user → kernel
             ▼
  ┌──────────────────────────────────────┐
  │          Kernel Space                 │
  │  Drivers, scheduler, memory mgmt,    │
  │  network stack, filesystem            │
  │                                       │
  │  Ejecuta la operación real            │
  │  Copia datos al buffer del user       │
  │  Retorna resultado                    │
  └──────────────────────────────────────┘
```

Cada syscall tiene un costo: ~100-200 ns de overhead (mode switch + parameter validation). En un servidor haciendo 100K syscalls/segundo, esto suma.

### Syscalls clave para backend

| Syscall | Propósito | Uso en backend |
|---------|----------|----------------|
| `socket()` | Crear un endpoint de comunicación | Crear el listening socket del servidor |
| `bind()` | Asociar socket a dirección + puerto | Bind a `0.0.0.0:8080` |
| `listen()` | Marcar socket como pasivo (acepta conexiones) | Configurar backlog |
| `accept()` | Aceptar una conexión entrante → nuevo fd | Cada nuevo cliente = nuevo fd |
| `connect()` | Iniciar conexión a un servidor remoto | Llamada a DB, otro microservicio |
| `read()` / `recv()` | Leer datos de un fd | Leer el request HTTP del cliente |
| `write()` / `send()` | Escribir datos a un fd | Enviar la response HTTP |
| `close()` | Cerrar un fd | Terminar conexión |
| `fork()` | Crear proceso hijo (copia del padre) | Pre-fork servers (Nginx, Gunicorn) |
| `exec()` | Reemplazar el proceso actual con otro programa | Iniciar workers |
| `epoll_create()`, `epoll_ctl()`, `epoll_wait()` | I/O multiplexing eficiente | Event loop (ver 1.6) |
| `sendfile()` | Transferir datos entre fds sin pasar por user space | Servir archivos estáticos (zero-copy) |

### File descriptors

Un fd es un entero no negativo que referencia una entrada en la tabla de fds del proceso:

```
  Proceso (fd table)
  ┌────┬────────────────────────┐
  │ 0  │ stdin  (terminal)      │
  │ 1  │ stdout (terminal)      │
  │ 2  │ stderr (terminal)      │
  │ 3  │ listening socket :8080 │
  │ 4  │ conexión cliente A     │
  │ 5  │ conexión cliente B     │
  │ 6  │ archivo access.log     │
  │ 7  │ conexión a PostgreSQL  │
  │ 8  │ pipe a worker          │
  └────┴────────────────────────┘
```

Cada conexión TCP aceptada consume un fd. El límite por defecto suele ser 1024 (`ulimit -n`). Servidores de producción necesitan 65K+ fds.

### Buffers de kernel

Cuando llegan datos por la red, el kernel los almacena en buffers internos antes de que la aplicación los lea:

```
  NIC (tarjeta de red)
      │
      ▼
  Ring Buffer (driver)          ← DMA directo desde NIC
      │
      ▼
  Socket Receive Buffer         ← Kernel buffer (SO_RCVBUF)
  (sk_buff / skb)                  Default: 128 KB - 4 MB
      │
      ▼ read() / recv()
  User Buffer                   ← Tu buffer en user space
  (el que pasas a read())
```

Si tu aplicación no lee lo suficientemente rápido:
1. Socket receive buffer se llena
2. Kernel anuncia `window = 0` al sender (TCP flow control)
3. El sender para de enviar → **backpressure**

## Modelo mental

Las syscalls son la ventanilla de atención de un banco (kernel). Tu programa (user space) hace fila, entrega un formulario (parámetros), espera a que el cajero (kernel) procese la operación y te devuelve el resultado. Cada visita tiene overhead (espera en fila, validación). Los file descriptors son los números de ticket que te dan para referenciar operaciones abiertas.

## Flujo: servidor HTTP mínimo (syscalls)

```
  socket(AF_INET, SOCK_STREAM, 0)     → fd=3
  bind(3, {port:8080}, ...)            → OK
  listen(3, backlog=128)               → OK

  loop:
    accept(3, ...)                     → fd=4 (nuevo cliente)
    read(4, buffer, 4096)              → "GET / HTTP/1.1\r\n..."
    // procesar request
    write(4, response, len)            → HTTP response
    close(4)                           → fin de conexión
```

## Modos de fallo

| Síntoma | Causa probable | Señal |
|---------|---------------|-------|
| `Too many open files` | fd leak (conexiones no cerradas) | `lsof -p <PID> | wc -l` crece indefinidamente |
| `Connection refused` | Accept queue llena (backlog overflow) | `ss -ltn` muestra `Recv-Q` alto |
| Alto uso de CPU en syscalls | Demasiadas syscalls pequeñas | `strace -c -p <PID>` muestra distribución |
| `EMFILE` / `ENFILE` | Límite de fds alcanzado | `ulimit -n` demasiado bajo |

## Implicaciones de rendimiento

- `sendfile()` evita 2 copies (kernel → user → kernel) y 2 context switches — hasta 3x más rápido para servir archivos estáticos
- `writev()` / scatter-gather I/O: escribir múltiples buffers en una sola syscall (evita concatenar en user space)
- Batching syscalls: `io_uring` permite enviar múltiples operaciones al kernel en una sola syscall, reduciendo overhead dramáticamente

## Herramientas de diagnóstico

```bash
# Ver syscalls de un proceso en tiempo real
strace -p <PID> -e trace=network       # Solo syscalls de red
strace -p <PID> -e trace=file           # Solo syscalls de archivo
strace -c -p <PID>                      # Resumen estadístico

# File descriptors abiertos
ls -la /proc/<PID>/fd                   # Lista de fds
lsof -p <PID>                          # Detalle de cada fd
lsof -i :8080                          # Qué proceso usa el puerto 8080

# Límites de fd
ulimit -n                               # Límite actual (soft)
cat /proc/<PID>/limits                  # Límites del proceso
cat /proc/sys/fs/file-nr               # fds usados / max del sistema

# Socket buffers
sysctl net.core.rmem_max               # Max receive buffer
sysctl net.core.wmem_max               # Max send buffer
ss -tnm                                # Memoria usada por sockets
```

## Insights de ingeniería

- **Trade-off**: cada abstracción de alto nivel (ORM, HTTP framework) agrega syscalls. Un request HTTP simple puede generar 10+ syscalls. Perfilar con `strace -c` revela si tu bottleneck es syscall overhead.
- **Pitfall**: olvidar cerrar fds en error paths. En Go/Java el GC eventualmente cierra, pero en C/Python manual, un fd leak en un error handler poco frecuente se acumula durante semanas hasta crashear.
- **Producción**: siempre configurar `ulimit -n 65536` o mayor para servidores. En systemd: `LimitNOFILE=65536`. En Docker: `--ulimit nofile=65536:65536`.
- **Zero-copy chain**: `mmap()` + `sendfile()` = servir archivos sin que los datos toquen user space. Nginx lo usa por defecto.
