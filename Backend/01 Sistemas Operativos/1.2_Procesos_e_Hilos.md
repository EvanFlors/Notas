# 1.2 Procesos e Hilos

## Definición

Un **proceso** es una instancia de un programa en ejecución con su propio espacio de memoria aislado. Un **hilo** (thread) es una unidad de ejecución dentro de un proceso que comparte el espacio de memoria con otros hilos del mismo proceso. La elección entre procesos e hilos define el modelo de concurrencia de tu backend.

## Mecánica interna

### Proceso

El kernel mantiene un **PCB (Process Control Block)** por cada proceso:

```
  PCB
  ├── PID (Process ID)
  ├── Estado (running, ready, blocked, zombie)
  ├── Registros de CPU guardados
  ├── Page table pointer (memoria virtual propia)
  ├── File descriptor table
  ├── Signal handlers
  ├── Credenciales (UID, GID)
  └── Scheduling info (prioridad, CPU time usado)
```

Cada proceso tiene:
- Su propio espacio de direcciones virtuales (aislamiento total)
- Su propia tabla de file descriptors
- Su propia copia de variables (copy-on-write en `fork()`)

### Hilo (Thread)

El kernel mantiene un **TCB (Thread Control Block)** más ligero:

```
  Proceso
  ├── Espacio de memoria compartido
  ├── File descriptors compartidos
  ├── Signal handlers compartidos
  │
  ├── Thread 1 (TCB)
  │   ├── Stack propio (~1-8 MB)
  │   ├── Registros propios
  │   ├── Thread-local storage
  │   └── Estado de scheduling
  │
  ├── Thread 2 (TCB)
  │   ├── Stack propio
  │   └── ...
  └── ...
```

Lo que comparten los threads: heap, code segment, data segment, file descriptors.
Lo que NO comparten: stack, registros, thread-local storage.

### Context switching

Cambiar de un proceso/thread a otro en la CPU:

```
  Thread A ejecutando
      │
      ├── Guardar registros de A en TCB-A
      ├── Guardar estado de FPU/SSE
      ├── Cambiar page table (solo si es otro proceso)
      ├── Flush TLB (solo si es otro proceso)          ← costoso
      ├── Cargar registros de B desde TCB-B
      ├── Restaurar estado de FPU/SSE
      │
      ▼
  Thread B ejecutando
```

Costo real:
- Thread switch (mismo proceso): ~1-5 µs
- Process switch: ~5-30 µs (TLB flush + page table switch)
- En un servidor con 10K requests/segundo, context switching puede consumir >10% de CPU

### User threads vs Kernel threads

| Tipo | Scheduling | Overhead | Blocking I/O |
|------|-----------|----------|--------------|
| **Kernel threads** | OS scheduler | Alto (syscall para crear) | Un thread bloqueado no bloquea otros |
| **User threads** | Runtime del lenguaje | Bajo (no syscall) | Un thread bloqueado puede bloquear todos |
| **Green threads** (Go, Erlang) | Runtime M:N mapping | Muy bajo | Runtime hace el multiplex |

### Green threads y coroutines

Go goroutines: el runtime de Go multiplexa miles de goroutines sobre un pool pequeño de OS threads (M:N scheduling):

```
  OS Thread 1          OS Thread 2          OS Thread 3
  ┌─────────────┐      ┌─────────────┐      ┌─────────────┐
  │ goroutine A │      │ goroutine D │      │ goroutine G │
  │ goroutine B │      │ goroutine E │      │ goroutine H │
  │ goroutine C │      │ goroutine F │      │ goroutine I │
  └─────────────┘      └─────────────┘      └─────────────┘

  Stack de goroutine: ~2 KB inicial (vs ~1 MB de un OS thread)
  → Puedes tener millones de goroutines
```

Python asyncio / Node.js: modelo de event loop con un solo thread + cooperativemultitasking (coroutines yield en puntos de I/O).

## Modelo mental

Un proceso es un departamento en una empresa: tiene su propia oficina, archivos y personal. Los hilos son empleados dentro del mismo departamento que comparten la oficina y archivos pero cada uno tiene su propio escritorio (stack). Las goroutines son micro-tareas que los empleados van rotando — tan livianas que un solo empleado puede gestionar miles.

## Flujo paso a paso: fork() + exec()

```bash
# Cómo un servidor como Nginx pre-fork funciona:
```

```
  Master Process (PID 1)
      │
      ├── fork() ──► Worker Process (PID 2)  [copia del master]
      │                   │
      │                   └── exec() no se llama
      │                       (hereda el listening socket)
      │                       (acepta conexiones independientemente)
      │
      ├── fork() ──► Worker Process (PID 3)
      │
      └── fork() ──► Worker Process (PID 4)

  Cada worker hereda el file descriptor del socket.
  El kernel balancea accept() entre workers.
```

## Modos de fallo

| Síntoma | Causa probable | Señal de detección |
|---------|---------------|-------------------|
| CPU al 100%, throughput estancado | Demasiados threads → context switching domina | `vmstat` muestra alto `cs` (context switches/seg) |
| `OOM Killed` | Cada thread usa ~1 MB stack × 10K threads = 10 GB | `dmesg` muestra OOM killer |
| Zombie processes acumulados | Parent no llama `wait()` / `waitpid()` | `ps aux | grep Z` muestra zombies |
| Race conditions | Threads accediendo a shared state sin locks | Resultados inconsistentes, crashes esporádicos |
| Deadlock | Dos threads esperando mutuamente por locks | Proceso colgado, no consume CPU, no responde |

## Impacto en sistemas distribuidos

- **Thread-per-request** no escala a miles de conexiones concurrentes en un microservicio que recibe tráfico de otros 20 servicios
- **Pre-fork** (múltiples procesos) da aislamiento de memoria — un crash en un worker no tumba el master
- **Event loop** (single thread) es vulnerable a operaciones CPU-bound que bloquean el loop entero

## Implicaciones de rendimiento

| Modelo | Conexiones concurrentes | Uso de memoria | CPU overhead | Caso de uso |
|--------|------------------------|----------------|-------------|-------------|
| Thread-per-connection | ~1K-10K | Alto (1 MB/thread) | Alto (context switch) | Apps legacy, Java tradicional |
| Thread pool | ~10K-50K | Moderado | Moderado | Tomcat, Spring Boot |
| Event loop | ~100K+ | Bajo | Bajo (sin switch) | Node.js, Nginx |
| Green threads (M:N) | ~1M+ | Muy bajo (2 KB/goroutine) | Bajo | Go, Erlang/Elixir |

## Herramientas de diagnóstico

```bash
# Ver procesos y threads
ps -eLf                                  # Todos los threads de todos los procesos
ps -T -p <PID>                           # Threads de un proceso específico
ls /proc/<PID>/task/                      # Thread IDs en el filesystem

# Context switches
vmstat 1                                 # Columna 'cs'
pidstat -w -p <PID> 1                    # Context switches por proceso

# Threads en uso por un proceso
cat /proc/<PID>/status | grep Threads

# Strace: ver syscalls de un proceso
strace -f -p <PID> -e trace=clone,fork   # Ver creación de threads/procesos

# Thread dump (Java)
kill -3 <PID>                            # Envía SIGQUIT → thread dump en stdout
jstack <PID>                             # Java thread dump
```

## Insights de ingeniería

- **Trade-off fundamental**: threads dan simplicidad de programación (código secuencial) a costa de uso de memoria y context switching. Event loops dan eficiencia a costa de complejidad (callbacks, async/await).
- **Pitfall**: crear un thread por request en un servidor HTTP que recibe 10K req/s = 10K threads = ~10 GB de stacks + context switching masivo. Usa thread pool con tamaño fijo.
- **Regla de producción**: número óptimo de threads ≈ número de cores para CPU-bound; para I/O-bound, threads = cores × (1 + wait_time/service_time).
- **Go en producción**: las goroutines resuelven el trade-off — código secuencial simple con eficiencia de event loop. Por eso Go domina en infraestructura (Docker, Kubernetes, Prometheus).
