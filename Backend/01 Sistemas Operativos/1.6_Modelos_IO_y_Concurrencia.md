# 1.6 Modelos de I/O y Concurrencia

## Definición

El modelo de I/O determina cómo tu servidor espera datos de la red y el disco. Es la decisión arquitectónica más importante para el rendimiento de un backend: define cuántas conexiones concurrentes puede manejar y cómo utiliza CPU y memoria.

## Mecánica interna

### Los 5 modelos de I/O

```
  1. Blocking I/O              2. Non-blocking I/O
  ┌──────────┐                 ┌──────────┐
  │ App      │                 │ App      │
  │ read()   │ ← bloquea      │ read()   │ → EAGAIN (no data)
  │ ...      │   hasta datos   │ read()   │ → EAGAIN
  │ datos!   │ ← retorna      │ read()   │ → datos!
  └──────────┘                 └──────────┘
  Simple pero un thread        Busy-wait: desperdicia CPU
  por conexión                 haciendo polling constante

  3. I/O Multiplexing          4. Signal-driven I/O
  ┌──────────┐                 ┌──────────┐
  │ App      │                 │ App      │
  │ epoll()  │ ← bloquea      │ trabaja  │
  │ (espera  │   hasta que     │ SIGIO!   │ ← signal cuando hay datos
  │  N fds)  │   alguno ready  │ read()   │ → datos
  │ read(fd) │ ← lee el ready  └──────────┘
  └──────────┘                 Poco usado en backend
  Un thread para miles
  de conexiones

  5. Async I/O (io_uring)
  ┌──────────┐
  │ App      │
  │ submit() │ → kernel ejecuta la operación
  │ trabaja  │
  │ poll()   │ → completion event con datos
  └──────────┘
  True async: kernel hace todo
```

### epoll (Linux)

`epoll` es la base de Node.js, Nginx, Redis, y cualquier servidor de alto rendimiento en Linux:

```c
// 1. Crear instancia de epoll
int epfd = epoll_create1(0);

// 2. Registrar file descriptors a monitorear
struct epoll_event ev;
ev.events = EPOLLIN;          // Monitorear lectura
ev.data.fd = client_fd;
epoll_ctl(epfd, EPOLL_CTL_ADD, client_fd, &ev);

// 3. Esperar eventos (bloquea hasta que haya actividad)
struct epoll_event events[MAX_EVENTS];
int n = epoll_wait(epfd, events, MAX_EVENTS, timeout);

// 4. Procesar solo los fds que tienen datos
for (int i = 0; i < n; i++) {
    handle(events[i].data.fd);
}
```

Complejidad: `epoll_wait` es O(1) respecto al número total de fds monitoreados (solo retorna los activos). `select`/`poll` son O(n) — escanean todos los fds.

| Mecanismo | Complejidad | Max fds | Edge-triggered | Plataforma |
|-----------|-------------|---------|----------------|------------|
| `select` | O(n) | 1024 (FD_SETSIZE) | No | Todas |
| `poll` | O(n) | Sin límite | No | Todas |
| `epoll` | O(ready) | Sin límite | Sí | Linux |
| `kqueue` | O(ready) | Sin límite | Sí | macOS, BSD |

### Edge-triggered vs Level-triggered

```
  Level-triggered (default):
  epoll_wait retorna MIENTRAS haya datos → puedes leer parcialmente

  Edge-triggered (EPOLLET):
  epoll_wait retorna UNA VEZ cuando llegan datos nuevos →
  DEBES leer todo (loop hasta EAGAIN) o pierdes la notificación
```

Edge-triggered es más eficiente (menos syscalls) pero requiere que leas todo el buffer en cada evento.

### Event loop (modelo Node.js / Nginx)

```
  ┌─────────────────────────────────┐
  │          Event Loop             │
  │                                 │
  │  while (running):               │
  │    events = epoll_wait(epfd)    │  ← bloquea hasta actividad
  │    for event in events:         │
  │      if event.type == ACCEPT:   │
  │        new_fd = accept()        │
  │        register(new_fd, epfd)   │
  │      if event.type == READ:     │
  │        data = read(event.fd)    │
  │        response = process(data) │  ← DEBE ser non-blocking
  │        write(event.fd, response)│
  │      if event.type == CLOSE:    │
  │        close(event.fd)          │
  └─────────────────────────────────┘

  Un solo thread maneja miles de conexiones.
  PERO: si process() tarda 100ms (CPU-bound),
  TODAS las demás conexiones esperan.
```

### io_uring (Linux 5.1+)

Verdadera async I/O con submission queue y completion queue en shared memory:

```
  User space                    Kernel space
  ┌──────────────┐              ┌──────────────┐
  │ Submission   │  ────────►   │  Procesa     │
  │ Queue (SQ)   │  (ring buf)  │  operaciones │
  │ [read, write,│              │  en paralelo │
  │  accept, ...]│              │              │
  └──────────────┘              └──────┬───────┘
                                       │
  ┌──────────────┐              ┌──────┴───────┐
  │ Completion   │  ◄────────   │  Resultados  │
  │ Queue (CQ)   │  (ring buf)  │              │
  └──────────────┘              └──────────────┘

  Zero syscalls para submit/complete (shared memory rings)
```

## Modelo mental

- **Blocking I/O** = un cajero por cliente. Simple pero ineficiente.
- **epoll** = un cajero con un panel de luces. Cada luz indica qué ventanilla tiene un cliente listo. El cajero solo va a ventanillas con luz encendida.
- **io_uring** = el cajero deja formularios en una bandeja, el banco los procesa en background, y los resultados aparecen en otra bandeja. El cajero nunca espera.

## Modos de fallo

| Síntoma | Causa probable | Señal |
|---------|---------------|-------|
| Event loop bloqueado, todas las conexiones pausadas | Operación CPU-bound en el event loop | Latencia uniforme alta en todas las requests |
| `EMFILE` bajo carga | Cada conexión = 1 fd, límite alcanzado | `ulimit -n` demasiado bajo |
| Datos perdidos con EPOLLET | No leer hasta EAGAIN en edge-triggered | Responses parciales o colgadas |
| Alta latencia con pocos fds | Usando `select`/`poll` en vez de `epoll` | `strace` muestra poll() con todos los fds |

## Impacto en sistemas distribuidos

Un microservicio recibe conexiones de N otros servicios + mantiene conexiones a DB + cache + message queue. Con blocking I/O, necesitarías threads separados para cada conexión. Con epoll, un solo thread maneja todas. Esto es por qué:
- **Envoy** (service mesh proxy) usa event loops
- **Redis** es single-threaded con epoll y procesa 100K+ ops/seg
- **Nginx** maneja 10K+ conexiones concurrentes con pocos workers

## Herramientas de diagnóstico

```bash
# Ver qué modelo de I/O usa un proceso
strace -e trace=epoll_wait,poll,select,io_uring_enter -p <PID>

# Verificar si un proceso usa epoll
ls -la /proc/<PID>/fd | grep eventpoll

# Estadísticas de epoll
cat /proc/<PID>/fdinfo/<epoll_fd>    # Muestra fds registrados
```

## Insights de ingeniería

- **Trade-off central**: event loop (epoll) da máximo throughput de I/O pero requiere que todo sea non-blocking. Una sola operación bloqueante (DNS sync, file read sync, CPU-heavy computation) congela todas las conexiones.
- **Solución híbrida**: event loop para I/O + thread pool para tareas CPU-bound (Node.js con worker_threads, Nginx con thread pool para disk I/O).
- **io_uring en producción**: adoptado por librerías como Tokio (Rust) y siendo evaluado por bases de datos. Reduce syscall overhead hasta 10x para workloads de alto throughput.
- **Regla**: si tu servicio es I/O-bound (mayoría de backends), usa event loop o green threads. Si es CPU-bound (encoding, crypto, ML inference), usa thread pool con tamaño = número de cores.
