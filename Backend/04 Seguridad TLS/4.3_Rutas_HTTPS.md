# 4.3 Rutas de HTTPS

## Definición

Una "ruta HTTPS" es la secuencia completa de handshakes (TCP + TLS) que ocurre antes de enviar el primer byte de datos de aplicación. La latencia de esta ruta define el tiempo mínimo hasta el primer byte útil (TTFB). Cada combinación de protocolo ofrece trade-offs diferentes.

## Mecánica interna

### Ruta 1: HTTP/1.1 + TLS 1.2 (3 RTT)

```
  Cliente                          Servidor
     │                                │
     │── SYN ────────────────────►   │  ┐
     │◄── SYN-ACK ───────────────   │  │ TCP: 1 RTT
     │── ACK ────────────────────►   │  ┘
     │                                │
     │── ClientHello ────────────►   │  ┐
     │◄── ServerHello + Cert ────   │  │ TLS 1.2: 2 RTT
     │── KeyExchange + Finished ─►   │  │
     │◄── Finished ──────────────   │  ┘
     │                                │
     │── GET /api/users ─────────►   │  Primer dato de app
     │◄── 200 OK ────────────────   │
     │                                │
  Total: 3 RTT antes del primer byte de datos.
  Con RTT de 50ms = 150ms mínimo de latencia.
```

### Ruta 2: HTTP/2 + TLS 1.3 (2 RTT)

```
  Cliente                          Servidor
     │                                │
     │── SYN ────────────────────►   │  ┐
     │◄── SYN-ACK ───────────────   │  │ TCP: 1 RTT
     │── ACK ────────────────────►   │  ┘
     │                                │
     │── ClientHello + KeyShare ──►  │  ┐
     │◄── ServerHello + Finished ─   │  │ TLS 1.3: 1 RTT
     │── Finished ────────────────►  │  ┘
     │                                │
     │── GET /api/users (HTTP/2) ──► │
     │◄── 200 OK ────────────────   │
     │                                │
  Total: 2 RTT. TLS 1.3 reduce 1 RTT completo.
```

### Ruta 3: HTTP/3 QUIC (1 RTT)

```
  Cliente                          Servidor
     │                                │
     │── QUIC Initial ───────────►   │  ┐ Crypto + transport
     │   (ClientHello + transport)   │  │ en un solo paso
     │◄── QUIC Handshake ────────   │  │ 1 RTT total
     │   (ServerHello + Finished)    │  ┘
     │                                │
     │── GET /api/users (HTTP/3) ──► │
     │◄── 200 OK ────────────────   │
     │                                │
  Total: 1 RTT. QUIC integra TCP + TLS en un solo handshake sobre UDP.
```

### Ruta 4: HTTP/3 QUIC 0-RTT (0 RTT)

```
  Cliente (ya conectó antes)        Servidor
     │                                │
     │── QUIC Initial ───────────►   │  Incluye:
     │   + Early Data (0-RTT)        │  - Crypto resumption
     │   + GET /api/users            │  - Request de app
     │                                │
     │◄── QUIC Handshake ────────   │  ┐ Handshake completa
     │◄── 200 OK ────────────────   │  ┘ + response
     │                                │
  Total: 0 RTT. Los datos de aplicación viajan en el primer paquete.
  PERO: vulnerable a replay attacks (solo para requests idempotentes/GET).
```

### TCP Fast Open (TFO)

```
  Primera conexión (normal):
  SYN → SYN-ACK → ACK → data

  Conexiones subsecuentes con TFO:
  SYN + data + TFO cookie → SYN-ACK + data → ACK

  Ahorra 1 RTT en TCP, pero:
  - No soportado universalmente
  - Middleboxes (firewalls, NAT) pueden descartarlo
  - Solo para la primera request
```

### Comparación de latencia

```
  RTT = 50ms (mismo continente)

  HTTP/1.1 + TLS 1.2:  3 × 50ms = 150ms  ████████████████
  HTTP/2 + TLS 1.3:    2 × 50ms = 100ms   ███████████
  HTTP/3 (QUIC):       1 × 50ms =  50ms   █████
  HTTP/3 (0-RTT):      0 × 50ms =   0ms*  █

  RTT = 150ms (cross-continent)

  HTTP/1.1 + TLS 1.2:  3 × 150ms = 450ms  ████████████████████████
  HTTP/2 + TLS 1.3:    2 × 150ms = 300ms  ████████████████
  HTTP/3 (QUIC):       1 × 150ms = 150ms  ████████
  HTTP/3 (0-RTT):      0 × 150ms =   0ms* █

  * 0-RTT real incluye 1 RTT para completar handshake,
    pero los datos ya viajan en el primer paquete.
```

## Modelo mental

Piensa en cada ruta como pasos para entrar a un edificio seguro:
- **HTTP/1.1 + TLS 1.2**: tocar la puerta (TCP), mostrar ID (TLS round 1), confirmar identidad (TLS round 2), hablar.
- **HTTP/2 + TLS 1.3**: tocar la puerta (TCP), mostrar ID y hablar (TLS 1.3 en 1 paso).
- **QUIC**: tocar la puerta y mostrar ID al mismo tiempo, hablar.
- **QUIC 0-RTT**: ya te conocen, entras hablando.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| TTFB alto en primera conexión | Handshake TCP+TLS completo | Connection pooling, keep-alive |
| QUIC cae a HTTP/2 | Firewall/middlebox bloquea UDP | Verificar que UDP 443 está abierto |
| 0-RTT replay attack | Atacante replica early data | Solo usar 0-RTT para GET/idempotent |
| TFO no funciona | Middlebox elimina TFO option | Fallback a handshake normal |

## Impacto en sistemas distribuidos

- **Entre microservicios en el mismo datacenter**: RTT < 1ms. La diferencia entre rutas es negligible. Lo importante es connection pooling para evitar handshakes repetidos.
- **Edge/CDN a usuario**: RTT 50-200ms. Aquí QUIC 0-RTT marca la diferencia. Por eso Cloudflare, Google, y Facebook usan QUIC agresivamente.
- **Cross-region replication**: RTT 100-300ms. Cada RTT duele. Las bases de datos distribuidas minimizan round trips usando protocolos optimizados.

## Insights de ingeniería

- **Connection pooling amortiza el handshake**: si mantienes 10 conexiones persistentes a un servicio downstream, el costo de handshake se paga solo 10 veces (al inicio), no en cada request.
- **QUIC 0-RTT con cuidado**: solo para requests safe/idempotent (GET). Un atacante podría capturar y replicar el 0-RTT data, causando un request duplicado. Para POST con side effects, usa 1-RTT.
- **Mide, no asumas**: usa `curl -w` con timing breakdown para medir exactamente dónde está el bottleneck: ¿DNS? ¿TCP? ¿TLS? ¿TTFB del backend?

```bash
curl -w "DNS:%{time_namelookup} TCP:%{time_connect} TLS:%{time_appconnect} TTFB:%{time_starttransfer} Total:%{time_total}\n" -o /dev/null -s https://api.example.com
```
