# 4.1 Criptografía Aplicada

## Definición

Criptografía aplicada al backend cubre los primitivos que protegen datos en tránsito (TLS), datos en reposo (encryption at rest), autenticación (HMAC, signatures), y almacenamiento de passwords (key derivation). No necesitas ser criptógrafo, pero debes saber qué primitivo usar y por qué.

## Mecánica interna

### Symmetric Encryption (una clave)

```
  Misma clave para cifrar y descifrar:

  plaintext ──► [AES-256-GCM + key] ──► ciphertext
  ciphertext ──► [AES-256-GCM + key] ──► plaintext

  Algoritmo estándar: AES (Advanced Encryption Standard)
  Modos seguros: GCM (autenticado), CBC + HMAC

  Velocidad: ~1 GB/s en hardware moderno (AES-NI)
  Uso: datos en tránsito (TLS), datos en reposo (disk encryption)
```

Problema: ¿cómo comparten la clave dos partes que nunca se han visto? → Asymmetric encryption.

### Asymmetric Encryption (par de claves)

```
  Key pair: public key + private key

  Cifrar: plaintext + public key → ciphertext
  Descifrar: ciphertext + private key → plaintext

  Firmar: message + private key → signature
  Verificar: message + signature + public key → valid/invalid

  Algoritmos:
  - RSA (2048+ bits): legacy pero aún usado
  - ECDSA (P-256): más compacto, mismo nivel de seguridad
  - Ed25519: moderno, rápido, seguro (SSH keys, JWT)

  Velocidad: ~1000x más lento que AES
  Uso: intercambio de claves (TLS handshake), firmas digitales (JWT, certificates)
```

### Hashing

```
  input (cualquier tamaño) → [hash function] → digest (tamaño fijo)

  "hello"     → SHA-256 → 2cf24dba5fb0a30e26e83b2ac5b9e29e...  (64 hex chars)
  "hello!"    → SHA-256 → ce06092fb948d9ffac7d1a376e404b26...  (completamente diferente)

  Propiedades:
  - Determinista: mismo input → mismo output
  - One-way: no puedes recuperar el input del hash
  - Avalanche: cambiar 1 bit del input cambia ~50% del output
  - Collision resistant: computacionalmente imposible encontrar dos inputs con mismo hash
```

| Algoritmo | Output | Seguro | Uso |
|-----------|--------|--------|-----|
| MD5 | 128 bits | No (colisiones) | Solo checksums no-crypto |
| SHA-1 | 160 bits | No (colisiones) | Legacy, no usar |
| SHA-256 | 256 bits | Sí | Integridad, blockchain, certificados |
| SHA-3 | Variable | Sí | Alternativa a SHA-2 |
| BLAKE3 | 256 bits | Sí | Rápido, moderno |

### HMAC (Hash-based Message Authentication Code)

```
  HMAC = Hash(key + message)  (simplificado)

  Servidor genera: HMAC-SHA256(secret_key, "user_id=42&role=admin")
  → "a1b2c3d4..."

  Cliente envía: data + HMAC
  Servidor verifica: recalcula HMAC con su secret_key, compara.

  Si alguien modifica los datos, el HMAC no coincide.
  Sin la secret_key, no puedes generar un HMAC válido.

  Uso: JWT signature (HS256), API request signing (AWS Signature V4),
       webhook verification (GitHub, Stripe).
```

### Key Derivation (passwords)

```
  NUNCA almacenes passwords en texto plano.
  NUNCA uses SHA-256 directo (demasiado rápido → brute force fácil).

  Password → [KDF + salt + cost factor] → derived key (almacenar esto)

  Algoritmos (de mejor a peor):
  1. Argon2id    ← ganador de Password Hashing Competition (2015)
  2. bcrypt      ← estándar de facto, amplio soporte
  3. scrypt      ← memory-hard
  4. PBKDF2      ← legacy, aceptable con suficientes iteraciones

  Salt: random bytes únicos por password. Previene rainbow tables.
  Cost factor: hace el hash lento intencionalmente (~100ms por intento).

  Ejemplo (bcrypt):
  password: "s3cret"
  salt: random 16 bytes
  cost: 12 (2^12 = 4096 iteraciones)
  → "$2b$12$LJ3m4ys3Lp...aBcDeFgHiJkLmNoPqR"
```

## Modelo mental

- **Symmetric** = candado con una copia de llave. Rápido, pero ambos necesitan la llave.
- **Asymmetric** = buzón público. Cualquiera puede meter cartas (public key), solo tú puedes abrirlo (private key).
- **Hashing** = triturar un documento. Puedes verificar que dos documentos producen las mismas tiras, pero no reconstruir el documento original.
- **HMAC** = sello de cera con tu anillo personal. Prueba que TÚ sellaste el mensaje y que nadie lo modificó.
- **KDF** = candado que tarda 1 segundo en abrirse. Demasiado lento para brute force.

## Modos de fallo

| Error | Consecuencia | Corrección |
|-------|-------------|------------|
| Passwords con SHA-256 sin salt | Vulnerable a rainbow tables | Usar bcrypt/Argon2 con salt |
| Clave simétrica hardcodeada en código | Compromiso de todos los datos cifrados | Secrets manager (Vault, AWS SM) |
| MD5/SHA-1 para integridad | Colisiones posibles | Migrar a SHA-256 |
| No verificar HMAC con constant-time comparison | Timing attack | Usar `hmac.compare_digest()` |
| Reutilizar nonce/IV en AES-GCM | Compromiso total de confidencialidad | Nonce único por operación |

## Impacto en sistemas distribuidos

- **TLS everywhere**: toda comunicación entre microservicios debe usar TLS. mTLS (mutual TLS) autentica ambos lados, eliminando la necesidad de API keys internos.
- **JWT signing**: HS256 (HMAC) requiere shared secret entre servicios. RS256/ES256 (asymmetric) permite que cualquiera verifique con la public key sin conocer la private key — mejor para microservicios.
- **Secrets rotation**: rotar claves de encripción sin downtime requiere soportar múltiples claves activas (key versioning). El sistema descifra con la clave que corresponda a la versión del ciphertext.

## Insights de ingeniería

- **No inventes tu propia criptografía**: usa librerías estándar (libsodium, OpenSSL). Un error sutil (reutilizar nonce, timing leak) rompe toda la seguridad.
- **AES-256-GCM para encryption**: proporciona confidencialidad + integridad + autenticación en un solo paso. Es el modo usado por TLS 1.3.
- **bcrypt cost factor**: ajústalo para que tome ~100ms en tu hardware. Demasiado bajo = brute force fácil. Demasiado alto = login lento para usuarios legítimos.
- **Encryption at rest vs in transit**: TLS protege datos en la red. Para datos en disco (backups de DB, S3), necesitas encryption at rest separada (AES-256).
