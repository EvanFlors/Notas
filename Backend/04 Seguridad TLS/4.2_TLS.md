# 4.2 TLS (Transport Layer Security)

## Definición

TLS cifra y autentica la comunicación entre cliente y servidor sobre TCP. Es la capa que convierte HTTP en HTTPS. TLS 1.3 simplifica el handshake a 1 RTT, elimina cipher suites inseguras, y obliga forward secrecy.

## Mecánica interna

### TLS 1.2 Handshake (2 RTT)

```
  Cliente                                 Servidor
     │                                       │
     │── ClientHello ──────────────────►     │  Ciphers soportados, random
     │                                       │
     │◄── ServerHello ─────────────────     │  Cipher elegido, random
     │◄── Certificate ─────────────────     │  Certificado X.509
     │◄── ServerKeyExchange ───────────     │  Parámetros DH/ECDH
     │◄── ServerHelloDone ─────────────     │
     │                                       │
     │── ClientKeyExchange ────────────►     │  Pre-master secret (cifrado)
     │── ChangeCipherSpec ─────────────►     │  "Ahora cifro"
     │── Finished ─────────────────────►     │  Verify handshake
     │                                       │
     │◄── ChangeCipherSpec ────────────     │  "Ahora yo también cifro"
     │◄── Finished ────────────────────     │
     │                                       │
     │         Encrypted data flow           │

  Total: 2 round trips antes de enviar datos de aplicación.
```

### TLS 1.3 Handshake (1 RTT)

```
  Cliente                                 Servidor
     │                                       │
     │── ClientHello + KeyShare ───────►     │  Cipher + DH public key
     │                                       │
     │◄── ServerHello + KeyShare ──────     │  DH public key
     │◄── EncryptedExtensions ─────────     │  (ya cifrado)
     │◄── Certificate ─────────────────     │  (ya cifrado)
     │◄── CertificateVerify ───────────     │
     │◄── Finished ────────────────────     │
     │                                       │
     │── Finished ─────────────────────►     │
     │                                       │
     │         Encrypted data flow           │

  Total: 1 round trip. El cliente envía su key share en el primer mensaje.

  0-RTT (session resumption):
  Si el cliente ya conectó antes, puede enviar datos cifrados
  en el primer mensaje (pre-shared key). Riesgo: replay attacks.
```

### Mejoras de TLS 1.3 sobre 1.2

| Aspecto | TLS 1.2 | TLS 1.3 |
|---------|---------|---------|
| Handshake | 2 RTT | 1 RTT (0-RTT con resumption) |
| Forward secrecy | Opcional | Obligatorio (ECDHE) |
| Cipher suites | Muchas (incluyendo inseguras) | Solo 5 seguras |
| RSA key exchange | Sí (sin forward secrecy) | Eliminado |
| Encryption del handshake | Parcial | Casi todo cifrado |

### Certificados X.509

```
  ┌──────────────────────────────────┐
  │        Root CA Certificate        │  Preinstalado en el OS/browser
  │        (self-signed)              │  ~150 root CAs globales
  └──────────┬───────────────────────┘
             │ firma
  ┌──────────▼───────────────────────┐
  │     Intermediate CA Certificate   │  Emitido por Root CA
  └──────────┬───────────────────────┘
             │ firma
  ┌──────────▼───────────────────────┐
  │     Server Certificate            │  Tu certificado
  │     CN: api.example.com           │  Let's Encrypt, DigiCert, etc.
  │     Valid: 2024-01-01 to 2024-12  │
  │     Public Key: [ECDSA P-256]     │
  └──────────────────────────────────┘

  Verificación: el cliente sube la cadena validando cada firma
  hasta llegar a un Root CA que ya confía.
```

### mTLS (Mutual TLS)

```
  TLS normal:
  - Servidor presenta certificado → cliente verifica
  - Cliente es anónimo (autenticación por JWT/API key)

  mTLS:
  - Servidor presenta certificado → cliente verifica
  - Cliente presenta certificado → servidor verifica

  Uso: comunicación entre microservicios (service mesh: Istio, Linkerd).
  Cada servicio tiene su propio certificado emitido por una CA interna.
  No necesitas JWT ni API keys para inter-service auth.
```

## Modelo mental

TLS es como verificar la identidad de alguien por teléfono antes de compartir información sensible. El certificado es su credencial oficial emitida por una autoridad de confianza (CA). El handshake es el proceso de verificación y acuerdo sobre cómo cifrar. Después del handshake, la conversación es privada.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| `certificate has expired` | Certificado no renovado | Automatizar con Let's Encrypt + certbot |
| `certificate verify failed` | CA no confiable o chain incompleta | Incluir intermediate certs en el servidor |
| Handshake lento (>200ms) | TLS 1.2 + high latency link | Upgrade a TLS 1.3, usar session resumption |
| `SSL: WRONG_VERSION_NUMBER` | Puerto no-TLS contactado con TLS | Verificar puerto y configuración |
| `no common cipher suite` | Cliente y servidor no comparten ciphers | Actualizar configuración TLS |

## Impacto en sistemas distribuidos

- **Certificate management at scale**: con 100 microservicios, cada uno necesita un certificado. Service meshes (Istio) automatizan emisión y rotación via SPIFFE/SPIRE.
- **TLS termination**: el reverse proxy (Nginx, Envoy) termina TLS y reenvía HTTP plano al backend. Reduce la carga de crypto en la app pero requiere red interna segura, o mTLS hasta el backend.
- **Latencia de TLS**: en conexiones de corta duración, el handshake puede ser el 50%+ de la latencia total. Connection pooling y HTTP/2 (una conexión, múltiples requests) amortizan este costo.

## Herramientas de diagnóstico

```bash
# Ver certificado de un servidor
openssl s_client -connect api.example.com:443 -showcerts

# Verificar cadena de certificados
openssl verify -CAfile chain.pem server.pem

# Ver detalles del certificado
openssl x509 -in cert.pem -text -noout

# Test de configuración TLS
curl -vI https://api.example.com 2>&1 | grep -E "SSL|TLS|subject|issuer|expire"

# Generar certificado self-signed (solo para desarrollo)
openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem -days 365 -nodes
```

## Insights de ingeniería

- **Let's Encrypt + auto-renewal**: certificados gratuitos con renovación automática cada 90 días. No hay excusa para no tener HTTPS en 2024.
- **HSTS (HTTP Strict Transport Security)**: header que dice al browser "siempre usa HTTPS para este dominio". Previene downgrade attacks. `Strict-Transport-Security: max-age=31536000; includeSubDomains`.
- **Certificate pinning**: en mobile apps, hardcodear el hash del certificado del servidor. Previene MITM incluso si una CA es comprometida. Difícil de mantener (rotación de certificados requiere app update).
- **Forward secrecy**: con ECDHE, cada conexión usa claves efímeras. Si la private key del servidor se compromete después, las sesiones pasadas NO se pueden descifrar. TLS 1.3 lo obliga.
