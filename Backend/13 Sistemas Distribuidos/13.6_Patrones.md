# 13.6 Patrones de Sistemas Distribuidos

## Definición

Patrones prácticos para resolver problemas recurrentes en sistemas distribuidos: transacciones sin 2PC (Saga), garantías de publicación de eventos (Outbox), retry safety (idempotency), y el mito de exactly-once delivery.

## Mecánica interna

### Saga Pattern

```
  Transacción distribuida sin lock global:

  1. Create Order (orders-service) ✅
  2. Reserve Inventory (inventory-service) ✅
  3. Charge Payment (payment-service) ❌ FAIL

  Compensación (rollback):
  3c. Refund Payment → N/A (nunca se cobró)
  2c. Release Inventory (inventory-service) ✅
  1c. Cancel Order (orders-service) ✅

  Tipos:
  - Choreography: cada servicio escucha eventos y reacciona
  - Orchestration: un coordinator central dirige los pasos

  Orchestration es más fácil de entender y debuggear.
```

### Outbox Pattern

```
  Problema: quieres escribir en DB Y publicar evento atómicamente.

  Sin outbox:
  1. INSERT order → DB commit ✅
  2. Publish event to Kafka → FAIL ❌
  → DB tiene el dato pero el evento se perdió.

  Con outbox:
  1. BEGIN transaction
     INSERT order INTO orders;
     INSERT event INTO outbox;  ← misma transacción
  2. COMMIT ✅
  3. Separate process: lee outbox → publica a Kafka → marca como publicado

  Garantía: si el dato está en DB, el evento también está (en la outbox).
  El publisher puede retrasarse pero nunca pierde el evento.
```

### Idempotency en Sistemas Distribuidos

```
  En redes no confiables, los mensajes pueden entregarse más de una vez.

  Consumer recibe: {event: "ChargePayment", amount: 100, idempotency_key: "pay-42"}

  1. ¿Ya procesé pay-42? → Buscar en tabla de idempotency
  2. No → procesar, almacenar resultado con key pay-42
  3. Sí → retornar resultado almacenado sin re-procesar

  Almacenamiento: tabla en PostgreSQL o key en Redis con TTL.
```

### Exactly-Once: el mito

```
  At-most-once:  fire and forget. Simple, puede perder.
  At-least-once: retry hasta ACK. Puede duplicar.
  Exactly-once:  cada mensaje procesado exactamente una vez.

  Exactly-once es imposible en redes no confiables (probar que un mensaje
  se entregó requiere confirmación, que también puede perderse → retry → duplicado).

  Solución práctica:
  At-least-once delivery + idempotent processing = exactly-once SEMANTICS

  El mensaje puede llegar 2 veces, pero el efecto es como si llegara 1 vez.
```

## Modelo mental

- **Saga** = un viaje con escalas. Si un vuelo se cancela, deshaces las reservas anteriores una por una.
- **Outbox** = escribir una nota adhesiva junto con tu trabajo. Un asistente recoge las notas y las envía por correo. Nunca pierdes la nota porque está junto al trabajo.
- **Idempotency** = un cerrojo con llave numerada. La primera vez que usas llave #42, la puerta abre. Las siguientes veces con la misma llave, nada cambia.

## Insights de ingeniería

- **Saga orchestrator vs choreography**: con 3 servicios, choreography funciona. Con 10 servicios, el flujo es imposible de seguir. Usa orchestration (servicio coordinator dedicado) para flujos complejos.
- **Outbox + CDC (Change Data Capture)**: en vez de un proceso que poll la tabla outbox, usa Debezium que lee el WAL de PostgreSQL y publica cambios a Kafka automáticamente. Más eficiente, menor latencia.
- **Idempotency key = entity ID + operation**: para un pago, la idempotency key puede ser `order_42_payment`. Esto previene cobros duplicados independientemente de cuántas veces se retry la operación.
