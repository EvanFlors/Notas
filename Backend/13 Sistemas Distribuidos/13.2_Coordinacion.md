# 13.2 Coordinación Distribuida

## Definición

En un sistema distribuido, los nodos necesitan coordinarse: elegir un líder, adquirir locks, y llegar a consenso. Estos problemas son fundamentalmente difíciles porque cualquier nodo puede fallar en cualquier momento y los mensajes pueden perderse o retrasarse.

## Mecánica interna

### Leader Election

```
  ¿Por qué necesitas un líder?
  - Solo un nodo debe escribir al primary DB (evitar split-brain)
  - Solo un nodo debe procesar un job scheduled
  - Solo un nodo debe hacer compaction/cleanup

  Lease-based election:
  1. Nodos compiten por un lock en etcd/ZooKeeper con TTL
  2. El que lo obtiene es el líder
  3. Líder renueva el lease periódicamente
  4. Si el líder muere → lease expira → nuevo nodo adquiere → nuevo líder

  Raft-based: el algoritmo de consenso incluye leader election.
```

### Distributed Locks

```
  Problema: dos instancias no deben procesar el mismo job simultáneamente.

  Redis lock (simple):
  SET lock:job42 owner_id NX EX 30  ← NX = solo si no existe, EX = TTL 30s
  → OK (adquirido) o nil (ya tomado)

  Redlock (más robusto):
  Adquirir lock en mayoría de nodos Redis (3 de 5).
  Si la mayoría confirma → lock adquirido.
  Previene que un nodo Redis fallido dé lock falso.

  etcd/ZooKeeper locks: basados en consenso distribuido.
  Más confiables que Redis para locks críticos.
```

### Consensus: Byzantine Generals Problem

```
  N generales deben acordar atacar o retirarse.
  Algunos generales pueden ser traidores (enviar mensajes contradictorios).

  En sistemas distribuidos: los "traidores" son:
  - Nodos con bugs que envían datos corruptos
  - Nodos comprometidos por atacantes
  - Mensajes alterados en tránsito

  BFT (Byzantine Fault Tolerance): tolera f fallas con 3f+1 nodos.
  Non-BFT (Raft, Paxos): asume nodos honestos, tolera f fallas con 2f+1 nodos.

  En backend normal: non-BFT es suficiente (confiamos en nuestros servidores).
  Blockchain: BFT necesario (no confías en los participantes).
```

## Modelo mental

- **Leader election** = elegir un capitán. Si el capitán se enferma, el equipo elige uno nuevo.
- **Distributed lock** = una llave física que solo una persona puede tener. Si la pierde (crash), la llave reaparece después de un timeout (TTL).
- **Consensus** = un grupo de amigos decidiendo dónde cenar por votación. Necesitan mayoría para decidir, y pueden manejar que algunos no respondan.

## Insights de ingeniería

- **Distributed locks son peligrosos**: si el proceso adquiere el lock, se pausa (GC, swap), y el lock expira, otro proceso lo adquiere. Ambos creen tener el lock. Solución: fencing tokens (incrementar un counter con cada lock, el recurso rechaza tokens viejos).
- **etcd > Redis para coordination**: Redis es AP (eventual consistency). etcd es CP (strong consistency via Raft). Para leader election y locks críticos, usa etcd o ZooKeeper.
- **Minimize coordination**: la coordinación distribuida es costosa (red + consenso). Diseña para minimizarla. Datos independientes por shard no necesitan coordinación.
