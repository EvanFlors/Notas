# 13.5 Escalabilidad Real

## Definición

Escalabilidad real va más allá de "agregar más servidores". Incluye horizontal scaling de servicios stateless, particionamiento de datos con consistent hashing, estrategias de replicación, y global load balancing.

## Mecánica interna

### Horizontal vs Vertical Scaling

```
  Vertical: más CPU/RAM en la misma máquina
  + Simple, sin cambios de arquitectura
  - Límite físico, single point of failure

  Horizontal: más máquinas
  + Sin límite teórico, fault tolerance
  - Requiere stateless services, distributed state
```

### Consistent Hashing

```
  Problema: hash(key) % N cambia cuando N cambia.
  Si agregas un nodo, TODAS las keys se redistribuyen.

  Consistent hashing:
  Ring de 0 a 2^32. Cada nodo se coloca en el ring por hash(node_id).
  Cada key se asigna al primer nodo clockwise.

       Node A (hash=100)
        ╱
  ─────●───── key1(hash=50) → Node A
  │         │
  │         │ key2(hash=150) → Node B
  │         │
  ─────●─────
        ╲
       Node B (hash=200)

  Agregar Node C (hash=120):
  Solo keys entre 100-120 se mueven de B a C.
  El resto NO se redistribuye.

  Virtual nodes: cada nodo físico tiene múltiples posiciones en el ring
  → distribución más uniforme.
```

### Replication Strategies

```
  Sync replication:  write confirmado después de que N replicas confirmen.
  Async replication: write confirmado inmediatamente, replicado en background.
  Semi-sync:         write confirmado después de al menos 1 replica.

  Quorum: write a W nodos, read de R nodos.
  Si W + R > N → strong consistency.
  Ejemplo (N=3): W=2, R=2 → al menos 1 nodo en común entre write y read.
```

### Global Load Balancing

```
  GeoDNS: DNS retorna la IP del datacenter más cercano al cliente.

  User en Europa → DNS → 10.0.1.1 (EU datacenter)
  User en APAC   → DNS → 10.0.2.1 (APAC datacenter)

  Anycast: misma IP anunciada desde múltiples datacenters.
  BGP routing envía al más cercano en hops.
  Usado por CDNs (Cloudflare) y DNS servers.
```

## Insights de ingeniería

- **Stateless es prerequisito**: no puedes escalar horizontalmente si cada instancia tiene estado diferente. Externaliza estado en Redis/DB. Los servicios deben ser intercambiables.
- **Consistent hashing para caching distribuido**: Memcached y Redis clusters usan consistent hashing. Agregar/quitar nodos redistribuye solo 1/N de las keys.
- **Scale reads con replicas, writes con sharding**: la mayoría de apps son read-heavy. Replicas resuelven el 90% de los problemas de scaling. Sharding es para write-heavy o datos > capacidad de un nodo.
