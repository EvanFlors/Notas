# 13.3 Algoritmos de Consenso

## Definición

Algoritmos de consenso permiten que un grupo de nodos acuerde un valor, incluso si algunos nodos fallan. Raft es el estándar moderno (usado por etcd, Consul, CockroachDB). Es equivalente a Paxos pero diseñado para ser comprensible.

## Mecánica interna

### Raft

```
  Roles: Leader, Follower, Candidate

  1. LEADER ELECTION
  - Followers tienen un election timeout (150-300ms random)
  - Si timeout sin heartbeat del leader → se convierte en Candidate
  - Candidate pide votos a todos → si mayoría vota → nuevo Leader
  - Leader envía heartbeats para mantener liderazgo

  2. LOG REPLICATION
  Client → Leader: "write X=5"
  Leader: append to log → replicate to followers
  Followers: append to log → ACK
  Leader: si mayoría ACK → commit → responder al client
  Leader: notificar commit a followers → followers aplican

  ┌────────┐    append    ┌──────────┐
  │ Leader │ ──────────── │Follower 1│ ACK
  │ [X=5]  │ ──────────── │Follower 2│ ACK  ← mayoría (2/3)
  │        │ ──────────── │Follower 3│ (network partition, no ACK)
  └────────┘              └──────────┘
  → commit! X=5 es persistente.

  3. SAFETY
  - Un nodo solo vota por un candidate con log al menos tan actualizado
  - Esto garantiza que el nuevo leader tiene todos los commits anteriores
```

### Paxos

```
  Teórico, más antiguo que Raft. Mismas garantías.
  Roles: Proposer, Acceptor, Learner.

  Fase 1 (Prepare): Proposer envía número de propuesta a Acceptors
  Fase 2 (Accept): si mayoría responde, Proposer envía valor

  Equivalente a Raft pero más difícil de implementar y entender.
  En la práctica: usa Raft (o una implementación probada como etcd).
```

### PBFT (Practical Byzantine Fault Tolerance)

```
  Tolera nodos maliciosos. Requiere 3f+1 nodos para tolerar f fallas.

  Fases: Pre-prepare → Prepare → Commit

  Uso: blockchain (Hyperledger), sistemas donde no confías en todos los participantes.
  No usado en backend normal (overhead alto, asumimos nodos honestos).
```

## Modelo mental

Raft es como una asamblea con un presidente. El presidente propone decisiones, la asamblea vota. Si el presidente se ausenta, la asamblea elige uno nuevo (el que tiene las notas más actualizadas). Ninguna decisión es final hasta que la mayoría aprueba.

## Insights de ingeniería

- **Raft requiere nodos impares**: 3 nodos tolera 1 falla, 5 nodos tolera 2 fallas. 2 o 4 nodos no mejoran la tolerancia a fallos pero agregan overhead.
- **etcd = Raft en producción**: etcd implementa Raft y es el backend de Kubernetes. Entender Raft te ayuda a debuggear problemas de etcd/K8s.
- **Latency cost of consensus**: cada write requiere que la mayoría de nodos confirme. Con nodos en múltiples regiones, la latencia es la del nodo más lento del quorum. Por eso CockroachDB recomienda regiones cercanas para un quorum.
