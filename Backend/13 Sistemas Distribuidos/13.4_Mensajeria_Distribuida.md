# 13.4 Mensajería Distribuida

## Definición

Kafka y RabbitMQ son las plataformas principales de mensajería distribuida. Kafka es un log distribuido optimizado para streaming de eventos. RabbitMQ es un message broker optimizado para task queues y routing flexible. Event sourcing y CQRS son patrones arquitectónicos que los utilizan.

## Mecánica interna

### Kafka Deep Dive

```
  Topic: "orders" con 3 particiones, replication factor 3

  Broker 1 (leader P0, follower P1, P2)
  Broker 2 (leader P1, follower P0, P2)
  Broker 3 (leader P2, follower P0, P1)

  Producer → partition key: hash(order_id) % 3 → partition

  Consumer Group "order-processor":
    Consumer A → P0
    Consumer B → P1
    Consumer C → P2

  Consumer Group "analytics":
    Consumer D → P0, P1, P2  (lee los mismos datos independientemente)

  Garantías:
  - Orden dentro de una partición (no entre particiones)
  - At-least-once delivery (default)
  - Exactly-once con transactional producer + consumer offset commit

  Retención: configurable (7 días default, puede ser infinita)
  Los mensajes no se borran al ser consumidos → re-lectura posible.
```

### Event Sourcing

```
  Tradicional: almacenar estado actual
  UPDATE accounts SET balance = 950 WHERE id = 1;
  → Solo sabes el saldo actual. No sabes cómo llegó ahí.

  Event sourcing: almacenar todos los eventos
  Event 1: {type: "AccountCreated", balance: 1000}
  Event 2: {type: "Withdrawal", amount: 50}
  Event 3: {type: "Deposit", amount: 100}
  Event 4: {type: "Withdrawal", amount: 100}

  Estado actual: replay events → balance = 1000 - 50 + 100 - 100 = 950

  Ventajas: audit trail completo, rebuild state, temporal queries.
  Desventajas: complejidad, event schema evolution, eventual consistency.
```

### CQRS (Command Query Responsibility Segregation)

```
  ┌──────────┐    Command    ┌──────────────┐
  │  Client   │ ────────────►│ Write Model  │ → Event Store (PostgreSQL/Kafka)
  └──────────┘              └──────┬───────┘
                                    │ events
                              ┌─────▼──────┐
                              │  Projector  │ → actualiza read model
                              └─────┬──────┘
  ┌──────────┐    Query      ┌─────▼──────┐
  │  Client   │ ◄────────────│ Read Model  │ (denormalized, optimizado para queries)
  └──────────┘              └────────────┘

  Write model: normalizado, optimizado para writes.
  Read model: denormalizado, optimizado para queries específicas.
  Eventual consistency entre write y read model.
```

## Insights de ingeniería

- **Kafka para event streaming, RabbitMQ para task queues**: no son intercambiables. Kafka retiene mensajes y permite re-lectura. RabbitMQ borra al consumir y ofrece routing flexible.
- **Event sourcing no es para todo**: agrega complejidad significativa. Úsalo solo cuando el audit trail y la capacidad de reconstruir estado son requisitos reales (finanzas, compliance, collaborative editing).
- **CQRS sin event sourcing**: puedes separar read/write models sin event sourcing. Simplemente usa una DB para writes y otra (con materialized views o denormalized tables) para reads.
