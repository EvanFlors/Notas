# 13.1 Fundamentos de Sistemas Distribuidos

## Definición

Un sistema distribuido es un grupo de computadoras que trabajan juntas y aparecen como un solo sistema para el usuario. Los desafíos fundamentales son: fallos parciales (un nodo puede fallar sin que los demás se enteren), latencia de red (comunicación no es instantánea), y la imposibilidad de tener consistencia perfecta + disponibilidad perfecta simultáneamente.

## Mecánica interna

### CAP Theorem

```
  En presencia de una partición de red (P), debes elegir entre:

  C (Consistency): todos los nodos ven los mismos datos al mismo tiempo
  A (Availability): toda request recibe una respuesta (no necesariamente la más reciente)
  P (Partition tolerance): el sistema funciona a pesar de pérdida de mensajes entre nodos

  P es inevitable en la realidad → la elección es C o A:

  CP (Consistency + Partition tolerance):
  Durante partición → rechaza requests (unavailable) antes que servir datos stale
  Ejemplo: etcd, ZooKeeper, bases de datos con quorum writes

  AP (Availability + Partition tolerance):
  Durante partición → sirve datos (posiblemente stale) antes que estar unavailable
  Ejemplo: Cassandra, DynamoDB (eventual consistency mode), DNS
```

### PACELC

```
  Extensión de CAP:

  If Partition:  choose A or C  (CAP)
  Else (normal): choose L or C  (Latency or Consistency)

  Ejemplos:
  - DynamoDB: PA/EL (disponible en partición, baja latencia en normalidad)
  - PostgreSQL sync replica: PC/EC (consistente siempre, mayor latencia)
  - Cassandra: PA/EL (pero configurable por query: ONE vs QUORUM vs ALL)
```

### Modelos de consistencia

| Modelo | Garantía | Ejemplo |
|--------|----------|---------|
| Strong (linearizable) | Reads siempre retornan el write más reciente | PostgreSQL single node, etcd |
| Sequential | Todos los nodos ven operaciones en el mismo orden | ZooKeeper |
| Causal | Reads respetan causalidad (si A causó B, ves A antes que B) | MongoDB causal sessions |
| Read-your-writes | Después de escribir, tus propios reads ven tu write | Session consistency en DynamoDB |
| Eventual | En algún momento, todos los nodos convergen al mismo valor | DNS, Cassandra (default) |

### Fallacies of Distributed Computing

```
  1. La red es confiable              → timeouts, retries, circuit breakers
  2. La latencia es cero              → async, caching, connection pooling
  3. El bandwidth es infinito         → pagination, compression, protobuf
  4. La red es segura                 → mTLS, encryption at rest
  5. La topología no cambia           → service discovery, DNS
  6. Hay un solo administrador        → IAM, RBAC
  7. El transporte no tiene costo     → minimizar network calls
  8. La red es homogénea              → protocol negotiation
```

## Modelo mental

CAP es como elegir entre un banco que siempre te da tu saldo exacto pero puede cerrar temporalmente (CP), o un banco que siempre está abierto pero puede mostrarte un saldo de hace 5 minutos (AP).

## Insights de ingeniería

- **CAP no es una elección permanente**: puedes elegir diferente por operación. En el mismo sistema: writes fuertes (CP) para pagos, reads eventuales (AP) para catálogo de productos.
- **Eventual consistency es más difícil de programar**: tu código debe manejar el caso de datos stale. ¿Qué pasa si el usuario ve su orden como "no creada" durante 2 segundos después de crearla?
- **Strong consistency tiene costo**: cada write debe esperar confirmación de la mayoría de nodos (quorum). Latencia = RTT al nodo más lento del quorum. Para la mayoría de aplicaciones, read-your-writes consistency es suficiente.
