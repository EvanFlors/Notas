# 14.3 CI/CD

## Definición

CI/CD (Continuous Integration / Continuous Delivery) es el pipeline automatizado que lleva código desde un commit hasta producción: build → test → deploy. Elimina pasos manuales, reduce errores humanos, y permite releases frecuentes con confianza.

## Mecánica interna

### Pipeline estándar

```
  commit → Build → Unit Tests → Integration Tests → Security Scan → Staging Deploy → E2E Tests → Production Deploy

  ┌─────────┐    ┌──────────┐    ┌──────────────┐    ┌─────────────┐    ┌────────────┐
  │ Build    │───▶│ Test     │───▶│ Scan         │───▶│ Staging     │───▶│ Production │
  │ compile  │    │ unit     │    │ SAST/DAST    │    │ deploy      │    │ deploy     │
  │ docker   │    │ integr.  │    │ CVE check    │    │ smoke tests │    │ canary/b-g │
  │ build    │    │ lint     │    │ license      │    │ e2e tests   │    │ rollback?  │
  └─────────┘    └──────────┘    └──────────────┘    └─────────────┘    └────────────┘

  Trigger: push a main, merge de PR, tag de versión
  Gate: cada etapa es un gate — si falla, el pipeline se detiene
```

### Estrategias de deploy

```
  Rolling Update:
  Reemplaza instancias gradualmente. Siempre hay capacidad disponible.
  Rollback: desplegar versión anterior.
  Riesgo: durante el update hay dos versiones coexistiendo.

  Blue/Green:
  ┌──────────┐     ┌──────────┐
  │ Blue     │     │ Green    │
  │ (v1)     │     │ (v2)     │
  │ ACTIVE   │     │ IDLE     │
  └────┬─────┘     └────┬─────┘
       │                 │
       └───── LB ────────┘  ← switch instantáneo

  Dos entornos idénticos. Deploy a Green (idle), test, switch tráfico.
  Rollback: switch de vuelta a Blue. Costo: doble infraestructura.

  Canary:
  100% tráfico → v1
  Canary deploy: 5% tráfico → v2, 95% → v1
  Observar métricas (latencia, error rate)
  Si OK: 25% → 50% → 100% gradualmente
  Si KO: rollback inmediato, solo 5% afectado

  Feature Flags:
  Deploy v2 con feature desactivada (flag=false).
  Activar feature para subset de usuarios sin re-deploy.
  Permite desacoplar deploy de release.
```

### GitOps

```
  Principio: Git es la single source of truth para infraestructura y deployments.

  Developer → push manifiesto K8s a Git
                    │
                    ▼
  ArgoCD/Flux (operator en K8s):
    - Observa repo Git continuamente
    - Compara estado del repo vs estado del cluster
    - Si hay drift → reconcilia automáticamente
    - Audit trail: todo cambio es un commit

  Beneficios:
  - Rollback = git revert
  - PR review para cambios de infra
  - Reproducibilidad total
  - Self-healing: si alguien modifica manualmente el cluster, GitOps lo revierte
```

### Infrastructure as Code

```
  Terraform:
  resource "aws_instance" "api" {
    ami           = "ami-0c55b159cbfafe1f0"
    instance_type = "t3.medium"
    tags = { Name = "api-server" }
  }

  terraform plan   → muestra cambios sin aplicar (diff)
  terraform apply  → aplica cambios
  terraform destroy → destruye recursos

  State file: Terraform mantiene un archivo de estado (.tfstate) que mapea
  recursos declarados a recursos reales en el cloud. Este archivo DEBE
  almacenarse en backend remoto (S3 + DynamoDB lock) para trabajo en equipo.

  Pulumi: alternativa que permite definir infra con lenguajes reales
  (TypeScript, Python, Go) en vez de HCL.
```

## Modelo mental

CI/CD es una línea de ensamblaje con quality gates. Cada etapa verifica un aspecto (compila, pasa tests, es seguro, funciona en staging). Si un gate falla, la línea se detiene. GitOps añade un loop de reconciliación: Git dice cómo debe estar el mundo, un operador se asegura de que el mundo coincida. El resultado es que deployar a producción es tan simple — y tan auditado — como mergear un PR.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Pipeline lento (30+ min) | Tests lentos, builds sin cache | Paralelizar tests, cache de deps/Docker layers |
| Flaky tests bloquean pipeline | Tests no determinísticos | Quarantine flaky tests, fix root cause |
| Deploy roto no detectado | Sin smoke tests post-deploy | Canary + métricas + rollback automático |
| Terraform state corruption | State local sin lock | Remote backend (S3 + DynamoDB) con locking |

## Insights de ingeniería

- **Pipeline speed importa**: si el pipeline tarda 45 minutos, los desarrolladores hacen commits grandes y poco frecuentes. Si tarda 5 minutos, hacen commits pequeños y frecuentes. Optimiza agresivamente: paraleliza, cachea, ejecuta solo tests afectados.
- **Canary > Blue/Green** para la mayoría de casos: Blue/Green necesita doble infraestructura y el switch es binario (100% de golpe). Canary permite rollout gradual con observabilidad real. Combinar con feature flags es lo más seguro.
- **GitOps elimina "kubectl apply" manual**: nadie debería tener `kubectl` con acceso a producción. Todo cambio pasa por Git → PR review → merge → ArgoCD aplica. El audit trail es completo y el rollback es un `git revert`.
