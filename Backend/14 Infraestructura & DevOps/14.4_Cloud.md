# 14.4 Cloud

## Definición

Cloud computing es el consumo de recursos computacionales (compute, storage, networking) bajo demanda, pagando por uso, sin gestionar hardware. Los tres grandes proveedores (AWS, GCP, Azure) ofrecen servicios primitivos (VMs, discos) y managed services (bases de datos, colas, ML) que abstraen la operación.

## Mecánica interna

### Compute

```
  Spectrum de abstracción (menos → más):

  Bare Metal → VMs → Containers (ECS/GKE) → Serverless (Lambda/Cloud Run)

  VMs (EC2, Compute Engine):
  Tú controlas el OS, runtime, y app. Pagas por hora/segundo.
  Tipos: general purpose (t3), compute-optimized (c5), memory-optimized (r5).

  Containers como servicio:
  ECS/Fargate (AWS), Cloud Run (GCP), ACI (Azure).
  Tú defines la imagen Docker, el cloud la ejecuta.
  No gestionas nodos/VMs directamente.

  Serverless (Lambda, Cloud Functions):
  Tú defines una función. El cloud la ejecuta bajo demanda.
  Pago por invocación + duración. Escala a cero.
  Cold start: primera invocación tarda más (inicializar runtime).
  Limitaciones: timeout (15 min Lambda), sin estado local persistente.
```

### Networking

```
  VPC (Virtual Private Cloud):
  Tu red privada aislada en el cloud.

  ┌─ VPC 10.0.0.0/16 ─────────────────────────────┐
  │                                                 │
  │  ┌─ Public Subnet 10.0.1.0/24 ──┐              │
  │  │ Internet Gateway ←→ ALB       │              │
  │  └────────────────────────────────┘              │
  │                                                 │
  │  ┌─ Private Subnet 10.0.2.0/24 ─┐              │
  │  │ App servers (EC2/ECS)          │              │
  │  │ NAT Gateway → Internet (salida)│              │
  │  └────────────────────────────────┘              │
  │                                                 │
  │  ┌─ Private Subnet 10.0.3.0/24 ─┐              │
  │  │ Databases (RDS)                │              │
  │  │ Sin acceso a Internet          │              │
  │  └────────────────────────────────┘              │
  └─────────────────────────────────────────────────┘

  Security Groups: firewall stateful por recurso (allow inbound 443 from ALB).
  NACLs: firewall stateless por subnet.
  VPC Peering / Transit Gateway: conectar VPCs entre sí.
```

### Storage

```
  Block Storage (EBS, Persistent Disk):
  Disco virtual para VMs. Un volumen → un VM (o Pod en K8s).
  IOPS provisionados para workloads de BD.

  Object Storage (S3, GCS, Blob Storage):
  Key-value store para archivos. Ilimitado, barato, durable (99.999999999%).
  Ideal para: backups, assets estáticos, data lake, logs.
  Tiers: Standard → Infrequent Access → Glacier (archival, retrieval en horas).

  File Storage (EFS, Filestore):
  NFS compartido entre múltiples VMs/containers.
  Más caro que S3, más lento que EBS, pero compartible.
```

### Managed Services

```
  Bases de datos:
  RDS / Cloud SQL ── PostgreSQL, MySQL managed
  DynamoDB / Firestore ── NoSQL managed
  ElastiCache / Memorystore ── Redis managed

  Mensajería:
  SQS / Pub/Sub ── colas de mensajes
  SNS / EventBridge ── pub/sub y event routing
  Kinesis / Dataflow ── streaming de datos

  Beneficio: el cloud gestiona backups, patching, failover, scaling.
  Costo: vendor lock-in, menos control sobre configuración.

  Regla: si no es tu core business operarlo, usa managed.
  Ejemplo: no operes tu propio PostgreSQL en EC2 salvo que necesites
  configuración extrema. RDS cubre el 95% de casos.
```

## Modelo mental

El cloud es un catálogo de LEGOs con diferentes niveles de abstracción. Los bloques primitivos (VMs, discos, redes) te dan control total pero requieren que tú operes todo. Los managed services (RDS, SQS, Lambda) te quitan operación a cambio de flexibilidad y portabilidad. La decisión es siempre un trade-off entre control y carga operacional.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Factura inesperada | Recursos olvidados corriendo | Tags obligatorios + alertas de billing |
| App inaccesible | Security Group sin regla de ingress | Verificar SG + NACLs + route tables |
| Lambda timeout | Cold start + procesamiento largo | Provisioned concurrency o containers |
| Datos inaccesibles en S3 Glacier | Retrieval tarda horas | Usar tier correcto según access pattern |

## Insights de ingeniería

- **Multi-AZ es el mínimo**: una Availability Zone puede caer. Distribuir app en al menos 2 AZs es el baseline. Multi-region es para disaster recovery real, pero añade complejidad significativa (replicación de datos, DNS failover).
- **Tagging es infraestructura**: sin tags consistentes (`team`, `env`, `service`) no puedes saber quién paga qué, qué se puede borrar, ni qué impacta un outage. Implementa tagging policies desde el día uno.
- **Serverless no es gratis a escala**: Lambda es barato para cargas esporádicas pero caro para tráfico constante y alto. A partir de cierto volumen, containers en Fargate o ECS son más cost-effective. Calcula siempre el break-even.
