# 14.2 Orquestación

## Definición

Un orquestador de contenedores (Kubernetes, Docker Swarm, Nomad) automatiza el deploy, escalado, networking, y self-healing de contenedores distribuidos en un cluster de máquinas. Kubernetes (K8s) es el estándar de facto.

## Mecánica interna

### Arquitectura K8s

```
  Control Plane:
  ┌────────────────────────────────────────────────┐
  │ API Server ── el "frontend" de K8s, todo pasa por aquí
  │ etcd ──────── key-value store: estado del cluster
  │ Scheduler ─── decide en qué nodo corre cada Pod
  │ Controller Manager ── loops de reconciliación:
  │   - Deployment controller: replica count deseado vs actual
  │   - Node controller: detecta nodos caídos
  │   - Job controller: gestiona workloads batch
  └────────────────────────────────────────────────┘

  Worker Node:
  ┌────────────────────────────────────────────────┐
  │ kubelet ──── agente que ejecuta Pods en el nodo
  │ kube-proxy ─ reglas iptables/IPVS para routing
  │ Container Runtime (containerd) ── corre containers
  │                                                │
  │  ┌─Pod─────────────────────────┐               │
  │  │ Container A  Container B    │ ← comparten   │
  │  │ (app)        (sidecar)      │   network/vol  │
  │  └─────────────────────────────┘               │
  └────────────────────────────────────────────────┘
```

### Recursos fundamentales

```yaml
# Deployment: gestiona ReplicaSets → gestiona Pods
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  replicas: 3                    # Desired state
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1                # Max pods extra durante update
      maxUnavailable: 0          # Cero downtime
  selector:
    matchLabels:
      app: api-server
  template:
    spec:
      containers:
      - name: api
        image: api:v2.1
        resources:
          requests:              # Scheduler reserva estos recursos
            cpu: "250m"
            memory: "256Mi"
          limits:                # OOM kill si excede
            cpu: "500m"
            memory: "512Mi"
        readinessProbe:          # ¿Puede recibir tráfico?
          httpGet:
            path: /healthz
            port: 8080
          initialDelaySeconds: 5
        livenessProbe:           # ¿Sigue vivo? Si falla → restart
          httpGet:
            path: /healthz
            port: 8080
          periodSeconds: 10
---
# Service: abstracción de networking estable
apiVersion: v1
kind: Service
metadata:
  name: api-server
spec:
  selector:
    app: api-server
  ports:
  - port: 80
    targetPort: 8080
  type: ClusterIP                # Solo accesible dentro del cluster
---
# Ingress: expone servicios al exterior con routing L7
apiVersion: networking.k8s.io/v1
kind: Ingress
spec:
  rules:
  - host: api.example.com
    http:
      paths:
      - path: /
        backend:
          service:
            name: api-server
            port:
              number: 80
```

### Auto-scaling

```
  HPA (Horizontal Pod Autoscaler):
  Observa métricas (CPU, memory, custom) → ajusta replica count.

  spec:
    minReplicas: 2
    maxReplicas: 10
    metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70   ← si CPU > 70%, escalar

  VPA (Vertical Pod Autoscaler):
  Ajusta requests/limits de CPU y memoria automáticamente.
  Requiere restart del Pod para aplicar cambios.

  Cluster Autoscaler:
  Si hay Pods Pending (no caben en ningún nodo) → añade nodos al cluster.
  Si hay nodos infrautilizados → los elimina.
```

### Helm y Operators

```
  Helm:
  Package manager para K8s. Un "chart" es un template de manifiestos.
  helm install my-app ./chart --set replicas=3
  Permite versionar, compartir, y reusar configuraciones complejas.

  Operators:
  Controllers custom que automatizan operaciones de software stateful.
  Ejemplo: PostgreSQL Operator → crea réplicas, hace backups, failover automático.
  Extienden la API de K8s con Custom Resource Definitions (CRDs).
```

## Modelo mental

K8s es un loop de reconciliación infinito: tú declaras el estado deseado ("quiero 3 réplicas de mi app"), los controllers observan el estado actual, y actúan para converger. Si un Pod muere, el controller crea otro. Si un nodo cae, los Pods se reschedulean. No ejecutas instrucciones imperativas; declaras intenciones y K8s las hace realidad continuamente.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Pod en CrashLoopBackOff | App crashea al iniciar | `kubectl logs` → fix app o config |
| Pod en Pending | Sin recursos en el cluster | Cluster Autoscaler o aumentar nodos |
| Service sin endpoints | Label mismatch entre Service y Pod | Verificar `selector` en Service |
| Rolling update stuck | readinessProbe nunca pasa | Verificar health endpoint y timeouts |

## Insights de ingeniería

- **Requests vs Limits**: requests es lo que el scheduler garantiza. Limits es el techo. Si no configuras requests, el scheduler no sabe dónde poner tu Pod. Si no configuras limits, un Pod puede consumir todo el nodo y matar otros Pods (noisy neighbor).
- **Probes son críticas**: sin readinessProbe, K8s envía tráfico a Pods que aún no están listos. Sin livenessProbe, un Pod en deadlock nunca se reinicia. Configúralas siempre.
- **Stateful workloads**: bases de datos en K8s son posibles (StatefulSets + PersistentVolumes), pero añaden complejidad significativa. Evalúa managed services (RDS, Cloud SQL) antes de operar bases de datos en K8s.
