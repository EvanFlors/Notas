# 14.1 Contenedores

## Definición

Un contenedor es un proceso aislado que cree que tiene su propio OS, filesystem, y red, pero comparte el kernel del host. Docker usa tres primitivas del kernel Linux: namespaces (aislamiento), cgroups (límites de recursos), y overlay filesystem (capas de imagen).

## Mecánica interna

### Primitivas del kernel

```
  Namespaces (aislamiento):
  ┌─────────────────────────────────────┐
  │ PID namespace:  el contenedor ve PID 1 (su proceso principal)
  │ NET namespace:  su propia interfaz de red, IP, puertos
  │ MNT namespace:  su propio filesystem root
  │ UTS namespace:  su propio hostname
  │ IPC namespace:  su propio shared memory
  │ USER namespace: su propio mapeo de UIDs
  └─────────────────────────────────────┘

  cgroups (límites):
  ┌─────────────────────────────────────┐
  │ CPU:    max 2 cores
  │ Memory: max 512MB (OOM kill si excede)
  │ I/O:    max 100MB/s de disco
  │ PIDs:   max 100 procesos
  └─────────────────────────────────────┘

  Overlay filesystem (capas):
  ┌─────────────────┐ ← Container layer (read-write)
  ├─────────────────┤
  │ App code         │ ← Image layer 3
  │ Dependencies     │ ← Image layer 2
  │ Base OS (Alpine) │ ← Image layer 1
  └─────────────────┘

  Las capas de imagen son read-only y compartidas entre contenedores.
  Solo la capa superior es read-write (copy-on-write).
```

### Dockerfile Best Practices

```dockerfile
# Multi-stage build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --production          # Instalar deps (cacheado si package.json no cambia)
COPY . .
RUN npm run build

FROM node:20-alpine
WORKDIR /app
RUN addgroup -S app && adduser -S app -G app  # Non-root user
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
USER app                        # No correr como root
EXPOSE 8080
CMD ["node", "dist/server.js"]
```

```
  Principios:
  1. Multi-stage: imagen final sin herramientas de build (menor tamaño, menor superficie de ataque)
  2. Order matters for caching: COPY package.json antes que COPY . (deps se cachean si no cambian)
  3. Non-root user: NUNCA correr como root en producción
  4. Alpine base: ~5MB vs ~100MB para debian-based
  5. .dockerignore: excluir node_modules, .git, tests
```

### Container Networking

```
  Bridge (default):
  Contenedores en la misma red virtual, se comunican por IP.
  Docker crea un bridge virtual (docker0) en el host.

  Host:
  Contenedor comparte la red del host. Sin aislamiento de red.
  Mejor rendimiento (sin NAT), pero sin aislamiento de puertos.

  Overlay (Docker Swarm / K8s):
  Red virtual que conecta contenedores en múltiples hosts.
  Encapsula tráfico en VXLAN.
```

## Modelo mental

Un contenedor no es una VM. Es un proceso con paredes invisibles: cree que está solo (namespaces), tiene un presupuesto limitado (cgroups), y ve un filesystem construido por capas (overlay). El kernel es compartido con el host y todos los demás contenedores.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Container OOM killed | Excede memory limit | Aumentar limit o optimizar memoria |
| Image pull lenta | Image demasiado grande (1GB+) | Multi-stage build, Alpine base |
| Permission denied | Root en Dockerfile, non-root en runtime | Verificar USER y permisos de archivos |
| Datos perdidos al restart | Datos en container layer (ephemeral) | Usar volumes para datos persistentes |

## Insights de ingeniería

- **Containers son efímeros**: no almacenes datos en el container layer. Usa volumes (Docker) o persistent volumes (K8s) para datos que deben sobrevivir restarts.
- **One process per container**: no corras nginx + app + cron en un solo contenedor. Un proceso principal, un propósito. Usa docker-compose o K8s pods para multi-container setups.
- **Image size importa**: imágenes grandes = pull lento = deploy lento. Alpine + multi-stage = imagen de 50MB en vez de 1GB. En K8s con auto-scaling, pull time es parte del time-to-ready.
