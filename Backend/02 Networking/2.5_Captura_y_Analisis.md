# 2.5 Captura y Análisis de Tráfico

## Definición

La captura de tráfico de red permite observar los paquetes reales que fluyen entre servicios. Es la herramienta definitiva de debugging: no depende de logs, no miente, muestra exactamente qué pasó a nivel de red. `tcpdump` captura, Wireshark analiza visualmente.

## Mecánica interna

### tcpdump: captura en línea de comandos

```bash
# Captura básica
tcpdump -i eth0                         # Todo el tráfico en la interfaz
tcpdump -i any                          # Todas las interfaces
tcpdump -i eth0 -n                      # No resolver DNS (más rápido)
tcpdump -i eth0 -c 100                  # Capturar solo 100 paquetes

# Filtros por protocolo
tcpdump -i eth0 tcp                     # Solo TCP
tcpdump -i eth0 udp port 53            # Solo DNS
tcpdump -i eth0 icmp                    # Solo ICMP (ping)
tcpdump -i eth0 arp                     # Solo ARP

# Filtros por host y puerto
tcpdump -i eth0 host 10.0.1.5          # Tráfico de/hacia un host
tcpdump -i eth0 dst port 443           # Tráfico hacia puerto 443
tcpdump -i eth0 src 10.0.1.5 and dst port 5432  # App → PostgreSQL

# Guardar para análisis posterior
tcpdump -i eth0 -w capture.pcap        # Guardar a archivo
tcpdump -r capture.pcap                 # Leer archivo

# Ver contenido de paquetes
tcpdump -i eth0 -A port 80             # ASCII (HTTP legible)
tcpdump -i eth0 -X port 80             # Hex + ASCII
tcpdump -i eth0 -v                      # Verbose (headers detallados)
```

### Filtros BPF (Berkeley Packet Filter)

```bash
# Combinaciones con and, or, not
tcpdump 'src host 10.0.1.5 and (dst port 80 or dst port 443)'
tcpdump 'tcp and not port 22'           # TCP excepto SSH

# Filtros por flags TCP
tcpdump 'tcp[tcpflags] & tcp-syn != 0'             # Paquetes SYN
tcpdump 'tcp[tcpflags] & (tcp-syn|tcp-fin) != 0'   # SYN o FIN
tcpdump 'tcp[tcpflags] & tcp-rst != 0'              # RST (resets)

# Filtrar por tamaño
tcpdump 'greater 1000'                  # Paquetes > 1000 bytes
tcpdump 'less 100'                      # Paquetes < 100 bytes
```

### Wireshark: análisis visual

```
  Flujo de trabajo:
  1. Capturar con tcpdump -w capture.pcap (en servidor)
  2. Transferir: scp capture.pcap local:
  3. Abrir en Wireshark (local)

  Features clave:
  - Follow TCP Stream: reconstruye la conversación completa
  - IO Graph: throughput/paquetes por segundo en el tiempo
  - Expert Info: Wireshark identifica anomalías automáticamente
  - Display Filters: filtrar lo capturado (diferente de capture filters)
```

Display filters de Wireshark (sintaxis diferente a tcpdump):

```
tcp.port == 8080                     # Tráfico en puerto 8080
http.request.method == "POST"        # Solo HTTP POST
tcp.analysis.retransmission          # Retransmisiones TCP
tcp.analysis.zero_window             # Zero window (receiver lento)
tcp.flags.reset == 1                 # Paquetes RST
dns.qry.name contains "api"         # Queries DNS que contienen "api"
tcp.time_delta > 0.5                 # Gaps mayores a 500ms entre paquetes
```

### Diagnóstico de problemas comunes

```
  1. RETRANSMISIONES
  ┌─────────┐         ┌─────────┐
  │ Cliente  │──data──►│ Servidor│
  │          │  (lost) │         │
  │          │  ...    │         │  Timeout (RTO)
  │          │──data──►│         │  Retransmisión
  │          │◄──ACK───│         │
  └─────────┘         └─────────┘

  Diagnóstico: tcpdump muestra paquetes duplicados con mismo seq number.
  Wireshark: tcp.analysis.retransmission
  Causa: pérdida de paquetes (red congestionada, cable malo, buffer overflow)

  2. TCP ZERO WINDOW
  ┌─────────┐         ┌─────────┐
  │ Sender   │──data──►│Receiver │
  │          │──data──►│(buffer  │
  │          │──data──►│ lleno)  │
  │          │◄─win=0──│         │  "Para, estoy lleno"
  │ (espera) │         │procesa..│
  │          │◄─win=64K│         │  "Ok, continúa"
  │          │──data──►│         │
  └─────────┘         └─────────┘

  Causa: el receiver no puede procesar datos tan rápido como llegan.
  Señal: tu aplicación es lenta o tiene backpressure.

  3. CONNECTION RESET (RST)
  Un lado envía RST = "esta conexión ya no existe, borra todo".
  Causas: app crasheó, firewall timeout, load balancer corta idle connections.
```

## Modelo mental

`tcpdump` es un estetoscopio para la red: escuchas exactamente lo que fluye por el cable. Wireshark es el laboratorio donde analizas las muestras con microscopio. No asumas qué pasa en la red — captúralo y míralo.

## Modos de fallo

| Síntoma que observas | Qué buscar en la captura | Filtro |
|---------------------|--------------------------|--------|
| Latencia alta | Gap grande entre request y response | `tcp.time_delta > 0.1` |
| Conexiones fallidas | SYN sin SYN-ACK | `tcp.flags.syn==1 && tcp.flags.ack==0` |
| Desconexiones aleatorias | RST inesperados | `tcp.flags.reset==1` |
| Throughput bajo | Zero window o retransmisiones | `tcp.analysis.zero_window` |
| Timeout de aplicación | Datos enviados pero sin respuesta | Follow TCP stream |

## Impacto en sistemas distribuidos

- **Debugging entre microservicios**: cuando el servicio A dice "B es lento" y B dice "yo soy rápido", la captura de red muestra la verdad: ¿es latencia de red? ¿retransmisiones? ¿DNS lento? ¿TLS handshake?
- **Service mesh debugging**: Envoy/Istio agregan un proxy entre cada servicio. Si hay latencia, ¿es el proxy o el servicio? Captura en ambos lados del proxy.
- **Kubernetes**: puedes capturar tráfico dentro de un pod con `kubectl exec` + `tcpdump`, o usar herramientas como `ksniff` que automatizan esto.

## Herramientas de diagnóstico

```bash
# Captura rápida de debugging en producción
# Ver handshakes TCP (¿se establecen las conexiones?)
tcpdump -i eth0 'tcp[tcpflags] & tcp-syn != 0' -n -c 50

# Ver RST (¿quién corta conexiones?)
tcpdump -i eth0 'tcp[tcpflags] & tcp-rst != 0' -n

# Medir latencia DNS
tcpdump -i eth0 udp port 53 -n -ttt    # -ttt muestra delta entre paquetes

# Captura con rotación (producción, largo plazo)
tcpdump -i eth0 -w /tmp/cap.pcap -C 100 -W 5   # 5 archivos de 100MB max

# En Kubernetes
kubectl exec -it <pod> -- tcpdump -i eth0 -c 100 -w /tmp/cap.pcap
kubectl cp <pod>:/tmp/cap.pcap ./cap.pcap
```

## Insights de ingeniería

- **Captura en producción con cuidado**: `tcpdump` consume CPU y puede llenar disco. Usa filtros específicos, limita con `-c` (count) y `-C` (file size), y captura solo lo necesario.
- **No captures en la interfaz equivocada**: en contenedores, la interfaz es `eth0` del contenedor, no del host. En K8s con CNI, el tráfico puede ir por interfaces virtuales (veth).
- **tcpdump vs aplicación logs**: los logs dicen "timeout al conectar a Redis". tcpdump te dice exactamente: "el SYN salió, el SYN-ACK nunca llegó porque un firewall rule lo bloqueó". Siempre que un problema de red sea ambiguo, captura.
- **mitmproxy para APIs**: cuando debugging HTTP/HTTPS entre servicios, `mitmproxy` es más práctico que Wireshark. Intercepta, muestra, y permite modificar requests/responses en tiempo real.
