# 2.1 Modelos de Red

## Definición

Los modelos de red definen capas de abstracción que separan responsabilidades en la comunicación entre máquinas. Cada capa resuelve un problema específico (direccionamiento, transporte confiable, formato de datos) y se comunica solo con las capas adyacentes.

## Mecánica interna

### OSI (7 capas) vs TCP/IP (4 capas)

```
  OSI Model                    TCP/IP Model
  ┌─────────────────┐
  │ 7. Application  │          ┌─────────────────┐
  │ 6. Presentation │ ───────► │ 4. Application  │  (HTTP, DNS, gRPC)
  │ 5. Session      │          └─────────────────┘
  ├─────────────────┤          ┌─────────────────┐
  │ 4. Transport    │ ───────► │ 3. Transport    │  (TCP, UDP)
  ├─────────────────┤          └─────────────────┘
  │ 3. Network      │ ───────► ┌─────────────────┐
  ├─────────────────┤          │ 2. Internet     │  (IP, ICMP)
  │ 2. Data Link    │ ───────► └─────────────────┘
  │ 1. Physical     │          ┌─────────────────┐
  └─────────────────┘          │ 1. Network      │  (Ethernet, WiFi)
                               │    Access       │
                               └─────────────────┘
```

En la práctica, TCP/IP es el modelo real. OSI es útil como referencia conceptual.

### Encapsulación

Cada capa envuelve los datos de la capa superior con su propio header:

```
  Application data:  [HTTP GET /api/users]
                          │
  Transport (TCP):   [TCP Header][HTTP GET /api/users]
                          │         ← segment
  Network (IP):      [IP Header][TCP Header][HTTP data]
                          │         ← packet
  Data Link (Eth):   [Eth Header][IP Header][TCP Header][HTTP data][Eth Trailer]
                                    ← frame
```

En el receptor, cada capa **desencapsula** (lee su header, lo remueve, pasa el payload a la capa superior).

### Multiplexing y Demultiplexing

```
  Servidor con 3 servicios:

  ┌───────────────────────────────────┐
  │          NIC (IP: 10.0.1.5)       │
  │                │                  │
  │    ┌───────────┼───────────┐      │
  │    ▼           ▼           ▼      │
  │  :80         :443        :5432    │
  │  nginx       nginx       postgres │
  └───────────────────────────────────┘

  Demultiplexing: el kernel lee el puerto destino del
  header TCP/UDP y entrega los datos al socket correcto.

  Multiplexing: múltiples apps comparten la misma NIC (Network Interface Card),
  el kernel agrega el puerto origen al header de salida.
```

## Modelo mental

La red es un sistema postal. La capa de aplicación es la carta. La capa de transporte es el sobre con el número de apartamento (puerto). La capa de red es la dirección de la ciudad (IP). La capa de enlace es el camión de correo local (Ethernet/WiFi). Cada intermediario solo mira la información que le corresponde.

## Modos de fallo

| Síntoma | Causa probable | Capa |
|---------|---------------|------|
| Servicio inaccesible | IP incorrecta o routing roto | Network (L3) |
| Conexión rechazada | Puerto cerrado o firewall | Transport (L4) |
| Datos corruptos | Checksum falla | Transport/Network |
| Request llega pero no la response | NAT (Network Address Translation) o firewall stateful descarta la respuesta | Network (L3) |

## Impacto en sistemas distribuidos

Entender las capas determina dónde operar:
- **Load balancer L4** (Transport): enruta por IP:puerto, no inspecciona contenido. Rápido.
- **Load balancer L7** (Application): lee HTTP headers, URL paths. Flexible, más lento.
- **Service mesh** (L7): proxy sidecar que opera en la capa de aplicación para mTLS, routing, observabilidad.

## Herramientas de diagnóstico

```bash
# Ver cada capa en acción
tcpdump -i eth0 -n             # Capa 2/3: paquetes IP
tcpdump -i eth0 port 80        # Capa 4: filtrar por puerto TCP
curl -v https://api.example.com # Capa 7: HTTP verbose

# Trazar la ruta a nivel de red
traceroute -n api.example.com   # Cada hop = un router L3
mtr api.example.com             # traceroute continuo con estadísticas
```

## Insights de ingeniería

- **OSI es para exámenes, TCP/IP es para producción**: nadie habla de "capa de presentación" en un incidente. Se habla de "nivel TCP" o "nivel HTTP".
- **Cada capa agrega overhead**: Ethernet header (14 bytes) + IP header (20 bytes) + TCP header (20 bytes) = 54 bytes mínimo antes de un solo byte de datos. Con MTU de 1500, tu payload máximo por paquete es ~1446 bytes.
- **La abstracción tiene fugas**: TCP promete entrega confiable, pero si la red se particiona, la conexión muere. Debes manejar reconexiones a nivel de aplicación.
