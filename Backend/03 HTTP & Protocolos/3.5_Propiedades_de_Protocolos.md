# 3.5 Propiedades de Protocolos

## Definición

Cada protocolo de comunicación se define por un conjunto de propiedades ortogonales: stateful/stateless, full/half duplex, reliable/unreliable, connection-oriented/connectionless. Entender estas propiedades permite elegir el protocolo correcto para cada caso de uso.

## Mecánica interna

### Stateful vs Stateless

```
  Stateless (HTTP):
  Request 1: GET /api/users (+ Authorization: Bearer token)
  Request 2: GET /api/users (+ Authorization: Bearer token)  ← debe re-autenticar

  El servidor no recuerda nada entre requests.
  Toda la información necesaria va en cada request.

  Stateful (WebSocket, TCP, database connection):
  connect() → autenticar → [sesión establecida]
  request 1 → response (servidor sabe quién eres)
  request 2 → response (sin re-autenticar)

  El servidor mantiene estado de la conexión.
```

| Propiedad | Stateless | Stateful |
|-----------|-----------|----------|
| Scaling | Fácil (cualquier servidor) | Difícil (sticky sessions) |
| Fault tolerance | Alta (retry a otro servidor) | Baja (estado se pierde) |
| Overhead por request | Mayor (contexto en cada req) | Menor (contexto persistente) |
| Ejemplo | HTTP, DNS | WebSocket, TCP, gRPC stream |

### Full Duplex vs Half Duplex

```
  Full Duplex (WebSocket, TCP):
  ┌─────────┐    ← datos →    ┌─────────┐
  │ Cliente  │    ← datos →    │ Servidor│
  └─────────┘                  └─────────┘
  Ambos envían simultáneamente.

  Half Duplex (HTTP/1.1):
  ┌─────────┐    → request →   ┌─────────┐
  │ Cliente  │                   │ Servidor│
  │          │    ← response ←  │         │
  └─────────┘                  └─────────┘
  Uno envía, el otro espera. Luego se invierte.

  Simplex (SSE):
  ┌─────────┐    ← datos ←    ┌─────────┐
  │ Cliente  │                  │ Servidor│
  └─────────┘                  └─────────┘
  Solo una dirección.
```

### Reliable vs Unreliable

| Propiedad | Reliable (TCP) | Unreliable (UDP) |
|-----------|----------------|------------------|
| Entrega garantizada | Sí (retransmissions) | No |
| Orden garantizado | Sí (sequence numbers) | No |
| Detección de errores | Checksum + retransmit | Solo checksum |
| Overhead | Alto | Bajo |
| Latencia | Mayor (espera ACKs) | Menor |
| Caso de uso | APIs, DB, archivos | Streaming, gaming, DNS |

### Connection-oriented vs Connectionless

```
  Connection-oriented (TCP):
  1. Establecer conexión (handshake)
  2. Transferir datos
  3. Cerrar conexión (teardown)
  Estado mantenido durante toda la sesión.

  Connectionless (UDP):
  1. Enviar datagrama
  Fin. No hay estado previo ni posterior.
```

### Matriz de protocolos

| Protocolo | Stateful | Duplex | Reliable | Connection |
|-----------|----------|--------|----------|------------|
| HTTP/1.1 | No | Half | Sí (TCP) | Sí (TCP) |
| HTTP/2 | No | Half* | Sí (TCP) | Sí (TCP) |
| WebSocket | Sí | Full | Sí (TCP) | Sí |
| gRPC | Depende | Full | Sí (HTTP/2) | Sí |
| SSE | Sí | Simplex | Sí (HTTP) | Sí |
| UDP | No | Full | No | No |
| TCP | Sí | Full | Sí | Sí |
| QUIC | Sí | Full | Sí | Sí |
| DNS | No | Half | No (UDP) | No |

*HTTP/2 es half-duplex por stream (request → response), pero full-duplex a nivel de conexión (múltiples streams simultáneos).

## Modelo mental

Piensa en las propiedades como dimensiones independientes de un protocolo:
- **State**: ¿el servidor recuerda quién eres? (memoria)
- **Duplex**: ¿pueden hablar los dos al mismo tiempo? (dirección)
- **Reliability**: ¿el cartero confirma que llegó? (garantía)
- **Connection**: ¿necesitas marcar un número antes de hablar? (setup)

## Impacto en sistemas distribuidos

- **Stateless = horizontal scaling**: si cada request es autosuficiente, cualquier instancia puede atenderla. Load balancer round-robin funciona perfecto.
- **Stateful = sticky sessions o estado externo**: WebSocket connections deben ir al mismo servidor. Alternativa: externalizar estado en Redis y hacer el servicio stateless.
- **Reliability trade-off**: TCP garantiza entrega pero agrega latencia. Para métricas internas (StatsD), UDP es preferible — perder un datapoint es mejor que agregar latencia al hot path.

## Insights de ingeniería

- **HTTP es stateless, pero las aplicaciones necesitan estado**: JWT tokens, session cookies, y API keys son mecanismos para agregar estado sobre un protocolo stateless. El estado se externaliza al token o al store (Redis, DB).
- **Full duplex no significa que lo necesitas**: la mayoría de APIs son request-response. Usar WebSocket para todo agrega complejidad innecesaria (scaling, reconnection, state management).
- **Elige el protocolo más simple que resuelva tu problema**: REST para CRUD, SSE para push unidireccional, WebSocket solo si necesitas bidireccional con baja latencia, gRPC para inter-service con schema.
