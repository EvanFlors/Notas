# 3.2 Serialización de Datos

## Definición

Serialización es convertir estructuras de datos en memoria a un formato transmisible por la red (bytes). La elección del formato afecta tamaño del payload, velocidad de parsing, compatibilidad entre lenguajes, y experiencia de debugging.

## Mecánica interna

### JSON

```json
{
  "id": 42,
  "name": "Alice",
  "roles": ["admin", "user"],
  "active": true,
  "metadata": null
}
```

- **Formato**: texto UTF-8, human-readable
- **Tipos**: string, number, boolean, null, array, object
- **Sin schema**: el receptor debe validar la estructura
- **Tamaño**: verbose (keys repetidas, comillas, whitespace)
- **Parsing**: O(n) donde n = tamaño del string. Rápido con parsers como simdjson.

### Protocol Buffers (Protobuf)

```protobuf
// user.proto — schema definition
syntax = "proto3";

message User {
  int32 id = 1;          // field number, no name, en el wire
  string name = 2;
  repeated string roles = 3;
  bool active = 4;
}
```

```
  Wire format (binario):
  [field_number + wire_type][value]

  08 2A        → field 1 (id), varint, value 42
  12 05 Alice  → field 2 (name), length-delimited, "Alice"

  Mismo objeto: JSON = ~80 bytes, Protobuf = ~20 bytes
```

- **Schema obligatorio**: `.proto` file compartido entre servicios
- **Backward/forward compatible**: agregar campos nuevos no rompe clientes viejos
- **Code generation**: genera clases/structs en Go, Java, Python, etc.
- **Tamaño**: 3-10x más compacto que JSON

### MessagePack

```
  JSON:        {"id":42,"name":"Alice"}     → 25 bytes (texto)
  MessagePack: 82 A2 69 64 2A A4 ...       → 18 bytes (binario)

  Misma semántica que JSON pero en binario.
  No requiere schema. Más compacto, más rápido de parsear.
```

### Comparación

| Aspecto | JSON | Protobuf | MessagePack | XML |
|---------|------|----------|-------------|-----|
| Formato | Texto | Binario | Binario | Texto |
| Schema | No | Sí (.proto) | No | Sí (XSD) |
| Human-readable | Sí | No | No | Sí (verbose) |
| Tamaño relativo | 1x | 0.3x | 0.7x | 1.5x |
| Parse speed | Medio | Rápido | Rápido | Lento |
| Backward compat | Manual | Built-in | Manual | Manual |
| Uso principal | REST APIs, config | gRPC, inter-service | Caching, logging | Legacy (SOAP) |

## Modelo mental

JSON es como escribir una carta: legible, pero ocupa espacio. Protobuf es como código morse: compacto y rápido, pero necesitas el codebook (schema). MessagePack es como taquigrafía: más compacto que la carta, sin necesitar un codebook formal.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Parse error en el receiver | Schema mismatch (Protobuf) o JSON malformado | Validación, schema registry |
| Payload demasiado grande | JSON con campos innecesarios | Projección de campos, Protobuf |
| Incompatibilidad al agregar campos | Breaking change en Protobuf (cambiar field number) | Nunca reutilizar field numbers |
| Números grandes pierden precisión | JSON number > 2^53 (JavaScript) | Enviar como string |

## Impacto en sistemas distribuidos

- **gRPC + Protobuf** entre microservicios: schema compartido como contrato, backward compatible, 3-10x más eficiente que JSON. El schema `.proto` es la API specification.
- **JSON para APIs públicas**: los clientes (browsers, mobile) esperan JSON. Fácil de debuggear, amplio soporte.
- **Schema evolution**: en Protobuf, campos nuevos usan field numbers nuevos. Clientes viejos ignoran campos desconocidos. Esta propiedad es crítica cuando tienes 50 microservicios con deploy independiente.

## Insights de ingeniería

- **No serialices lo que no necesitas**: si solo necesitas id y name, no envíes el objeto completo con 30 campos. Reduce latencia y uso de red.
- **JSON parsing es más caro de lo que crees**: en hot paths con alto throughput, el parsing de JSON puede consumir 10-30% del CPU. Considera Protobuf o MessagePack para comunicación interna.
- **Protobuf field numbers son para siempre**: si asignas `id = 1`, nunca puedes usar el field number 1 para otra cosa, incluso si eliminas el campo. Marca como `reserved`.
- **XML en 2024**: solo lo encuentras en SOAP, integraciones enterprise legacy, y configuraciones de Java (Maven, Spring XML). No lo elijas para servicios nuevos.
