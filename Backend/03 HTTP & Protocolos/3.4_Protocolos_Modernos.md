# 3.4 Protocolos Modernos de Comunicación

## Definición

Más allá de HTTP request-response, existen protocolos especializados para comunicación bidireccional, streaming, y RPC de alto rendimiento. La elección depende del patrón de datos: ¿unidireccional o bidireccional? ¿latencia crítica? ¿schema tipado?

## Mecánica interna

### WebSockets

```
  1. HTTP Upgrade Handshake:
  GET /chat HTTP/1.1
  Upgrade: websocket
  Connection: Upgrade
  Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==

  HTTP/1.1 101 Switching Protocols
  Upgrade: websocket
  Connection: Upgrade

  2. Conexión full-duplex establecida:
  ┌─────────┐                    ┌─────────┐
  │ Cliente  │ ◄── frame ──────► │ Servidor│
  │          │ ◄── frame ──────► │         │
  │          │ ◄── frame ──────► │         │
  └─────────┘                    └─────────┘

  - Persistente: una sola conexión TCP para toda la sesión
  - Full-duplex: ambos lados envían cuando quieren
  - Frames binarios o texto
  - Ping/pong frames para keep-alive
```

Casos de uso: chat en tiempo real, trading, dashboards live, gaming, collaborative editing.

### gRPC

```
  Client                              Server
    │                                    │
    │── HTTP/2 stream ──────────────────►│
    │   [Protobuf-encoded request]       │
    │                                    │
    │◄── HTTP/2 stream ─────────────────│
    │   [Protobuf-encoded response]      │
    │                                    │

  4 tipos de RPC:
  1. Unary:              request → response (como REST)
  2. Server streaming:   request → stream de responses
  3. Client streaming:   stream de requests → response
  4. Bidirectional:      stream ↔ stream
```

```protobuf
service UserService {
  rpc GetUser (GetUserRequest) returns (User);                    // Unary
  rpc ListUsers (ListRequest) returns (stream User);              // Server stream
  rpc UploadUsers (stream User) returns (UploadResponse);         // Client stream
  rpc Chat (stream Message) returns (stream Message);             // Bidi stream
}
```

Ventajas sobre REST: schema tipado (Protobuf), code generation, multiplexing (HTTP/2), streaming nativo, más eficiente en tamaño y parsing.

### Server-Sent Events (SSE)

```
  GET /events HTTP/1.1
  Accept: text/event-stream

  HTTP/1.1 200 OK
  Content-Type: text/event-stream

  data: {"price": 42.50}       ← evento 1

  data: {"price": 42.75}       ← evento 2

  event: alert                 ← evento con tipo custom
  data: {"msg": "threshold"}

  La conexión queda abierta indefinidamente.
  Si se corta → browser reconecta automáticamente (Last-Event-ID).
```

Casos de uso: notifications, price feeds, log streaming. Más simple que WebSocket cuando solo necesitas server → client.

### WebRTC

```
  Peer A ◄──────── direct ────────► Peer B
         (UDP, peer-to-peer)

  Signaling (via server): intercambio de SDP/ICE candidates
  Media: audio/video directo entre peers
  Data channels: datos arbitrarios peer-to-peer

  Uso: video calls, screen sharing, peer-to-peer file transfer.
  No es un protocolo de backend típico, pero el signaling server sí lo es.
```

### Comparación

| Aspecto | WebSocket | gRPC | SSE | WebRTC |
|---------|-----------|------|-----|--------|
| Dirección | Bidireccional | Bidireccional | Server → Client | Peer-to-peer |
| Transporte | TCP | HTTP/2 (TCP) | HTTP/1.1+ | UDP (DTLS) |
| Formato | Texto/binario | Protobuf | Texto (event-stream) | Media/datos |
| Schema | No | Sí (.proto) | No | SDP |
| Browser support | Sí | No (necesita grpc-web) | Sí | Sí |
| Reconexión auto | No (manual) | No | Sí (built-in) | ICE restart |
| Caso ideal | Chat, gaming | Inter-service RPC | Notifications | Video calls |

## Modelo mental

- **REST**: envías una carta, esperas respuesta. Para cada pregunta, nueva carta.
- **WebSocket**: abres una línea telefónica permanente. Ambos hablan cuando quieren.
- **SSE**: te suscribes a un canal de radio. Solo recibes, no transmites.
- **gRPC**: como una llamada telefónica con un protocolo estricto (ambos saben exactamente qué decir, en qué formato).

## Modos de fallo

| Protocolo | Síntoma | Causa |
|-----------|---------|-------|
| WebSocket | Conexión se corta cada 60s | Proxy/LB con idle timeout. Solución: ping/pong |
| WebSocket | No conecta detrás de proxy | Proxy no soporta HTTP Upgrade. Usar fallback a long polling |
| gRPC | No funciona desde browser | gRPC requiere HTTP/2 trailers, browsers no soportan. Usar grpc-web |
| SSE | Reconexiones constantes | Servidor cierra conexión. Verificar keep-alive y proxy timeouts |

## Impacto en sistemas distribuidos

- **gRPC para inter-service**: en una arquitectura de microservicios, gRPC con Protobuf es el estándar de facto para comunicación interna. Schema como contrato, backward compatible, eficiente.
- **WebSocket scaling**: cada conexión WebSocket consume un fd y memoria. 100K usuarios conectados = 100K sockets persistentes. Requiere sticky sessions o un pub/sub backend (Redis) para broadcasting.
- **SSE vs WebSocket**: si solo necesitas push del server (notifications, feeds), SSE es más simple, funciona con HTTP estándar, y reconecta automáticamente. No necesitas la complejidad de WebSocket.

## Insights de ingeniería

- **WebSocket a través de load balancers**: necesitas sticky sessions o un mecanismo de routing (ej: por user ID). Si el load balancer redistribuye, la conexión se pierde.
- **gRPC deadline propagation**: cuando servicio A llama a B que llama a C, el deadline (timeout) se propaga. Si quedan 200ms cuando B llama a C, C sabe que tiene máximo 200ms. Esto previene cascading timeouts.
- **No uses WebSocket para todo**: si tu patrón es request-response, HTTP es más simple, cacheable, y compatible con toda la infraestructura existente (proxies, CDNs, monitoring).
