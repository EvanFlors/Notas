# 3.1 Fundamentos HTTP

## Definición

HTTP (HyperText Transfer Protocol) es un protocolo request-response sin estado en la capa de aplicación. El cliente envía un request con method, URL, headers y body; el servidor responde con status code, headers y body. Es la base de toda API REST y web.

## Mecánica interna

### Anatomía de un HTTP Request

```
  POST /api/users HTTP/1.1          ← Request line (method, path, version)
  Host: api.example.com             ← Header obligatorio en HTTP/1.1
  Content-Type: application/json    ← Formato del body
  Authorization: Bearer eyJhb...    ← Autenticación
  Accept: application/json          ← Formato deseado de respuesta
  Content-Length: 42                 ← Tamaño del body en bytes
                                    ← Línea vacía (separa headers del body)
  {"name": "Alice", "role": "admin"}  ← Body
```

### Anatomía de un HTTP Response

```
  HTTP/1.1 201 Created              ← Status line (version, code, reason)
  Content-Type: application/json
  Location: /api/users/42           ← URL del recurso creado
  Cache-Control: no-cache
  X-Request-Id: abc-123             ← Custom header para tracing
  Content-Length: 58

  {"id": 42, "name": "Alice", "role": "admin", "created": true}
```

### HTTP Methods

| Method | Semántica | Idempotente | Safe | Body |
|--------|-----------|-------------|------|------|
| GET | Obtener recurso | Sí | Sí | No |
| POST | Crear recurso / acción | No | No | Sí |
| PUT | Reemplazar recurso completo | Sí | No | Sí |
| PATCH | Modificar parcialmente | No* | No | Sí |
| DELETE | Eliminar recurso | Sí | No | Opcional |
| HEAD | Como GET pero solo headers | Sí | Sí | No |
| OPTIONS | Métodos soportados / CORS preflight | Sí | Sí | No |

*PATCH puede ser idempotente si usas JSON Merge Patch, pero no está garantizado por el spec.

**Idempotente**: ejecutar N veces produce el mismo resultado que ejecutar 1 vez.
**Safe**: no modifica estado en el servidor.

### Status Codes

```
  1xx — Informational
  100 Continue              Cliente puede enviar el body
  101 Switching Protocols   WebSocket upgrade

  2xx — Success
  200 OK                    Request exitosa
  201 Created               Recurso creado (POST)
  204 No Content            Éxito sin body (DELETE)

  3xx — Redirection
  301 Moved Permanently     URL cambió permanentemente (cacheable)
  302 Found                 Redirect temporal
  304 Not Modified          Cache válida, no enviar body

  4xx — Client Error
  400 Bad Request           Payload inválido
  401 Unauthorized          No autenticado
  403 Forbidden             Autenticado pero sin permisos
  404 Not Found             Recurso no existe
  409 Conflict              Conflicto de estado (ej: duplicado)
  422 Unprocessable Entity  Validación fallida
  429 Too Many Requests     Rate limit alcanzado

  5xx — Server Error
  500 Internal Server Error Error no manejado
  502 Bad Gateway           Upstream no responde (proxy)
  503 Service Unavailable   Servidor sobrecargado
  504 Gateway Timeout       Upstream timeout (proxy)
```

### Headers esenciales para backend

| Header | Dirección | Propósito |
|--------|-----------|-----------|
| `Content-Type` | Ambos | Formato del body (application/json, multipart/form-data) |
| `Authorization` | Request | Credenciales (Bearer token, Basic) |
| `Cache-Control` | Response | Política de cache (max-age, no-store, no-cache) |
| `ETag` | Response | Versión del recurso para conditional requests |
| `If-None-Match` | Request | Conditional GET (304 si ETag coincide) |
| `Accept` | Request | Formatos que acepta el cliente |
| `X-Request-Id` | Ambos | Correlation ID para distributed tracing |
| `Retry-After` | Response | Cuándo reintentar (con 429 o 503) |

### Conditional Requests y Caching

```
  Primera request:
  GET /api/users/42 → 200 OK
  ETag: "v5"
  Cache-Control: max-age=60

  Dentro de 60s:
  GET /api/users/42 → (no sale del browser, cache hit)

  Después de 60s:
  GET /api/users/42
  If-None-Match: "v5"
  → 304 Not Modified (datos no cambiaron, no envía body)

  Si los datos cambiaron:
  → 200 OK, ETag: "v6" (envía body nuevo)
```

## Modelo mental

HTTP es como llenar un formulario en una ventanilla: escribes qué quieres (method + URL), adjuntas documentos (body), el empleado procesa y te da un ticket con el resultado (status code + response body). Es stateless — cada vez que regresas, debes presentar todo de nuevo (o llevar un token que te identifique).

## Modos de fallo

| Síntoma | Causa probable | Acción |
|---------|---------------|--------|
| 502 Bad Gateway | Backend caído o no responde al proxy | Verificar health del upstream |
| 504 Gateway Timeout | Backend demasiado lento | Aumentar timeout o optimizar query |
| 429 Too Many Requests | Rate limit del API | Respetar `Retry-After`, implementar backoff |
| CORS error en browser | Falta `Access-Control-Allow-Origin` | Configurar CORS headers en el backend |

## Impacto en sistemas distribuidos

- **Idempotency**: en una red no confiable, el cliente puede retry un POST. Si no es idempotente, creas duplicados. Solución: idempotency keys (header `Idempotency-Key`).
- **Status codes como contratos**: en microservicios, el status code indica al caller si debe retry (503), corregir su request (400), o escalar como incidente (500).
- **Content negotiation**: permite evolucionar APIs sin romper clientes. El server puede responder JSON o Protobuf según el `Accept` header.

## Herramientas de diagnóstico

```bash
# Request completa con timing
curl -w "\nDNS: %{time_namelookup}s\nTCP: %{time_connect}s\nTLS: %{time_appconnect}s\nTTFB: %{time_starttransfer}s\nTotal: %{time_total}s\n" \
  -o /dev/null -s https://api.example.com

# Ver headers de response
curl -I https://api.example.com

# Request con headers custom
curl -H "Authorization: Bearer token" -H "Accept: application/json" \
  https://api.example.com/users
```

## Insights de ingeniería

- **PUT vs PATCH**: PUT reemplaza el recurso completo (debes enviar todos los campos). PATCH modifica solo los campos enviados. La mayoría de APIs realmente necesitan PATCH, no PUT.
- **204 vs 200 para DELETE**: usar 204 No Content evita enviar un body vacío y es semánticamente correcto para "eliminé el recurso, no hay nada que devolver".
- **Cache-Control en APIs**: para APIs dinámicas, usa `Cache-Control: no-store` (ni browser ni proxy cachean). Para recursos semi-estáticos (catálogos), `max-age` + `ETag` reduce carga significativamente.
- **No inventes status codes**: usa los estándar. 200 para éxito, 201 para creación, 400 para input malo, 404 para no encontrado, 500 para error interno. No uses 200 con `{"error": true}` en el body.
