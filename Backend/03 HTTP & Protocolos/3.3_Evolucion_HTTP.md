# 3.3 Evolución de HTTP

## Definición

Cada versión de HTTP resuelve limitaciones de rendimiento de la anterior. Entender la evolución explica por qué HTTP/2 multiplexa, por qué HTTP/3 usa UDP, y cómo estas decisiones impactan la latencia de tu API.

## Mecánica interna

### HTTP/1.0

```
  Cada request = nueva conexión TCP:

  [TCP handshake] → GET /index.html → response → [TCP close]
  [TCP handshake] → GET /style.css  → response → [TCP close]
  [TCP handshake] → GET /app.js     → response → [TCP close]

  3 requests = 3 handshakes = 3 RTT de overhead solo en TCP.
  Slow start reinicia en cada conexión.
```

### HTTP/1.1

```
  Keep-alive (conexión persistente):
  [TCP handshake] → GET /index → resp → GET /style → resp → GET /app → resp

  1 conexión, múltiples requests secuenciales.

  PERO: Head-of-Line (HOL) Blocking
  Request 1 tarda 2s → requests 2 y 3 esperan.

  Workaround del browser: abrir 6 conexiones TCP paralelas al mismo dominio.
  Workaround del server: domain sharding (static.example.com, img.example.com).
```

### HTTP/2

```
  Multiplexing: múltiples streams sobre UNA conexión TCP.

  ┌─── Conexión TCP ────────────────────────┐
  │                                          │
  │  Stream 1: GET /api/users   ─────►      │
  │  Stream 2: GET /api/orders  ─────►      │
  │  Stream 3: POST /api/cart   ─────►      │
  │                                          │
  │  Frames intercalados:                    │
  │  [S1:HEADERS][S2:HEADERS][S1:DATA]      │
  │  [S3:HEADERS][S2:DATA][S3:DATA]         │
  │  [S1:DATA]                               │
  └──────────────────────────────────────────┘

  Cada stream es independiente a nivel HTTP.
  PERO: si se pierde 1 paquete TCP, TODOS los streams esperan
  la retransmisión (HOL blocking a nivel TCP).
```

Features de HTTP/2:
- **Binary framing**: no es texto como HTTP/1.1, es binario (más eficiente de parsear)
- **Header compression (HPACK)**: tabla de headers comunes. `Content-Type: application/json` se envía como un índice de 1 byte en requests subsecuentes.
- **Server Push**: el servidor envía recursos antes de que el cliente los pida (poco usado en la práctica, deprecado en Chrome).
- **Stream priorities**: el cliente indica qué streams son más importantes.

### HTTP/3 (QUIC)

```
  HTTP/3 usa QUIC en vez de TCP:

  ┌─── QUIC (sobre UDP) ───────────────────┐
  │                                          │
  │  Stream 1: GET /api/users               │
  │  Stream 2: GET /api/orders              │
  │  Stream 3: POST /api/cart               │
  │                                          │
  │  Si se pierde un paquete del Stream 2:  │
  │  - Stream 2 espera retransmisión        │
  │  - Streams 1 y 3 NO se afectan         │  ← Sin HOL blocking
  │                                          │
  └──────────────────────────────────────────┘

  QUIC integra TLS 1.3 en el handshake:
  HTTP/1.1 + TLS 1.2: TCP(1 RTT) + TLS(2 RTT) = 3 RTT
  HTTP/2 + TLS 1.3:   TCP(1 RTT) + TLS(1 RTT) = 2 RTT
  HTTP/3 (QUIC):      QUIC(1 RTT, incluye TLS) = 1 RTT
  HTTP/3 (0-RTT):     Session resumption        = 0 RTT
```

### Comparación directa

| Aspecto | HTTP/1.1 | HTTP/2 | HTTP/3 |
|---------|----------|--------|--------|
| Transporte | TCP | TCP | QUIC (UDP) |
| Multiplexing | No (1 req/resp a la vez) | Sí (streams) | Sí (streams independientes) |
| HOL blocking | Sí (aplicación) | Parcial (TCP level) | No |
| Header compression | No | HPACK | QPACK |
| Handshake | TCP + TLS separados | TCP + TLS separados | Integrado (1 RTT) |
| Formato | Texto | Binario | Binario |
| Adopción | Universal | Mayoría de CDNs/servers | Creciendo (CDNs, browsers) |

## Modelo mental

HTTP/1.1 es una carretera de un carril: los autos van uno detrás de otro. HTTP/2 agrega carriles virtuales (streams) pero sobre el mismo puente (TCP): si hay un bache, todos paran. HTTP/3 construye un puente separado por carril (QUIC streams independientes): un bache en un carril no afecta a los demás.

## Modos de fallo

| Síntoma | Versión | Causa |
|---------|---------|-------|
| Latencia alta en carga de página | HTTP/1.1 | HOL blocking, muchas conexiones TCP |
| Stall momentáneo en todos los streams | HTTP/2 | Packet loss → TCP retransmission afecta todo |
| HTTP/3 fallback a HTTP/2 | HTTP/3 | Firewall bloquea UDP |
| Primera conexión lenta, subsecuentes rápidas | Todas | TLS handshake. HTTP/3 con 0-RTT lo resuelve |

## Impacto en sistemas distribuidos

- **Inter-service**: gRPC usa HTTP/2 multiplexing. Una sola conexión TCP entre dos servicios maneja cientos de RPCs concurrentes.
- **Edge/CDN**: Cloudflare, AWS CloudFront soportan HTTP/3. La ventaja es mayor en redes con packet loss (mobile, regiones con mala conectividad).
- **Elección pragmática**: entre microservicios en la misma red (baja latencia, bajo packet loss), HTTP/2 es suficiente. HTTP/3 brilla en edge/client-facing con redes variables.

## Insights de ingeniería

- **HTTP/2 no elimina la necesidad de connection pooling**: aunque multiplexa, una sola conexión comparte un cwnd de TCP. Para máximo throughput, múltiples conexiones HTTP/2 pueden ser necesarias.
- **HTTP/3 fallback**: los clientes intentan QUIC primero y hacen fallback a HTTP/2 si UDP está bloqueado. Tu servidor debe soportar ambos.
- **No hagas domain sharding con HTTP/2**: en HTTP/1.1, abrir múltiples conexiones a subdominios diferentes mejoraba rendimiento. En HTTP/2, esto es contraproducente (pierdes multiplexing y header compression).
