# 12.1 Estrategia de Testing

## Definición

La testing pyramid define la distribución óptima de tests: muchos unit tests rápidos en la base, menos integration tests en el medio, y pocos end-to-end tests costosos en la cima. Contract tests verifican interfaces entre servicios.

## Mecánica interna

### Testing Pyramid

```
        /  E2E  \         Pocos, lentos, frágiles
       / ─────── \        (Cypress, Playwright)
      / Contract  \       Interfaz entre servicios
     / ─────────── \      (Pact)
    /  Integration  \     Componentes reales (DB, API)
   / ─────────────── \    (testcontainers, real HTTP)
  /     Unit Tests    \   Muchos, rápidos, aislados
 / ─────────────────── \  (jest, pytest, go test)
```

### Unit Tests

```
  Testan lógica aislada. Mocks para dependencias externas.

  // Test de lógica de negocio
  test("discount applies for orders over $100", () => {
    const order = createOrder({subtotal: 150});
    expect(calculateDiscount(order)).toBe(15); // 10% discount
  });

  Rápidos (<1s), ejecutables en cada commit.
  Mocks: simulan DB, HTTP clients, external services.
```

### Integration Tests

```
  Testan componentes reales conectados.

  // Test con base de datos real (testcontainers)
  test("creates user in database", async () => {
    const user = await userService.create({name: "Alice", email: "a@b.com"});
    const found = await db.query("SELECT * FROM users WHERE id = $1", [user.id]);
    expect(found.name).toBe("Alice");
  });

  Más lentos (seconds), ejecutables en CI.
  Testcontainers: levanta PostgreSQL, Redis en Docker para tests.
```

### Contract Tests

```
  En microservicios: ¿el API del service B sigue siendo compatible
  con lo que service A espera?

  Consumer (A) define: "espero GET /users/42 → {id: 42, name: string}"
  Provider (B) verifica: "¿mi API cumple con este contrato?"

  Herramienta: Pact
  Previene: deploy de B que rompe A sin que nadie se dé cuenta.
```

### End-to-End Tests

```
  Flujo completo del usuario:
  1. Abrir browser
  2. Login
  3. Crear orden
  4. Verificar confirmación

  Lentos (minutos), frágiles (cualquier cambio de UI los rompe).
  Pocos pero críticos (happy paths principales).
```

## Insights de ingeniería

- **70/20/10 rule**: 70% unit, 20% integration, 10% e2e. Si inviertes la pirámide (muchos e2e, pocos unit), tu CI será lento y frágil.
- **Testcontainers > mocks para DB**: mockear SQL queries es frágil y no testa la query real. Usar una DB real en Docker es más lento pero más confiable.
- **Contract tests para microservicios**: sin contract tests, solo descubres incompatibilidades cuando deployeas y el servicio falla en producción.
- **Test lo que importa, no el coverage %**: 80% coverage con tests significativos > 100% coverage con tests triviales.
