# 12.4 Network Performance Engineering

## Definición

Entender el overhead de la red (MTU, handshakes, keep-alive, HOL blocking) permite optimizar la latencia entre servicios y entre servidor y cliente.

## Mecánica interna

### MSS, MTU, Path MTU

```
  MTU = Maximum Transmission Unit (Ethernet: 1500 bytes)
  MSS = Maximum Segment Size = MTU - IP header(20) - TCP header(20) = 1460 bytes

  Si envías más que el MSS → fragmentación → overhead.
  Path MTU Discovery: enviar con DF=1, reducir si ICMP "too big".

  En tunnels (VPN, VXLAN): MTU efectivo menor.
  K8s overlay networks: MTU típico 1450 (50 bytes de overhead de tunnel).
```

### Handshake Overhead

```
  Costo de establecer una nueva conexión:
  TCP:        1 RTT   (SYN → SYN-ACK → ACK)
  TCP + TLS:  2-3 RTT (dependiendo de TLS version)

  En un backend que llama a 5 servicios por request:
  Sin pooling: 5 × 2 RTT = 10 RTT de overhead = 500ms (con 50ms RTT)
  Con pooling: 0 RTT (conexiones ya establecidas)
```

### Keep-alive vs New Connections

```
  HTTP/1.1 keep-alive: reutiliza conexión TCP para múltiples requests.
  Connection: keep-alive (default en HTTP/1.1)

  Cuándo cerrar:
  - Timeout: si no hay actividad por N segundos
  - Max requests: después de N requests por conexión
  - Server: Connection: close header

  En microservicios: SIEMPRE keep-alive + connection pooling.
  Cada nueva conexión = handshake + slow start = latencia innecesaria.
```

### Head-of-Line Blocking

```
  HTTP/1.1: request 1 bloquea request 2 (mismo connection)
  HTTP/2:   packet loss bloquea TODOS los streams (TCP level)
  HTTP/3:   packet loss solo bloquea el stream afectado (QUIC)

  Impacto real: en redes con >1% packet loss (mobile, WiFi),
  HTTP/3 tiene ventaja significativa sobre HTTP/2.
  Entre microservicios (datacenter, <0.01% loss): diferencia negligible.
```

## Insights de ingeniería

- **Connection pooling es la optimización #1 de red**: amortiza handshake, mantiene cwnd caliente, reduce latencia a near-zero para requests subsecuentes.
- **MTU mismatch causa problemas silenciosos**: si un path tiene MTU menor al esperado y ICMP está bloqueado, Path MTU Discovery falla → paquetes descartados silenciosamente → connections hang. Common en VPN/tunnels.
- **Cada proxy hop suma latencia**: cliente → CDN → LB → API Gateway → sidecar → service → sidecar → DB. Cada hop agrega ~0.1-1ms. Con 6 hops: 0.6-6ms de overhead puro de proxying.
