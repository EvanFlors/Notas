# 8.6 Caching

## Definición

Caching almacena resultados previamente computados para evitar recalcularlos o re-leerlos de fuentes lentas. Es la optimización más efectiva en backend: reduce latencia de ms a μs, reduce carga en la base de datos, y mejora throughput.

## Mecánica interna

### Cache Patterns

```
  Cache-Aside (Lazy Loading):
  1. App busca en cache → miss
  2. App lee de DB
  3. App escribe en cache
  4. Próxima request: cache hit → responde sin tocar DB

  Problema: primera request siempre es lenta (cold cache).

  Write-Through:
  1. App escribe en cache Y en DB (síncrono)
  2. Reads siempre del cache

  Problema: write latency aumenta (doble escritura).

  Write-Behind (Write-Back):
  1. App escribe en cache
  2. Cache escribe en DB asincrónicamente (batch)

  Problema: si cache crashea antes de flush, datos perdidos.

  Read-Through:
  1. App busca en cache → miss → cache LEE de DB automáticamente
  2. Cache almacena y retorna

  Similar a cache-aside pero la lógica de fetch está en el cache layer.
```

### Redis como Cache

```
  SET user:42 '{"name":"Alice","role":"admin"}' EX 3600  ← TTL 1 hora
  GET user:42 → cache hit o nil

  Data structures útiles:
  - String: cache simple key-value
  - Hash: campos de un objeto (HGET user:42 name)
  - Sorted Set: leaderboards, rate limiting (sliding window)
  - List: queues, recent activity feeds
  - Set: tags, unique visitors

  Latencia: ~0.1ms (local), ~1ms (cross-AZ)
  Throughput: 100K+ ops/seg por instancia
```

### Cache Invalidation

```
  "There are only two hard things in Computer Science:
   cache invalidation and naming things." — Phil Karlton

  Estrategias:
  1. TTL (Time-To-Live): expira después de N segundos
     Simple, pero datos stale durante el TTL.

  2. Event-based: cuando cambia el dato en DB, invalidar cache
     Preciso, pero requiere pub/sub o change data capture.

  3. Write-through: actualiza cache al escribir en DB
     Siempre consistente, pero doble write.

  4. Versioned keys: cache:user:42:v5
     Cambiar la versión invalida implícitamente.
```

### Multi-Level Caching

```
  Request → CDN (edge, ms) → Reverse Proxy cache (Nginx, ms)
         → Application cache (Redis, <1ms) → DB cache (buffer pool, <1ms)
         → Disk (10ms HDD, 0.1ms SSD)

  Cada nivel tiene trade-offs:
  CDN: global, estático, TTL-based
  Nginx: local, semi-dinámico
  Redis: shared, dinámico, invalidatable
  Buffer pool: transparente, managed by DB
```

### Cache Stampede

```
  Cache key expires.
  100 concurrent requests para la misma key.
  Todas van a la DB simultáneamente → DB overloaded.

  Soluciones:
  1. Lock: primera request adquiere lock, las demás esperan
  2. Probabilistic early expiration: refrescar antes del TTL real
  3. Stale-while-revalidate: retornar dato stale mientras se refresca
```

## Modelo mental

Cache es como tener un post-it con el teléfono de alguien. Si lo tienes (hit), lo lees en 1 segundo. Si no (miss), buscas en la agenda (DB) y lo anotas en el post-it para la próxima vez. El reto es saber cuándo tirar el post-it (invalidation) porque el número puede haber cambiado.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Datos stale | TTL demasiado largo, no invalidation | Event-based invalidation |
| Cache miss storm → DB overload | Cache expiry masiva (misma hora) | Jitter en TTL, staggered expiration |
| Cache stampede | Key popular expira, N requests a DB | Locking, probabilistic early refresh |
| Redis OOM | Sin max-memory policy | maxmemory + eviction policy (allkeys-lru) |

## Insights de ingeniería

- **Cache las queries, no los objetos**: `cache_key = hash(SQL + params)`. Invalidar es más difícil pero los savings son mayores.
- **TTL + event invalidation**: usa TTL como safety net (máximo stale de 5 min) + event-based invalidation para consistency cuando es posible.
- **No cachees todo**: solo cachea datos que se leen mucho más de lo que se escriben (read-heavy). Cachear datos que cambian cada segundo agrega complejidad sin beneficio.
- **Redis eviction policies**: cuando Redis llena su maxmemory, `allkeys-lru` evicta la key menos usada recientemente. `volatile-ttl` evicta la key con menor TTL restante. Configura según tu caso de uso.
