# 8.5 Escalabilidad de Datos

## Definición

Cuando una sola instancia de base de datos no es suficiente (reads, writes, o storage), las técnicas de scaling entran en juego: replication para reads, sharding para writes y storage, partitioning para queries, y connection pooling para concurrencia.

## Mecánica interna

### Replication

```
  Master-Slave (Read Replicas):
  ┌────────┐    WAL stream    ┌──────────┐
  │ Primary │ ──────────────► │ Replica 1 │  ← reads
  │ (R+W)  │ ──────────────► │ Replica 2 │  ← reads
  └────────┘                  └──────────┘

  - Writes van al primary
  - Reads se distribuyen entre replicas
  - Replication lag: replicas pueden estar ms-seconds detrás

  Sync vs Async:
  - Sync: primary espera ACK de replica antes de COMMIT. Durabilidad máxima, latencia alta.
  - Async: primary no espera. Rápido, pero replica puede perder últimos writes si primary muere.
  - Semi-sync: primary espera ACK de al menos 1 replica.
```

### Sharding (Horizontal Partitioning)

```
  Distribuir datos entre múltiples bases de datos:

  Shard 1 (users 1-1M)    Shard 2 (users 1M-2M)    Shard 3 (users 2M-3M)
  ┌──────────────┐         ┌──────────────┐           ┌──────────────┐
  │ PostgreSQL   │         │ PostgreSQL   │           │ PostgreSQL   │
  │ users, orders│         │ users, orders│           │ users, orders│
  └──────────────┘         └──────────────┘           └──────────────┘

  Shard key: determina a qué shard va cada fila.
  - Hash: hash(user_id) % num_shards → distribución uniforme
  - Range: user_id 1-1M → shard 1 → hotspots si rangos desiguales
  - Directory: lookup table que mapea key → shard → flexible pero bottleneck

  Consistent hashing: minimiza redistribución al agregar/quitar shards.
```

### Table Partitioning (dentro de una instancia)

```sql
  -- PostgreSQL native partitioning
  CREATE TABLE orders (
    id SERIAL, user_id INT, created_at TIMESTAMP, amount DECIMAL
  ) PARTITION BY RANGE (created_at);

  CREATE TABLE orders_2024_q1 PARTITION OF orders
    FOR VALUES FROM ('2024-01-01') TO ('2024-04-01');
  CREATE TABLE orders_2024_q2 PARTITION OF orders
    FOR VALUES FROM ('2024-04-01') TO ('2024-07-01');

  -- Query automáticamente solo escanea partición relevante:
  SELECT * FROM orders WHERE created_at = '2024-02-15';
  -- → solo lee orders_2024_q1
```

### Connection Pooling

```
  Sin pooler:
  App (100 connections) → PostgreSQL (100 processes = 100 × ~10MB = 1GB RAM)

  Con PgBouncer:
  App (100 connections) → PgBouncer → PostgreSQL (20 connections)

  PgBouncer mantiene un pool de conexiones reales a PostgreSQL.
  App connections se multiplexan sobre menos DB connections.

  Modos:
  - Session: 1 app conn = 1 db conn mientras dure la session
  - Transaction: 1 app conn = 1 db conn solo durante una TX
  - Statement: 1 app conn = 1 db conn por statement (más agresivo)
```

## Modelo mental

- **Replication** = fotocopias del libro. Más copias = más lectores simultáneos, pero solo el original se puede editar.
- **Sharding** = dividir una enciclopedia en tomos. Tomo A-F en un estante, G-M en otro. Cada tomo es independiente.
- **Partitioning** = un archivador con cajones por año. Buscar algo de 2024 solo abre el cajón de 2024.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Read-after-write inconsistency | Replication lag + read from replica | Read from primary para critical reads |
| Cross-shard query lento | Query necesita datos de múltiples shards | Denormalizar, o evitar cross-shard queries |
| Hotspot en un shard | Shard key con distribución desigual | Mejor shard key, consistent hashing |
| Connection exhaustion | Demasiadas app instances × connections | PgBouncer/ProxySQL |

## Insights de ingeniería

- **No shardees prematuramente**: sharding agrega complejidad masiva (cross-shard queries, distributed transactions, resharding). PostgreSQL maneja bien hasta ~1TB con hardware adecuado. Replica primero, shardea después.
- **Read replicas para analytics**: envía queries pesadas (reports, analytics) a replicas para no impactar el primary (OLTP).
- **PgBouncer es casi obligatorio en producción**: PostgreSQL fork un proceso por conexión. Con 200 app pods × 10 connections = 2000 procesos PostgreSQL. Con PgBouncer: 50 conexiones reales.
