# 8.2 Internals de Bases de Datos

## Definición

Entender la estructura interna de una base de datos — cómo almacena datos en disco, cómo indexa, cómo garantiza durabilidad — te permite tomar decisiones informadas sobre schema design, indexación, y tuning. Los conceptos clave son B-Trees, LSM Trees, WAL, MVCC, y el buffer pool.

## Mecánica interna

### B-Tree (índices en bases relacionales)

```
  B-Tree de orden 3 (max 3 keys por nodo):

              [10 | 20]
             /    |    \
        [3|7]  [12|15]  [25|30]
        / | \   / | \    / | \
       datos   datos    datos

  - Cada nodo = una página de disco (~8KB en PostgreSQL)
  - Búsqueda: O(log n) — sigue el árbol desde la raíz
  - Inserción: O(log n) — busca posición, split si nodo lleno
  - Lecturas dominan: optimizado para read-heavy workloads

  PostgreSQL B-Tree index:
  - Cada hoja contiene punteros a las filas en la tabla (heap)
  - Index scan: lee el índice → sigue punteros → lee filas
```

### LSM Tree (Log-Structured Merge Tree)

```
  Writes: SIEMPRE van a un memtable en RAM (sorted)

  ┌─────────────────┐
  │   Memtable (RAM) │  ← escrituras aquí (red-black tree)
  └────────┬────────┘
           │ flush cuando lleno
  ┌────────▼────────┐
  │  SSTable L0      │  ← archivo sorted en disco (immutable)
  └────────┬────────┘
           │ compaction (merge sort)
  ┌────────▼────────┐
  │  SSTable L1      │  ← archivos más grandes, menos overlap
  └────────┬────────┘
           │
  ┌────────▼────────┐
  │  SSTable L2      │
  └─────────────────┘

  Writes: O(1) amortizado (append to memtable). Ultra rápido.
  Reads: O(log n) pero puede necesitar buscar en múltiples SSTables.
  Bloom filters: evitan reads innecesarios de SSTables que no contienen la key.

  Usado por: Cassandra, RocksDB, LevelDB, CockroachDB.
```

### WAL (Write-Ahead Log)

```
  Sin WAL:
  UPDATE → modifica página en buffer pool → crash → datos perdidos

  Con WAL:
  UPDATE → escribe cambio al WAL (disco, sequential) → modifica buffer pool
  → crash → replay WAL → estado recuperado

  WAL = log secuencial append-only en disco.
  Sequential write es 100x más rápido que random write.

  PostgreSQL: WAL en pg_wal/
  MySQL: redo log (mismo concepto)
  Checkpoint: periódicamente, flush dirty pages a disco y truncar WAL.
```

### MVCC (Multi-Version Concurrency Control)

```
  Transacción A (read)          Transacción B (write)
  BEGIN                         BEGIN
  SELECT * FROM users           │
  WHERE id = 1                  │
  → ve version V1               UPDATE users SET name='Bob' WHERE id=1
  │                              → crea version V2, V1 sigue visible para A
  SELECT * FROM users           │
  WHERE id = 1                  │
  → sigue viendo V1             COMMIT
  COMMIT                        → V1 puede ser garbage collected

  Lecturas NUNCA bloquean escrituras.
  Escrituras NUNCA bloquean lecturas.
  Cada transacción ve un snapshot consistente.

  PostgreSQL: cada fila tiene xmin (tx que la creó) y xmax (tx que la borró).
  Vacuum: limpia versiones viejas que ya nadie necesita.
```

### Buffer Pool / Page Cache

```
  ┌──────────────────────────────────────┐
  │          Buffer Pool (RAM)            │
  │                                      │
  │  [Page 1: users] [Page 2: orders]   │
  │  [Page 3: idx_name] [Page 4: ...]   │
  │                                      │
  │  Hit ratio ideal: >99%               │
  │  Si una página no está en pool:      │
  │    → disk read (10ms HDD, 0.1ms SSD) │
  └──────────────────────────────────────┘

  PostgreSQL: shared_buffers (default 128MB, producción: 25% RAM)
  MySQL: innodb_buffer_pool_size (producción: 70-80% RAM)
```

## Modelo mental

- **B-Tree** = índice de un libro. Buscas la palabra, te dice la página. Rápido para leer.
- **LSM Tree** = pila de notas. Escribes una nota nueva rápido (memtable). Periodicamente las organizas (compaction). Leer requiere buscar en varias pilas.
- **WAL** = escribir en el diario antes de hacer el trabajo real. Si te interrumpen, lees el diario y retomas.
- **MVCC** = fotocopias del documento. Cada lector tiene su propia copia, el escritor modifica el original sin interrumpir a los lectores.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Queries lentas progresivamente | Buffer pool demasiado pequeño (cache miss) | Aumentar shared_buffers |
| Tabla bloated, espacio no se recupera | MVCC dead tuples sin vacuum | Configurar autovacuum agresivamente |
| Write amplification en LSM | Demasiados niveles de compaction | Tuning de compaction strategy |
| Crash recovery lento | WAL demasiado grande entre checkpoints | Reducir checkpoint_timeout |

## Insights de ingeniería

- **B-Tree vs LSM**: B-Tree para read-heavy (OLTP, PostgreSQL). LSM para write-heavy (logs, metrics, Cassandra). Esta decisión está en el core de cada base de datos.
- **Buffer pool hit ratio**: monitorea `cache hit ratio` en PostgreSQL (`pg_stat_user_tables`). Si baja de 99%, necesitas más RAM o menos datos activos.
- **Vacuum es crítico en PostgreSQL**: sin vacuum, dead tuples se acumulan, la tabla crece, los queries se ralentizan. Autovacuum debe estar configurado y monitoreado. Es la causa #1 de degradación de rendimiento en PostgreSQL.
