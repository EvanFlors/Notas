### Key-Value (Redis, DynamoDB)

```
  Simplemente: key вҶ’ value

  "user:42" вҶ’ {"name": "Alice", "role": "admin"}
  "session:abc" вҶ’ {"user_id": 42, "expires": 1704067200}
  "cache:product:99" вҶ’ "{...product data...}"

  Operaciones: GET, SET, DELETE. O(1).
  Redis: in-memory, data structures (lists, sets, sorted sets, hashes).
  DynamoDB: distributed, persistent, auto-scaling.

  Fortalezas: latencia sub-ms (Redis), escala horizontal (DynamoDB).
  Debilidades: no joins, no queries complejas, datos simples.
```

### Wide-Column (Cassandra, HBase)

```
  Row key: "user:42"
  Column families: profile, activity

  profile:name = "Alice"    profile:email = "a@b.com"

  Fortalezas: escala linealmente, multi-datacenter.
  Debilidades: modelo de queries limitado, no joins.
```

## Modelo mental

- **Relacional** = hoja de cГЎlculo con mГәltiples pestaГұas enlazadas.
- **Key-Value** = diccionario: busca por palabra, obtГ©n definiciГіn. Ultra-rГЎpido.
- **Document** = carpetas con documentos: cada documento puede tener estructura diferente.
- **Graph** = red social: importan las conexiones entre nodos, no los nodos aislados.

## Insights de ingenierГӯa

- **Start with PostgreSQL**: es relacional, soporta JSON (document-like), tiene extensiones para time-series (TimescaleDB), full-text search, y geospatial. Agrega bases especializadas solo cuando el rendimiento de PostgreSQL no sea suficiente.
- **Polyglot persistence**: usa la base correcta para cada caso. PostgreSQL para transacciones, Redis para cache/sessions, Kafka para event log, Elasticsearch para full-text search.
- **Schema-on-write vs schema-on-read**: relacional valida al escribir (schema-on-write, datos siempre vГЎlidos). Document valida al leer (schema-on-read, flexibilidad pero riesgo de datos inconsistentes).
