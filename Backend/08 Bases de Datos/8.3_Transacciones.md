# 8.3 Transacciones

## Definición

Una transacción es una unidad atómica de trabajo: o todas las operaciones se ejecutan, o ninguna. ACID (Atomicity, Consistency, Isolation, Durability) son las garantías que una base de datos relacional provee. Los isolation levels definen qué tan "aislada" está una transacción de las demás.

## Mecánica interna

### ACID

```
  Atomicity:    Todo o nada. Si falla una operación, se revierte todo.
  Consistency:  La DB pasa de un estado válido a otro estado válido.
  Isolation:    Transacciones concurrentes no se interfieren.
  Durability:   Una vez committed, los datos sobreviven a crashes.

  Transferencia bancaria:
  BEGIN
    UPDATE accounts SET balance = balance - 100 WHERE id = 1;  ← restar
    UPDATE accounts SET balance = balance + 100 WHERE id = 2;  ← sumar
  COMMIT

  Si crashea entre las dos operaciones → ROLLBACK automático (atomicity).
  Después de COMMIT → datos en disco via WAL (durability).
```

### Isolation Levels

| Level | Dirty Read | Non-repeatable Read | Phantom Read | Performance |
|-------|-----------|-------------------|-------------|-------------|
| Read Uncommitted | Sí | Sí | Sí | Más rápido |
| Read Committed | No | Sí | Sí | Rápido (PostgreSQL default) |
| Repeatable Read | No | No | Sí* | Medio (MySQL default) |
| Serializable | No | No | No | Más lento |

*PostgreSQL's Repeatable Read también previene phantom reads (usa MVCC snapshots).

```
  Dirty Read: leer datos que otra transacción aún no ha committed.
  Non-repeatable Read: leer la misma fila 2 veces y obtener valores diferentes.
  Phantom Read: un query retorna filas diferentes al ejecutarse 2 veces
               (otra transacción insertó/borró filas).
```

### Locking

```
  Pessimistic locking (row-level locks):
  SELECT * FROM products WHERE id = 42 FOR UPDATE;
  → Nadie más puede UPDATE/DELETE esta fila hasta que termines.

  Optimistic locking (versioning):
  SELECT *, version FROM products WHERE id = 42;  → version = 5
  UPDATE products SET stock = stock - 1, version = 6
  WHERE id = 42 AND version = 5;
  → Si affected_rows = 0, alguien más modificó primero → retry.
```

### Deadlocks

```
  TX A: lock row 1, espera lock row 2
  TX B: lock row 2, espera lock row 1
  → Deadlock. La DB detecta y aborta una transacción.

  Prevención:
  - Siempre adquirir locks en el mismo orden
  - Timeouts cortos (lock_timeout)
  - Minimizar duración de transacciones
```

### Distributed Transactions

```
  2PC (Two-Phase Commit):
  Coordinator: "¿Todos listos?"
  DB1: "Sí"  DB2: "Sí"  → Coordinator: "COMMIT todos"
  DB1: "Sí"  DB2: "No"  → Coordinator: "ROLLBACK todos"

  Problema: si coordinator crashea después de "COMMIT todos"
  pero antes de que DB2 reciba → DB2 queda bloqueado.

  Saga pattern (alternativa):
  Step 1: Create order → success
  Step 2: Reserve inventory → success
  Step 3: Charge payment → FAIL
  → Compensate: cancel inventory, cancel order

  Cada paso tiene una acción compensatoria. No hay lock global.
```

## Modelo mental

- **Isolation levels** = niveles de privacidad en una oficina. Read Uncommitted = oficina abierta, ves todo. Serializable = oficina privada con puerta cerrada.
- **Pessimistic lock** = reservar un libro en la biblioteca. Nadie más puede tomarlo hasta que lo devuelvas.
- **Optimistic lock** = anotar el número de versión del documento. Si alguien lo modificó antes que tú, re-lees y re-intentas.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Deadlock detected | Locks en orden diferente | Ordenar acceso a recursos, reducir TX scope |
| Lost update | Read committed sin lock | FOR UPDATE o optimistic locking |
| Long-running TX blocking others | TX abierta por mucho tiempo | Timeout, minimizar scope |
| 2PC coordinator failure | Single point of failure | Saga pattern en su lugar |

## Insights de ingeniería

- **Read Committed es suficiente para la mayoría**: PostgreSQL default. Si necesitas repeatable read, evalúa si puedes resolver con optimistic locking en vez de subir el isolation level.
- **Transacciones cortas**: abre TX, haz el mínimo trabajo, commit. No hagas HTTP calls a servicios externos dentro de una transacción (si el servicio tarda, la TX retiene locks).
- **Saga > 2PC para microservicios**: 2PC requiere coordinación síncrona entre bases de datos. Sagas usan eventos asincrónicos con compensaciones. Más resiliente, más complejo de implementar correctamente.
