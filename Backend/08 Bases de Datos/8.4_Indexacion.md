# 8.4 Indexación

## Definición

Un índice es una estructura de datos que acelera queries evitando full table scans. Sin índice, PostgreSQL lee todas las filas (sequential scan). Con índice, salta directamente a las filas relevantes. Elegir los índices correctos es la optimización más impactante en bases de datos.

## Mecánica interna

### Tipos de índice

| Tipo | Estructura | Operaciones | Uso |
|------|-----------|-------------|-----|
| B-Tree | Árbol balanceado | =, <, >, <=, >=, BETWEEN, IN | Default, la mayoría de queries |
| Hash | Hash table | Solo = | Igualdad exacta (raro en práctica) |
| GIN | Inverted index | @>, ?, &&, full-text search | Arrays, JSONB, tsvector |
| GiST | Generalized search tree | Geospatial, range types | PostGIS, range queries |
| BRIN | Block range index | <, >, BETWEEN | Datos naturalmente ordenados (timestamps) |

### Tipos de index scan

```
  Sequential Scan: lee TODA la tabla. O(n).
  Index Scan: busca en el índice, sigue punteros a la tabla. O(log n + k).
  Index-Only Scan: responde solo con datos del índice, sin tocar la tabla. O(log n).
  Bitmap Index Scan: crea bitmap de filas, luego lee las páginas. Para selectividad media.
```

### Covering Index (Index-Only Scan)

```sql
  -- Query frecuente
  SELECT name, email FROM users WHERE email = 'alice@example.com';

  -- Índice normal: busca email en índice → puntero → lee fila en tabla
  CREATE INDEX idx_email ON users(email);

  -- Covering index: incluye name Y email en el índice
  CREATE INDEX idx_email_cover ON users(email) INCLUDE (name);
  -- Index-Only Scan: responde directamente del índice. Zero table access.
```

### Composite Index (orden importa)

```sql
  CREATE INDEX idx_user_status ON orders(user_id, status);

  -- Usa el índice (leftmost prefix):
  WHERE user_id = 42                    ✅
  WHERE user_id = 42 AND status = 'paid' ✅

  -- NO usa el índice (status no es leftmost):
  WHERE status = 'paid'                 ❌
```

### Partial Index

```sql
  -- Indexa solo filas que cumplen condición
  CREATE INDEX idx_active ON users(email) WHERE active = true;

  -- Más pequeño que indexar toda la tabla.
  -- Útil cuando queries filtran siempre por la misma condición.
```

### EXPLAIN ANALYZE

```sql
  EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 42;

  -- Output:
  Index Scan using idx_user_id on orders  (cost=0.29..8.31 rows=5 width=48)
    (actual time=0.015..0.020 rows=5 loops=1)
    Index Cond: (user_id = 42)
  Planning Time: 0.085 ms
  Execution Time: 0.040 ms

  Campos clave:
  - Scan type: Index Scan, Seq Scan, Bitmap Scan
  - cost: estimado por el planner
  - actual time: tiempo real en ms
  - rows: filas estimadas vs reales
  - loops: cuántas veces se ejecutó este nodo
```

## Modelo mental

Sin índice = buscar un nombre en un libro leyendo cada página. Con B-Tree = ir al índice alfabético al final del libro. Covering index = el índice tiene toda la información que necesitas, ni siquiera abres el libro.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Query lenta con índice existente | Baja selectividad (índice retorna 50% de filas) | Seq scan es más eficiente, agregar más filtros |
| Inserts lentos | Demasiados índices (cada INSERT actualiza todos) | Eliminar índices no usados |
| Index bloat | Dead tuples en el índice (vacuum no corrió) | REINDEX o vacuum + monitoring |
| Wrong index chosen | Estadísticas desactualizadas | ANALYZE table |

## Insights de ingeniería

- **No indexes todo**: cada índice consume espacio y ralentiza writes. Crea índices solo para queries que realmente ejecutas. Usa `pg_stat_user_indexes` para encontrar índices no usados.
- **EXPLAIN ANALYZE siempre**: nunca asumas que tu query usa el índice correcto. Siempre verifica con EXPLAIN ANALYZE. El planner puede elegir seq scan si estima que es más eficiente.
- **Partial indexes para queries frecuentes**: si el 95% de tus queries filtran `WHERE active = true` y solo 5% de filas son active, un partial index es 20x más pequeño y rápido.
- **BRIN para datos ordenados cronológicamente**: logs, eventos, métricas insertados cronológicamente. BRIN index ocupa ~1000x menos espacio que B-Tree con rendimiento aceptable para range queries.
