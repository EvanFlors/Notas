# 6.4 Interrupciones

## Definición

Una interrupción (interrupt) es una señal que un dispositivo hardware envía al CPU para notificar un evento que requiere atención inmediata (paquete de red recibido, operación de disco completada, timer expirado). El CPU suspende la ejecución actual, ejecuta un handler (ISR - Interrupt Service Routine), y luego retoma la ejecución interrumpida. Las interrupciones son el mecanismo fundamental de I/O asíncrono en hardware.

## Mecánica interna

```
  Flujo de una interrupción:

  1. Dispositivo genera señal de interrupción
  2. Interrupt controller (APIC) recibe y prioriza
  3. APIC envía interrupción al CPU core apropiado
  4. CPU termina la instrucción actual
  5. CPU guarda estado (registros, PC, flags) en stack
  6. CPU consulta IDT (Interrupt Descriptor Table) para encontrar el handler
  7. CPU ejecuta el ISR (Interrupt Service Routine)
  8. ISR procesa el evento (lee datos del dispositivo, actualiza buffers)
  9. ISR envía EOI (End of Interrupt) al APIC
  10. CPU restaura estado y retoma ejecución interrumpida


  Tipos:
  ┌──────────────────────────────────────────────────┐
  │ Hardware Interrupts (async, del dispositivo):    │
  │ - NIC: paquete recibido                         │
  │ - Disk: I/O completado                          │
  │ - Timer: tick del reloj del sistema             │
  │ - Keyboard: tecla presionada                    │
  │                                                 │
  │ Software Interrupts (sync, del programa):       │
  │ - INT 0x80 / SYSCALL: system call               │
  │ - INT 3: breakpoint (debugger)                  │
  │                                                 │
  │ Exceptions (sync, del CPU):                     │
  │ - Division by zero (#DE)                        │
  │ - Page fault (#PF)                              │
  │ - General protection fault (#GP)                │
  │ - Double fault (#DF)                            │
  └──────────────────────────────────────────────────┘
```

### Interrupt Controllers

```
  APIC (Advanced Programmable Interrupt Controller):

  Local APIC (uno por core):
  - Recibe interrupciones del I/O APIC y de otros cores (IPI)
  - Prioriza interrupciones locales
  - Genera timer interrupts (APIC timer)

  I/O APIC (uno o más por sistema):
  - Recibe interrupciones de dispositivos externos
  - Rutea a Local APICs según configuración
  - Puede distribuir interrupciones entre cores (load balancing)

  MSI/MSI-X (Message Signaled Interrupts):
  En vez de una línea física de interrupción, el dispositivo
  escribe un mensaje a una dirección de memoria específica.

  MSI: 1-32 vectores por dispositivo
  MSI-X: hasta 2048 vectores por dispositivo

  Ventajas de MSI-X:
  - Cada queue de un NIC/NVMe puede tener su propio vector
  - Cada vector puede dirigirse a un core diferente
  - Elimina sharing de interrupciones (menos overhead)

  NIC con 8 queues + MSI-X:
  Queue 0 → IRQ → Core 0
  Queue 1 → IRQ → Core 1
  ...
  Queue 7 → IRQ → Core 7
  → Procesamiento de paquetes distribuido entre 8 cores
```

### Interrupt Coalescing

```
  Problema: un NIC de 100Gbps puede generar ~8 millones de paquetes/segundo.
  Si cada paquete genera una interrupción:
  8M interrupts/s × ~2μs por interrupt = 16 segundos de CPU por segundo.
  → Imposible, el CPU estaría 100% en interrupt handling.

  Solución: interrupt coalescing (agrupación):
  En vez de interrumpir por cada paquete, el NIC espera:
  - N paquetes acumulados (e.g., 64 paquetes), O
  - T microsegundos transcurridos (e.g., 100μs)
  Lo que ocurra primero → genera UNA interrupción.

  El ISR procesa los 64 paquetes de una vez (NAPI polling en Linux).

  Trade-off:
  Más coalescing → menos interrupciones → menos overhead de CPU
  Más coalescing → mayor latencia (el paquete espera en el buffer)

  ethtool -C eth0 rx-usecs 50 rx-frames 64
  → Interrumpir después de 50μs O 64 frames


  NAPI (New API) en Linux:
  Modo híbrido interrupt + polling:
  1. Primer paquete → interrupción → deshabilitar interrupciones
  2. Kernel entra en modo polling: lee paquetes directamente del ring buffer
  3. Cuando no hay más paquetes → rehabilitar interrupciones

  Esto evita el overhead de interrupciones bajo carga alta
  (el kernel hace polling activo) mientras mantiene baja latencia
  bajo carga baja (interrupciones para notificación inmediata).
```

### Afinidad de interrupciones

```
  /proc/interrupts muestra qué core procesa cada interrupción:

  CPU0  CPU1  CPU2  CPU3
  IRQ 24: 1000    0     0     0  eth0-rx-0
  IRQ 25:    0 1000     0     0  eth0-rx-1
  IRQ 26:    0    0  1000     0  eth0-rx-2
  IRQ 27:    0    0     0  1000  eth0-rx-3

  Configurar afinidad:
  echo 1 > /proc/irq/24/smp_affinity   ← Core 0
  echo 2 > /proc/irq/25/smp_affinity   ← Core 1

  irqbalance: daemon que distribuye interrupciones automáticamente.
  Para workloads de alto rendimiento: desactivar irqbalance
  y fijar manualmente IRQs a cores específicos.

  Regla: el core que procesa la interrupción debería ser el mismo
  que procesa los datos. Esto maximiza cache locality
  (los datos del paquete están en L1/L2 del core que los recibió).
```

## Modelo mental

Las interrupciones son el timbre de la puerta del CPU. Cuando un dispositivo tiene algo que entregar (paquete de red, bloque de disco), toca el timbre. El CPU deja lo que está haciendo, abre la puerta, recibe el paquete, y vuelve a lo suyo. Interrupt coalescing es como el buzón: en vez de tocar el timbre por cada carta, el cartero acumula varias y toca una vez. NAPI es el modo "voy yo mismo a buscar al buzón periódicamente cuando sé que hay mucho correo".

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| CPU saturado con softirqs | Demasiadas interrupciones (alta tasa de paquetes) | Interrupt coalescing, NAPI, RSS |
| Latencia alta en un core | Todas las IRQs van al mismo core | Distribuir con smp_affinity o RSS |
| Interrupt storm | Dispositivo defectuoso genera IRQs continuamente | Deshabilitar IRQ del dispositivo |
| Packet drops bajo carga | IRQ handler no puede procesar a tiempo | Aumentar ring buffer, RSS, XDP |

## Insights de ingeniería

- **RSS (Receive Side Scaling)** distribuye paquetes entre múltiples colas de la NIC basándose en un hash del flow (src IP + dst IP + src port + dst port). Cada cola tiene su propia interrupción dirigida a un core diferente. Esto escala el procesamiento de red linealmente con el número de cores.
- **La latencia de interrupción determina la latencia mínima de I/O**: desde que un paquete llega a la NIC hasta que la aplicación lo ve, pasa: NIC DMA (~1μs) → interrupción (~1-2μs) → ISR + NAPI (~1-2μs) → kernel protocol stack (~2-5μs) → wakeup de la aplicación (~1-2μs). Total: ~5-10μs. Para latencia sub-microsegundo, se necesita kernel bypass (DPDK, tema 10.5).
- **Los context switches de interrupciones tienen costo**: cada interrupción guarda y restaura registros, contamina la I-cache y D-cache del core (el ISR desplaza datos de la aplicación), y puede causar branch mispredictions al volver. Por esto, minimizar interrupciones (coalescing, polling) mejora el rendimiento de la aplicación, no solo el overhead de las interrupciones mismas.
