# 2.1 Modelos Arquitectónicos

## Definición

Un modelo arquitectónico define la organización fundamental de una computadora: cómo se conectan la CPU, la memoria, y los dispositivos de I/O, y cómo fluyen las instrucciones y los datos entre ellos. Los dos modelos dominantes — Von Neumann y Harvard — establecen contratos fundamentales que determinan las capacidades y limitaciones de todo sistema computacional.

## Mecánica interna

### Arquitectura Von Neumann

```
  Propuesta por John von Neumann (1945).
  Idea central: instrucciones y datos comparten la MISMA memoria.

  ┌──────────────────────────────────────────┐
  │                  CPU                      │
  │  ┌──────┐  ┌──────────┐  ┌──────────┐   │
  │  │ ALU  │  │ Unidad   │  │Registros │   │
  │  │      │  │ Control  │  │          │   │
  │  └──────┘  └──────────┘  └──────────┘   │
  └──────────────────┬───────────────────────┘
                     │ Bus único
                     │ (datos + instrucciones)
  ┌──────────────────┴───────────────────────┐
  │              Memoria                      │
  │  ┌────────────────────────────────────┐  │
  │  │ Instrucciones │ Datos              │  │
  │  │ (programa)    │ (variables, heap)  │  │
  │  └────────────────────────────────────┘  │
  └──────────────────────────────────────────┘

  Concepto de programa almacenado:
  Las instrucciones del programa se almacenan en la misma memoria
  que los datos. Esto permite:
  - Modificar el programa en runtime (self-modifying code)
  - Cargar diferentes programas sin cambiar hardware
  - Tratar código como datos (compiladores, interpreters, JIT)

  Von Neumann Bottleneck:
  Como instrucciones y datos comparten el MISMO bus de memoria,
  no pueden transferirse simultáneamente.

  Ciclo: fetch instrucción → decode → fetch datos → execute → write back
                 ↑                         ↑
          usa el bus                 usa el bus (conflicto)

  El bus es el cuello de botella: la CPU es más rápida que
  la memoria puede alimentarla. Este bottleneck es LA razón
  por la que existen las caches.
```

### Arquitectura Harvard

```
  Idea central: memorias SEPARADAS para instrucciones y datos,
  con buses independientes.

  ┌──────────────────────────────────────────┐
  │                  CPU                      │
  │  ┌──────┐  ┌──────────┐  ┌──────────┐   │
  │  │ ALU  │  │ Unidad   │  │Registros │   │
  │  │      │  │ Control  │  │          │   │
  │  └──────┘  └──────────┘  └──────────┘   │
  └───────┬──────────────────────┬───────────┘
          │ Bus de instrucciones │ Bus de datos
  ┌───────┴──────────┐  ┌───────┴──────────┐
  │ Memoria de       │  │ Memoria de       │
  │ instrucciones    │  │ datos            │
  └──────────────────┘  └──────────────────┘

  Ventaja: puede fetch instrucción y datos SIMULTÁNEAMENTE.
  No hay Von Neumann bottleneck.

  Desventaja: más complejo, más caro (dos buses, dos memorias).
  No puede tratar código como datos fácilmente.

  Usado en: microcontroladores (PIC, AVR), DSPs (procesadores de señal digital).
  El Harvard puro es raro en procesadores de propósito general.
```

### Modified Harvard (lo que usan los procesadores modernos)

```
  Los procesadores modernos (x86, ARM, RISC-V) usan un HÍBRIDO:

  - Externamente: Von Neumann (una memoria principal unificada)
  - Internamente: Harvard (caches separadas para instrucciones y datos)

  ┌─────────────────────────────────────────────────┐
  │                     CPU                          │
  │  ┌─────────────┐         ┌─────────────┐        │
  │  │ L1 I-Cache  │         │ L1 D-Cache  │        │
  │  │(instrucciones)│        │ (datos)     │        │
  │  └──────┬──────┘         └──────┬──────┘        │
  │         └───────────┬───────────┘                │
  │              ┌──────┴──────┐                     │
  │              │  L2 Cache   │ ← unificada         │
  │              │ (inst+datos)│                     │
  │              └──────┬──────┘                     │
  └─────────────────────┼───────────────────────────┘
                        │
                 ┌──────┴──────┐
                 │  L3 Cache   │ ← compartida entre cores
                 └──────┬──────┘
                        │
                 ┌──────┴──────┐
                 │    RAM      │ ← Von Neumann (unificada)
                 └─────────────┘

  Esto da lo mejor de ambos mundos:
  - L1 separada: la CPU puede fetch instrucción y dato en paralelo
  - Memoria unificada: programas pueden tratar código como datos
    (necesario para JIT compilation, dynamic linking, etc.)

  El "Modified Harvard" es tan universal que cuando decimos
  "arquitectura Von Neumann" en el contexto de PCs modernos,
  en realidad es un Modified Harvard.
```

### Stored Program Concept

```
  La idea más importante de Von Neumann:

  El programa es DATOS en memoria, no hardware fijo.

  Antes de Von Neumann:
  - ENIAC: programado reconectando cables físicos
  - Cambiar el programa = días de recableado

  Después de Von Neumann:
  - El programa se almacena en memoria como cualquier dato
  - Cambiar el programa = cargar nuevos bytes en memoria
  - Un programa puede generar y ejecutar otro programa (compilador)
  - Un programa puede modificarse a sí mismo (JIT, patching)

  Consecuencias profundas:
  1. Operating Systems: un programa que carga y gestiona otros programas
  2. Compiladores: un programa que transforma texto en instrucciones ejecutables
  3. Interpreters/VMs: un programa que simula otra máquina
  4. JIT compilation: un programa que observa su propia ejecución y
     genera código optimizado en runtime
  5. Buffer overflow attacks: si código y datos comparten memoria,
     un atacante puede inyectar código disfrazado de datos
```

## Modelo mental

Von Neumann dice: "todo es memoria, el programa y los datos viven juntos, y un solo bus los conecta con la CPU". Harvard dice: "separemos los caminos para instrucciones y datos para no competir por el bus". Los procesadores modernos usan ambos: memoria unificada (Von Neumann) para flexibilidad, y caches separadas (Harvard) para rendimiento. El Von Neumann bottleneck — la CPU es más rápida que la memoria — es el problema fundamental que la jerarquía de memoria (caches L1/L2/L3, prefetching, etc.) intenta resolver.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| CPU stall esperando datos | Von Neumann bottleneck (bus saturado) | Cache más grande, prefetching |
| I-cache y D-cache thrashing | Working set excede cache | Optimizar localidad de datos/código |
| JIT code execution falla | Protecciones W^X (write XOR execute) | mprotect() para cambiar permisos de página |
| Self-modifying code inconsistente | I-cache no ve cambios en D-cache | Cache flush/invalidation explícita |

## Insights de ingeniería

- **El Von Neumann bottleneck es EL problema central del rendimiento**: la CPU puede procesar datos 100x más rápido de lo que la RAM puede entregarlos. Toda la jerarquía de memoria (cache L1 a 1ns, L2 a 4ns, L3 a 10ns, RAM a 100ns) existe para mitigar este bottleneck. Cuando optimizas rendimiento, la primera pregunta es: "¿mis datos caben en cache?"
- **La separación I-cache/D-cache es invisible pero crucial**: cuando un JIT compiler (V8 para JavaScript, HotSpot para Java) genera código en runtime, escribe en memoria de datos (D-cache) y luego ejecuta desde I-cache. Si la I-cache no se invalida, ejecuta código stale. Los ISAs modernos tienen instrucciones específicas para esto (ICACHE invalidate en ARM).
- **El stored program concept habilita toda la ingeniería de software moderna**: compiladores, linkers, loaders, VMs, containers, dynamic libraries — todos dependen de que el programa sea datos manipulables. Sin esto, cada programa requeriría hardware dedicado.
