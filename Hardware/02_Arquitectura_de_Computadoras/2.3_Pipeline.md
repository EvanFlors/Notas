# 2.3 Pipeline

## Definición

El pipeline es una técnica de diseño de CPU que permite ejecutar múltiples instrucciones simultáneamente, cada una en una etapa diferente del ciclo de instrucción. En lugar de esperar a que una instrucción complete las 5 etapas antes de empezar la siguiente, el pipeline alimenta una nueva instrucción por ciclo. El throughput ideal es 1 instrucción por ciclo (IPC=1), pero los hazards (dependencias de datos, saltos, y conflictos de recursos) reducen la eficiencia real.

## Mecánica interna

### Pipeline básico de 5 etapas

```
  Sin pipeline (ejecución secuencial):

  Instr 1: [IF][ID][EX][ME][WB]
  Instr 2:                      [IF][ID][EX][ME][WB]
  Instr 3:                                           [IF][ID][EX][ME][WB]

  3 instrucciones = 15 ciclos


  Con pipeline:

  Ciclo:    1    2    3    4    5    6    7
  Instr 1: [IF] [ID] [EX] [ME] [WB]
  Instr 2:      [IF] [ID] [EX] [ME] [WB]
  Instr 3:           [IF] [ID] [EX] [ME] [WB]

  3 instrucciones = 7 ciclos (vs 15 sin pipeline)

  En régimen estable, una instrucción se completa CADA CICLO.
  Throughput ideal: 1 instrucción/ciclo (IPC = 1).
  Latencia por instrucción: sigue siendo 5 ciclos.
  El pipeline mejora throughput, NO latencia individual.


  Profundidad del pipeline:
  Más etapas = más instrucciones en vuelo = mayor throughput teórico.
  Pero más etapas = mayor penalización por misprediction.

  ┌────────────────────────┬──────────┬──────────────────┐
  │ Procesador             │ Etapas   │ Penalty por flush │
  ├────────────────────────┼──────────┼──────────────────┤
  │ MIPS clásico           │ 5        │ 5 ciclos         │
  │ ARM Cortex-A53         │ 8        │ 8 ciclos         │
  │ Intel Skylake          │ 14-19    │ ~15-20 ciclos    │
  │ Intel Pentium 4 (2004) │ 31       │ 31 ciclos        │
  └────────────────────────┴──────────┴──────────────────┘

  Pentium 4 fue un ejemplo extremo: pipeline de 31 etapas para
  alcanzar frecuencias altas (3.8GHz), pero las penalizaciones
  por branch misprediction eran devastadoras. Intel abandonó
  este enfoque con la arquitectura Core (14 etapas).
```

### Data Hazards

```
  Ocurren cuando una instrucción depende del resultado de otra
  que aún no ha completado el pipeline.

  RAW (Read After Write) — el más común:
  ADD X1, X2, X3    ← escribe X1 en etapa WB (ciclo 5)
  SUB X4, X1, X5    ← lee X1 en etapa ID (ciclo 3) ← X1 aún no actualizado!

  Ciclo:    1    2    3    4    5
  ADD:     [IF] [ID] [EX] [ME] [WB] ← X1 se escribe aquí
  SUB:          [IF] [ID]            ← SUB lee X1 aquí (valor viejo!)

  Soluciones:

  1. Stall (burbuja):
  Insertar ciclos vacíos hasta que el dato esté disponible.
  ADD:     [IF] [ID] [EX] [ME] [WB]
  bubble:       [  ] [  ]
  SUB:               [IF] [ID] [EX] [ME] [WB]
  Penalización: 2 ciclos perdidos. Simple pero lento.

  2. Forwarding (bypass):
  El resultado de la ALU se "adelanta" directamente al input
  de la siguiente instrucción, sin esperar a WB.

  ADD:     [IF] [ID] [EX] [ME] [WB]
  SUB:          [IF] [ID] [EX] [ME] [WB]
                      ↑
                      El resultado de ADD se forward desde
                      la salida de EX al input de EX del SUB

  Sin stalls. Requiere hardware adicional (bypass paths).
  Los procesadores modernos implementan forwarding extensivo.

  3. Reordenamiento por compilador:
  El compilador reordena instrucciones para llenar los "huecos"
  con instrucciones independientes.

  Antes:                  Después:
  ADD X1, X2, X3          ADD X1, X2, X3
  SUB X4, X1, X5          MUL X7, X8, X9    ← independiente, llena el hueco
  MUL X7, X8, X9          SUB X4, X1, X5    ← ahora X1 ya está disponible


  WAW (Write After Write):
  ADD X1, X2, X3    ← escribe X1
  SUB X1, X4, X5    ← escribe X1 (¿cuál llega primero?)
  Solucionado con register renaming en OoO CPUs.

  WAR (Write After Read):
  ADD X1, X2, X3    ← lee X2
  SUB X2, X4, X5    ← escribe X2 (¿ya leyó ADD el valor original?)
  También solucionado con register renaming.
```

### Control Hazards

```
  Ocurren con instrucciones de branch (salto condicional).
  El problema: no sabemos a dónde va el flujo hasta que
  la condición se evalúa en la etapa EX.

  CMP X1, X2         ← compara
  BEQ target         ← salta si son iguales
  ADD X3, X4, X5     ← ¿se ejecuta esta o no?
  SUB X6, X7, X8     ← ¿y esta?
  target:
  MUL X9, X10, X11

  Sin pipeline: no hay problema, cada instrucción espera.
  Con pipeline: cuando BEQ está en EX, ADD y SUB ya están
  en IF e ID. Si el branch se toma, esas instrucciones
  deben descartarse (pipeline flush).

  Penalización = profundidad del pipeline entre IF y resolución del branch.
  En un pipeline de 5 etapas: ~2 ciclos.
  En un pipeline de 14 etapas (Intel Skylake): ~15 ciclos.

  Soluciones:

  1. Branch Prediction (tema 3.3):
  Predecir si el branch se toma o no, y empezar a ejecutar
  especulativamente. Si la predicción es correcta: 0 penalización.
  Si es incorrecta: flush del pipeline (~15 ciclos perdidos).

  Tasa de acierto típica: 95-97% en CPUs modernos.
  El 3-5% de mispredictions tiene impacto significativo en código
  con muchos branches (parsers, interpreters, decision trees).

  2. Branch delay slot (MIPS):
  La instrucción DESPUÉS del branch siempre se ejecuta,
  independientemente del resultado del branch.
  El compilador coloca una instrucción útil en ese slot.
  Técnica arcaica, no usada en procesadores modernos.

  3. Predicated execution (ARM):
  Instrucciones condicionales que se ejecutan o no basándose en flags,
  sin necesidad de branch.
  CSEL X1, X2, X3, EQ  ← X1 = X2 si EQ, sino X1 = X3
  Elimina el branch por completo. Útil para if-else simples.
  x86 tiene CMOV (conditional move) con el mismo propósito.
```

### Structural Hazards

```
  Ocurren cuando dos instrucciones necesitan el MISMO recurso
  hardware en el mismo ciclo.

  Ejemplo: una sola memoria (sin caches separadas)
  Instr 1 en etapa MEM: accediendo a memoria para datos
  Instr 4 en etapa IF:  accediendo a memoria para instrucción
  → Conflicto: ambas quieren usar el bus de memoria

  Solución: caches separadas (L1 I-cache y L1 D-cache)
  → Harvard modificado elimina este structural hazard.

  Ejemplo: una sola ALU
  Si dos instrucciones necesitan la ALU en el mismo ciclo,
  una debe esperar.
  Solución: múltiples unidades de ejecución (2+ ALUs, FPU, etc.)
  Los CPUs modernos tienen 4-8 unidades de ejecución en paralelo.
```

### Superscalar: más allá de IPC=1

```
  Un pipeline escalar emite 1 instrucción por ciclo (IPC max = 1).
  Un pipeline superscalar emite MÚLTIPLES instrucciones por ciclo.

  Superscalar de 4 vías (Intel Skylake):

  Ciclo:    1    2    3    4    5    6
  Instr 1: [IF] [ID] [EX] [ME] [WB]
  Instr 2: [IF] [ID] [EX] [ME] [WB]
  Instr 3: [IF] [ID] [EX] [ME] [WB]
  Instr 4: [IF] [ID] [EX] [ME] [WB]
  Instr 5:      [IF] [ID] [EX] [ME] [WB]
  ...

  IPC teórico máximo: 4 instrucciones/ciclo.
  IPC real en workloads típicos: 2-3 (limitado por dependencias y hazards).

  Requiere:
  - Múltiples unidades de ejecución (ALU, FPU, Load, Store, Branch)
  - Register renaming (para eliminar WAW/WAR hazards)
  - Out-of-order execution (para encontrar instrucciones independientes)
  - Estos temas se cubren en detalle en el módulo 3.
```

## Modelo mental

El pipeline es una cadena de montaje: cada estación (etapa) trabaja en una pieza diferente (instrucción) simultáneamente. La velocidad de la cadena se mide en piezas completadas por unidad de tiempo (throughput = IPC), no en cuánto tarda una pieza individual (latencia). Los hazards son interrupciones en la cadena: un data hazard es como una estación que necesita una pieza que aún no está lista; un control hazard es como una bifurcación donde no sabes qué camino tomar; un structural hazard es como dos estaciones peleando por la misma herramienta. Toda la ingeniería de CPUs modernos (forwarding, branch prediction, out-of-order, superscalar) existe para mantener la cadena alimentada y productiva.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| IPC real << IPC teórico | Data/control hazards frecuentes | Mejor branch prediction, forwarding |
| Branch misprediction alta en loops | Código con patterns impredecibles | Branchless code (CMOV, predication) |
| Pipeline flush frecuente | Exceptions, interrupts, mispredictions | Reducir branches, unroll loops |
| Front-end bottleneck | I-cache misses, decode stalls | Código más compacto, DSB (micro-op cache) |

## Insights de ingeniería

- **Pipeline depth es un trade-off**: más etapas permiten mayor frecuencia (cada etapa hace menos trabajo → ciclo de reloj más corto), pero aumentan la penalización por misprediction. Intel Pentium 4 (31 etapas, 3.8GHz) perdió contra Athlon 64 (12 etapas, 2.4GHz) porque las mispredictions destruían su throughput. El sweet spot moderno está en 14-19 etapas.
- **El compilador es tu primer optimizador de pipeline**: `-O2` y `-O3` en GCC/Clang reordenan instrucciones para evitar data hazards, desenrollan loops para reducir control hazards, y usan instrucciones sin branch (CMOV) cuando es beneficioso. Escribir código "pipeline-friendly" a mano es innecesario en la mayoría de casos; el compilador lo hace mejor.
- **IPC × frecuencia = rendimiento**: un CPU con IPC=4 a 3GHz ejecuta 12 billones de instrucciones/segundo, igual que uno con IPC=3 a 4GHz. Apple Silicon (M1/M2/M3) logra IPC muy alto (~6-8 en su pipeline de 8 vías) a frecuencias moderadas (~3.5GHz), compitiendo con x86 a ~5GHz pero menor IPC. El rendimiento real es el producto, no un factor individual.
