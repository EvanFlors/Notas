# 2.2 Ciclo de Instrucción

## Definición

El ciclo de instrucción es la secuencia de pasos que la CPU repite para ejecutar cada instrucción del programa. En su forma básica consta de cinco etapas: Fetch (buscar la instrucción en memoria), Decode (interpretar qué operación es), Execute (ejecutar la operación en la ALU), Memory Access (leer/escribir datos en memoria si es necesario), y Write-back (almacenar el resultado en un registro). Este ciclo se repite billones de veces por segundo.

## Mecánica interna

### Las cinco etapas

```
  1. FETCH (IF - Instruction Fetch)
  ┌───────────────────────────────────────────────────┐
  │ - PC (Program Counter) contiene la dirección de    │
  │   la próxima instrucción                           │
  │ - La CPU envía PC al bus de direcciones            │
  │ - La memoria devuelve la instrucción               │
  │ - La instrucción se almacena en IR (Instruction Register)│
  │ - PC se incrementa: PC = PC + tamaño_instrucción  │
  │   (4 bytes en ARM/RISC-V, variable en x86)        │
  └───────────────────────────────────────────────────┘

  2. DECODE (ID - Instruction Decode)
  ┌───────────────────────────────────────────────────┐
  │ - La unidad de control interpreta los bits del IR  │
  │ - Identifica: opcode, registros fuente, registro   │
  │   destino, valor inmediato                         │
  │ - Genera señales de control para la ALU, MUXes,    │
  │   memoria, y register file                         │
  │                                                    │
  │ Ejemplo (ARM64):                                   │
  │ ADD X1, X2, X3                                     │
  │ Opcode: ADD (suma)                                 │
  │ Fuente: X2, X3                                     │
  │ Destino: X1                                        │
  │ Señales: ALU=suma, leer X2 y X3, escribir en X1   │
  │                                                    │
  │ En x86 (CISC): decode es más complejo porque las   │
  │ instrucciones tienen longitud variable (1-15 bytes).│
  │ Se descomponen en micro-ops (μops) internamente.   │
  └───────────────────────────────────────────────────┘

  3. EXECUTE (EX - Execute)
  ┌───────────────────────────────────────────────────┐
  │ - La ALU realiza la operación indicada:            │
  │   ADD: suma los operandos                          │
  │   SUB: resta                                       │
  │   AND/OR/XOR: operación lógica                     │
  │   CMP: compara (resta sin almacenar resultado,     │
  │         solo actualiza flags)                      │
  │   SHIFT: desplazamiento de bits                    │
  │                                                    │
  │ - Para instrucciones de branch:                    │
  │   Evalúa la condición y calcula la dirección       │
  │   de salto (PC + offset)                           │
  │                                                    │
  │ - Para instrucciones de memoria (LOAD/STORE):      │
  │   Calcula la dirección efectiva:                   │
  │   addr = base_register + offset                    │
  └───────────────────────────────────────────────────┘

  4. MEMORY ACCESS (MEM)
  ┌───────────────────────────────────────────────────┐
  │ - Solo para instrucciones LOAD y STORE             │
  │                                                    │
  │ LOAD (leer de memoria):                            │
  │   Envía dirección al bus → recibe dato de memoria  │
  │   LDR X1, [X2]  →  X1 = memoria[X2]              │
  │                                                    │
  │ STORE (escribir en memoria):                       │
  │   Envía dirección + dato al bus → memoria almacena │
  │   STR X1, [X2]  →  memoria[X2] = X1              │
  │                                                    │
  │ Para instrucciones que no acceden a memoria         │
  │ (ADD, SUB, etc.), esta etapa es un no-op.          │
  │                                                    │
  │ En la práctica, el acceso va primero a L1 D-cache: │
  │   Cache hit: ~1-4ns                                │
  │   Cache miss → L2: ~4-10ns                         │
  │   L2 miss → L3: ~10-40ns                           │
  │   L3 miss → RAM: ~50-100ns                         │
  │   Un cache miss puede stall la CPU por 100+ ciclos │
  └───────────────────────────────────────────────────┘

  5. WRITE-BACK (WB)
  ┌───────────────────────────────────────────────────┐
  │ - El resultado se escribe en el registro destino   │
  │                                                    │
  │ ADD X1, X2, X3:  WB escribe el resultado en X1    │
  │ LDR X1, [X2]:    WB escribe el dato leído en X1   │
  │                                                    │
  │ Para STORE: no hay write-back (el dato ya se       │
  │ escribió en memoria en la etapa MEM)               │
  └───────────────────────────────────────────────────┘
```

### Flujo completo de una instrucción

```
  ADD X1, X2, X3   (X1 = X2 + X3)

  Ciclo 1 (IF):   PC → bus → memoria devuelve "ADD X1,X2,X3" → IR
                   PC += 4
  Ciclo 2 (ID):   Decodifica opcode=ADD, src=X2,X3, dst=X1
                   Lee X2=5, X3=3 del register file
  Ciclo 3 (EX):   ALU calcula 5 + 3 = 8
  Ciclo 4 (MEM):  No-op (ADD no accede a memoria)
  Ciclo 5 (WB):   Escribe 8 en X1

  Sin pipeline: 5 ciclos por instrucción.
  A 4GHz = 4×10⁹ ciclos/s → 800 millones de instrucciones/segundo.
  Con pipeline (tema 2.3): hasta 1 instrucción por ciclo
  → 4 billones de instrucciones/segundo (5x mejora).
```

### CISC vs RISC

```
  CISC (Complex Instruction Set Computer) — x86:
  - Instrucciones de longitud variable (1-15 bytes)
  - Una instrucción puede hacer LOAD + operación + STORE
    Ejemplo: ADD [mem], reg  (lee de memoria, suma, escribe en memoria)
  - Decode complejo: se descompone en micro-ops (μops) internamente
  - Más código denso (menos instrucciones por programa)

  RISC (Reduced Instruction Set Computer) — ARM, RISC-V:
  - Instrucciones de longitud fija (4 bytes)
  - Load/Store architecture: SOLO LOAD y STORE acceden a memoria
    Las operaciones aritméticas SOLO operan sobre registros
    Ejemplo: LDR X1,[mem]; ADD X1,X1,X2; STR X1,[mem] (3 instrucciones)
  - Decode simple: longitud fija facilita pipelining
  - Más instrucciones por programa pero cada una es más rápida

  En la práctica moderna:
  x86 CPUs (Intel, AMD) son externamente CISC pero internamente RISC:
  el front-end decodifica instrucciones CISC en micro-ops RISC,
  y el back-end ejecuta esas micro-ops en un pipeline RISC.
```

## Modelo mental

La CPU es una línea de montaje que procesa instrucciones. Cada instrucción pasa por cinco estaciones: se busca en memoria (fetch), se interpreta (decode), se ejecuta (execute), se accede a datos si es necesario (memory), y se guarda el resultado (write-back). Sin pipeline, cada instrucción espera a que la anterior termine todas las etapas. Con pipeline, mientras una instrucción está en execute, la siguiente ya está en decode y la tercera en fetch — como una fábrica donde múltiples productos están en diferentes estaciones simultáneamente.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| CPI >> 1 (muchos ciclos por instrucción) | Cache misses en etapa MEM | Mejorar localidad de datos |
| Decode bottleneck en x86 | Instrucciones complejas de longitud variable | Micro-op cache (DSB en Intel) |
| Branch misprediction penalty | Pipeline se vacía al predecir mal | Branch prediction mejorada (tema 3.3) |
| Instruction fetch stall | I-cache miss | Código más compacto, mejor I-cache locality |

## Insights de ingeniería

- **La etapa MEM es la más variable en latencia**: fetch y decode son relativamente predecibles. Execute es rápido (1 ciclo para ADD). Pero memory access puede tardar de 1 ciclo (L1 hit) a 300+ ciclos (RAM miss). Esta variabilidad es por lo que la optimización de acceso a memoria domina la performance engineering.
- **x86 micro-ops son el secreto de la compatibilidad**: Intel y AMD mantienen compatibilidad con instrucciones x86 de los años 80, pero internamente las descomponen en micro-ops RISC que ejecutan en pipelines modernos. El micro-op cache (Decoded Stream Buffer) cachea micro-ops ya decodificadas para evitar el costo de decode repetido en loops calientes.
- **El ciclo de instrucción es el latido del software**: cada línea de tu código se compila a instrucciones de máquina, y cada instrucción pasa por este ciclo. Un `for` loop de un millón de iteraciones es un millón de veces fetch-decode-execute-mem-writeback. Entender este ciclo es entender qué realmente ejecuta la CPU cuando corre tu programa.
