# 2.4 Paralelismo

## Definición

El paralelismo es la ejecución simultánea de múltiples operaciones para aumentar el throughput. Existen tres niveles fundamentales: Instruction-Level Parallelism (ILP) — ejecutar múltiples instrucciones de un mismo hilo en paralelo; Data-Level Parallelism (DLP/SIMD) — aplicar la misma operación a múltiples datos simultáneamente; y Thread-Level Parallelism (TLP) — ejecutar múltiples hilos de ejecución independientes en paralelo.

## Mecánica interna

### Instruction-Level Parallelism (ILP)

```
  ILP explota el hecho de que muchas instrucciones consecutivas
  NO dependen entre sí y pueden ejecutarse en paralelo.

  Código:
  a = b + c     ← instrucción 1
  d = e * f     ← instrucción 2 (independiente de 1)
  g = a + d     ← instrucción 3 (depende de 1 Y 2)

  Grafo de dependencias:
  [1: a=b+c] ──────┐
                    ├──▶ [3: g=a+d]
  [2: d=e*f] ──────┘

  1 y 2 pueden ejecutarse EN PARALELO.
  3 debe esperar a que ambas terminen.


  Mecanismos hardware que explotan ILP:

  1. Superscalar execution:
     Múltiples instrucciones emitidas por ciclo (4-8 en CPUs modernos).
     El hardware detecta instrucciones independientes y las despacha
     a diferentes unidades de ejecución simultáneamente.

  2. Out-of-order execution (OoO):
     La CPU reordena instrucciones dinámicamente para encontrar
     instrucciones independientes que puedan ejecutarse mientras
     otras esperan datos.

     Programa original:        Ejecución OoO:
     LOAD X1, [mem]  (100ns)   LOAD X1, [mem]     ← empieza
     ADD X2, X1, X3  (espera)  MUL X5, X6, X7     ← independiente, ejecuta
     MUL X5, X6, X7            OR  X8, X9, X10    ← independiente, ejecuta
     OR  X8, X9, X10           ADD X2, X1, X3     ← X1 listo, ahora ejecuta

     Sin OoO: la CPU se detiene en ADD esperando el LOAD.
     Con OoO: la CPU ejecuta MUL y OR mientras espera.

  3. Speculative execution:
     La CPU predice el resultado de branches y ejecuta
     instrucciones especulativamente. Si la predicción es correcta,
     el trabajo ya está hecho. Si es incorrecta, se descarta.

  Límite de ILP (wall of ILP):
  En la práctica, el ILP explotable está limitado por dependencias
  de datos y control. Los CPUs modernos extraen IPC de ~3-6 con
  ventanas de instrucciones de 200-400 entries. Más ILP requiere
  ventanas exponencialmente más grandes con rendimientos decrecientes.
```

### Data-Level Parallelism (SIMD)

```
  SIMD: Single Instruction, Multiple Data.
  Una instrucción opera sobre MÚLTIPLES datos en paralelo.

  Escalar (sin SIMD):
  a[0] = b[0] + c[0]    ← 1 ciclo
  a[1] = b[1] + c[1]    ← 1 ciclo
  a[2] = b[2] + c[2]    ← 1 ciclo
  a[3] = b[3] + c[3]    ← 1 ciclo
  Total: 4 ciclos

  SIMD (vectorizado):
  a[0:3] = b[0:3] + c[0:3]    ← 1 ciclo (4 sumas en paralelo)
  Total: 1 ciclo (4× speedup)


  Registro SIMD:
  ┌────────┬────────┬────────┬────────┐
  │ dato 0 │ dato 1 │ dato 2 │ dato 3 │  ← 128 bits (4 × 32-bit float)
  └────────┴────────┴────────┴────────┘

  Extensiones SIMD en x86:
  ┌─────────────┬────────┬──────────────────┐
  │ Extensión   │ Ancho  │ Datos por op     │
  ├─────────────┼────────┼──────────────────┤
  │ SSE         │ 128bit │ 4× float32       │
  │ AVX         │ 256bit │ 8× float32       │
  │ AVX-512     │ 512bit │ 16× float32      │
  └─────────────┴────────┴──────────────────┘

  ARM NEON: 128 bits (4× float32)
  ARM SVE: longitud variable (128-2048 bits, configurable por implementación)


  Auto-vectorización:
  El compilador puede convertir loops escalares en instrucciones SIMD
  automáticamente (con -O2 o -O3):

  // El compilador puede vectorizar esto:
  for (int i = 0; i < N; i++)
      a[i] = b[i] + c[i];

  Condiciones para auto-vectorización:
  - Sin dependencias entre iteraciones
  - Datos contiguos en memoria (arrays, no linked lists)
  - Loop trip count conocido o estimable
  - Sin branches complejos dentro del loop


  Aplicaciones de SIMD:
  - Procesamiento de imagen/video (cada pixel = operación independiente)
  - Machine learning (multiplicación de matrices, dot products)
  - Compresión/descompresión
  - Criptografía (AES-NI: AES implementado en SIMD)
  - Base de datos: filtrado de columnas, hash joins
```

### Thread-Level Parallelism (TLP)

```
  TLP ejecuta múltiples hilos de ejecución en paralelo,
  cada uno con su propio program counter, stack, y registros.

  Niveles de TLP:

  1. Multicore:
  Múltiples cores físicos en un solo chip, cada uno ejecutando
  un hilo independiente.

  ┌────────────────────────────────────────────┐
  │                 CPU Package                 │
  │  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  │
  │  │Core 0│  │Core 1│  │Core 2│  │Core 3│  │
  │  │Thread │  │Thread │  │Thread │  │Thread │  │
  │  │  A    │  │  B    │  │  C    │  │  D    │  │
  │  │L1 I/D│  │L1 I/D│  │L1 I/D│  │L1 I/D│  │
  │  │L2    │  │L2    │  │L2    │  │L2    │  │
  │  └──────┘  └──────┘  └──────┘  └──────┘  │
  │           ┌──────────────────────┐         │
  │           │    L3 Cache (shared) │         │
  │           └──────────────────────┘         │
  └────────────────────────────────────────────┘

  Cada core tiene su propio pipeline, registros, L1, y L2.
  L3 es compartida entre todos los cores.
  La comunicación entre cores requiere coherencia de cache (MESI).


  2. Simultaneous Multithreading (SMT) / Hyper-Threading:
  Un core físico ejecuta DOS hilos lógicos compartiendo
  los recursos de ejecución (ALU, FPU, caches).

  ┌────────────────────────────────┐
  │           Core 0               │
  │  ┌──────────┐ ┌──────────┐    │
  │  │ Thread 0 │ │ Thread 1 │    │
  │  │ (regs,PC)│ │ (regs,PC)│    │
  │  └────┬─────┘ └────┬─────┘    │
  │       └──────┬──────┘          │
  │        Shared execution units  │
  │        (ALU, FPU, Load/Store)  │
  │        Shared L1, L2 cache     │
  └────────────────────────────────┘

  Beneficio: cuando Thread 0 espera un cache miss,
  Thread 1 puede usar las unidades de ejecución ociosas.
  Mejora típica: 20-30% de throughput extra (NO 2×).
  No es un segundo core; es mejor utilización del mismo core.


  3. MIMD (Multiple Instruction, Multiple Data):
  Múltiples procesadores ejecutando programas diferentes
  sobre datos diferentes. Es el modelo de un cluster de servidores
  o un sistema multi-socket.
```

### Taxonomía de Flynn

```
  Clasificación de arquitecturas paralelas:

  ┌────────┬──────────────────┬────────────────────────────────┐
  │        │ Single Data      │ Multiple Data                  │
  ├────────┼──────────────────┼────────────────────────────────┤
  │ Single │ SISD             │ SIMD                           │
  │ Instr. │ CPU clásico      │ Instrucciones vectoriales      │
  │        │ Von Neumann      │ SSE, AVX, NEON                 │
  ├────────┼──────────────────┼────────────────────────────────┤
  │ Multi  │ MISD             │ MIMD                           │
  │ Instr. │ (raro, fault     │ Multicore, multi-socket,       │
  │        │  tolerance)      │ clusters, GPUs (sort of)       │
  └────────┴──────────────────┴────────────────────────────────┘

  MIMD es el modelo dominante en servidores y datacenters.
  SIMD es el modelo dominante para ML y procesamiento multimedia.
```

## Modelo mental

El paralelismo tiene tres escalas. ILP es microscopio: dentro de un solo hilo, el hardware encuentra instrucciones independientes y las ejecuta simultáneamente — es invisible para el programador. SIMD es un rodillo: aplica la misma operación a una fila entera de datos de una vez — requiere datos regulares y contiguos. TLP es una fábrica con múltiples líneas de montaje: cada core/hilo es una línea independiente — requiere que el trabajo sea particionable. El rendimiento máximo combina los tres: múltiples cores (TLP), cada uno ejecutando instrucciones fuera de orden (ILP) sobre datos vectorizados (SIMD).

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Single-core saturado, otros idle | Programa single-threaded | Paralelizar con threads o async |
| Speedup sub-lineal al añadir cores | Ley de Amdahl (parte serial) | Reducir la fracción serial |
| SIMD no activado por compilador | Loop con dependencias o branches | Refactorizar para auto-vectorización |
| Hyper-threading degrada rendimiento | Threads compitiendo por L1/L2 cache | Deshabilitar HT para workloads cache-sensitive |

## Insights de ingeniería

- **La Ley de Amdahl limita el paralelismo**: si el 10% de tu programa es serial, el speedup máximo con infinitos cores es 10×. No importa cuántos cores tengas; la parte serial domina. La optimización del rendimiento debe empezar por reducir la fracción serial.
- **SIMD es free performance si tus datos lo permiten**: el compilador auto-vectoriza loops simples sin esfuerzo del programador. Pero las condiciones son estrictas: datos contiguos en memoria, sin dependencias entre iteraciones, sin branches. Por esto los arrays son más rápidos que las linked lists: no solo por cache locality, sino porque son vectorizables.
- **TLP requiere pensar en concurrencia**: paralelizar con threads introduce complejidad (race conditions, deadlocks, false sharing en cache lines). El hardware provee los cores; el software debe descomponer el problema en tareas independientes sin compartir estado mutable. Modelos como actors (Erlang), CSP (Go channels), o async/await simplifican esto a nivel de lenguaje.
