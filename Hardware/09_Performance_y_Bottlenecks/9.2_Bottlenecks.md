# 9.2 Bottlenecks

## Definición

Un bottleneck (cuello de botella) es el recurso que limita el rendimiento total del sistema. Todo sistema tiene exactamente un bottleneck a la vez. Identificarlo es el primer paso para optimizar. Los tres bottlenecks fundamentales son: CPU-bound (procesador saturado), memory-bound (acceso a memoria como limitante), e IO-bound (disco o red como limitante).

## Mecánica interna

### CPU-bound

```
  Síntomas:
  - CPU al 100% en uno o más cores
  - Latencia proporcional a la carga de CPU
  - Añadir más disco o red no mejora nada

  Causas comunes:
  - Algoritmo ineficiente (O(n²) cuando existe O(n log n))
  - Serialización/deserialización costosa (JSON parsing, protobuf)
  - Compresión/descompresión (gzip, zstd)
  - Cifrado (TLS handshake, AES sin instrucciones AES-NI)
  - Regex complejas, XML parsing
  - GC (Garbage Collection) en lenguajes managed

  Diagnóstico:
  top / htop: ver %CPU por proceso y por core
  perf top: ver qué funciones consumen CPU
  perf stat: IPC, cache misses, branch mispredictions

  Si IPC < 1: probablemente memory-bound disfrazado de CPU-bound.
  La CPU espera datos de memoria, pero el core muestra 100% porque
  está "ocupado" esperando (stall cycles cuentan como CPU usage).
```

### Memory-bound

```
  Síntomas:
  - CPU aparece al 100% pero IPC es bajo (< 1)
  - Alto porcentaje de stall cycles
  - Muchos cache misses (L3 miss rate alto)
  - perf stat muestra alto LLC-load-misses

  Causas comunes:
  - Working set mayor que L3 cache → constantes cache misses
  - Acceso aleatorio a estructuras grandes (hash tables, B-trees)
  - Pointer chasing (linked lists, árboles desbalanceados)
  - False sharing entre threads
  - NUMA remote access (acceso a memoria de otro socket)

  Diagnóstico:
  perf stat -e LLC-loads,LLC-load-misses,cycles,instructions
  → Si LLC-load-misses / LLC-loads > 10%, es memory-bound

  numastat: ver si hay accesos NUMA remotos
  perf c2c: detectar false sharing


  Bandwidth de memoria como bottleneck:
  DDR5 dual-channel: ~100 GB/s
  Un workload que escanea arrays grandes (analytics, ML training)
  puede saturar el bandwidth de memoria antes que la CPU.
  Diagnóstico: perf stat -e mem_bw_read,mem_bw_write (si disponible)
```

### IO-bound

```
  Síntomas:
  - CPU en su mayoría idle (baja utilización)
  - Procesos en estado D (uninterruptible sleep) esperando I/O
  - Alto iowait en top/vmstat
  - Disk utilization al 100% (iostat)

  Disk IO-bound:
  iostat -x 1
  → Métricas clave:
    %util: si >90%, disco saturado
    await: latencia promedio de I/O (ms)
    r/s, w/s: IOPS actuales
    rkB/s, wkB/s: throughput actual

  Si await es alto y %util es bajo:
  → Queue depth bajo, el disco podría manejar más en paralelo.
  → Aumentar iodepth (io_uring, aio) o usar más threads de I/O.

  Si %util es 100% y await es alto:
  → Disco saturado. Solución: SSD más rápido, RAID, cache layer.


  Network IO-bound:
  - Socket buffers llenos (ss -tnp muestra Recv-Q/Send-Q)
  - Bandwidth de red saturado (iftop, nload)
  - Latencia de red alta (RTT × window size limita throughput)

  Ley de bandwidth-delay product:
  Max throughput = Window size / RTT
  Con window de 64KB y RTT de 100ms:
  Max = 64KB / 0.1s = 640 KB/s (miserable)

  Solución: TCP window scaling (RFC 7323) permite windows de hasta 1GB.
  Con window de 4MB y RTT de 100ms:
  Max = 4MB / 0.1s = 40 MB/s (aceptable)
```

### Identificación del bottleneck

```
  Metodología USE (Brendan Gregg):
  Para CADA recurso (CPU, memoria, disco, red):
  - Utilization: ¿qué porcentaje del recurso se usa?
  - Saturation: ¿hay cola de espera? (queue depth, run queue)
  - Errors: ¿hay errores? (disk errors, network drops, OOM)

  El recurso con mayor utilización + saturación es el bottleneck.

  Ejemplo:
  CPU util: 30%, saturation: 0      → no es bottleneck
  Memory util: 60%, saturation: 0   → no es bottleneck
  Disk util: 98%, saturation: 15    → BOTTLENECK (queue depth 15)
  Network util: 5%, saturation: 0   → no es bottleneck

  → Optimizar disco: NVMe, caching, reducir I/O innecesario.
```

## Modelo mental

Un sistema es una cadena de producción: CPU procesa, memoria alimenta, disco/red transporta. El throughput total es el de la estación más lenta. Si la CPU es rápida pero espera datos de disco, el disco es el bottleneck. Si el disco es rápido pero la CPU no puede procesar lo que recibe, la CPU es el bottleneck. Siempre hay exactamente UN bottleneck; resolver uno mueve el bottleneck al siguiente recurso más lento.

## Insights de ingeniería

- **"CPU-bound" a menudo es memory-bound disfrazado**: un proceso que muestra 100% CPU puede estar gastando 70% de los ciclos en stalls esperando datos de L3 o RAM. `perf stat` con IPC < 1 lo confirma. La solución no es un CPU más rápido, sino mejor locality de datos (arrays en vez de linked lists, access patterns secuenciales, reducir working set).
- **IO-bound se resuelve con arquitectura, no solo hardware**: pasar de HDD a NVMe da 1000× más IOPS, pero si la aplicación hace 1 I/O sincrónico a la vez, solo usa 1 IOP del millón disponible. La solución es I/O asíncrono (io_uring, epoll) con queue depth alto para saturar el dispositivo. El mejor hardware del mundo no ayuda si el software no lo aprovecha.
