# 9.1 Métricas de Rendimiento

## Definición

Las métricas fundamentales de rendimiento de hardware son latencia (tiempo que tarda una operación individual), throughput (cantidad de operaciones o datos por unidad de tiempo), e IOPS (operaciones de entrada/salida por segundo, específico para almacenamiento). Estas tres métricas están interrelacionadas pero no son intercambiables: un sistema puede tener alto throughput con alta latencia (pipeline profundo) o baja latencia con bajo throughput (un solo canal estrecho).

## Mecánica interna

### Latencia

```
  Latencia = tiempo desde que se inicia una operación hasta que se completa.

  Latencias típicas (órdenes de magnitud):
  ┌──────────────────────────┬──────────────┬──────────────┐
  │ Operación                │ Latencia     │ Referencia   │
  ├──────────────────────────┼──────────────┼──────────────┤
  │ L1 cache hit             │ ~1 ns        │ 1 ciclo      │
  │ L2 cache hit             │ ~4 ns        │ 4 ciclos     │
  │ L3 cache hit             │ ~10-20 ns    │ 10-20 ciclos │
  │ RAM access               │ ~50-100 ns   │              │
  │ NVMe SSD read (4KB)      │ ~10-20 μs    │              │
  │ SATA SSD read (4KB)      │ ~50-100 μs   │              │
  │ HDD read (4KB)           │ ~5-10 ms     │              │
  │ Network same DC          │ ~100-500 μs  │ RTT          │
  │ Network cross-continent  │ ~50-150 ms   │ RTT          │
  └──────────────────────────┴──────────────┴──────────────┘

  Distribución de latencia:
  - Mean (promedio): engañosa, no muestra la cola
  - Median (p50): la mitad de las requests son más rápidas
  - p99: 99% de las requests son más rápidas
  - p99.9: 99.9% son más rápidas
  - Max: el peor caso observado

  En producción, p99 importa más que el promedio.
  Si el promedio es 5ms pero p99 es 500ms, 1 de cada 100 usuarios
  tiene una experiencia 100× peor.

  Causas de tail latency:
  - Garbage collection pauses
  - Page faults (minor o major)
  - Context switches
  - Cache misses (TLB, L3)
  - Disk I/O en path inesperado
  - Network retransmissions
```

### Throughput

```
  Throughput = cantidad de trabajo completado por unidad de tiempo.

  Puede expresarse como:
  - Bytes/segundo (bandwidth): cuántos datos se transfieren
  - Requests/segundo (RPS): cuántas operaciones se completan
  - Transactions/segundo (TPS): cuántas transacciones DB se completan

  Relación con latencia:

  Sistema serial (una operación a la vez):
  Throughput = 1 / Latencia
  Si latencia = 10ms → throughput = 100 ops/s

  Sistema paralelo (N operaciones simultáneas):
  Throughput = N / Latencia (Ley de Little)
  Con 100 conexiones concurrentes y 10ms latencia:
  Throughput = 100 / 0.01s = 10,000 ops/s

  → Se puede aumentar throughput sin reducir latencia,
    simplemente aumentando el paralelismo (hasta saturar un recurso).
```

### IOPS

```
  IOPS = Input/Output Operations Per Second.
  Métrica principal para almacenamiento.

  ┌──────────────────┬────────────────┬──────────────────┐
  │ Dispositivo      │ IOPS (4KB rand)│ Throughput seq.  │
  ├──────────────────┼────────────────┼──────────────────┤
  │ HDD 7200 RPM     │ ~100-200       │ ~150 MB/s        │
  │ SATA SSD          │ ~50K-100K      │ ~550 MB/s        │
  │ NVMe SSD          │ ~500K-1M       │ ~3-7 GB/s        │
  │ Intel Optane      │ ~500K-1M       │ ~2.5 GB/s        │
  │ RAM (tmpfs)       │ ~10M+          │ ~50+ GB/s        │
  └──────────────────┴────────────────┴──────────────────┘

  IOPS depende del tamaño del bloque:
  - 4KB random read: la métrica más exigente (peor caso para storage)
  - 128KB sequential read: mucho más fácil (prefetch, menos seeks)

  IOPS vs Throughput:
  Throughput = IOPS × block_size
  100K IOPS × 4KB = 400 MB/s
  100K IOPS × 64KB = 6.4 GB/s (si el dispositivo lo soporta)

  Un workload puede ser IOPS-bound (muchas operaciones pequeñas: DB random reads)
  o throughput-bound (pocas operaciones grandes: streaming video, backups).
```

## Modelo mental

Latencia es cuánto tarda un carro en ir de A a B. Throughput es cuántos carros llegan a B por hora. IOPS es cuántas entregas completa un servicio de mensajería por segundo. Puedes tener alta latencia (ruta larga) pero alto throughput (muchos carros en paralelo). La clave es saber cuál métrica importa para tu workload: bases de datos necesitan baja latencia (p99); streaming necesita alto throughput; OLTP necesita alto IOPS.

## Herramientas de diagnóstico

```bash
# Latencia de disco
ioping -c 10 /dev/sda                    # latencia de I/O
fio --name=test --rw=randread --bs=4k \
    --iodepth=1 --numjobs=1 --size=1G    # IOPS y latencia

# Throughput de disco
fio --name=test --rw=read --bs=1M \
    --iodepth=32 --numjobs=1 --size=1G   # throughput secuencial

# Latencia de red
ping -c 100 <host>                        # RTT básico
mtr <host>                                # traceroute con estadísticas

# Throughput de red
iperf3 -c <host>                          # bandwidth TCP

# Latencia de memoria
mlc --latency_matrix                      # Intel Memory Latency Checker
```

## Insights de ingeniería

- **Optimizar para la métrica equivocada es el error más común**: una base de datos OLTP necesita baja latencia en p99 para random reads de 4KB. Comprar un SSD con 7 GB/s de throughput secuencial pero 50μs de latencia random no ayuda si el workload es 100% random. Inversamente, un sistema de backup solo necesita throughput secuencial y no importa la latencia.
- **La Ley de Little conecta todo**: L = λ × W (items en sistema = tasa de llegada × tiempo en sistema). Si tienes 1000 requests concurrentes (L) con latencia promedio de 10ms (W), tu throughput es λ = L/W = 100,000 req/s. Si la latencia sube a 100ms, con las mismas 1000 conexiones, throughput baja a 10,000 req/s. Para mantener throughput ante mayor latencia, necesitas más concurrencia.
