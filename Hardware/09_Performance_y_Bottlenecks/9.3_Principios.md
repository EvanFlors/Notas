# 9.3 Principios: Ley de Amdahl y Ley de Little

## Definición

La Ley de Amdahl establece el límite teórico de speedup al paralelizar un programa: la porción serial del programa limita la mejora máxima, sin importar cuántos procesadores se añadan. La Ley de Little relaciona tres métricas fundamentales de cualquier sistema en estado estable: L = λ × W (items en el sistema = tasa de llegada × tiempo de permanencia). Ambas leyes son herramientas de razonamiento esenciales para entender los límites del rendimiento.

## Mecánica interna

### Ley de Amdahl

```
  Speedup = 1 / (S + P/N)

  Donde:
  S = fracción serial del programa (no paralelizable)
  P = fracción paralela (P = 1 - S)
  N = número de procesadores

  Ejemplo: programa con 10% serial (S=0.1), 90% paralelo (P=0.9):

  N=1:   Speedup = 1 / (0.1 + 0.9/1)  = 1.0×
  N=2:   Speedup = 1 / (0.1 + 0.9/2)  = 1.82×
  N=4:   Speedup = 1 / (0.1 + 0.9/4)  = 3.08×
  N=8:   Speedup = 1 / (0.1 + 0.9/8)  = 4.71×
  N=16:  Speedup = 1 / (0.1 + 0.9/16) = 6.40×
  N=64:  Speedup = 1 / (0.1 + 0.9/64) = 8.77×
  N=∞:   Speedup = 1 / (0.1 + 0)      = 10.0× ← MÁXIMO

  Con solo 10% serial, el máximo speedup es 10×.
  No importa si usas 64, 1000, o un millón de cores.


  Implicación: con 5% serial → máximo 20×.
  Con 1% serial → máximo 100×.
  Con 0.1% serial → máximo 1000×.

  La parte serial incluye:
  - Inicialización y finalización
  - Sincronización entre threads (locks, barriers)
  - Acceso serial a recursos compartidos (I/O, memoria)
  - Merge/reduce de resultados parciales


  Ley de Amdahl en sistemas distribuidos:
  Aplica igualmente: si una request tiene una llamada serial
  a una DB que tarda 50ms en un pipeline de 100ms total,
  no importa cuántos servidores de aplicación tengas:
  el tiempo mínimo es 50ms (la parte serial de la DB).
```

### Ley de Little

```
  L = λ × W

  L = número promedio de items en el sistema (concurrencia)
  λ = tasa de llegada (throughput, en items/segundo)
  W = tiempo promedio en el sistema (latencia)

  Esto se aplica a CUALQUIER sistema en estado estable:
  - Colas de supermercado
  - Connection pools de bases de datos
  - Servidores web
  - Queues de disco
  - Pipelines de CPU


  Ejemplo 1: Servidor web
  Latencia promedio (W) = 50ms
  Conexiones concurrentes (L) = 200
  → Throughput (λ) = L/W = 200 / 0.05 = 4,000 req/s

  Si la latencia sube a 200ms (DB lenta):
  → Throughput = 200 / 0.2 = 1,000 req/s (4× peor)
  Para mantener 4,000 req/s: necesitas L = 4000 × 0.2 = 800 conexiones


  Ejemplo 2: Connection pool
  Pool size (L) = 50 conexiones
  Query time (W) = 10ms promedio
  → Max throughput (λ) = 50 / 0.01 = 5,000 queries/s

  Si necesitas 10,000 queries/s:
  Con W=10ms: necesitas L = 10,000 × 0.01 = 100 conexiones
  O reducir W a 5ms con pool de 50.


  Ejemplo 3: Disk queue
  Queue depth (L) = 32
  IO latency (W) = 20μs (NVMe)
  → IOPS (λ) = 32 / 0.00002 = 1,600,000 IOPS

  Con queue depth 1:
  → IOPS = 1 / 0.00002 = 50,000 IOPS

  → Queue depth 32 da 32× más IOPS que queue depth 1,
    con la MISMA latencia por operación.
    Esto es por qué io_uring y iodepth importan.
```

### Combinando ambas leyes

```
  Ley de Amdahl dice CUÁNTO puedes mejorar con paralelismo.
  Ley de Little dice CÓMO el paralelismo se traduce en throughput.

  Sistema con:
  - 90% paralelo, 10% serial (Amdahl: máximo 10× speedup)
  - Latencia: 100ms, Pool: 50 conexiones

  Con 1 worker:
  λ = 1 / 0.1 = 10 req/s

  Con 10 workers (Amdahl: ~5.3× speedup):
  Latencia efectiva ≈ 100ms / 5.3 ≈ 19ms (para la parte paralela)
  Pero la parte serial sigue siendo 10ms
  Latencia total ≈ 10 + 9 = 19ms
  λ = 50 / 0.019 ≈ 2,632 req/s

  Con 100 workers (Amdahl: ~9.2× speedup):
  λ = 50 / 0.011 ≈ 4,545 req/s (mejora marginal)

  Conclusión: la parte serial domina, más workers no ayudan mucho
  después de cierto punto.
```

## Modelo mental

La Ley de Amdahl es como una autopista con un puente de un carril: no importa cuántos carriles tenga la autopista, el throughput total está limitado por el puente (la parte serial). La Ley de Little es el principio del estacionamiento: si cada carro (request) permanece 2 horas (latencia) y el estacionamiento tiene 100 espacios (concurrencia), entonces pasan 50 carros por hora (throughput). Para más carros por hora: más espacios o menor tiempo de permanencia.

## Insights de ingeniería

- **Amdahl muestra por qué "just add more servers" tiene límites**: si hay una base de datos centralizada que serializa el 5% de la request pipeline, el máximo speedup es 20×. Añadir 100 servidores de aplicación no supera 20×. La solución es reducir la fracción serial: caching, sharding, reads de réplicas, CRDTs.
- **Little explica por qué connection pools se agotan**: un pool de 20 conexiones a una DB con queries de 50ms da 400 QPS máximo. Si un query lento (1 segundo) ocupa una conexión, esa conexión produce solo 1 QPS en vez de 20. Unos pocos queries lentos pueden colapsar el pool entero. Solución: timeouts agresivos, circuit breakers, separar pools para queries lentos y rápidos.
- **Queue depth es throughput gratis**: un NVMe con iodepth=1 da ~50K IOPS. Con iodepth=32 da ~1M IOPS. El hardware es el mismo; la diferencia es que el software permite al dispositivo procesar múltiples operaciones en paralelo internamente. Si tu aplicación hace I/O sincrónico (un read a la vez), estás desperdiciando 95% de la capacidad del SSD.
