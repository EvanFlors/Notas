# 11.1 GPUs

## Definición

Una GPU (Graphics Processing Unit) es un procesador masivamente paralelo diseñado originalmente para renderizado gráfico, pero adoptado ampliamente para cómputo general (GPGPU). Una GPU moderna contiene miles de cores simples optimizados para ejecutar la misma operación sobre muchos datos simultáneamente (SIMT — Single Instruction, Multiple Threads). Su arquitectura la hace ideal para álgebra lineal, deep learning, y cualquier workload con paralelismo de datos masivo.

## Mecánica interna

### Arquitectura

```
  CPU vs GPU:

  CPU (pocos cores potentes):       GPU (miles de cores simples):
  ┌───────────────────┐             ┌───────────────────────────┐
  │ Core 0 │ Core 1  │             │ SM SM SM SM SM SM SM SM   │
  │ (OoO,  │ (OoO,  │             │ SM SM SM SM SM SM SM SM   │
  │ branch │ branch │             │ SM SM SM SM SM SM SM SM   │
  │ pred,  │ pred,  │             │ SM SM SM SM SM SM SM SM   │
  │ 512KB  │ 512KB  │             │ ...                       │
  │ L2)    │ L2)    │             │ (128 SMs, cada uno con    │
  ├────────┴────────┤             │  128 CUDA cores)          │
  │   L3 Cache      │             │  = 16,384 cores total     │
  │   (32 MB)       │             ├───────────────────────────┤
  └─────────────────┘             │    L2 Cache (50 MB)       │
                                  ├───────────────────────────┤
                                  │   HBM3 VRAM (80 GB)       │
                                  │   ~3 TB/s bandwidth       │
                                  └───────────────────────────┘

  SM (Streaming Multiprocessor) — NVIDIA:
  Cada SM contiene:
  - 128 CUDA cores (FP32)
  - 64 FP64 cores (double precision)
  - 4 Tensor Cores (matrix multiply, FP16/BF16/INT8)
  - 256 KB register file
  - 256 KB L1 cache / shared memory (configurable)
  - Warp scheduler (schedula grupos de 32 threads = 1 warp)

  Ejecución SIMT:
  32 threads ejecutan la MISMA instrucción simultáneamente (1 warp).
  Si un branch diverge (if-else), ambos paths se ejecutan serialmente
  con threads inactivos enmascarados → pérdida de eficiencia.


  NVIDIA H100:
  - 132 SMs × 128 cores = 16,896 CUDA cores
  - 528 Tensor Cores (4 por SM)
  - 80 GB HBM3 @ 3.35 TB/s
  - PCIe Gen5 ×16 o NVLink (900 GB/s entre GPUs)
  - FP16 Tensor: 1,979 TFLOPS
  - TDP: 700W
```

### CUDA y modelo de programación

```
  Jerarquía de threads CUDA:

  Grid (todo el kernel)
  └─ Block (ejecuta en 1 SM)
     └─ Warp (32 threads, unidad de scheduling)
        └─ Thread (1 CUDA core)

  // Kernel CUDA: suma de vectores
  __global__ void add(float *a, float *b, float *c, int n) {
      int i = blockIdx.x * blockDim.x + threadIdx.x;
      if (i < n) c[i] = a[i] + b[i];
  }

  // Launch: 256 threads por block, ceil(n/256) blocks
  add<<<ceil(n/256), 256>>>(d_a, d_b, d_c, n);

  Flujo de datos:
  1. CPU aloca memoria en GPU: cudaMalloc(&d_a, size)
  2. CPU copia datos a GPU: cudaMemcpy(d_a, h_a, size, H2D)
     (transferencia PCIe: ~25 GB/s PCIe 5.0 ×16)
  3. CPU lanza kernel en GPU
  4. GPU ejecuta (miles de threads en paralelo)
  5. CPU copia resultado de vuelta: cudaMemcpy(h_c, d_c, size, D2H)

  Bottleneck común: la transferencia PCIe es lenta (~25 GB/s)
  comparada con el bandwidth de HBM (~3 TB/s).
  → Los datos deben permanecer en la GPU tanto como sea posible.
```

### Jerarquía de memoria GPU

```
  ┌────────────────────────────────┐
  │ Registers (256 KB por SM)      │  ~0 cycles, per-thread
  ├────────────────────────────────┤
  │ Shared Memory (up to 228 KB)  │  ~20 cycles, per-block
  │ / L1 Cache                    │
  ├────────────────────────────────┤
  │ L2 Cache (50 MB)              │  ~200 cycles, global
  ├────────────────────────────────┤
  │ HBM3 VRAM (80 GB)             │  ~400 cycles, 3.35 TB/s
  ├────────────────────────────────┤
  │ CPU RAM (via PCIe/NVLink)     │  ~10,000 cycles, ~25-900 GB/s
  └────────────────────────────────┘
```

## Modelo mental

Una CPU es un profesor experto que resuelve problemas complejos uno a la vez muy rápido. Una GPU es un aula con 10,000 estudiantes que resuelven problemas simples simultáneamente. Si tienes 1 problema complejo → CPU. Si tienes 10,000 problemas simples e idénticos → GPU. Deep learning es multiplicación de matrices masivas: el problema perfecto para 10,000 estudiantes haciendo multiplicaciones en paralelo.

## Insights de ingeniería

- **La memoria es el bottleneck, no el cómputo**: una H100 puede hacer 2 PFLOPS de FP16, pero solo puede leer 3.35 TB/s de HBM. Para operaciones de baja intensidad aritmética (elementwise ops, batch norm), la GPU está memory-bound. Optimizar acceso a memoria (coalesced access, shared memory tiling) importa más que optimizar cómputo.
- **NVLink vs PCIe para multi-GPU**: PCIe 5.0 ×16 da ~64 GB/s bidireccional. NVLink 4.0 (H100) da 900 GB/s. Para training distribuido donde las GPUs intercambian gradientes constantemente, NVLink es ~14× más rápido. Un servidor DGX H100 con 8 GPUs conectadas por NVLink puede hacer all-reduce de 80 GB de gradientes en ~100ms. Con PCIe, tomaría ~1.4 segundos.
