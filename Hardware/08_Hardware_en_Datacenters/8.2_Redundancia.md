# 8.2 Redundancia

## Definición

La redundancia en hardware de datacenter es la duplicación de componentes críticos para que la falla de uno no cause downtime. Se aplica en múltiples niveles: fuentes de alimentación (PSU), almacenamiento (RAID), red (NICs duales, paths redundantes), y sistemas completos (clusters activo-pasivo o activo-activo).

## Mecánica interna

### Niveles de redundancia

```
  Componente         Redundancia típica        Failover
  ──────────────────────────────────────────────────────
  PSU                2× PSU (N+1)              Automático, <1ms
  Ventiladores       N+1 fans                  Automático
  NIC                2× NIC (bonding/teaming)  Automático, <1s
  Uplink de red      2× switches ToR           Automático (LACP, ECMP)
  Almacenamiento     RAID 1/5/10               Automático
  Servidor completo  Cluster activo-pasivo     Segundos a minutos
  Datacenter         Multi-DC (geo-redundant)  Minutos (DNS failover)


  PSU Redundancy (N+1):
  ┌──────────┐  ┌──────────┐
  │ PSU A    │  │ PSU B    │
  │ 750W     │  │ 750W     │
  │ Feed A   │  │ Feed B   │
  └────┬─────┘  └────┬─────┘
       │              │
       └──────┬───────┘
              │
       ┌──────┴──────┐
       │  Server     │
       │  (consume   │
       │   500W)     │
       └─────────────┘

  Cada PSU conectada a un feed eléctrico diferente (A y B).
  Cada feed viene de un PDU diferente, idealmente de un UPS diferente.
  Si PSU A falla o Feed A cae → PSU B sostiene la carga.


  NIC Bonding (Linux):
  ┌──────┐      ┌──────────┐      ┌──────────┐
  │ eth0 │──────│ bond0    │      │ Switch A │
  │ eth1 │──────│ (active- │──────│ Switch B │
  └──────┘      │  backup) │      └──────────┘
                └──────────┘

  Modos:
  - mode 1 (active-backup): un NIC activo, otro standby
  - mode 2 (balance-xor): distribuye por hash de MAC
  - mode 4 (802.3ad/LACP): link aggregation, requiere soporte del switch

  LACP: combina 2× 25Gbps en 50Gbps lógico + failover automático.
```

### Disponibilidad

```
  Nines de disponibilidad:
  ┌────────────┬──────────────────┬─────────────────────┐
  │ Nivel      │ Uptime %         │ Downtime/año        │
  ├────────────┼──────────────────┼─────────────────────┤
  │ 99%        │ "two nines"      │ 3.65 días           │
  │ 99.9%      │ "three nines"    │ 8.76 horas          │
  │ 99.99%     │ "four nines"     │ 52.6 minutos        │
  │ 99.999%    │ "five nines"     │ 5.26 minutos        │
  │ 99.9999%   │ "six nines"      │ 31.5 segundos       │
  └────────────┴──────────────────┴─────────────────────┘

  Para lograr 99.99%:
  - Hardware redundante en TODOS los niveles
  - Failover automático (no manual)
  - Mantenimiento sin downtime (hot-swap de componentes)
  - Monitoreo 24/7 con alertas automatizadas

  MTBF (Mean Time Between Failures):
  Tiempo promedio entre fallas de un componente.
  HDD: ~1.5M horas, SSD: ~2M horas, PSU: ~100K horas.

  MTTR (Mean Time To Repair):
  Tiempo promedio para reparar/reemplazar.
  Hot-swap disk: ~5 min, reboot servidor: ~5 min,
  reemplazo servidor completo: ~30 min (con spare).

  Disponibilidad = MTBF / (MTBF + MTTR)
```

## Modelo mental

La redundancia es como un avión con dos motores: puede volar con uno. Cada componente redundante es un motor adicional. Más motores = más caro, más pesado, pero mayor probabilidad de llegar a destino. La clave es que el failover sea automático (el avión ajusta empuje automáticamente si un motor falla) y que el componente fallido se pueda reemplazar sin aterrizar (hot-swap).

## Insights de ingeniería

- **La redundancia de red es donde más se falla en la práctica**: servidores con PSU dual y RAID son estándar, pero muchas configuraciones tienen un solo switch ToR o un solo uplink. Un fallo de switch deja offline a todo el rack. La configuración correcta es dual-ToR con MLAG o ECMP, donde cada servidor tiene un NIC conectado a cada switch.
- **La redundancia a nivel de datacenter (multi-DC) es la más costosa y compleja**: requiere replicación de datos en tiempo real (o near-real-time), DNS failover o anycast, y la capacidad de cada DC de absorber la carga completa. El "split-brain" (ambos DCs creen ser el primario) es el fallo más peligroso y requiere mecanismos de consenso distribuido (quorum).
