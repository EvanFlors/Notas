# 12.1 Flujo de Ejecución End-to-End

## Definición

Este tema integra todos los módulos anteriores en un recorrido completo de una request HTTP desde el cliente hasta la respuesta, identificando cada componente de hardware involucrado en cada paso. Es el modelo mental unificado que un ingeniero de backend debe tener del hardware subyacente.

## Mecánica interna

```
  CLIENTE ENVÍA REQUEST: curl https://api.example.com/users/42

  ═══════════════════════════════════════════════════════════
  PASO 1: SALIDA DEL CLIENTE
  ═══════════════════════════════════════════════════════════

  CPU del cliente:
  ├─ Aplicación llama send() → syscall → kernel mode
  ├─ TCP stack: segmentación, headers, checksum
  ├─ IP stack: routing table lookup → interfaz de salida
  ├─ L2: ARP para gateway MAC → Ethernet frame

  NIC del cliente:
  ├─ DMA: lee frame del TX ring buffer
  ├─ TSO offload: segmenta si es necesario
  ├─ Checksum offload: calcula CRC
  └─ PHY: transmite señal eléctrica/óptica

  ═══════════════════════════════════════════════════════════
  PASO 2: TRÁNSITO POR LA RED
  ═══════════════════════════════════════════════════════════

  Switch L2 (acceso):
  ├─ ASIC: MAC lookup en CAM table → forwarding al puerto del router
  ├─ Latencia: ~1-5 μs

  Router:
  ├─ ASIC/TCAM: longest prefix match en FIB
  ├─ Decrementa TTL, recalcula checksum
  ├─ NAT: reescribe IP origen (si aplica)
  ├─ Reenvía al siguiente hop
  ├─ Latencia: ~1-5 μs

  [Múltiples routers en Internet: ~5-15 hops]

  Firewall del datacenter:
  ├─ ASIC o FPGA: inspección de paquete
  ├─ Stateful: conntrack lookup
  ├─ Si permitido → forwarding

  Load Balancer L4:
  ├─ ASIC/DPDK/XDP: consistent hash del 5-tuple
  ├─ Selecciona backend server
  ├─ DNAT: reescribe IP destino al backend elegido
  ├─ Latencia: ~1-10 μs

  Switch spine-leaf del datacenter:
  ├─ ASIC: ECMP forwarding entre spines
  ├─ Leaf ToR: entrega al puerto del servidor
  ├─ Latencia: ~1-2 μs por hop, 2 hops = ~2-4 μs

  ═══════════════════════════════════════════════════════════
  PASO 3: LLEGADA AL SERVIDOR
  ═══════════════════════════════════════════════════════════

  NIC del servidor:
  ├─ PHY: recibe señal, decodifica
  ├─ MAC: verifica CRC, filtra por MAC
  ├─ RSS: hash del 5-tuple → RX queue 3
  ├─ DMA: escribe frame en ring buffer (RAM)
  ├─ Checksum offload: verifica IP/TCP checksum
  ├─ MSI-X: interrupción al core 3

  CPU del servidor (kernel):
  ├─ ISR: marca NAPI scheduled
  ├─ softirq: NAPI poll, construye sk_buff
  ├─ L2→L3→L4 processing
  ├─ conntrack: registra conexión
  ├─ Socket buffer: enqueue datos
  ├─ Wake up del proceso de la aplicación

  CPU del servidor (userspace):
  ├─ recv(): copia datos de socket buffer a user buffer
  ├─ HTTP parsing: identifica GET /users/42
  ├─ Lógica de aplicación: consultar base de datos

  ═══════════════════════════════════════════════════════════
  PASO 4: ACCESO A BASE DE DATOS
  ═══════════════════════════════════════════════════════════

  CPU:
  ├─ Aplicación envía query SQL via send() al socket de la DB
  ├─ [Si DB es remota: repite pasos de red]
  ├─ [Si DB es local:]

  DB Engine (PostgreSQL):
  ├─ Parser → Planner → Executor
  ├─ Buffer Pool lookup (shared_buffers en RAM):
  │   ├─ HIT: lee página de RAM → ~1 μs
  │   └─ MISS: necesita leer de disco
  │
  ├─ Lectura de disco:
  │   CPU: read() syscall → kernel → block layer
  │   Page Cache lookup:
  │     ├─ HIT: ~1-2 μs
  │     └─ MISS: I/O al dispositivo
  │
  │   NVMe Controller:
  │   ├─ CPU escribe SQ entry (en RAM)
  │   ├─ Doorbell write al controlador
  │   ├─ Controller lee SQ via DMA
  │   ├─ Controller lee NAND flash (~10 μs)
  │   ├─ Controller escribe datos a RAM via DMA
  │   ├─ Controller escribe CQ entry, MSI-X interrupt
  │   ├─ Datos en page cache → buffer pool → resultado
  │
  ├─ CPU ejecuta query (joins, filters, sorts)
  │   ├─ Datos en L1/L2 cache: ~1-4 ns por acceso
  │   ├─ L3 cache miss → RAM: ~50-100 ns
  │   ├─ NUMA remote access: ~100-150 ns
  │
  ├─ Resultado → send() al socket de la aplicación

  ═══════════════════════════════════════════════════════════
  PASO 5: RESPUESTA
  ═══════════════════════════════════════════════════════════

  CPU del servidor (userspace):
  ├─ Serializa respuesta (JSON)
  ├─ send(): copia a kernel socket buffer

  CPU del servidor (kernel):
  ├─ TCP: headers, segmentación (o TSO offload)
  ├─ IP: routing, netfilter
  ├─ L2: Ethernet header
  ├─ TX ring buffer enqueue

  NIC del servidor:
  ├─ DMA: lee frame de TX ring buffer
  ├─ CRC, transmite

  [Tránsito inverso por la red]

  NIC del cliente → kernel → aplicación → usuario ve la respuesta


  ═══════════════════════════════════════════════════════════
  PRESUPUESTO DE LATENCIA (misma región, best case):
  ═══════════════════════════════════════════════════════════

  Cliente NIC TX:                   ~1 μs
  Red (LB + switches + router):    ~10-50 μs
  Servidor NIC RX + kernel:        ~5-10 μs
  App processing + DB query:       ~1-50 ms (domina)
  Servidor NIC TX:                 ~2-5 μs
  Red (retorno):                   ~10-50 μs
  Cliente NIC RX + kernel:         ~5-10 μs
  ──────────────────────────────────────────
  Total overhead hardware:         ~30-130 μs
  Total con app:                   ~1-50 ms

  → El hardware añade <1% del tiempo total en la mayoría de requests.
  → El bottleneck es casi siempre la lógica de aplicación y los accesos a DB.
  → Pero en sistemas de ultra-baja latencia (trading, gaming, ad-serving),
     cada μs del hardware importa.
```

## Modelo mental

Una HTTP request es un viaje transcontinental. Sale de una ciudad (cliente), pasa por autopistas (switches), cruza fronteras (routers/firewalls), llega al aeropuerto (load balancer), aterriza en la terminal correcta (servidor), va a una oficina (aplicación), consulta un archivo (base de datos), y envía la respuesta de vuelta por el mismo camino. Cada tramo tiene su propio vehículo (hardware), velocidad, y posibles retrasos. El viajero (dato) nunca ve la infraestructura, solo experimenta el tiempo total.

## Insights de ingeniería

- **Conocer el data path completo permite diagnosticar problemas en cualquier capa**: si la latencia p99 sube de 5ms a 50ms, ¿es la red? ¿el disco? ¿la CPU? ¿un GC pause? Con el modelo mental del flujo completo, puedes razonar sobre dónde se introduce la latencia y usar las herramientas correctas (tcpdump para red, iostat para disco, perf para CPU, strace para syscalls) en la capa correcta.
