# 12.2 Principios Clave

## Definición

Este documento sintetiza los principios fundamentales de hardware que todo ingeniero de backend debe internalizar. Son las reglas que gobiernan el rendimiento, la fiabilidad, y la escalabilidad de los sistemas, independientemente del lenguaje de programación o framework utilizado.

## Principios

### 1. Latencia: los números que debes conocer

```
  Operación                        Latencia        Factor
  ─────────────────────────────────────────────────────────
  L1 cache reference               1 ns            1×
  L2 cache reference               4 ns            4×
  Branch misprediction             5 ns            5×
  L3 cache reference               10-20 ns        10-20×
  RAM reference                    50-100 ns       50-100×
  NVMe SSD random 4K read          10-20 μs        10,000-20,000×
  SATA SSD random 4K read          50-100 μs       50,000-100,000×
  HDD random 4K read               5-10 ms         5,000,000-10,000,000×
  Network roundtrip same DC        100-500 μs      100,000-500,000×
  Network roundtrip cross-region   50-150 ms       50,000,000-150,000,000×

  Regla: cada salto en la jerarquía cuesta ~10-100× más latencia.
  Diseña tus estructuras de datos para maximizar cache hits.
  Diseña tu arquitectura para minimizar network roundtrips.
```

### 2. Throughput: paralelismo es la respuesta

```
  Ley de Little: Throughput = Concurrency / Latency

  Para aumentar throughput:
  a) Reducir latencia (hardware más rápido, algoritmos mejores)
  b) Aumentar concurrencia (más threads, más conexiones, más queue depth)
  c) Reducir la fracción serial (Ley de Amdahl)

  Paralelismo existe en múltiples niveles:
  - ILP: el CPU ejecuta múltiples instrucciones por ciclo (superscalar)
  - SIMD: una instrucción opera sobre múltiples datos (AVX-512)
  - TLP: múltiples threads en múltiples cores
  - DLP: múltiples datos en GPU (miles de threads)
  - Request-level: múltiples requests en múltiples servidores

  El software debe EXPONER paralelismo para que el hardware lo aproveche.
  Una aplicación single-threaded en un servidor de 128 cores usa <1% del hardware.
```

### 3. Locality: el principio más importante

```
  Temporal locality: si accediste un dato, probablemente lo accederás pronto otra vez.
  → Caches funcionan por esto. Buffer pools, connection pools, DNS cache.

  Spatial locality: si accediste un dato, probablemente accederás datos cercanos.
  → Prefetch funciona por esto. Arrays > linked lists. Sequential I/O > random I/O.

  Aplicaciones prácticas:
  - Arrays de structs vs struct de arrays (AoS vs SoA)
  - B-tree (nodos grandes, secuenciales) vs binary tree (nodos dispersos)
  - Batch processing (procesar N items juntos) vs one-at-a-time
  - Column stores (datos del mismo tipo contiguos) para analytics
  - Hot/cold data separation en bases de datos
```

### 4. Networking: el costo de la distribución

```
  Red = latencia + unreliability + bandwidth limitado.

  Falacias de la computación distribuida (Peter Deutsch):
  1. La red es confiable             → NO (packet loss, partitions)
  2. La latencia es cero             → NO (100μs-150ms)
  3. El bandwidth es infinito        → NO (saturación de uplinks)
  4. La red es segura                → NO (MITM, sniffing)
  5. La topología no cambia          → NO (failovers, re-routing)
  6. Hay un solo administrador       → NO (multi-team, multi-cloud)
  7. El costo de transporte es cero  → NO (CPU para TCP, egress costs)
  8. La red es homogénea             → NO (diferentes MTUs, velocidades)

  Implicaciones para el diseño:
  - Minimizar roundtrips (batching, pipelining, caching)
  - Diseñar para fallos de red (timeouts, retries, circuit breakers)
  - Locality de datos: procesar donde están los datos, no moverlos
  - Compresión en tránsito para reducir bandwidth (gzip, zstd)
```

### 5. Persistencia: el storage es el fundamento

```
  La jerarquía de storage refleja el trade-off velocidad-capacidad-costo:

  Registros (KB) → Cache (MB) → RAM (GB) → SSD (TB) → HDD (PB)

  Reglas:
  - RAM es volátil. Si se apaga → se pierde. Siempre fsync() lo importante.
  - SSD tiene vida limitada (write endurance). Monitorear SMART.
  - HDD es lento pero barato. Ideal para backups y cold storage.
  - RAID no es backup. Backup es copia en OTRO lugar.
  - La regla 3-2-1: 3 copias, 2 medios diferentes, 1 offsite.

  Para bases de datos:
  - WAL (Write-Ahead Log) en SSD rápido para durabilidad
  - Data files pueden estar en storage más lento
  - Buffer pool en RAM para hot data
  - Backups en object storage (S3) para cold storage
```

### 6. Observabilidad: medir antes de optimizar

```
  "Si no puedes medirlo, no puedes mejorarlo."

  Herramientas por capa:

  CPU:        perf stat, perf top, perf record + flamegraph
  Memoria:    perf mem, numastat, valgrind (cachegrind)
  Disco:      iostat, fio, blktrace
  Red:        tcpdump, ss, ethtool -S, iperf3
  Kernel:     bpftrace, eBPF, strace, ftrace
  Aplicación: profilers del lenguaje, tracing distribuido

  Metodología:
  1. Medir (USE method: Utilization, Saturation, Errors)
  2. Identificar bottleneck
  3. Formar hipótesis
  4. Cambiar UNA variable
  5. Medir de nuevo
  6. Repetir

  Anti-pattern: optimizar sin medir → "optimizar" cosas que no son el bottleneck.
```

## Modelo mental integrado

```
  El backend engineer debe ver el sistema como:

  ┌─────────────────────────────────────────────────────┐
  │                  APLICACIÓN                         │
  │  (lógica de negocio, el 99% del tiempo aquí)       │
  ├─────────────────────────────────────────────────────┤
  │                  RUNTIME / OS                       │
  │  (syscalls, scheduling, memory management)          │
  ├─────────────────────────────────────────────────────┤
  │                  HARDWARE                           │
  │  CPU ← cache ← RAM ← SSD ← NIC ← Red             │
  │  (cada flecha es un salto de 10-100× en latencia)  │
  └─────────────────────────────────────────────────────┘

  No necesitas diseñar hardware.
  Necesitas diseñar SOFTWARE que RESPETE el hardware:
  - Estructuras de datos cache-friendly
  - I/O asíncrono para saturar dispositivos
  - Batching para amortizar overhead de syscalls
  - Locality de datos para minimizar network roundtrips
  - Concurrencia para aprovechar múltiples cores
  - Medición para encontrar el bottleneck real
```

## Insights de ingeniería

- **El hardware más rápido no compensa software mal diseñado**: un NVMe de 1M IOPS con una aplicación que hace I/O sincrónico (queue depth 1) da 50K IOPS. Un HDD con una aplicación que hace I/O asíncrono con prefetch inteligente puede superar a ese NVMe mal utilizado para workloads secuenciales. El software debe estar diseñado para aprovechar las capacidades del hardware.
- **Los principios son eternos, las implementaciones cambian**: las velocidades de DDR5 vs DDR4, PCIe 5.0 vs 4.0, NVMe vs SATA cambiarán cada 2-3 años. Pero los principios (locality, paralelismo, jerarquía de latencia, Amdahl, Little) son fundamentales de la física y la ingeniería. Aprende los principios y las implementaciones específicas se derivan naturalmente.
