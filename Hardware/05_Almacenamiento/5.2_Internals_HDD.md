# 5.2 Internals HDD

## Definición

Un HDD almacena datos como orientaciones magnéticas en platos giratorios. Un brazo mecánico con cabezales de lectura/escritura se posiciona sobre la pista correcta (seek) y espera a que el sector deseado pase debajo (rotational latency). La naturaleza mecánica del HDD hace que los accesos aleatorios sean extremadamente lentos (~10ms) mientras que los accesos secuenciales son razonablemente rápidos (~200 MB/s).

## Mecánica interna

```
  Estructura física:
  ┌───────────────────────────────────────────┐
  │            Vista superior                  │
  │                                           │
  │         ╭─── Plato (disco) ───╮           │
  │        ╱   ╭───────────╮       ╲          │
  │       │   │  ╭───────╮ │        │         │
  │       │   │  │Spindle│ │ Track  │         │
  │       │   │  ╰───────╯ │        │         │
  │        ╲   ╰───────────╯       ╱          │
  │         ╰─────────────────────╯           │
  │                   ↑                        │
  │              Arm / Head ──→               │
  │              (se mueve radialmente)        │
  └───────────────────────────────────────────┘

  Componentes:
  - Platos: discos de aluminio/vidrio con recubrimiento magnético
    Típicamente 1-5 platos apilados, ambas caras grabables
  - Cabezal: flota a ~10nm del plato (levitación aerodinámica)
  - Spindle: motor que gira los platos a velocidad constante
  - Actuator: motor que mueve el brazo radialmente

  Organización lógica:
  - Track: anillo concéntrico en la superficie del plato
  - Sector: arco de un track (~512 bytes o 4KB con Advanced Format)
  - Cylinder: todos los tracks en la misma posición radial en todos los platos


  Velocidades de rotación:
  ┌──────────┬──────────────────┬───────────────┐
  │ RPM      │ Rotational lat.  │ Uso           │
  ├──────────┼──────────────────┼───────────────┤
  │ 5400 RPM │ ~5.5ms promedio  │ Laptops, NAS  │
  │ 7200 RPM │ ~4.2ms promedio  │ Desktop       │
  │ 10000 RPM│ ~3.0ms promedio  │ Enterprise    │
  │ 15000 RPM│ ~2.0ms promedio  │ Enterprise SAS│
  └──────────┴──────────────────┴───────────────┘

  Rotational latency promedio = (60 / RPM) / 2 × 1000 ms
  (promedio = media revolución)
```

### Latencia de acceso

```
  Acceso aleatorio a un sector:

  Total = Seek time + Rotational latency + Transfer time

  Seek time: tiempo para mover el cabezal al track correcto
  - Full stroke (track 0 a track N): ~15-20ms
  - Promedio: ~5-8ms
  - Track-to-track: ~0.5-1ms

  Rotational latency: esperar a que el sector pase bajo el cabezal
  - Promedio: media revolución
  - 7200 RPM: ~4.2ms promedio

  Transfer time: leer los datos del sector
  - Un sector de 4KB a 200 MB/s: ~0.02ms (despreciable)

  Total acceso aleatorio: ~5ms + 4.2ms + 0.02ms ≈ 9.2ms
  IOPS random = 1000ms / 9.2ms ≈ 108 IOPS

  Total acceso secuencial (seek ya hecho, track ya abierto):
  Solo transfer time: 200 MB/s sostenido → excelente para streams


  Esto explica por qué los HDDs son buenos para:
  - Lectura secuencial grande (video streaming, backup restore)
  - Escritura secuencial (logs append-only)

  Y malos para:
  - Random read/write (bases de datos, index lookups)
  - Acceso a archivos pequeños dispersos
```

### Optimizaciones de firmware

```
  NCQ (Native Command Queuing):
  El HDD reordena los comandos en cola para minimizar movimiento
  del cabezal. Si hay 32 reads pendientes, los ejecuta en el orden
  que minimiza seek distance (elevator algorithm / SCAN).

  Track caching:
  Lee tracks completos en un buffer interno (~256MB en HDDs modernos).
  Si la siguiente lectura es del mismo track → servida del buffer.

  Shingled Magnetic Recording (SMR):
  Tracks se superponen como tejas para aumentar densidad.
  + Mayor capacidad por plato.
  - Escrituras aleatorias requieren reescribir tracks adyacentes → muy lento.
  - Solo viable para workloads write-once/read-many (archival).
```

## Modelo mental

Un HDD es un tocadiscos: el brazo busca el surco correcto (seek), espera a que la aguja llegue al punto exacto (rotational latency), y luego lee los datos en secuencia (transfer). Mover el brazo es lo más caro. Leer en secuencia una vez posicionado es barato. Todo el rendimiento de un HDD se resume en: "minimiza los movimientos del brazo".

## Insights de ingeniería

- **HDDs deberían estar en ext4/XFS con journal en SSD**: el journaling genera escrituras aleatorias frecuentes. Si el journal está en HDD, cada commit es un seek. Poner el journal en un SSD pequeño mejora dramáticamente el rendimiento.
- **Para bases de datos, los HDDs son obsoletos**: una base de datos relacional con índices B-tree genera accesos aleatorios por diseño. 100 IOPS vs 500K IOPS de NVMe es la diferencia entre un query de 100ms y uno de 0.02ms. Si tu base de datos está en HDD y es lenta, cámbiala a NVMe antes de optimizar queries.
