# 5.3 Internals SSD

## Definición

Un SSD almacena datos en chips de memoria NAND Flash, organizados en pages (4-16KB) y blocks (256KB-4MB). A diferencia del HDD, no tiene partes móviles. Sin embargo, tiene características únicas: las pages solo pueden escribirse una vez antes de necesitar un borrado del block completo (erase-before-write), el controlador debe redistribuir escrituras para evitar desgaste desigual (wear leveling), y la recolección de basura (garbage collection) puede causar latency spikes.

## Mecánica interna

### NAND Flash

```
  Jerarquía de organización:

  SSD
  └── Controller
      ├── Channel 0
      │   ├── Die 0
      │   │   ├── Plane 0
      │   │   │   ├── Block 0
      │   │   │   │   ├── Page 0 (4-16KB) ← unidad mínima de lectura/escritura
      │   │   │   │   ├── Page 1
      │   │   │   │   └── ... (256-512 pages per block)
      │   │   │   ├── Block 1
      │   │   │   └── ... (1000-4000 blocks per plane)
      │   │   └── Plane 1
      │   └── Die 1
      ├── Channel 1
      └── ... (4-8 channels típico)


  Operaciones fundamentales:
  ┌────────────┬─────────────┬────────────────────────────┐
  │ Operación  │ Granularidad│ Latencia                   │
  ├────────────┼─────────────┼────────────────────────────┤
  │ Read       │ Page (4KB)  │ 25-50μs                    │
  │ Write      │ Page (4KB)  │ 200-500μs                  │
  │ Erase      │ Block       │ 1.5-3ms                    │
  └────────────┴─────────────┴────────────────────────────┘

  Restricción crítica:
  Una page solo puede escribirse si está BORRADA (estado "empty").
  Para sobrescribir datos, hay que borrar el BLOCK entero (256KB-4MB)
  que contiene la page. Esto se llama erase-before-write.


  Tipos de NAND:
  ┌──────┬────────────┬──────────────┬────────────────┐
  │ Tipo │ Bits/celda │ Durabilidad  │ Velocidad      │
  ├──────┼────────────┼──────────────┼────────────────┤
  │ SLC  │ 1          │ 100K cycles  │ Más rápido     │
  │ MLC  │ 2          │ 10K cycles   │ Rápido         │
  │ TLC  │ 3          │ 3K cycles    │ Moderado       │
  │ QLC  │ 4          │ 1K cycles    │ Más lento      │
  └──────┴────────────┴──────────────┴────────────────┘

  Más bits por celda = más capacidad, menos durabilidad y velocidad.
  Enterprise SSDs usan SLC/MLC. Consumer SSDs usan TLC/QLC.
```

### Wear Leveling

```
  Problema: cada celda NAND solo soporta un número finito de
  ciclos de escritura/borrado (P/E cycles). Si siempre escribes
  en las mismas celdas, se desgastan y fallan prematuramente.

  Solución: wear leveling distribuye escrituras uniformemente
  entre todos los blocks del SSD.

  Static wear leveling:
  Incluso datos que NO se modifican se mueven periódicamente
  para que los blocks que los contienen acumulen ciclos P/E
  al mismo ritmo que los blocks con datos hot.

  Dynamic wear leveling:
  Solo redistribuye entre blocks con datos "calientes" (frecuentemente
  escritos). Más simple pero blocks con datos fríos no se desgastan,
  desperdiciando su vida útil.


  Overprovisioning (OP):
  Los SSDs reservan ~7-28% de su capacidad total como espacio
  no visible al usuario. Este espacio se usa para:
  - Wear leveling (mover datos)
  - Garbage collection (blocks de trabajo)
  - Reemplazar blocks defectuosos (bad block replacement)

  Un SSD de "1TB" tiene ~1.1TB de NAND; 100GB son overprovisioning.
  Enterprise SSDs tienen más OP (~28%) para mayor durabilidad.
```

### Garbage Collection

```
  Escenario:
  Block A: [Page 0: valid][Page 1: stale][Page 2: valid][Page 3: stale]

  "Stale" = datos obsoletos (sobreescritos con FTL redirect, ver abajo).
  El block tiene espacio desperdiciado pero no se puede reusar
  sin borrar el block entero.

  Garbage Collection (GC):
  1. Seleccionar un block con muchas pages stale
  2. Copiar las pages VÁLIDAS a otro block
  3. Borrar el block original (ahora vacío y reutilizable)

  Block A: [V][S][V][S]     Block B: [V][V][ ][ ]
  → Copiar V pages a B     → Erase Block A
  Block A: [ ][ ][ ][ ]     Block B: [V][V][ ][ ]

  Problema: GC consume IOPS y bandwidth internamente.
  Si el SSD está muy lleno (>80-90%), GC se vuelve agresivo
  y compite con I/O del usuario → latency spikes.

  TRIM:
  Cuando el filesystem borra un archivo, el SSD no lo sabe
  (el filesystem solo marca el espacio como libre en sus metadatos).
  TRIM es un comando que informa al SSD: "estas pages ya no son válidas".
  El SSD puede marcarlas como stale inmediatamente → GC más eficiente.

  Sin TRIM: el SSD cree que tiene menos espacio libre del real → GC agresivo.
  Con TRIM: el SSD sabe exactamente qué pages son válidas → GC eficiente.

  Linux: fstrim (manual) o discard mount option (automático).
```

### Write Amplification

```
  Write Amplification (WA) = datos escritos a NAND / datos escritos por host

  Ideal: WA = 1.0 (cada byte escrito por el host se escribe 1 vez a NAND)
  Real: WA = 1.5 - 3.0+ (por GC, wear leveling, y journaling interno)

  Ejemplo: WA = 3.0
  El host escribe 100GB. El SSD escribe 300GB internamente.
  Un SSD con 300 TBW (Total Bytes Written) de vida útil
  realmente solo soporta ~100TB de escrituras del host.

  Factores que aumentan WA:
  - SSD muy lleno (>80%): GC copia más datos válidos
  - Escrituras aleatorias pequeñas (4KB): cada write puede triggear GC
  - No usar TRIM: el SSD no sabe qué pages son stale

  Factores que reducen WA:
  - Escrituras secuenciales (llenan blocks completos, no necesitan GC)
  - TRIM habilitado
  - Overprovisioning generoso
  - Mantener el SSD <70-80% lleno
```

### Flash Translation Layer (FTL)

```
  El FTL es el firmware del controlador que traduce
  direcciones lógicas (LBA) a direcciones físicas (PBA) en NAND.

  Cuando el host escribe al LBA 1000:
  1. FTL elige una page vacía en NAND (e.g., Block 5, Page 3)
  2. Escribe los datos ahí
  3. Actualiza la mapping table: LBA 1000 → Block 5, Page 3
  4. La page antigua (si existía) se marca como stale

  Esto hace que las escrituras sean "append-like" internamente:
  siempre se escribe a pages vacías, nunca se sobrescribe in-place.
  Es por esto que los SSDs no necesitan defragmentación.
```

## Modelo mental

Un SSD es como un notebook con páginas que solo se pueden escribir con tinta indeleble. Para "borrar" algo, tienes que arrancar toda la hoja (block) y copiar lo que aún necesitas a una hoja nueva. Wear leveling es rotar qué hojas usas para que todas se desgasten parejo. Garbage collection es reorganizar tus notas periódicamente: copiar las notas útiles a hojas limpias y tirar las hojas llenas de tachaduras.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Latency spikes periódicos | GC en background | Mantener <70% lleno, TRIM habilitado |
| Write performance degrada con el tiempo | NAND wear, WA alto | Monitorear SMART: Media Wearout Indicator |
| SSD falla prematuramente | TBW excedido | Enterprise SSD con más OP y SLC/MLC |
| Random write lento | GC compitiendo con writes | Secuencializar escrituras donde sea posible |

## Insights de ingeniería

- **No llenes un SSD a más del 70-80%**: la garbage collection necesita blocks libres para mover datos. Si el SSD está >90% lleno, GC no tiene espacio para trabajar y las escrituras causan latency spikes de 10-100ms (vs 10-20μs normales). Para bases de datos en producción, reserva 30%+ de espacio libre.
- **Las escrituras secuenciales son 5-10× más rápidas que las aleatorias**: las escrituras secuenciales llenan blocks completos, minimizando GC. Por esto los WAL (Write-Ahead Log) de bases de datos son append-only: convierten escrituras aleatorias de datos en escrituras secuenciales de log.
- **SMART monitoring es obligatorio en producción**: los SSDs reportan su estado de salud via SMART. `Media_Wearout_Indicator`, `Reallocated_Sector_Count`, y `Total_LBAs_Written` te dicen cuánta vida útil queda. Alertar cuando el wearout supera el 80% para planificar reemplazo.
