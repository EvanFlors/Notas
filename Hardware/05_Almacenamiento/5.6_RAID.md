# 5.6 RAID

## Definición

RAID (Redundant Array of Independent Disks) combina múltiples discos en una unidad lógica para mejorar rendimiento, redundancia, o ambos. Los niveles RAID (0, 1, 5, 10) ofrecen diferentes trade-offs entre capacidad, rendimiento de lectura/escritura, y tolerancia a fallos de disco.

## Mecánica interna

### Niveles RAID

```
  RAID 0 (Striping):
  ┌────────┐ ┌────────┐
  │Disco 0 │ │Disco 1 │
  │ A1  A3 │ │ A2  A4 │
  │ A5  A7 │ │ A6  A8 │
  └────────┘ └────────┘

  Los datos se dividen en stripes y se distribuyen entre discos.
  + Rendimiento: N× bandwidth (N discos escriben/leen en paralelo)
  + Capacidad: 100% (ningún overhead de redundancia)
  - Redundancia: CERO. Si un disco falla, TODOS los datos se pierden.
  Uso: scratch space, temp files, donde se puede reconstruir desde fuente.


  RAID 1 (Mirroring):
  ┌────────┐ ┌────────┐
  │Disco 0 │ │Disco 1 │
  │ A1  A2 │ │ A1  A2 │  (copia exacta)
  │ A3  A4 │ │ A3  A4 │
  └────────┘ └────────┘

  Cada dato se escribe en AMBOS discos (espejo).
  + Redundancia: tolera fallo de 1 disco
  + Lecturas: 2× (puede leer de cualquier disco)
  - Capacidad: 50% (la mitad es redundancia)
  - Escrituras: igual a 1 disco (ambos deben escribir)
  Uso: OS boot drive, bases de datos pequeñas, alta disponibilidad.


  RAID 5 (Striping + Parity distribuida):
  ┌────────┐ ┌────────┐ ┌────────┐
  │Disco 0 │ │Disco 1 │ │Disco 2 │
  │ A1     │ │ A2     │ │ Ap     │  (Ap = A1 XOR A2)
  │ B1     │ │ Bp     │ │ B2     │  (Bp = B1 XOR B2)
  │ Cp     │ │ C1     │ │ C2     │  (Cp = C1 XOR C2)
  └────────┘ └────────┘ └────────┘

  Datos striped + un bloque de paridad por stripe, rotado entre discos.
  Si un disco falla, los datos se reconstruyen con XOR:
  A1 = A2 XOR Ap, o A2 = A1 XOR Ap

  + Capacidad: (N-1)/N (con 3 discos: 66%, con 5: 80%)
  + Lecturas: buenas (N-1 discos en paralelo)
  + Redundancia: tolera fallo de 1 disco
  - Escrituras: lentas (write penalty: cada write requiere
    leer paridad vieja, calcular nueva, escribir dato + paridad)
  - Rebuild lento y peligroso: si otro disco falla durante rebuild → data loss
  Uso: almacenamiento de lectura-heavy, file servers.
  Mínimo: 3 discos.


  RAID 10 (1+0: Mirror + Stripe):
  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐
  │Disco 0 │ │Disco 1 │ │Disco 2 │ │Disco 3 │
  │ A1     │ │ A1     │ │ A2     │ │ A2     │
  │ A3     │ │ A3     │ │ A4     │ │ A4     │
  └────────┘ └────────┘ └────────┘ └────────┘
    Mirror pair 0           Mirror pair 1
    ←── Stripe ──────────────────────→

  Primero mirror (cada par es un espejo), luego stripe entre pares.
  + Rendimiento: excelente lectura Y escritura (stripes de mirrors)
  + Redundancia: tolera fallo de 1 disco por mirror pair
  + Rebuild rápido: solo copiar de un disco, no recalcular paridad
  - Capacidad: 50% (como RAID 1)
  Uso: bases de datos de producción, workloads write-heavy.
  Mínimo: 4 discos.
```

### Comparación

```
  ┌────────┬───────────┬──────────┬────────────┬────────────────┐
  │ RAID   │ Capacidad │ Tolerancia│ Read perf  │ Write perf     │
  ├────────┼───────────┼──────────┼────────────┼────────────────┤
  │ 0      │ 100%      │ 0 discos │ N×         │ N×             │
  │ 1      │ 50%       │ 1 disco  │ 2×         │ 1×             │
  │ 5      │ (N-1)/N   │ 1 disco  │ (N-1)×     │ Lento (parity) │
  │ 10     │ 50%       │ 1/pair   │ N×         │ (N/2)×         │
  └────────┴───────────┴──────────┴────────────┴────────────────┘
```

### Hardware vs Software RAID

```
  Hardware RAID:
  Controlador RAID dedicado (con su propio CPU y cache).
  + Transparente al OS, mejor rendimiento (cache con batería).
  + Battery-Backed Write Cache (BBWC): protege datos en cache durante power loss.
  - Vendor lock-in (el array solo funciona con el mismo controlador).
  - Caro.

  Software RAID (mdadm, ZFS, LVM):
  El OS gestiona el RAID usando la CPU del servidor.
  + Barato, portable, flexible.
  + Visible al OS (mejor monitoreo).
  - Consume CPU (marginal en CPUs modernos).
  - Sin BBWC nativo (pero SSDs con power loss protection compensan).

  ZFS RAID (RAIDZ1/Z2/Z3):
  Software RAID integrado con el filesystem.
  + Checksums en cada bloque → detecta silent data corruption.
  + No necesita hardware RAID controller.
  + RAIDZ2 tolera 2 fallos (vs 1 en RAID 5).
```

## Modelo mental

RAID 0 es velocidad sin seguridad: repartir trabajo entre discos pero sin respaldo. RAID 1 es seguridad sin eficiencia: tener una copia exacta de todo. RAID 5 es un compromiso: guardar una "receta" (paridad) para reconstruir datos perdidos, pero calcularla en cada escritura. RAID 10 es lo mejor de ambos mundos: espejos para seguridad, stripes para velocidad, pero usas el 50% del espacio.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Data loss en RAID 0 | Un disco falla | RAID 0 no tiene redundancia → backup externo |
| Rebuild lento (días) en RAID 5 | Discos grandes (10TB+), rebuild read-intensive | Preferir RAID 10 o RAIDZ2 para discos grandes |
| Segundo disco falla durante rebuild | URE (Unrecoverable Read Error) en disco durante rebuild | RAID 6/RAIDZ2 (tolera 2 fallos) |
| Silent data corruption | Bit rot no detectado | ZFS con checksums, scrubbing periódico |

## Insights de ingeniería

- **RAID 5 es peligroso con discos grandes**: durante el rebuild de un disco de 10TB (que puede tardar 24+ horas), la probabilidad de un URE en otro disco es significativa. Un URE durante rebuild = data loss. Para discos >2TB, usar RAID 6 (doble paridad) o RAID 10.
- **RAID no es backup**: RAID protege contra fallo de disco, no contra eliminación accidental, ransomware, o corrupción de software. Siempre necesitas backups separados (en otro dispositivo/ubicación). La regla 3-2-1: 3 copias, 2 tipos de medio, 1 off-site.
- **Para bases de datos en producción, RAID 10 es el estándar**: combina el mejor rendimiento de escritura (stripes) con redundancia rápida de recuperar (mirrors). PostgreSQL, MySQL, y Oracle en producción casi siempre corren sobre RAID 10 con SSDs.
