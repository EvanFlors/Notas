# 5.5 Sistemas de Archivos

## Definición

Un sistema de archivos (filesystem) organiza datos en un dispositivo de almacenamiento en una estructura jerárquica de archivos y directorios. Internamente, gestiona la asignación de bloques (qué bloques del disco pertenecen a qué archivo), metadatos (permisos, timestamps) mediante inodos, y la integridad de datos mediante journaling o copy-on-write.

## Mecánica interna

### Inodos

```
  Un inodo (inode) es la estructura de metadatos que describe un archivo.
  Todo archivo y directorio tiene exactamente un inodo.

  Contenido de un inodo:
  ┌────────────────────────────────────────────────┐
  │ Inode #12345                                   │
  │ ─────────────────────────────────              │
  │ Type:        regular file                      │
  │ Permissions: rwxr-xr-x (755)                   │
  │ Owner:       uid=1000, gid=1000                │
  │ Size:        1,048,576 bytes (1MB)              │
  │ Timestamps:                                    │
  │   atime: 2024-01-15 10:30:00 (last access)    │
  │   mtime: 2024-01-10 14:20:00 (last modify)    │
  │   ctime: 2024-01-10 14:20:00 (last change)    │
  │ Link count:  1                                 │
  │ Block pointers:                                │
  │   Direct[0..11]: 12 bloques directos           │
  │   Indirect: apunta a bloque de punteros        │
  │   Double indirect: puntero a punteros          │
  │   Triple indirect: puntero a punteros a punt.  │
  └────────────────────────────────────────────────┘

  Nota: el inodo NO contiene el nombre del archivo.
  El nombre está en el directorio (que es una tabla nombre→inode#).
  Un archivo puede tener múltiples nombres (hard links).

  ls -i: muestra el número de inodo
  stat file.txt: muestra toda la info del inodo
```

### Bloques

```
  El disco se divide en bloques de tamaño fijo (típicamente 4KB).
  Un archivo ocupa uno o más bloques.

  Archivo de 10KB con bloques de 4KB:
  Bloque 0: bytes 0-4095
  Bloque 1: bytes 4096-8191
  Bloque 2: bytes 8192-10239 (parcialmente lleno)
  → 3 bloques usados, 12KB ocupados, 2KB desperdiciados (fragmentación interna)

  Archivos pequeños: desperdician espacio relativo.
  Un directorio con 10,000 archivos de 100 bytes cada uno:
  Datos reales: 1MB. Espacio en disco: 40MB (4KB × 10,000 bloques).

  Fragmentación externa:
  Bloques de un archivo dispersos por el disco.
  En HDD: fragmentación causa seeks adicionales → lento.
  En SSD: fragmentación no afecta rendimiento significativamente
  (acceso aleatorio es rápido). No defragmentar SSDs.
```

### Journaling

```
  Problema: si el sistema crashea durante una escritura,
  el filesystem puede quedar en estado inconsistente.

  Ejemplo: crear un archivo requiere:
  1. Asignar un inodo
  2. Escribir datos en bloques
  3. Actualizar el directorio
  4. Actualizar el bitmap de bloques libres

  Si el crash ocurre entre el paso 2 y 3:
  - Los bloques están usados pero ningún directorio los referencia → lost blocks
  - Sin journaling: fsck debe escanear TODO el disco para detectar inconsistencias
    → puede tardar horas en discos grandes


  Journaling:
  Antes de modificar el filesystem, escribe un "plan" en un journal (log):

  1. JOURNAL: "Voy a: crear inode 12345, escribir bloques 100-102,
     actualizar directorio, actualizar bitmap"
  2. Ejecutar las operaciones
  3. JOURNAL: "Completado. Borrar journal entry."

  Si crash durante paso 2:
  - Al boot, leer el journal
  - Replay las operaciones incompletas o revertirlas
  - Filesystem consistente en segundos (no horas)


  Modos de journaling (ext4):
  ┌──────────────┬─────────────────────────────────────────────┐
  │ Modo         │ Qué se journalea                            │
  ├──────────────┼─────────────────────────────────────────────┤
  │ journal      │ Metadatos + datos (más seguro, más lento)   │
  │ ordered (def)│ Solo metadatos, datos escritos antes del    │
  │              │ commit del journal (buen balance)            │
  │ writeback    │ Solo metadatos (más rápido, datos pueden     │
  │              │ perderse en crash)                           │
  └──────────────┴─────────────────────────────────────────────┘


  Copy-on-Write (COW) filesystems:
  ZFS, Btrfs: en vez de sobrescribir datos, SIEMPRE escriben en
  nuevas ubicaciones. El puntero al bloque viejo se actualiza
  atómicamente al final. Si hay crash, el puntero viejo sigue válido.
  No necesitan journal. Soportan snapshots nativamente.
```

### Filesystems principales

```
  ext4 (Linux default):
  - Journaling, extents (rangos de bloques contiguos)
  - Max file: 16TB, max volume: 1EB
  - Estable, bien probado, buena performance general

  XFS (Red Hat default, ideal para archivos grandes):
  - Journaling solo de metadatos
  - Excelente para archivos grandes y I/O paralelo
  - Usado en servidores de producción con bases de datos

  ZFS (Sun/Oracle, OpenZFS):
  - COW, checksums en todos los datos (detecta bit rot)
  - RAID integrado (RAIDZ), snapshots, compression
  - Ideal para storage servers y NAS

  Btrfs (Linux, experimental):
  - COW, snapshots, subvolumes, compression
  - Checksums de datos y metadatos
  - Aún no se recomienda para producción crítica (RAID5/6 inestable)
```

## Modelo mental

Un filesystem es una biblioteca con un catálogo (inodos) y estanterías (bloques). El catálogo dice dónde está cada libro (punteros a bloques), quién lo posee (permisos), y cuándo fue modificado (timestamps). El nombre del libro está en el índice del pasillo (directorio), no en el catálogo del libro. Journaling es como un cuaderno de "tareas pendientes": antes de mover un libro, anotas el plan. Si te interrumpen, puedes retomar desde el cuaderno.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Datos corruptos tras crash | Writeback mode sin barrier | Usar ordered mode, habilitar barriers |
| Inodos agotados (no space pero df muestra espacio) | Muchos archivos pequeños | Reformatear con más inodos o usar XFS |
| Rendimiento degradado en filesystem lleno | Fragmentación, no hay bloques contiguos | Mantener <80% lleno, usar extents |

## Insights de ingeniería

- **ext4 con data=ordered es el default seguro**: protege metadatos con journal y garantiza que los datos se escriben antes del commit. Para bases de datos que hacen su propio journaling (PostgreSQL WAL), writeback mode puede ser aceptable (la BD maneja su propia consistencia).
- **XFS es superior para workloads con archivos grandes y I/O paralelo**: maneja mejor archivos de GBs (logs, dumps, backups) y tiene mejor escalabilidad en directorios con millones de archivos.
- **O_DIRECT bypassa el page cache**: las bases de datos (PostgreSQL, MySQL) a menudo usan O_DIRECT para controlar su propia cache (buffer pool) sin interferencia del page cache del OS. Esto evita double caching (dato en buffer pool + page cache).
