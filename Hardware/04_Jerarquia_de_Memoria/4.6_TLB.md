# 4.6 TLB (Translation Lookaside Buffer)

## Definición

El TLB es una cache hardware especializada que almacena traducciones recientes de direcciones virtuales a físicas. Sin TLB, cada acceso a memoria requeriría 4 accesos adicionales para recorrer la page table (page table walk). El TLB reduce este costo a 0-1 ciclos para traducciones cacheadas (TLB hit), haciendo la memoria virtual prácticamente gratuita en rendimiento para la mayoría de accesos.

## Mecánica interna

### Funcionamiento

```
  CPU genera dirección virtual:

  ┌─────────────┐    TLB hit     ┌──────────────┐
  │ Virtual Addr ├──────(1ns)────▶│ Physical Addr│──▶ Cache/RAM
  │              │                └──────────────┘
  │              │    TLB miss
  │              ├──────(~30ns)──▶ Page Table Walk
  │              │                4 accesos a memoria
  │              │                → Physical Addr
  └─────────────┘                → Actualizar TLB

  TLB hit: traducción en 1 ciclo (solapada con acceso a L1 cache).
  TLB miss: page table walk = ~30-100ns (4 niveles × ~7-25ns cada uno).

  Estructura del TLB:
  ┌─────────────────────────────────────────────────┐
  │                    TLB Entry                     │
  │ ┌──────────────┬──────────────┬────────────────┐ │
  │ │ Virtual Page │ Physical     │ Flags          │ │
  │ │ Number (VPN) │ Frame (PFN)  │ R/W/X/U/G/D   │ │
  │ └──────────────┴──────────────┴────────────────┘ │
  └─────────────────────────────────────────────────┘

  Flags:
  R (Read), W (Write), X (Execute) → permisos
  U (User/Supervisor) → accesible desde user space o solo kernel
  G (Global) → no se invalida en context switch (páginas de kernel)
  D (Dirty) → la página fue modificada


  Niveles de TLB:
  ┌─────────────────────────────────────────────────────┐
  │ TLB L1 (ITLB + DTLB):                              │
  │   ITLB: 128 entries (instrucciones)                 │
  │   DTLB: 64 entries para 4KB, 32 entries para 2MB   │
  │   Latencia: 1 ciclo                                 │
  │   Fully associative                                 │
  │                                                     │
  │ TLB L2 (STLB - Second-Level TLB):                  │
  │   1536-2048 entries (unificado)                     │
  │   Latencia: ~7-8 ciclos                             │
  │   12-way set-associative                            │
  └─────────────────────────────────────────────────────┘

  Cobertura:
  64 DTLB entries × 4KB = 256KB cubiertos (con páginas de 4KB)
  32 DTLB entries × 2MB = 64MB cubiertos (con huge pages)

  → Huge pages cubren 250× más memoria con el mismo TLB.
```

### TLB miss y page table walk

```
  Cuando el TLB no contiene la traducción:

  1. Hardware page table walker (x86, ARM):
     La MMU recorre la page table automáticamente.
     4 accesos a memoria (PML4 → PDPT → PD → PT).
     Cada acceso puede hacer hit en cache de datos (L1/L2/L3).
     Si las tablas están en L1: ~4 × 4ns = 16ns
     Si las tablas están en RAM: ~4 × 100ns = 400ns (raro, worst case)
     Promedio: ~30-50ns

  2. Software-managed TLB (MIPS, RISC-V):
     TLB miss genera una excepción.
     El OS maneja el miss en software → más flexible pero más lento.


  Page Table Walk Caches (PWC):
  Los CPUs modernos cachean los niveles intermedios de la page table:
  PML4 cache, PDPT cache, PD cache.
  Esto reduce el walk de 4 accesos a 1-2 en el caso común.


  Costo real de TLB miss:
  ┌─────────────────────┬──────────────────────────────┐
  │ Escenario           │ Latencia de traducción       │
  ├─────────────────────┼──────────────────────────────┤
  │ DTLB hit            │ 1 ciclo (~0.3ns)             │
  │ STLB hit            │ ~7-8 ciclos (~2ns)           │
  │ Walk (in cache)     │ ~30 ciclos (~7ns)            │
  │ Walk (in RAM)       │ ~200 ciclos (~50ns)          │
  │ Page fault (minor)  │ ~5,000 ciclos (~1μs)         │
  │ Page fault (major)  │ ~50,000,000 ciclos (~10ms)   │
  └─────────────────────┴──────────────────────────────┘
```

### TLB shootdown

```
  Problema: en multicore, cada core tiene su propio TLB.
  Cuando el OS cambia un mapeo de página (e.g., munmap, mprotect),
  debe invalidar la traducción en TODOS los TLBs.

  TLB shootdown:
  1. Core 0 cambia la page table.
  2. Core 0 envía IPI (Inter-Processor Interrupt) a todos los otros cores.
  3. Cada core recibe la IPI → invalida la TLB entry específica.
  4. Cada core envía acknowledgment a Core 0.
  5. Core 0 continúa.

  Costo: ~1-10μs (dependiendo del número de cores).

  En un servidor de 64 cores, un munmap() envía 63 IPIs.
  Esto es significativo para workloads con mmap/munmap frecuente
  (JIT compilers, memory allocators que devuelven páginas al OS).

  PCID (Process Context ID):
  Hardware tag que identifica al proceso en las TLB entries.
  Con PCID, un context switch NO necesita flush del TLB completo:
  las entries del proceso anterior permanecen (tagged con su PCID)
  y pueden reutilizarse si el proceso vuelve a ejecutarse.

  Sin PCID: context switch → flush TLB → cold start para nuevo proceso.
  Con PCID: context switch → el TLB mantiene entries de ambos procesos.
```

## Modelo mental

El TLB es la "libreta de direcciones" de la MMU. En vez de buscar en el directorio completo (page table de 4 niveles) cada vez que necesitas traducir una dirección, primero miras tu libreta. Si la dirección está anotada (TLB hit, 1 ciclo), la usas directamente. Si no (TLB miss), recorres el directorio (page table walk, ~30ns) y anotas la traducción en la libreta para la próxima vez. Huge pages son como anotar rangos enteros en vez de direcciones individuales: una sola anotación cubre 2MB en vez de 4KB.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Alto TLB miss rate | Working set > TLB coverage (256KB con 4KB pages) | Huge pages (2MB), mejorar data locality |
| Latency spikes por TLB shootdown | Frecuente mmap/munmap/mprotect | Reducir operaciones de VM, memory pools |
| Context switch lento | TLB flush sin PCID | Habilitar PCID en kernel |
| Page table walk lento | Page tables no en cache | Reducir niveles de indirección, huge pages |

## Insights de ingeniería

- **Huge pages son la optimización más impactante para datasets grandes**: pasar de 4KB a 2MB pages aumenta la cobertura del TLB de ~256KB a ~64MB (con el mismo número de entries). Para bases de datos con buffer pools de GBs, esto elimina millones de TLB misses por segundo. PostgreSQL, MySQL, y Redis soportan huge pages; siempre habilitarlas en producción.
- **TLB misses son invisibles en muchos profilers**: un TLB miss aparece como "memory access latency" genérica. Usa `perf stat -e dTLB-load-misses,dTLB-store-misses` para medir TLB misses directamente. Si el porcentaje es > 1-2%, huge pages tendrán impacto significativo.
- **KPTI (Kernel Page Table Isolation)** — mitigación de Meltdown — separa las page tables de kernel y user space. Cada syscall requiere cambiar de page table → flush parcial del TLB (las entries de kernel ya no están). PCID mitiga parcialmente este costo. Impacto de rendimiento: 2-30% dependiendo de la frecuencia de syscalls.
