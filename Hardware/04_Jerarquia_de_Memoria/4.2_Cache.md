# 4.2 Cache

## Definición

La cache es una memoria SRAM pequeña y extremadamente rápida que almacena copias de datos frecuentemente accedidos de la memoria principal. Su funcionamiento se basa en la localidad temporal y espacial de los programas. Las políticas de escritura (write-through, write-back) determinan cuándo se propagan los cambios a niveles inferiores. Las políticas de reemplazo (LRU, FIFO) determinan qué dato se descarta cuando la cache está llena.

## Mecánica interna

### Estructura de la cache

```
  La cache se organiza en sets y ways (set-associative).

  Dirección de memoria (64 bits):
  ┌──────────────┬──────────┬──────────┐
  │     Tag      │  Index   │  Offset  │
  │  (identifica │ (selecciona│(byte    │
  │   el bloque) │  el set)  │dentro del│
  │              │           │  bloque) │
  └──────────────┴──────────┴──────────┘

  Offset: 6 bits (para 64 bytes = 2⁶ bytes por cache line)
  Index: selecciona uno de los sets
  Tag: verifica que el dato en el set es el correcto


  Tipos de asociatividad:

  Direct-mapped (1-way):
  Cada dirección de memoria mapea a EXACTAMENTE una posición en cache.
  + Simple y rápido.
  - Conflictos: dos direcciones que mapean al mismo set se desalojan mutuamente.

  Fully associative:
  Un dato puede ir a CUALQUIER posición de cache.
  + Sin conflictos.
  - Costoso: debe comparar el tag con TODAS las entradas (N comparadores).

  N-way set-associative (el estándar):
  La cache se divide en sets. Cada set tiene N ways (posiciones).
  Un dato puede ir a cualquier way de su set asignado.

  ┌─────────────────────────────────────────────────┐
  │ 8-way set-associative cache (típico de L1):     │
  │                                                 │
  │ Set 0: [way0][way1][way2][way3][way4]...[way7]  │
  │ Set 1: [way0][way1][way2][way3][way4]...[way7]  │
  │ ...                                             │
  │ Set N: [way0][way1][way2][way3][way4]...[way7]  │
  └─────────────────────────────────────────────────┘

  Configuraciones típicas:
  ┌────────┬─────────┬────────┬─────────────────┐
  │ Cache  │ Tamaño  │ Ways   │ Line size       │
  ├────────┼─────────┼────────┼─────────────────┤
  │ L1 D   │ 48 KB   │ 12-way │ 64 bytes        │
  │ L1 I   │ 32 KB   │ 8-way  │ 64 bytes        │
  │ L2     │ 1.25 MB │ 20-way │ 64 bytes        │
  │ L3     │ 32 MB   │ 16-way │ 64 bytes        │
  └────────┴─────────┴────────┴─────────────────┘
```

### Localidad temporal y espacial

```
  Localidad temporal:
  Dato accedido recientemente → probablemente se accederá de nuevo.

  for (int i = 0; i < 1000; i++)
      sum += array[i];
  //  ^^^
  //  'sum' se accede 1000 veces → permanece en cache (temporal locality)


  Localidad espacial:
  Dato accedido → datos cercanos en memoria probablemente se accederán.

  for (int i = 0; i < 1000; i++)
      sum += array[i];
  //              ^^^^^^
  //  array[0], array[1], ... son contiguos en memoria
  //  Acceder a array[0] trae array[0..15] en una cache line
  //  Los próximos 15 accesos son gratis (cache hits)


  Prefetching:
  El hardware detecta patterns de acceso secuencial y trae
  cache lines ANTES de que se soliciten.

  Accesos: array[0], array[1], array[2], ...
  Hardware prefetcher: "patrón secuencial detectado"
  → trae array[16..31] mientras CPU procesa array[0..15]
  → cuando CPU llega a array[16], ya está en cache

  Funciona bien para: recorridos de arrays, streams de datos.
  Falla para: accesos aleatorios (hash tables, pointer chasing).
```

### Políticas de escritura

```
  Write-through:
  Cada escritura va INMEDIATAMENTE a cache Y al nivel inferior (L2/RAM).

  CPU escribe X=5:
  1. Escribe en L1: X=5  ✓
  2. Escribe en L2: X=5  ✓ (simultáneamente o en el próximo ciclo)

  + Coherencia simple: todos los niveles están siempre actualizados.
  + Recovery simple: si L1 se pierde, los datos están en L2.
  - Lento: cada escritura genera tráfico al nivel inferior.
  - Bandwidth: muchas escrituras saturan el bus.

  Usado en: L1 de algunos procesadores embebidos, caches de I/O.


  Write-back:
  La escritura va SOLO a cache. Se propaga al nivel inferior
  solo cuando la cache line es desalojada (evicted).

  CPU escribe X=5:
  1. Escribe en L1: X=5, marca como "dirty"  ✓
  2. L2 sigue con el valor viejo (stale)

  Cuando la cache line se desaloja:
  3. Si dirty → escribe a L2/RAM (write-back)
  4. Si clean → simplemente descarta

  + Rápido: escrituras múltiples al mismo bloque no generan tráfico.
  + Bandwidth: reduce tráfico al bus significativamente.
  - Complejidad: necesita dirty bits, coherencia más compleja.
  - Recovery: si L1 se pierde antes del write-back, datos perdidos
    (no aplica a caches on-chip que no fallan, pero sí a NVRAM).

  Usado en: L1 y L2 de todos los CPUs modernos de propósito general.


  Write-allocate vs No-write-allocate:
  Write-allocate (típico con write-back):
    Si se escribe a un dato que NO está en cache → traer a cache primero, luego escribir.
  No-write-allocate (típico con write-through):
    Si no está en cache → escribir directamente al nivel inferior, no traer a cache.
```

### Políticas de reemplazo

```
  Cuando la cache está llena y llega un dato nuevo,
  ¿qué cache line se descarta?

  LRU (Least Recently Used):
  Descarta la cache line que no se ha accedido hace más tiempo.
  Asume: "lo que no se usó recientemente, no se usará pronto".

  Accesos: A B C D A B E
  Cache (4 ways):
  Paso 1: [A][ ][ ][ ]
  Paso 2: [A][B][ ][ ]
  Paso 3: [A][B][C][ ]
  Paso 4: [A][B][C][D]     ← cache llena
  Paso 5: [A][B][C][D]     ← A accedido, se mueve a MRU
  Paso 6: [A][B][C][D]     ← B accedido, se mueve a MRU
  Paso 7: [A][B][E][D]     ← E entra, C descartado (LRU)

  + Buen rendimiento general.
  - Costoso de implementar exactamente para muchos ways
    (necesita tracking de orden de acceso).
  - En la práctica se usa Pseudo-LRU (Tree-PLRU):
    aproximación con bits de árbol, mucho más barato.


  FIFO (First In, First Out):
  Descarta la cache line más antigua (por orden de inserción).
  + Simple.
  - Ignora frecuencia de uso: puede descartar datos calientes.


  Random:
  Descarta una cache line al azar.
  + Extremadamente simple (un contador pseudoaleatorio).
  + Rendimiento sorprendentemente bueno en promedio.
  - No adaptativo: no explota patterns de acceso.
  Usado en: algunos caches ARM.
```

### Cache misses: las tres C's

```
  Compulsory miss (cold miss):
  Primera vez que se accede a un dato. Inevitable.
  No importa el tamaño de la cache.

  Capacity miss:
  El working set es más grande que la cache.
  Si tu programa accede a 2MB de datos con L1 de 48KB,
  los datos se desalojan constantemente.
  Solución: reducir working set o aumentar cache.

  Conflict miss:
  Dos datos mapean al mismo set y se desalojan mutuamente,
  incluso si la cache tiene espacio total libre.
  Solución: mayor asociatividad (más ways).

  Coherence miss (4th C):
  En sistemas multicore: un dato se invalida porque
  otro core lo modificó (protocolo MESI).
```

## Modelo mental

La cache es un almacén inteligente que mantiene copias de los datos que probablemente necesitarás. Cuando pides un dato, primero busca en el almacén (hit = rápido). Si no lo tiene (miss = lento), lo trae de la memoria principal y lo guarda para la próxima vez, descartando algo viejo si no hay espacio. Write-back es como escribir en un borrador y pasar a limpio solo cuando el borrador se necesita para otra cosa. LRU decide qué tirar basándose en "lo que no usaste últimamente probablemente no lo necesites".

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| L1 miss rate alto | Working set >> 48KB | Tiling, blocking, reducir struct size |
| Cache thrashing | Conflict misses (datos aligned a power of 2) | Padding, cambiar offset de arrays |
| Performance cliff al crecer datos | Working set cruza L2/L3 boundary | Profile para identificar boundary |
| Write-intensive slow | Write-back evictions saturan bus | Batch writes, write-combining buffers |

## Insights de ingeniería

- **Cache-oblivious algorithms**: algoritmos diseñados para ser eficientes en cache sin conocer el tamaño de la cache. Ejemplo: merge sort recursivo divide el problema hasta que los subproblemas caben en cualquier nivel de cache. En contraste, cache-aware algorithms (tiling con bloques de tamaño específico) requieren conocer el tamaño de la cache.
- **Data-oriented design (DOD)**: en vez de arrays of structs (AoS), usa structs of arrays (SoA). Si solo necesitas el campo `x` de 1000 objetos, AoS trae `x, y, z, w` de cada objeto (desperdiciando 75% de cada cache line). SoA agrupa todos los `x` juntos → 100% de cada cache line es útil. Usado extensivamente en game engines y databases.
- **La cache line de 64 bytes es la unidad de pensamiento**: toda optimización de cache debe razonar en unidades de 64 bytes. Un `struct` de 72 bytes ocupa 2 cache lines. Un `bool` de 1 byte trae 63 bytes adicionales. Mutex padding a 64 bytes evita false sharing. Las estructuras de datos eficientes en cache (B-trees, van Emde Boas trees) se diseñan alrededor del tamaño de la cache line.
