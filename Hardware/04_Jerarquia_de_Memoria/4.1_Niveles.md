# 4.1 Niveles de la Jerarquía de Memoria

## Definición

La jerarquía de memoria organiza el almacenamiento en niveles de velocidad, tamaño, y costo decrecientes: registros (más rápidos, más pequeños, más caros por bit) → caches L1/L2/L3 → memoria principal (RAM) → almacenamiento secundario (SSD/HDD). Cada nivel actúa como cache del nivel inferior. Esta jerarquía existe porque no es física ni económicamente viable tener toda la memoria tan rápida como los registros.

## Mecánica interna

### La pirámide

```
  Velocidad ▲                                    ▲ Costo/bit
            │  ┌─────────┐                       │
            │  │Registros│  ~0.3ns, ~KB           │
            │  │ (CPU)   │  SRAM (6 transistores) │

  Ratios de latencia (normalizados):
  ┌──────────────────┬──────────┬──────────────────┐

  ├──────────────────┼──────────┼──────────────────┤
  │ L1 Cache         │ 1ns      │ 1 segundo        │

  La diferencia entre L1 cache y HDD es como 1 segundo vs 115 días.
```

### Por qué funciona: localidad de referencia

```
  La jerarquía funciona porque los programas exhiben LOCALIDAD:

  Localidad temporal:
  Si un dato fue accedido recientemente, es probable que se acceda de nuevo pronto.
  Ejemplo: variables de loop (i, sum), datos calientes en un web server (session data).
  → Mantener datos recientes en cache funciona bien.

  Localidad espacial:
  Si un dato fue accedido, es probable que datos cercanos se accedan pronto.
  Ejemplo: recorrer un array secuencialmente, campos de un struct.
  → Traer bloques completos (cache lines de 64 bytes) es efectivo.


  Cache hit rate típico:
  ┌────────┬──────────────┐
  │ Nivel  │ Hit rate      │
  ├────────┼──────────────┤
  │ L1     │ 95-99%        │
  │ L2     │ 90-97%        │
  │ L3     │ 80-95%        │
  └────────┴──────────────┘

  Un hit rate del 95% en L1 significa que solo 5% de accesos
  van a L2. De esos, 95% hacen hit en L2. Solo 0.25% llegan a L3.
  Solo ~0.05% llegan a RAM. La jerarquía filtra exponencialmente.
```

### Inclusividad de caches

```
  Inclusive: todo dato en L1 también está en L2 y L3.
  - Ventaja: para buscar un dato, basta con verificar L3.
  - Desventaja: espacio desperdiciado (datos duplicados).
  - Usado en: Intel pre-Skylake.

  Exclusive: un dato está en SOLO un nivel de cache.
  - Ventaja: capacidad total = L1 + L2 + L3.
  - Desventaja: más complejo de gestionar.
  - Usado en: AMD Zen.

  Non-inclusive (NINE): L3 no garantiza contener todo lo de L1/L2.
  - Compromiso: menos duplicación que inclusive, más simple que exclusive.
  - Usado en: Intel Skylake+ (L3 es non-inclusive, L2 es inclusive de L1).
```

## Modelo mental

La jerarquía de memoria es como un sistema de almacenamiento personal. Los registros son tu memoria de trabajo inmediata (lo que estás pensando ahora). L1 es tu escritorio (lo que tienes a la mano). L2 es el cajón del escritorio (a unos pasos). L3 es la estantería de la oficina (compartida con colegas). RAM es el archivo de la empresa (hay que caminar hasta allá). SSD es el almacén del edificio (hay que tomar el ascensor). HDD es un almacén en otra ciudad. La razón por la que funciona es que casi siempre necesitas lo que ya tenías a la mano (localidad temporal) o lo que está justo al lado (localidad espacial).

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| IPC < 1.0, muchos stalls | Working set > cache size | Reducir working set, mejorar data layout |
| Rendimiento cae abruptamente al aumentar datos | Datos exceden L2/L3 | Tiling/blocking para que chunks quepan en cache |
| Latencia alta en acceso a RAM | DRAM latency ~100ns | Prefetching (hardware/software) |

## Insights de ingeniería

- **El "memory wall" es el problema central del rendimiento moderno**: la velocidad de los procesadores ha crecido exponencialmente, pero la latencia de DRAM solo ha mejorado ~2× en 20 años (~100ns → ~50ns). Las caches mitigan esto para datos con localidad, pero para working sets grandes o accesos aleatorios, la RAM es el bottleneck. Todo el diseño de software de alto rendimiento (databases, game engines, HPC) gira alrededor de mantener datos en cache.
- **Cache line size (64 bytes) es la unidad mínima de transferencia**: cuando accedes a 1 byte, el hardware trae 64 bytes completos. Si tu struct tiene 65 bytes, dos accesos a campos distintos pueden traer 2 cache lines. Alinear structs a 64 bytes y agrupar campos accedidos juntos (hot/cold splitting) mejora la eficiencia de cache.
- **La jerarquía se aplica también a software**: OS page cache es una "cache" de disco en RAM. CDN es una "cache" del servidor de origen geográficamente distribuida. Redis/Memcached es una "cache" de la base de datos en RAM. El principio es siempre el mismo: mantener los datos más accedidos en el nivel más rápido.
