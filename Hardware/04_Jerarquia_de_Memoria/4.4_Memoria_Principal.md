# 4.4 Memoria Principal

## Definición

La memoria principal (RAM) es el almacenamiento volátil de acceso aleatorio donde residen los programas en ejecución y sus datos. Se implementa con DRAM (Dynamic RAM), que almacena cada bit como carga en un capacitor. SRAM (Static RAM), más rápida pero más cara, se usa exclusivamente para caches. La latencia (~50-100ns) y el ancho de banda (~50-100 GB/s) de la RAM son factores limitantes del rendimiento.

## Mecánica interna

### DRAM vs SRAM

```
  SRAM (Static RAM):
  ┌──────────────────────────────────────────────┐
  │ 6 transistores por bit (6T cell)             │
  │ Retiene dato mientras haya alimentación      │
  │ No necesita refresh                          │
  │ Latencia: ~1-10ns                            │
  │ Densidad: baja (6 transistores/bit)          │
  │ Costo: ~$10/MB (estimado)                    │
  │ Uso: caches (L1, L2, L3), register files     │
  └──────────────────────────────────────────────┘

  DRAM (Dynamic RAM):
  ┌──────────────────────────────────────────────┐
  │ 1 transistor + 1 capacitor por bit (1T1C)    │
  │ El capacitor pierde carga → necesita refresh │
  │ Refresh: cada 32-64ms, fila por fila         │
  │ Latencia: ~50-100ns                          │
  │ Densidad: alta (1 transistor/bit)            │
  │ Costo: ~$0.003/MB (~3$/GB)                   │
  │ Uso: memoria principal (DDR4, DDR5)          │
  └──────────────────────────────────────────────┘

  Ratio densidad: DRAM es ~6× más densa que SRAM.
  Ratio costo: DRAM es ~3000× más barata por bit.
  Ratio velocidad: SRAM es ~10-100× más rápida.

  Por esto la jerarquía existe: SRAM para cache (rápida, cara, pequeña),
  DRAM para memoria principal (lenta, barata, grande).
```

### Organización interna de DRAM

```
  Un chip DRAM se organiza como una matriz 2D de celdas:

  ┌─────────────────────────────────────┐
  │           DRAM Bank                  │
  │                                     │
  │  Filas (rows):     65536 (64K)      │
  │  Columnas (cols):  1024-8192        │
  │  Cada celda:       1 bit            │
  │                                     │
  │  Row address ──▶ Row Decoder        │
  │                       │             │
  │                  ┌────┴────┐        │
  │  Sense Amps:     │ Row     │        │
  │  (row buffer)    │ Buffer  │        │
  │                  └────┬────┘        │
  │                       │             │
  │  Col address ──▶ Column Mux ──▶ Data│
  └─────────────────────────────────────┘

  Acceso a DRAM:
  1. Activate (ACT): abrir una fila → cargar en row buffer (~13ns = tRCD)
  2. Read/Write: acceder columnas desde el row buffer (~13ns = CL)
  3. Precharge: cerrar la fila, preparar para la siguiente (~13ns = tRP)

  Total latencia (miss): tRCD + CL + tRP ≈ 40ns
  Si la fila ya está abierta (row buffer hit): solo CL ≈ 13ns


  DDR (Double Data Rate):
  Transfiere datos en AMBOS flancos del reloj (rising + falling).
  DDR5-6400: frecuencia real = 3200 MHz, tasa efectiva = 6400 MT/s

  ┌──────────┬───────────┬───────────┬───────────────┐
  │ Tipo     │ Clock real│ Data rate │ BW (por canal) │
  ├──────────┼───────────┼───────────┼───────────────┤
  │ DDR4-3200│ 1600 MHz  │ 3200 MT/s │ 25.6 GB/s     │
  │ DDR5-4800│ 2400 MHz  │ 4800 MT/s │ 38.4 GB/s     │
  │ DDR5-6400│ 3200 MHz  │ 6400 MT/s │ 51.2 GB/s     │
  └──────────┴───────────┴───────────┴───────────────┘

  Bandwidth total = BW por canal × número de canales:
  Dual channel DDR5-6400: 2 × 51.2 = 102.4 GB/s
  Quad channel DDR5-6400: 4 × 51.2 = 204.8 GB/s (server)
```

### Latencia vs Ancho de banda

```
  Latencia = tiempo para obtener el PRIMER byte de un acceso.
  Ancho de banda = bytes por segundo en transferencia sostenida.

  Analogía: latencia es el tiempo para que llegue el primer vagón
  del tren. Ancho de banda es cuánta carga transporta el tren por hora.

  DRAM latencia: ~50-100ns (apenas ha mejorado en 20 años)
  DRAM bandwidth: ha mejorado ~10× en 20 años (DDR→DDR5)

  ¿Cuándo importa cada una?

  Latencia-sensitive:
  - Pointer chasing (linked lists, trees, hash table probes)
  - Cada acceso depende del anterior → no se puede paralelizar
  - Random access to large dataset

  Bandwidth-sensitive:
  - Stream processing (recorrer arrays grandes)
  - Cada acceso es independiente → se pueden solapar
  - Multiple outstanding requests via prefetching

  Latency-Bandwidth Product:
  ¿Cuántos bytes "en vuelo" necesitas para saturar el ancho de banda?
  LBP = Latencia × Bandwidth
  LBP = 100ns × 100 GB/s = 10,000 bytes ≈ 156 cache lines

  Necesitas ~156 cache misses concurrentes en vuelo para saturar
  el bandwidth. Los CPUs logran esto con:
  - Hardware prefetcher: detecta patterns secuenciales
  - Out-of-order execution: múltiples loads en vuelo
  - MSHRs (Miss Status Holding Registers): ~10-16 misses concurrentes
```

### Memory channels e interleaving

```
  Canales de memoria multiplican el bandwidth:

  Single channel:  [DIMM] ──── Memory Controller ──── CPU
  Dual channel:    [DIMM] ──┬─ Memory Controller ──── CPU
                   [DIMM] ──┘
  Quad channel:    [DIMM] ──┬─ Memory Controller ──── CPU (server)
                   [DIMM] ──┤
                   [DIMM] ──┤
                   [DIMM] ──┘

  Interleaving:
  Las direcciones consecutivas se distribuyen entre canales:
  Dirección 0-63: Canal 0
  Dirección 64-127: Canal 1
  Dirección 128-191: Canal 0
  ...

  Un acceso secuencial grande usa AMBOS canales en paralelo → 2× bandwidth.
  Single channel limita severamente la performance (~50% en algunos benchmarks).
```

## Modelo mental

DRAM es un almacén gigante organizado en estanterías (banks) con cajones (rows). Para sacar un ítem, primero abres el cajón correcto (activate — 13ns), luego buscas el ítem en el cajón abierto (column access — 13ns). Si el cajón ya está abierto, solo necesitas buscar. Multi-channel es tener múltiples puertas de entrada al almacén: más gente puede entrar y salir simultáneamente (más bandwidth), pero el tiempo para sacar un ítem individual (latencia) no cambia.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Single-channel bandwidth | DIMM en un solo canal | Instalar DIMMs en ambos canales |
| High DRAM latency (~100ns+) | Row buffer misses frecuentes | Mejorar data locality en memoria |
| Memory bandwidth saturado | Todos los cores accediendo a RAM | Optimizar cache utilization, menos misses |
| ECC errors | Bit flips por radiación cósmica o defecto | ECC RAM (server-grade), scrubbing |

## Insights de ingeniería

- **La latencia de DRAM es el "speed of light" del software**: ~50-100ns por acceso aleatorio, y no mejora significativamente entre generaciones. Si tu programa tiene accesos aleatorios a un dataset mayor que L3, estás limitado a ~10-20 millones de accesos/segundo por core, sin importar qué CPU uses. Por esto las bases de datos diseñan todo alrededor de acceso secuencial: B-trees con nodos de 4-16KB, columnar storage, prefetching.
- **ECC (Error Correcting Code) es obligatorio en servidores**: los rayos cósmicos y la radiación natural causan bit flips a una tasa de ~1 error por GB por ~100 horas. Con 512GB de RAM en un servidor, un bit flip cada ~12 minutos. Sin ECC, eso es data corruption silenciosa. ECC detecta y corrige errores de 1 bit, y detecta errores de 2 bits. Todo server-grade RAM es ECC.
- **DDR5 duplica el ancho de banda pero apenas mejora la latencia**: DDR4-3200 tiene ~15ns CL. DDR5-6400 tiene ~15ns CL también (compensado por frecuencia más alta). El beneficio de DDR5 es bandwidth para workloads streaming-heavy y más capacidad por DIMM, no menor latencia para accesos aleatorios.
