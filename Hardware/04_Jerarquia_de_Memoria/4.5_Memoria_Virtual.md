# 4.5 Memoria Virtual

## Definición

La memoria virtual es una abstracción del OS que da a cada proceso la ilusión de tener su propio espacio de direcciones contiguo y privado, independiente de la memoria física disponible. Se implementa mediante paginación: el espacio virtual se divide en páginas (típicamente 4KB) que se mapean a frames físicos en RAM. Las páginas no usadas pueden residir en disco (swap). La MMU (Memory Management Unit) del hardware traduce direcciones virtuales a físicas en cada acceso a memoria.

## Mecánica interna

### Paginación

```
  Dirección virtual → MMU → Dirección física

  Espacio virtual (por proceso):
  ┌──────────────────────────────┐ 0xFFFFFFFFFFFFFFFF
  │ Kernel space                 │ (mapeo compartido)
  ├──────────────────────────────┤ 0xFFFF800000000000 (Linux)
  │ User space                   │
  │ ┌──────────┐                 │
  │ │ Stack ↓  │                 │ (crece hacia abajo)
  │ │          │                 │
  │ │ (gap)    │                 │
  │ │          │                 │
  │ │ Heap ↑   │                 │ (crece hacia arriba)
  │ ├──────────┤                 │
  │ │ BSS      │                 │ (variables no inicializadas)
  │ │ Data     │                 │ (variables inicializadas)
  │ │ Text     │                 │ (código ejecutable)
  │ └──────────┘                 │
  └──────────────────────────────┘ 0x0000000000000000


  Traducción de dirección:
  ┌──────────────────────────────────────────────────┐
  │ Dirección virtual (48 bits en x86-64):           │
  │                                                  │
  │ ┌────────┬────────┬────────┬────────┬──────────┐ │
  │ │ PML4   │ PDPT   │  PD    │  PT    │ Offset   │ │
  │ │ (9 bit)│ (9 bit)│ (9 bit)│ (9 bit)│ (12 bit) │ │
  │ └───┬────┴───┬────┴───┬────┴───┬────┴────┬─────┘ │
  │     │        │        │        │         │       │
  │     ▼        ▼        ▼        ▼         │       │
  │   PML4    Page Dir  Page Dir  Page      │       │
  │   Table   Ptr Table Directory Table      │       │
  │   (512    (512      (512      (512       │       │
  │   entries) entries)  entries)  entries)   │       │
  │                                │         │       │
  │                         Physical Frame # │       │
  │                                + Offset ─┘       │
  │                                = Physical Address │
  └──────────────────────────────────────────────────┘

  4 niveles de tablas de páginas (x86-64):
  Cada nivel tiene 512 entries (9 bits), 12 bits de offset = 4KB páginas
  Espacio virtual total: 2^48 = 256 TB (solo 48 bits usados de 64)

  Costo de traducción sin TLB:
  4 accesos a memoria (uno por nivel de tabla) + 1 acceso al dato
  = 5 accesos × 100ns = 500ns por acceso (inaceptable)
  → Por esto existe el TLB (tema 4.6)
```

### Page faults

```
  Un page fault ocurre cuando la CPU accede a una dirección virtual
  cuya página NO está mapeada a un frame físico en RAM.

  Tipos:

  Minor page fault (soft):
  La página existe pero no tiene frame asignado.
  El OS asigna un frame, actualiza la page table, y continúa.
  Costo: ~1-10μs (solo operación de kernel, sin I/O de disco).
  Ejemplos:
  - Primera escritura a una página (copy-on-write)
  - Acceso a memoria recién mmaped (lazy allocation)

  Major page fault (hard):
  La página fue swapped out a disco. El OS debe leerla del swap.
  Costo: ~1-10ms (acceso a disco, 1000-10000× más lento que RAM).
  En SSD: ~100μs-1ms.
  En HDD: ~5-10ms.

  Invalid page fault:
  La dirección virtual no tiene mapeo válido.
  → Segmentation fault (SIGSEGV). El proceso se termina.
  Causas: null pointer dereference, acceso fuera de bounds,
  use-after-free.


  Copy-on-Write (COW):
  Optimización para fork():
  1. El proceso hijo comparte las mismas páginas físicas que el padre
  2. Ambas páginas se marcan como read-only
  3. Cuando uno escribe → page fault → el OS copia la página
     y asigna una nueva para el escritor

  fork() sin COW: copiar 1GB de memoria → ~100ms
  fork() con COW: compartir y copiar bajo demanda → ~1ms
  (solo copia las páginas que realmente se modifican)
```

### Huge Pages

```
  Páginas estándar: 4KB (2^12)
  Huge Pages: 2MB (2^21) o 1GB (2^30)

  Beneficios:
  1. Menos entries en TLB: 1 TLB entry cubre 2MB en vez de 4KB
     → 512× menos TLB misses para el mismo working set
  2. Menos niveles de page table walk: 2MB pages usan 3 niveles, no 4
  3. Menos overhead de page tables: 512× menos entries

  Cuándo usar:
  - Bases de datos con buffer pools grandes (PostgreSQL, MySQL)
  - VMs (cada VM tiene GBs de memoria)
  - Aplicaciones HPC con datasets grandes

  Linux:
  - Transparent Huge Pages (THP): el kernel promociona páginas
    automáticamente a 2MB cuando detecta acceso secuencial.
  - HugeTLB pages: reserva explícita con hugetlbfs.

  Problemas de THP:
  - Compaction: el kernel necesita 512 páginas contiguas de 4KB
    para crear una de 2MB → puede causar latency spikes.
  - Memory waste: una página de 2MB medio vacía desperdicia más
    que una de 4KB medio vacía.
  - Para bases de datos latency-sensitive, HugeTLB explícito
    es preferido sobre THP (evita compaction stalls).
```

## Modelo mental

La memoria virtual es una tabla de traducción: cada proceso tiene su propio "directorio telefónico" (page table) que mapea direcciones virtuales a direcciones físicas. Dos procesos pueden usar la misma dirección virtual (0x7FFF0000) apuntando a frames físicos diferentes → aislamiento total. Las páginas son "billetes" que pueden estar en RAM (disponibles) o en disco (swap, necesitan "cobrarse"). El TLB es la cache de este directorio: evita mirar el directorio completo en cada llamada.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Thrashing (swap in/out constante) | RAM insuficiente, swapping agresivo | Más RAM, reducir working set, swap off |
| Latency spikes | THP compaction | Deshabilitar THP, usar HugeTLB explícito |
| OOM kill | Proceso excede memoria disponible | cgroups memory limits, más RAM |
| SEGFAULT | Acceso a dirección no mapeada | Fix null pointer, bounds checking |

## Insights de ingeniería

- **Overcommit es la norma en Linux**: Linux permite que los procesos reserven más memoria virtual que la RAM física disponible (overcommit). Asume que no todos usarán toda la memoria reservada. Si se equivoca → OOM killer mata un proceso. Controlable con `vm.overcommit_memory` y `vm.overcommit_ratio`.
- **mmap es la interfaz entre archivo y memoria**: `mmap()` mapea un archivo directamente al espacio virtual del proceso. Leer de esa dirección → page fault → el OS lee el bloque del archivo a un page frame → transparent access. Es como si el archivo fuera un array en memoria. Bases de datos (LMDB, SQLite WAL) y el page cache del OS usan mmap extensivamente.
- **ASLR (Address Space Layout Randomization)** aleatoriza la posición del stack, heap, y librerías en el espacio virtual para dificultar exploits. Cada ejecución del programa tiene un layout diferente. Un atacante no puede predecir la dirección de una función o variable.
