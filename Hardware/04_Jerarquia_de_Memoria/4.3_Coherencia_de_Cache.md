# 4.3 Coherencia de Cache

## Definición

La coherencia de cache es el problema de mantener una vista consistente de la memoria cuando múltiples cores tienen copias del mismo dato en sus caches privadas. Si Core 0 modifica un dato en su L1, Core 1 (que tiene una copia del mismo dato) debe ser informado. El protocolo MESI (Modified, Exclusive, Shared, Invalid) es el mecanismo hardware estándar que garantiza coherencia entre caches de un sistema multicore.

## Mecánica interna

### El problema

```
  Core 0 L1: X = 5
  Core 1 L1: X = 5        (ambos tienen copia de X)

  Core 0 escribe: X = 10
  Core 0 L1: X = 10       ← actualizado
  Core 1 L1: X = 5        ← STALE! Dato inconsistente

  Sin coherencia: Core 1 lee X = 5 (incorrecto).
  Con coherencia (MESI): Core 1 es notificado de la invalidación,
  y cuando lee X, obtiene el valor actualizado (10).
```

### Protocolo MESI

```
  Cada cache line tiene un estado de 2 bits:

  ┌───────────┬────────────────────────────────────────────────────┐
  │ Estado    │ Significado                                        │
  ├───────────┼────────────────────────────────────────────────────┤
  │ Modified  │ Solo este core tiene la línea.                     │
  │ (M)       │ Ha sido modificada (dirty).                        │
  │           │ La memoria principal tiene un valor STALE.         │
  │           │ Debe escribirse a memoria antes de compartirse.    │
  ├───────────┼────────────────────────────────────────────────────┤
  │ Exclusive │ Solo este core tiene la línea.                     │
  │ (E)       │ NO ha sido modificada (clean).                     │
  │           │ Puede pasar a Modified sin bus transaction.        │
  ├───────────┼────────────────────────────────────────────────────┤
  │ Shared    │ Múltiples cores tienen la línea.                   │
  │ (S)       │ Read-only. Para escribir, debe invalidar las demás.│
  ├───────────┼────────────────────────────────────────────────────┤
  │ Invalid   │ La línea no es válida (no tiene dato útil).        │
  │ (I)       │ Un acceso a esta línea causa un cache miss.        │
  └───────────┴────────────────────────────────────────────────────┘


  Transiciones:

  Core 0 lee X (miss):
  1. Envía "Read" al bus
  2. Si ningún otro core tiene X → trae de RAM, estado = Exclusive
  3. Si otro core tiene X (Shared o Exclusive) → ambos pasan a Shared
  4. Si otro core tiene X en Modified → ese core escribe a RAM (write-back),
     ambos pasan a Shared

  Core 0 escribe X (hit, estado Exclusive):
  - Pasa directamente a Modified (sin bus transaction — nadie más lo tiene)

  Core 0 escribe X (hit, estado Shared):
  1. Envía "Invalidate" al bus
  2. Todos los otros cores invalidan su copia (I)
  3. Core 0 pasa a Modified

  Core 0 escribe X (miss):
  1. Envía "Read-Invalidate" al bus
  2. Trae la línea y la marca como Modified
  3. Otros cores invalidan su copia


  Diagrama de estados:

       Read hit          Write hit
    ┌───────────┐     ┌───────────┐
    │           │     │           │
    ▼           │     ▼           │
  ┌───┐  Read  ┌───┐ Write ┌───┐
  │ I │───────▶│ E │──────▶│ M │
  └───┘        └───┘       └───┘
    ▲            │           │
    │   Snoop    │  Snoop    │
    │  Invalidate│  Read     │
    └────────────┘  ┌───┐   │
                    │ S │◀──┘ (write-back + share)
                    └───┘
```

### Bus snooping vs Directory-based

```
  Bus snooping (Intel pre-Nehalem, sistemas pequeños):
  Cada core escucha (snoop) TODAS las transacciones del bus.
  Si ve una invalidación para una línea que tiene → la invalida.

  + Simple.
  - No escala: con 64+ cores, el bus se satura con tráfico de snoop.


  Directory-based (Intel Nehalem+, AMD, ARM):
  Un directorio central mantiene un registro de qué core tiene
  qué cache line y en qué estado.

  Core 0 quiere escribir X:
  1. Consulta el directorio: "¿quién tiene X?"
  2. Directorio responde: "Core 3 y Core 7 en Shared"
  3. Core 0 envía invalidación SOLO a Core 3 y Core 7
  4. No molesta a los otros 62 cores

  + Escala a muchos cores (solo tráfico punto a punto).
  - Más complejo, directorio ocupa espacio (tags por línea).
  - Latencia ligeramente mayor para lookup del directorio.

  En Intel Skylake+:
  L3 cache actúa como directorio (snoop filter).
  L3 sabe qué core tiene qué línea → envía invalidaciones dirigidas.
```

### False sharing

```
  False sharing: dos cores escriben a variables DIFERENTES que
  comparten la MISMA cache line (64 bytes).

  struct Counters {
      int counter_thread0;    // offset 0
      int counter_thread1;    // offset 4
  };

  Thread 0 en Core 0: counter_thread0++
  Thread 1 en Core 1: counter_thread1++

  Ambos modifican la MISMA cache line → protocolo MESI invalida
  la línea en el otro core en CADA escritura.

  Core 0: M → (invalidation from Core 1) → I → (read) → S → (write) → M
  Core 1: M → (invalidation from Core 0) → I → (read) → S → (write) → M

  Esto se repite en cada escritura: "ping-pong" de la cache line
  entre los dos cores. Latencia: ~40-100ns por acceso
  (L3 roundtrip + invalidation) en vez de 1ns (L1 hit).

  Resultado: 40-100× más lento de lo esperado.

  Solución: padding
  struct Counters {
      alignas(64) int counter_thread0;   // cache line 0
      alignas(64) int counter_thread1;   // cache line 1 (separada)
  };

  Ahora cada counter está en su propia cache line → no hay coherence traffic.

  En Java: @sun.misc.Contended
  En C++: alignas(std::hardware_destructive_interference_size)
```

## Modelo mental

La coherencia de cache es un contrato entre cores: "todos vemos la misma memoria". MESI implementa ese contrato con un protocolo de estados. Cuando un core modifica un dato, le dice a los demás "tu copia ya no vale". El costo de ese mensaje es ~40ns (L3 roundtrip). False sharing es pagar ese costo innecesariamente: los cores no comparten datos, pero el hardware no lo sabe porque comparten cache line. Es como dos personas en una biblioteca que piden el mismo libro solo porque sus notas están en la misma página — ninguno usa las notas del otro, pero ambos tienen que esperar.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Throughput no escala con cores | False sharing entre threads | Padding a cache line boundaries |
| Latencia alta en datos compartidos | Cache line ping-pong (MESI invalidations) | Reducir sharing, RCU para read-heavy |
| Memory bandwidth saturado | Exceso de coherence traffic | Partición de datos por core |
| Performance degrada con más threads | Lock contention + coherence overhead | Lock-free data structures, per-thread data |

## Insights de ingeniería

- **False sharing es uno de los bugs de rendimiento más insidiosos**: el programa es correctamente funcional, pero el rendimiento es 10-100× peor de lo esperado. No aparece en profiles como "cache miss" convencional; aparece como high L3 latency o high HITM (Hit Modified) en `perf c2c`. Siempre padea estructuras compartidas entre threads.
- **Read-heavy workloads son baratos en MESI**: si todos los cores leen un dato (Shared), no hay invalidaciones. El costo solo aparece en escrituras. Esto es por qué patrones como RCU (Read-Copy-Update) en el kernel de Linux son eficientes: los lectores nunca invalidan caches; solo el escritor (que es raro) paga el costo.
- **Atomic operations son costosas por coherencia**: un `atomic_increment` en una variable compartida entre cores requiere: (1) adquirir la línea en Modified, (2) invalidar en todos los otros cores, (3) ejecutar la operación atómica. El costo es ~40-100ns por operación. Para contadores de alta frecuencia, usa contadores per-thread y suma al final.
