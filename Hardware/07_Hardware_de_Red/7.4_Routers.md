# 7.4 Routers

## Definición

Un router opera en capa 3 (red) del modelo OSI. Recibe paquetes IP, examina la dirección IP de destino, consulta su tabla de enrutamiento, y reenvía el paquete hacia el siguiente salto (next hop) en el camino hacia el destino. Los routers conectan redes diferentes (subnets) y toman decisiones de forwarding basadas en prefijos IP, a diferencia de los switches que operan con direcciones MAC dentro de una misma red.

## Mecánica interna

### Forwarding vs Control Plane

```
  ┌─────────────────────────────────────────────┐
  │              Control Plane                   │
  │  ┌───────────────────────────────────────┐  │
  │  │ Routing Protocols (OSPF, BGP, IS-IS)  │  │
  │  │ → Construyen la Routing Table          │  │
  │  │ → Software, CPU general               │  │
  │  │ → Tiempo: segundos a minutos          │  │
  │  └─────────────┬─────────────────────────┘  │
  │                │ instala rutas               │
  │                ▼                             │
  │  ┌───────────────────────────────────────┐  │
  │  │           FIB (Forwarding Table)      │  │
  │  └───────────────────────────────────────┘  │
  │                │                             │
  │ ───────────────┼─────────────────────────── │
  │                ▼                             │
  │              Data Plane                      │
  │  ┌───────────────────────────────────────┐  │
  │  │ Forwarding Engine (ASIC/FPGA)         │  │
  │  │ → Longest prefix match en FIB         │  │
  │  │ → Hardware, nanosegundos              │  │
  │  │ → Millones de paquetes/segundo        │  │
  │  └───────────────────────────────────────┘  │
  └─────────────────────────────────────────────┘

  Separación clave:
  - Control Plane: LENTO, inteligente. Decide las rutas.
    Ejecuta protocolos de routing, responde a cambios de topología.
  - Data Plane: RÁPIDO, simple. Reenvía paquetes.
    Lookup en FIB, decrementa TTL, recalcula checksum, envía.
```

### Routing Table y Longest Prefix Match

```
  Routing Table:
  ┌──────────────────┬─────────────┬───────────┬─────────┐
  │ Destino (prefix) │ Next Hop    │ Interfaz  │ Métrica │
  ├──────────────────┼─────────────┼───────────┼─────────┤
  │ 10.0.1.0/24      │ directly    │ eth0      │ 0       │
  │ 10.0.2.0/24      │ 10.0.1.1    │ eth0      │ 10      │
  │ 172.16.0.0/16    │ 10.0.1.254  │ eth0      │ 20      │
  │ 0.0.0.0/0        │ 10.0.1.1    │ eth0      │ 100     │
  └──────────────────┴─────────────┴───────────┴─────────┘

  Longest Prefix Match:
  Paquete destino: 172.16.5.10
  Coincide con: 172.16.0.0/16 (/16 = 16 bits)
  Coincide con: 0.0.0.0/0 (/0 = 0 bits, default route)
  → Gana 172.16.0.0/16 (prefijo más largo = más específico)

  Implementación en hardware:
  TCAM (Ternary Content-Addressable Memory):
  Busca TODOS los prefijos en paralelo en un solo ciclo de reloj.
  Cada entrada TCAM puede ser 0, 1, o X (don't care).
  Resultado: lookup en O(1) independiente del tamaño de la tabla.
  Limitación: TCAM es caro y consume mucha energía.
  Tabla BGP completa (~1M prefijos) requiere TCAM grande.
```

### NAT (Network Address Translation)

```
  NAT traduce direcciones IP privadas a públicas:

  Red interna:          Router (NAT):         Internet:
  192.168.1.10:5000 ──▶ 203.0.113.1:40000 ──▶ 8.8.8.8:53
  192.168.1.11:5001 ──▶ 203.0.113.1:40001 ──▶ 8.8.8.8:53

  NAT Table:
  ┌──────────────────┬──────────────────────┐
  │ Interno          │ Externo              │
  ├──────────────────┼──────────────────────┤
  │ 192.168.1.10:5000│ 203.0.113.1:40000    │
  │ 192.168.1.11:5001│ 203.0.113.1:40001    │
  └──────────────────┴──────────────────────┘

  El router modifica los headers IP/TCP de cada paquete
  (source IP + port en outbound, dest IP + port en inbound).
  Recalcula checksums de IP y TCP.

  Tipos:
  - SNAT (Source NAT): cambia IP origen (lo más común, home routers)
  - DNAT (Destination NAT): cambia IP destino (port forwarding, load balancing)
  - PAT (Port Address Translation): muchos internos → 1 IP pública (overloaded NAT)

  Limitaciones:
  - Rompe end-to-end connectivity (hosts internos no son alcanzables directamente)
  - Estado: el router debe mantener la NAT table en memoria
  - Protocolos que embeben IPs en payload (SIP, FTP) necesitan ALG
```

## Modelo mental

Un router es un despachador de correo internacional. Recibe paquetes con dirección de destino (IP), consulta su mapa de rutas (routing table), y los envía por la puerta correcta (interfaz) hacia el siguiente punto de distribución (next hop). El control plane es el departamento de cartografía que actualiza los mapas cuando se abren o cierran caminos. El data plane es el operario que clasifica y reenvía sobres mecánicamente a alta velocidad.

## Insights de ingeniería

- **El data plane de routers modernos es puro hardware**: un router de core (Cisco NCS, Juniper PTX) puede reenviar billones de paquetes/segundo usando ASICs dedicados. El CPU general solo se usa para el control plane (protocolos de routing, management). Esto es lo que permite que Internet funcione a la escala actual.
- **NAT a alta escala es un bottleneck**: cada conexión a través de NAT requiere una entrada en la tabla de estado. Un carrier-grade NAT (CGNAT) con millones de suscriptores puede tener decenas de millones de entradas. El lookup y la modificación de paquetes por conexión consume recursos significativos, especialmente para tráfico UDP sin estado claro.
