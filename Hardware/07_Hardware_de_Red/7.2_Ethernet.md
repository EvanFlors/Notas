# 7.2 Ethernet

## Definición

Ethernet es el estándar de capa 2 (enlace de datos) dominante para redes locales (LAN). Define el formato del frame, el método de acceso al medio, y las velocidades de transmisión. Especificado en IEEE 802.3, opera sobre cable de cobre (twisted pair) o fibra óptica, con velocidades desde 10 Mbps hasta 400 Gbps.

## Mecánica interna

### Frame Ethernet

```
  ┌──────────┬──────────┬──────┬──────────┬──────────────┬─────┐
  │ Preamble │ Dest MAC │ Src  │EtherType │   Payload    │ FCS │
  │ 8 bytes  │ 6 bytes  │ MAC  │ 2 bytes  │ 46-1500 bytes│4 B  │
  │          │          │6 B   │          │              │     │
  └──────────┴──────────┴──────┴──────────┴──────────────┴─────┘

  Preamble (8 bytes): sincronización de reloj entre transmisor y receptor.
  7 bytes de 10101010 + 1 byte SFD (Start Frame Delimiter) 10101011.

  Destination MAC (6 bytes): dirección del receptor.
  Source MAC (6 bytes): dirección del transmisor.

  EtherType (2 bytes): identifica el protocolo de capa 3.
  0x0800 = IPv4
  0x86DD = IPv6
  0x0806 = ARP
  0x8100 = VLAN tag (802.1Q)

  Payload (46-1500 bytes): datos de capa 3+.
  Mínimo 46 bytes (padding si el dato es menor).
  Máximo 1500 bytes = MTU (Maximum Transmission Unit).

  FCS (4 bytes): Frame Check Sequence (CRC-32).
  Detecta errores de transmisión. Si CRC no coincide → frame descartado.
```

### MTU y Jumbo Frames

```
  MTU estándar: 1500 bytes de payload.

  Overhead por frame:
  - Ethernet header: 14 bytes (sin VLAN) o 18 bytes (con VLAN)
  - IP header: 20 bytes
  - TCP header: 20 bytes
  - Payload útil: 1500 - 40 = 1460 bytes (TCP MSS)

  Eficiencia: 1460/1538 = 94.9% (incluyendo preamble + IFG)

  Para transferir 1 GB:
  1GB / 1460 bytes = 718,000 paquetes
  718K paquetes × overhead por paquete (interrupciones, headers, processing)

  Jumbo Frames: MTU de 9000 bytes.
  1 GB / 8960 bytes = 116,000 paquetes
  → 6× menos paquetes → 6× menos overhead de CPU.

  Requisito: TODOS los dispositivos en el path deben soportar jumbo frames.
  Si un switch intermedio tiene MTU 1500 → fragmentación o drops.

  Configurar:
  ip link set eth0 mtu 9000

  Uso típico: redes de almacenamiento (iSCSI, NFS), clusters HPC,
  comunicación intra-datacenter.
```

```
  ┌──────────────┬────────────┬───────────────────────────┐
  │ Estándar     │ Velocidad  │ Medio típico              │
  ├──────────────┼────────────┼───────────────────────────┤
  │ 10BASE-T     │ 10 Mbps    │ Cat3 UTP                  │
  │ 100BASE-TX   │ 100 Mbps   │ Cat5 UTP                  │
  │ 1000BASE-T   │ 1 Gbps     │ Cat5e/Cat6 UTP            │

  │ 25GBASE-SR   │ 25 Gbps    │ Fibra multimodo           │
  │ 40GBASE-SR4  │ 40 Gbps    │ 4× fibra multimodo        │
  │ 100GBASE-SR4 │ 100 Gbps   │ 4×25G fibra               │

  En datacenters modernos:
  - Servidor a ToR switch: 25 Gbps o 100 Gbps
  - ToR a spine switch: 100 Gbps o 400 Gbps
  - Spine a spine: 400 Gbps
```

## Modelo mental

Ethernet es el sistema postal de la red local. Cada frame es un sobre con dirección de destino (MAC), remitente, tipo de contenido (EtherType), el contenido (payload), y un sello de verificación (CRC). El MTU es el tamaño máximo del sobre. Jumbo frames son sobres tamaño A3 en vez de carta estándar: caben más datos, pero todos los buzones del camino deben aceptar ese tamaño.

## Insights de ingeniería

- **MTU mismatch es un bug silencioso**: si un enlace en el path tiene MTU menor, los paquetes se fragmentan (IPv4) o se descartan con ICMP "Packet Too Big" (IPv6). Con PMTUD (Path MTU Discovery) bloqueado por firewalls, los paquetes grandes simplemente desaparecen. Síntoma: conexiones TCP se establecen (SYN/ACK son pequeños) pero transferencias grandes fallan.
- **El overhead de paquetes pequeños domina en alta velocidad**: a 100 Gbps, el máximo teórico es 148.8M paquetes/segundo (frames mínimos de 64 bytes). Procesar 148M interrupciones/segundo es imposible para cualquier CPU. Por eso NICs de alta velocidad dependen de batching (NAPI, GRO), jumbo frames, y offloading.
