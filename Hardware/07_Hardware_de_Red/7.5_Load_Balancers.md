# 7.5 Load Balancers

## Definición

Un load balancer distribuye tráfico de red entre múltiples servidores backend para maximizar throughput, minimizar latencia, y garantizar disponibilidad. Opera en capa 4 (L4, transporte: TCP/UDP) o capa 7 (L7, aplicación: HTTP/HTTPS). La diferencia fundamental es la profundidad de inspección del paquete y la granularidad de las decisiones de distribución.

## Mecánica interna

### L4 vs L7

```
  L4 Load Balancer:
  ┌────────┐      ┌──────────┐      ┌──────────┐
  │ Client │─TCP─▶│ L4 LB    │─TCP─▶│ Backend1 │
  │        │      │          │      │ Backend2 │
  │        │      │ Decide   │      │ Backend3 │
  └────────┘      │ por:     │      └──────────┘
                  │ - IP src │
                  │ - IP dst │
                  │ - Port   │
                  │ - Proto  │
                  └──────────┘

  - No inspecciona payload (no entiende HTTP, solo TCP/UDP)
  - Decisión por conexión (no por request)
  - Muy rápido: puede implementarse en hardware/kernel (IPVS, DPDK)
  - Latencia: ~microsegundos
  - Throughput: millones de conexiones/segundo

  Modos:
  - DSR (Direct Server Return): LB solo toca tráfico de entrada,
    respuesta va directamente del backend al cliente.
    → Ideal para tráfico asimétrico (request pequeño, response grande)
  - NAT: LB reescribe IP destino (DNAT) en entrada,
    IP origen (SNAT) en salida. Todo pasa por el LB.
  - Tunneling (IP-in-IP): LB encapsula paquete y lo envía al backend.


  L7 Load Balancer:
  ┌────────┐      ┌──────────┐      ┌──────────┐
  │ Client │─TLS─▶│ L7 LB    │─HTTP▶│ Backend1 │
  │        │      │          │      │ Backend2 │
  │        │      │ Decide   │      │ Backend3 │
  └────────┘      │ por:     │      └──────────┘
                  │ - URL    │
                  │ - Headers│
                  │ - Cookie │
                  │ - Method │
                  └──────────┘

  - Inspecciona HTTP (termina TLS, parsea headers)
  - Decisión por REQUEST (no por conexión)
  - Puede rutear /api → pool A, /static → pool B
  - Puede añadir/modificar headers (X-Forwarded-For)
  - Más lento que L4 (TLS termination + HTTP parsing)
  - Latencia: ~cientos de microsegundos a milisegundos
  - Ejemplos: NGINX, HAProxy, Envoy, ALB (AWS)
```

### Algoritmos de distribución

```
  Round Robin: 1→2→3→1→2→3→...
  Simple, no considera carga actual. Funciona bien si backends son homogéneos.

  Weighted Round Robin: backend con peso 3 recibe 3× más que peso 1.
  Útil cuando backends tienen diferente capacidad.

  Least Connections: envía al backend con menos conexiones activas.
  Mejor para requests de duración variable.

  IP Hash: hash(client IP) % N → siempre al mismo backend.
  Sticky sessions sin cookies.
  Problema: si un backend cae, se redistribuye todo (no solo el tráfico de ese backend).

  Consistent Hashing: hash ring con N puntos por backend.
  Si un backend cae, solo su tráfico se redistribuye (K/N).
  Usado en L4 (Maglev de Google, Katran de Facebook).

  Maglev Hashing (Google):
  Tabla de lookup de tamaño primo. Cada backend "llena" posiciones.
  Lookup O(1). Mínima redistribución cuando backends cambian.
  Usado en Google Front End (GFE) para millones de conexiones/segundo.
```

### Health Checks

```
  El LB monitorea la salud de los backends:

  - TCP check: ¿puede establecer conexión TCP? (básico)
  - HTTP check: GET /health → ¿responde 200? (aplicación)
  - gRPC check: rpc Check → ¿responde SERVING?

  Si un backend falla health checks:
  1. Se marca como unhealthy
  2. Se deja de enviar tráfico nuevo
  3. Conexiones existentes se drenan (drain) o se cierran
  4. Cuando vuelve a pasar health checks → se reincorpora gradualmente
```

## Modelo mental

Un L4 load balancer es un portero de hotel que asigna habitaciones basándose solo en el número de reserva (IP + puerto). Un L7 load balancer es un concierge que lee la solicitud del huésped ("quiero spa" → ala este, "quiero restaurante" → ala oeste) y lo dirige al lugar apropiado.

## Insights de ingeniería

- **L4 en el kernel escala masivamente**: Facebook Katran (basado en XDP/eBPF) maneja millones de paquetes/segundo por core sin pasar por el network stack del kernel. Google Maglev opera a 10 Gbps por máquina con consistent hashing. Para tráfico volumétrico (DDoS mitigation, CDN frontend), L4 es obligatorio.
- **L7 es necesario para routing inteligente**: microservicios modernos necesitan routing basado en path, headers, o contenido. Envoy (usado en Istio service mesh) hace L7 load balancing con observabilidad (tracing, métricas por ruta). El costo es mayor latencia y consumo de CPU por la terminación TLS y parsing HTTP.
