# 7.6 Firewalls

## Definición

Un firewall es un dispositivo (hardware o software) que filtra tráfico de red basándose en reglas de seguridad. Examina paquetes entrantes y salientes y decide permitirlos, rechazarlos (con respuesta), o descartarlos (sin respuesta) según criterios como IP origen/destino, puerto, protocolo, y estado de la conexión.

## Mecánica interna

### Tipos de firewalls

```
  1. Packet Filter (stateless):
  Examina cada paquete individualmente contra una lista de reglas.
  No tiene memoria de conexiones anteriores.

  Regla ejemplo:
  ALLOW TCP src=any dst=10.0.1.5 dport=443    ← permitir HTTPS entrante
  ALLOW TCP src=10.0.1.5 sport=443 dst=any     ← permitir respuestas
  DENY ALL                                      ← denegar todo lo demás

  Problema: necesita reglas explícitas para tráfico de respuesta.
  Difícil de mantener con aplicaciones complejas.


  2. Stateful Firewall:
  Mantiene una tabla de estado (connection tracking) con todas
  las conexiones activas.

  Connection Table:
  ┌──────────┬──────────┬───────┬───────┬────────┐
  │ Src IP   │ Dst IP   │ SPort │ DPort │ State  │
  ├──────────┼──────────┼───────┼───────┼────────┤
  │10.0.1.10 │93.184.2.1│ 45000 │ 443   │ ESTAB  │
  │10.0.1.11 │8.8.8.8   │ 52000 │ 53    │ NEW    │
  └──────────┴──────────┴───────┴───────┴────────┘

  Reglas simplificadas:
  ALLOW TCP dst=10.0.1.5 dport=443 state=NEW    ← nueva conexión HTTPS
  ALLOW state=ESTABLISHED,RELATED                ← tráfico de conexiones existentes
  DENY ALL

  El tráfico de respuesta se permite automáticamente porque la conexión
  está en estado ESTABLISHED en la tabla.

  Linux: conntrack (nf_conntrack) implementa connection tracking.
  Cada entrada consume ~300 bytes. Default: 65536 entradas.
  Servidores con muchas conexiones: sysctl net.nf_conntrack_max=1000000


  3. Next-Generation Firewall (NGFW):
  - Deep Packet Inspection (DPI): inspecciona payload
  - Application awareness: identifica aplicaciones (YouTube, Slack)
  - IPS integrado (Intrusion Prevention System)
  - TLS inspection (man-in-the-middle con CA propia)
  - URL filtering, antimalware
```

### Linux: iptables/nftables

```
  iptables (legacy) / nftables (moderno):

  Chains (puntos de inspección):

  Paquete entrante:
  → PREROUTING → routing decision → INPUT → proceso local
                                  → FORWARD → POSTROUTING → sale

  Paquete saliente:
  proceso local → OUTPUT → routing decision → POSTROUTING → sale

  Tables:
  - filter: accept/drop (la más común)
  - nat: SNAT/DNAT
  - mangle: modificar headers
  - raw: bypass conntrack

  Ejemplo con nftables:
  nft add rule inet filter input tcp dport 22 accept
  nft add rule inet filter input tcp dport 443 accept
  nft add rule inet filter input ct state established,related accept
  nft add rule inet filter input drop
```

### Firewall en hardware vs software

```
  Hardware (appliance):
  - ASICs dedicados para packet inspection
  - Throughput: 10-100+ Gbps
  - Latencia: ~microsegundos
  - Cisco Firepower, Palo Alto, Fortinet
  - Caro, pero necesario para perímetro de red de alto throughput

  Software (host-based):
  - iptables/nftables (Linux), pf (BSD), Windows Firewall
  - Ejecuta en CPU general
  - Throughput limitado por CPU
  - Flexible, gratuito, ideal para host-level security

  Cloud:
  - Security Groups (AWS), NSG (Azure): stateful, por instancia
  - Network ACL (AWS): stateless, por subnet
  - Implementados en el hypervisor/SDN, no hay appliance física
```

## Modelo mental

Un firewall stateless es un guardia de seguridad que revisa cada persona contra una lista de criterios (ID, destino) sin recordar quién ya pasó. Un firewall stateful es un guardia con memoria: si dejó salir a alguien, recuerda su cara y lo deja volver sin preguntar. Un NGFW es un guardia que además abre los paquetes, lee el contenido, y decide si el material es aceptable.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Conexiones timeout aleatorias | conntrack table llena | Aumentar `nf_conntrack_max` |
| Throughput degradado | DPI en firewall sobrecargado | Bypass para tráfico trusted, hardware upgrade |
| App funciona local pero no remoto | Firewall bloquea puerto | Verificar reglas, `tcpdump` para confirmar drops |

## Insights de ingeniería

- **conntrack es el bottleneck oculto de Linux networking**: cada paquete que pasa por iptables/nftables hace lookup en la tabla conntrack. A alta tasa de paquetes (millones/segundo), el lock contention en conntrack se vuelve un cuello de botella. Para servidores con millones de conexiones concurrentes (load balancers, proxies), se usa `NOTRACK` para bypass selectivo o se reemplaza con soluciones eBPF/XDP que evitan conntrack completamente.
- **Firewalls cloud son fundamentalmente diferentes**: Security Groups en AWS son stateful pero implementados en el Nitro hypervisor, no como appliances. No tienen throughput limit visible porque operan a nivel de vNIC. Pero Network ACLs son stateless y aplican por subnet, no por instancia. Entender la diferencia evita configuraciones inseguras.
