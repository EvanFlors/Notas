# 10.1 Camino de Datos

## Definición

El camino de datos (data path) describe el recorrido físico y lógico que siguen los datos desde que llegan a un sistema (por red o disco) hasta que son procesados y la respuesta es enviada de vuelta. Entender este camino end-to-end es esencial para identificar dónde se introduce latencia y dónde ocurren copias innecesarias de datos.

## Mecánica interna

### Request de red completo

```
  Cliente envía HTTP request → Servidor procesa → Responde

  1. LLEGADA DEL PAQUETE:

  NIC (hardware):
  ├─ Señal eléctrica/óptica llega al PHY
  ├─ PHY decodifica → frame Ethernet
  ├─ MAC engine verifica CRC, filtra por MAC address
  ├─ RSS calcula hash del 5-tuple → selecciona RX queue
  ├─ DMA engine escribe frame en ring buffer (RAM del host)
  ├─ Checksum offload verifica IP/TCP checksum
  └─ Genera interrupción MSI-X al core asignado a esa queue

  Kernel (software):
  ├─ ISR: marca NAPI como scheduled, deshabilita IRQ de esa queue
  ├─ softirq (NET_RX_SOFTIRQ): NAPI poll loop
  │   ├─ Lee paquetes del ring buffer (batch de hasta 64)
  │   ├─ Construye sk_buff (socket buffer) por paquete
  │   ├─ Network stack processing:
  │   │   ├─ L2: Ethernet header → extrae EtherType
  │   │   ├─ L3: IP header → routing decision, netfilter/iptables
  │   │   ├─ L4: TCP header → lookup de socket, enqueue a socket buffer
  │   │   └─ conntrack: actualiza tabla de estado
  │   └─ Si socket tiene proceso bloqueado → wake up
  ├─ Rehabilita IRQ si no hay más paquetes

  Userspace (aplicación):
  ├─ recv()/read() copia datos del kernel socket buffer a user buffer
  │   (COPIA #1: kernel → userspace)
  ├─ Aplicación procesa (parse HTTP, lógica de negocio)
  ├─ Puede hacer I/O a disco (read/write) → otra travesía
  └─ Prepara respuesta en user buffer


  2. ENVÍO DE RESPUESTA:

  Userspace:
  ├─ send()/write() copia datos de user buffer a kernel socket buffer
  │   (COPIA #2: userspace → kernel)

  Kernel:
  ├─ TCP segmentation (o TSO offload a la NIC)
  ├─ IP header, routing
  ├─ netfilter/iptables (POSTROUTING)
  ├─ L2: añade Ethernet header
  ├─ Enqueue en TX ring buffer de la NIC
  └─ Doorbell write al NIC para iniciar transmisión

  NIC (hardware):
  ├─ DMA lee frame del TX ring buffer (RAM)
  ├─ Añade CRC
  ├─ PHY transmite señal
  └─ DMA escribe completion entry, genera IRQ (opcional)


  Latencia total típica (localhost eliminando red):
  NIC DMA + IRQ:           ~1-2 μs
  Kernel network stack:    ~3-5 μs
  Kernel → userspace copy: ~0.5-1 μs
  Application processing:  variable
  Userspace → kernel copy: ~0.5-1 μs
  Kernel TX + NIC DMA:     ~2-3 μs
  ─────────────────────────────────
  Overhead del sistema:    ~7-12 μs (sin contar app logic)
```

### Request de disco

```
  Aplicación: read(fd, buf, 4096)

  1. Syscall: user mode → kernel mode (context switch ~1μs)
  2. VFS: identifica filesystem (ext4, XFS, etc.)
  3. Page cache lookup:
     - HIT: copia datos de page cache a user buffer → return (RÁPIDO)
     - MISS: continúa a paso 4
  4. Filesystem: traduce offset del archivo → bloque lógico → bloque físico
  5. Block layer: construye bio request, merge con requests adyacentes
  6. Device driver: envía comando al controlador NVMe
     - Escribe Submission Queue entry (en memoria)
     - Doorbell write al NVMe controller
  7. NVMe controller:
     - Lee SQ entry via DMA
     - Lee datos de NAND Flash
     - Escribe datos a RAM via DMA
     - Escribe Completion Queue entry via DMA
     - Genera interrupción MSI-X
  8. Kernel: ISR procesa completion
  9. Datos en page cache → copia a user buffer
  10. Return a userspace

  Latencia:
  Page cache hit: ~1-2 μs (solo copia memoria → memoria)
  NVMe read: ~10-20 μs (pasos 4-10)
  SATA SSD: ~50-100 μs
  HDD: ~5-10 ms
```

## Modelo mental

El data path es la cadena de montaje completa: desde que la materia prima (paquete) llega al muelle (NIC), pasa por aduanas (kernel network stack), se almacena temporalmente (socket buffer), se transporta a la fábrica (userspace), se procesa, y el producto terminado (respuesta) recorre el camino inverso. Cada estación añade latencia. Cada copia de datos es un peaje. Minimizar estaciones y peajes = menor latencia.

## Insights de ingeniería

- **Cada copia de datos cuesta tiempo y bandwidth de memoria**: en el path estándar, los datos se copian al menos 2 veces (kernel→user, user→kernel). Con un payload de 1KB a 1M paquetes/segundo, eso es 2 GB/s de bandwidth de memoria solo en copias. Zero-copy (sendfile, mmap, io_uring) elimina estas copias.
- **El kernel network stack es optimizado pero no gratis**: ~5μs por paquete a través del stack completo (netfilter, conntrack, routing, TCP). A 10M pps, necesitas 50 segundos de CPU por segundo — más de lo que un core puede dar. Para >1M pps por core se necesita bypass parcial (XDP) o total (DPDK).
