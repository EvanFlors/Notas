# 10.5 Kernel Bypass

## Definición

Kernel bypass es la técnica de eliminar completamente el kernel del sistema operativo del data path de red o almacenamiento. En vez de que los paquetes pasen por el network stack del kernel (netfilter, TCP/IP, socket layer), la aplicación en userspace accede directamente al hardware de la NIC o del dispositivo de almacenamiento. Esto elimina overhead de syscalls, copias, interrupciones, y locks del kernel, permitiendo latencias de ~1μs y tasas de millones de paquetes por segundo por core.

## Mecánica interna

### DPDK (Data Plane Development Kit)

```
  Arquitectura DPDK:

  Stack estándar:                    DPDK:
  ┌───────────┐                     ┌───────────┐
  │Application│                     │Application│
  ├───────────┤                     │ + DPDK    │
  │  TCP/IP   │                     │ (poll mode│
  │  Stack    │                     │  driver)  │
  ├───────────┤                     └─────┬─────┘
  │  Kernel   │                           │ userspace
  │  Driver   │                     ══════╪══════════
  ├───────────┤                           │ hardware
  │   NIC     │                     ┌─────┴─────┐
  └───────────┘                     │    NIC     │
                                    └───────────┘

  Cómo funciona:
  1. DPDK toma control exclusivo de la NIC (unbind del kernel driver)
     - La NIC desaparece del sistema operativo (no hay eth0)
     - Se usa UIO (Userspace I/O) o VFIO para acceso directo

  2. Poll Mode Driver (PMD):
     - NO usa interrupciones
     - Un thread dedicado hace busy-polling del ring buffer de la NIC
     - while(true) { check_rx_ring(); process_packets(); }
     - 100% CPU usage en ese core (siempre polling)

  3. Hugepages:
     - DPDK usa hugepages (2MB/1GB) para buffers
     - Elimina TLB misses
     - Memoria pinned (no paginable)

  4. Zero-copy + batch processing:
     - Paquetes accedidos directamente en memoria compartida
     - Procesamiento en batches de 32-64 paquetes

  Rendimiento:
  - Latencia: ~1-2 μs (vs ~10-15 μs con kernel stack)
  - Throughput: 10-20 millones de paquetes/segundo por core
  - Un servidor con 8 cores DPDK puede manejar 100 Gbps line rate

  Desventajas:
  - Pierde TODA la funcionalidad del kernel (TCP, firewall, routing)
  - Debe reimplementar TCP/IP en userspace si se necesita
  - Core dedicado 100% a polling (desperdicio si hay poco tráfico)
  - Debugging más difícil (no tcpdump, no iptables)

  Uso: load balancers (F5), firewalls de alto rendimiento, NFV,
  switches virtuales (OVS-DPDK), CDN edge nodes.
```

### RDMA (Remote Direct Memory Access)

```
  RDMA permite a una máquina escribir directamente en la memoria
  de otra máquina remota SIN involucrar el CPU de la máquina remota.

  Operación RDMA Write:

  Máquina A (initiator):           Máquina B (target):
  ┌─────────────┐                  ┌─────────────┐
  │ Application │                  │ Application │
  │ rdma_write()│                  │ (no se      │
  └──────┬──────┘                  │  entera)    │
         │                         └─────────────┘
  ┌──────▼──────┐                  ┌─────────────┐
  │ RDMA NIC    │════════════════▶│ RDMA NIC    │
  │ (RNIC/HCA)  │    network      │ DMA → RAM   │
  └─────────────┘                  └─────────────┘

  Los datos van de la RAM de A → NIC de A → red → NIC de B → RAM de B.
  El CPU de B NO participa. Cero interrupciones, cero copias, cero syscalls.

  Implementaciones:
  - InfiniBand: red dedicada RDMA (HPC, clusters de almacenamiento)
    Latencia: ~1-2 μs, Bandwidth: 200-400 Gbps
  - RoCE (RDMA over Converged Ethernet): RDMA sobre Ethernet estándar
    Requiere lossless Ethernet (PFC/ECN)
  - iWARP: RDMA sobre TCP/IP (compatible con cualquier red)
    Mayor latencia que InfiniBand/RoCE

  Uso:
  - Almacenamiento distribuido: Ceph, Windows SMB Direct
  - Bases de datos distribuidas: Oracle RAC, SAP HANA
  - HPC: MPI (Message Passing Interface) sobre RDMA
  - Machine Learning: NCCL (NVIDIA) usa RDMA para comunicación entre GPUs
```

### XDP (eXpress Data Path)

```
  XDP es un punto intermedio entre kernel stack completo y DPDK:
  Ejecuta programas eBPF directamente en el driver de la NIC,
  ANTES de que el paquete entre al kernel network stack.

  ┌──────────────┐
  │ Application  │  (solo ve paquetes que XDP permite pasar)
  ├──────────────┤
  │ Kernel Stack │  (TCP/IP, netfilter — bypass para paquetes filtrados)
  ├──────────────┤
  │ XDP Program  │  ← eBPF program aquí (decisión en ~100ns)
  │ (eBPF)       │  Acciones: XDP_PASS, XDP_DROP, XDP_TX, XDP_REDIRECT
  ├──────────────┤
  │ NIC Driver   │
  └──────────────┘

  Ventajas sobre DPDK:
  - No requiere core dedicado (no busy-polling)
  - Coexiste con el kernel stack (paquetes pueden pasar al stack normal)
  - Se puede adjuntar/detach en caliente sin reiniciar
  - Usa herramientas estándar del kernel (bpftool, bpftrace)

  Rendimiento: 5-10 millones de paquetes/segundo de drop rate por core.

  Uso:
  - DDoS mitigation: Facebook usa XDP para filtrar ataques volumétricos
  - Load balancing: Facebook Katran (L4 LB basado en XDP)
  - Monitoring: packet sampling sin overhead
  - Firewall de alto rendimiento: Cloudflare usa XDP
```

## Modelo mental

El kernel network stack es un aeropuerto con check-in, seguridad, y puerta de embarque: seguro y organizado, pero lento. DPDK es un avión privado: vas directamente a la pista, sin pasar por ningún control, máxima velocidad pero sin los servicios del aeropuerto. XDP es un fast-track: pasas por un filtro rápido y si todo está bien, vas directo a la puerta; si no, pasas por el proceso normal. RDMA es teleportación: los datos aparecen directamente en la memoria del destino sin que nadie intervenga.

## Insights de ingeniería

- **Kernel bypass es para el 1% de los workloads**: la vasta mayoría de aplicaciones no necesitan <5μs de latencia de red ni >5M pps. El kernel network stack es extremadamente optimizado (NAPI, GRO, TSO, RSS) y suficiente para >99% de los casos. El costo de DPDK (reimplementar TCP, perder herramientas del kernel, cores dedicados) solo se justifica en infraestructura de red especializada.
- **RDMA está creciendo en datacenters**: con la explosión de AI/ML, la comunicación GPU-to-GPU entre nodos usa RDMA (NCCL sobre InfiniBand o RoCE). Un cluster de 1000 GPUs H100 necesita transferir terabytes de gradientes por minuto. RDMA a 400 Gbps con ~1μs de latencia hace esto posible. Sin RDMA, el overhead del kernel TCP haría el entrenamiento distribuido inviablemente lento.
