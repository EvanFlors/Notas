# 10.4 Zero-Copy

## Definición

Zero-copy es un conjunto de técnicas que eliminan copias de datos entre kernel y userspace durante operaciones de I/O. En el path estándar, enviar un archivo por red requiere al menos 2 copias (disco→kernel, kernel→userspace, userspace→kernel, kernel→NIC). Con zero-copy, los datos van directamente del page cache a la NIC sin pasar por userspace, reduciendo uso de CPU y bandwidth de memoria.

## Mecánica interna

### Path estándar (con copias)

```
  Enviar un archivo de 1MB por TCP (sin zero-copy):

  read(file_fd, buffer, 1MB):
  1. Disco → Page Cache (DMA)         ← transferencia #1
  2. Page Cache → User Buffer (CPU)   ← COPIA #1 (kernel → user)

  send(socket_fd, buffer, 1MB):
  3. User Buffer → Socket Buffer (CPU) ← COPIA #2 (user → kernel)
  4. Socket Buffer → NIC (DMA)         ← transferencia #2

  Total: 4 transferencias, 2 copias CPU, 2 context switches.

  A 1 Gbps con paquetes de 1MB:
  ~125 envíos/segundo × 2 copias × 1MB = 250 MB/s de bandwidth de memoria
  consumido solo en copias.
```

### sendfile()

```
  sendfile(socket_fd, file_fd, offset, count):

  1. Disco → Page Cache (DMA)              ← transferencia #1
  2. Page Cache → NIC (DMA, via SG-DMA)    ← transferencia #2

  Total: 2 transferencias DMA, 0 copias CPU, 1 context switch.

  El kernel envía directamente del page cache al socket buffer
  (o directamente a la NIC si el hardware soporta scatter-gather DMA).
  Los datos NUNCA pasan por userspace.

  ┌──────┐    DMA     ┌────────────┐  DMA    ┌─────┐
  │ Disk │──────────▶│ Page Cache │────────▶│ NIC │
  └──────┘           └────────────┘         └─────┘
                    (sin pasar por userspace)

  Uso típico: servir archivos estáticos.
  NGINX usa sendfile por defecto para archivos estáticos.
  Configuración: sendfile on; en nginx.conf.

  Limitación: solo funciona para file→socket.
  No permite modificar los datos (no puedes comprimir o cifrar en el medio).
  Para HTTPS: sendfile envía al kernel TLS (kTLS) que cifra sin copia a userspace.
```

### mmap()

```
  void *ptr = mmap(NULL, size, PROT_READ, MAP_PRIVATE, fd, 0);
  send(socket_fd, ptr, size, 0);

  mmap() mapea el archivo directamente al espacio de direcciones del proceso.
  No copia datos — crea un mapeo de memoria virtual al page cache.

  Cuando la app accede a ptr, se genera un page fault (si la página
  no está en page cache), el kernel carga la página del disco,
  y la app accede directamente al page cache.

  Para I/O:
  1. Disco → Page Cache (DMA, on demand via page faults)
  2. send() copia Page Cache → Socket Buffer (CPU) ← 1 copia
  3. Socket Buffer → NIC (DMA)

  Ventaja sobre read(): elimina 1 copia (la de kernel→user buffer).
  Desventaja: page faults tienen overhead, TLB pressure con muchos mappings.

  Uso en bases de datos:
  MongoDB (WiredTiger), LMDB, RocksDB usan mmap para acceder a archivos.
  Ventaja: el kernel gestiona la paginación automáticamente.
  Desventaja: no control sobre eviction, no control sobre I/O ordering.
  Algunas DBs (PostgreSQL, MySQL) prefieren read() + buffer pool propio.
```

### splice() y tee()

```
  splice(): mueve datos entre dos file descriptors usando pipe como intermediario,
  sin copia a userspace.

  pipe(pipefd);
  splice(file_fd, offset, pipefd[1], NULL, len, flags);  // file → pipe
  splice(pipefd[0], NULL, socket_fd, NULL, len, flags);   // pipe → socket

  Útil cuando sendfile no aplica (e.g., proxy que reenvía datos
  de un socket a otro sin inspeccionarlos).
```

## Modelo mental

El path estándar es fotocopiar un documento dos veces para enviarlo: original → fotocopia en tu escritorio → fotocopia en el sobre de correo → enviar. sendfile() es dar el original directamente al cartero sin fotocopiarlo. mmap() es poner el original en un estante compartido donde todos pueden leerlo sin copiarlo, pero necesitas ir al estante cada vez.

## Insights de ingeniería

- **Zero-copy es esencial para servidores de alto throughput**: un CDN sirviendo archivos estáticos a 10 Gbps con el path estándar consumiría ~20 GB/s de bandwidth de memoria solo en copias. Con sendfile, ese costo baja a ~0 (solo DMA). NGINX con sendfile puede saturar una NIC de 10G con un solo core.
- **Para datos que necesitan transformación, zero-copy no aplica**: si necesitas comprimir (gzip), cifrar (TLS), o modificar los datos, necesitas acceder a ellos en userspace. kTLS (kernel TLS) resuelve parcialmente esto: el kernel hace cifrado TLS directamente sobre datos del page cache, permitiendo sendfile sobre HTTPS. Linux 4.13+ soporta kTLS, NGINX 1.21+ lo usa.
