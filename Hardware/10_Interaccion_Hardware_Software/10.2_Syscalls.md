# 10.2 Syscalls de I/O

## Definición

Las syscalls (system calls) son la interfaz entre programas de userspace y el kernel del sistema operativo. Para operaciones de I/O, las syscalls fundamentales son read()/write() para archivos y disco, send()/recv() para red, y accept() para conexiones entrantes. Cada syscall implica una transición de user mode a kernel mode (context switch), con un costo de ~1μs.

## Mecánica interna

### Syscalls fundamentales

```
  ARCHIVOS / DISCO:

  int fd = open("/path/file", O_RDONLY);
  ssize_t n = read(fd, buffer, size);
  ssize_t n = write(fd, buffer, size);
  close(fd);

  read():
  1. Transición user → kernel (~200ns con SYSCALL instruction)
  2. Lookup de fd en la tabla de file descriptors del proceso
  3. VFS dispatch al filesystem correspondiente
  4. Page cache check:
     - Hit: copia directa a user buffer
     - Miss: bloquea proceso, inicia I/O al dispositivo, despierta cuando completa
  5. Copia datos de kernel buffer a user buffer
  6. Retorna al userspace con bytes leídos

  write():
  Por defecto, escribe al page cache (write-back) y retorna inmediatamente.
  Los datos se flushean a disco después (pdflush/writeback thread).
  Con O_DIRECT: bypass page cache, escribe directo al dispositivo.
  Con O_SYNC: no retorna hasta que los datos están en disco persistente.


  RED:

  int sockfd = socket(AF_INET, SOCK_STREAM, 0);
  bind(sockfd, &addr, sizeof(addr));
  listen(sockfd, backlog);
  int client = accept(sockfd, &client_addr, &len);
  ssize_t n = recv(client, buffer, size, 0);
  ssize_t n = send(client, buffer, size, 0);

  accept():
  Bloquea hasta que un cliente completa el TCP 3-way handshake.
  Retorna un nuevo fd para la conexión establecida.
  El kernel mantiene dos colas:
  - SYN queue: conexiones en progreso (SYN recibido, SYN-ACK enviado)
  - Accept queue: conexiones completas esperando accept()
  Backlog controla el tamaño de la accept queue.

  recv():
  Bloquea hasta que haya datos en el socket buffer del kernel.
  Copia datos del kernel socket buffer al user buffer.
  Si el socket buffer está vacío y el peer cerró → retorna 0.

  send():
  Copia datos del user buffer al kernel socket buffer.
  Si el socket buffer está lleno → bloquea (TCP flow control).
  Retorna cuando los datos están en el kernel buffer, NO cuando llegan al peer.
```

### Multiplexación de I/O

```
  Problema: un servidor con 10,000 conexiones no puede tener 10,000 threads
  (cada thread consume ~8MB de stack + overhead de context switch).

  Solución: I/O multiplexing — un thread monitorea múltiples fds.

  select() / poll() — legacy:
  Pasa una lista de fds al kernel.
  Kernel escanea TODOS los fds, marca los que tienen datos.
  Retorna. Userspace escanea la lista para encontrar los activos.
  Complejidad: O(n) por cada llamada. Ineficiente con miles de fds.

  epoll() — Linux moderno:
  int epfd = epoll_create1(0);
  epoll_ctl(epfd, EPOLL_CTL_ADD, fd, &event);   // registrar fd
  int n = epoll_wait(epfd, events, max, timeout); // esperar eventos

  El kernel mantiene un árbol rojo-negro de fds registrados.
  Cuando un fd tiene datos, el kernel lo añade a una ready list.
  epoll_wait() solo retorna los fds que TIENEN eventos.
  Complejidad: O(1) por fd con evento. Escala a millones de conexiones.

  Modo edge-triggered vs level-triggered:
  - Level-triggered (default): epoll_wait reporta fd si hay datos disponibles
  - Edge-triggered: epoll_wait reporta fd solo cuando LLEGAN datos nuevos
  Edge-triggered es más eficiente pero requiere leer todo el buffer en cada evento.
```

### io_uring — I/O asíncrono moderno

```
  io_uring (Linux 5.1+): interfaz de I/O asíncrono basada en ring buffers
  compartidos entre kernel y userspace.

  ┌─────────────────────────────────────────────────┐
  │                 Userspace                        │
  │  ┌───────────────────┐  ┌────────────────────┐  │
  │  │ Submission Queue   │  │ Completion Queue   │  │
  │  │ (SQ) — requests   │  │ (CQ) — results     │  │
  │  └────────┬──────────┘  └─────────▲──────────┘  │
  │           │                       │              │
  ├───────────┼───────────────────────┼──────────────┤
  │           │        Kernel         │              │
  │           ▼                       │              │
  │  ┌────────────────────────────────┘              │
  │  │ Procesa requests, ejecuta I/O,               │
  │  │ escribe resultados en CQ                      │
  │  └──────────────────────────────────────────────┘│
  └─────────────────────────────────────────────────┘

  Ventajas:
  - ZERO syscalls en el hot path (SQ/CQ en memoria compartida)
  - Batching: enviar múltiples I/Os en una sola operación
  - Polling mode: kernel hace polling del dispositivo (no interrupciones)
  - Soporta: read, write, send, recv, accept, connect, fsync, etc.

  Resultado: latencia de ~2-3μs por I/O (vs ~5-10μs con syscalls clásicas).
  Throughput: >1M IOPS por core con NVMe.
```

## Modelo mental

Las syscalls son ventanillas de atención del kernel. Cada vez que la aplicación necesita algo del hardware (leer disco, recibir red), debe hacer fila en la ventanilla (context switch). Con select/poll, pregunta en cada ventanilla "¿hay algo para mí?" (ineficiente). Con epoll, deja su número de teléfono y el kernel lo llama cuando hay algo (eficiente). Con io_uring, hay un buzón compartido donde la aplicación deja pedidos y el kernel deja respuestas sin necesidad de hacer fila.

## Insights de ingeniería

- **El costo del syscall no es solo el context switch**: la transición user→kernel→user cuesta ~200ns, pero el verdadero costo es la contaminación de cache (la I-cache y D-cache se llenan con código del kernel, desplazando el código de la aplicación) y la serialización (solo un thread puede ejecutar en un core a la vez). io_uring evita esto al eliminar syscalls del hot path.
- **epoll es suficiente para la mayoría de aplicaciones de red**: NGINX, Redis, Node.js — todos usan epoll. io_uring brilla en disk I/O donde el overhead por operación importa (databases, storage engines). Para networking puro, la diferencia entre epoll y io_uring es marginal porque el bottleneck suele ser el protocol stack, no el mecanismo de notificación.
