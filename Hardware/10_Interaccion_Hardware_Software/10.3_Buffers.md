# 10.3 Buffers

## Definición

Los buffers son regiones de memoria temporal que almacenan datos en tránsito entre componentes del sistema. En I/O, hay buffers en múltiples niveles: buffers de la aplicación (userspace), socket buffers y page cache (kernel), ring buffers de la NIC, y buffers internos del dispositivo (SSD controller, NIC SRAM). Cada buffer introduce latencia potencial pero permite desacoplar productores y consumidores que operan a velocidades diferentes.

## Mecánica interna

### Niveles de buffering

```
  Red (recepción):

  ┌──────────────────┐
  │ NIC SRAM Buffer  │  Buffer interno de la NIC (~MB)
  │ (on-chip)        │  Si se llena antes de DMA → drop en hardware
  └────────┬─────────┘
           │ DMA
  ┌────────▼─────────┐
  │ Ring Buffer       │  Descriptores en RAM del host
  │ (driver/kernel)   │  Tamaño: 256-4096 entries (configurable)
  │                   │  Si kernel no consume a tiempo → drop
  └────────┬─────────┘
           │
  ┌────────▼─────────┐
  │ Socket Buffer     │  sk_buff en kernel (por socket)
  │ (kernel, per-     │  TCP: rmem (receive buffer), wmem (send buffer)
  │  socket)          │  Tamaño: net.core.rmem_max (default ~200KB)
  │                   │  Si app no lee a tiempo → TCP window = 0 → flow control
  └────────┬─────────┘
           │ recv() / read()
  ┌────────▼─────────┐
  │ Application      │  Buffer en userspace (malloc'd)
  │ Buffer           │  Tamaño: controlado por la app
  └──────────────────┘


  Disco (lectura):

  ┌──────────────────┐
  │ Device Buffer     │  DRAM del SSD controller (~256MB-4GB)
  │ (SSD/HDD)        │  Cache de lectura, write buffer
  └────────┬─────────┘
           │ DMA (via NVMe/AHCI)
  ┌────────▼─────────┐
  │ Page Cache        │  RAM del host, gestionado por kernel
  │ (kernel)          │  Cache de bloques de disco
  │                   │  Tamaño: toda la RAM libre
  │                   │  Eviction: LRU (Least Recently Used)
  └────────┬─────────┘
           │ read()
  ┌────────▼─────────┐
  │ Application      │  (o mmap: app accede page cache directamente)
  │ Buffer           │
  └──────────────────┘
```

### Tuning de buffers

```
  Socket buffers (TCP):

  # Ver valores actuales
  sysctl net.core.rmem_max        # max receive buffer (bytes)
  sysctl net.core.wmem_max        # max send buffer
  sysctl net.ipv4.tcp_rmem        # min default max (per-socket)
  sysctl net.ipv4.tcp_wmem

  # Aumentar para conexiones de alta latencia (CDN, cross-region)
  sysctl -w net.core.rmem_max=16777216      # 16MB
  sysctl -w net.core.wmem_max=16777216
  sysctl -w net.ipv4.tcp_rmem="4096 131072 16777216"

  TCP autotuning: el kernel ajusta automáticamente el buffer
  de cada socket basado en RTT y bandwidth observado.
  Respeta rmem_max como límite superior.


  Page cache:
  No se configura explícitamente — el kernel usa toda la RAM libre.

  # Ver uso de page cache
  free -h    # "buff/cache" column
  vmstat 1   # cache, buffer columns

  # Ratio dirty pages (cuánto se acumula antes de flush)
  sysctl vm.dirty_ratio              # % de RAM (default 20%)
  sysctl vm.dirty_background_ratio   # % para iniciar writeback (default 10%)

  # Para bases de datos con O_DIRECT: page cache no se usa
  # La DB maneja su propio buffer pool (InnoDB buffer pool, PostgreSQL shared_buffers)
```

## Modelo mental

Los buffers son tanques de agua entre las etapas de una planta de procesamiento. Cada tanque desacopla la velocidad del productor y consumidor: si la red produce datos más rápido de lo que la app los consume, el tanque (socket buffer) absorbe el exceso temporalmente. Si el tanque se llena, el flujo se frena (TCP flow control) o se derrama (packet drop en UDP/ring buffer).

## Insights de ingeniería

- **Buffer bloat es latencia oculta**: buffers demasiado grandes absorben ráfagas pero esconden congestión. Un socket buffer de 16MB con un peer lento puede acumular 16MB de datos "en vuelo" — eso son segundos de latencia en una conexión de 10Mbps. TCP depende de la señal de congestión (packet loss o ECN) para ajustar su tasa. Buffers grandes retrasan esa señal.
- **Las bases de datos bypasean el page cache a propósito**: InnoDB (MySQL) y PostgreSQL usan O_DIRECT para escribir directamente al disco, bypasseando el page cache del kernel. La razón es que la DB tiene su propio buffer pool con políticas de eviction optimizadas para su workload (clock sweep en PostgreSQL, LRU adaptativo en InnoDB). Usar page cache Y buffer pool de la DB duplica el uso de RAM sin beneficio.
