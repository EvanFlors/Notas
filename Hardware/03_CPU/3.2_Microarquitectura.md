# 3.2 Microarquitectura

## Definición

La microarquitectura es la implementación interna de una CPU: cómo se traduce el ISA (Instruction Set Architecture) en circuitos reales. Los procesadores modernos usan ejecución fuera de orden (Out-of-Order, OoO): las instrucciones se despachan y ejecutan no en el orden del programa sino en el orden en que sus operandos estén disponibles, maximizando la utilización de las unidades de ejecución. El reorder buffer (ROB) garantiza que los resultados se hagan visibles en orden correcto.

## Mecánica interna

### Front-end: Fetch y Decode

```
  El front-end se encarga de alimentar instrucciones al back-end.

  ┌──────────────────────────────────────────────┐
  │                FRONT-END                      │
  │                                              │
  │  Branch Predictor ──▶ Fetch Unit             │
  │                       │                      │
  │                       ▼                      │
  │                  I-Cache (L1)                 │
  │                       │                      │
  │                       ▼                      │
  │              Pre-decode / Decode              │
  │              (x86 → micro-ops)               │
  │                       │                      │
  │                       ▼                      │
  │              Micro-op Cache (DSB)            │
  │              (decoded stream buffer)          │
  │                       │                      │
  │                       ▼                      │
  │              Allocation / Rename              │
  │              (register renaming)              │
  └──────────────────────────────────────────────┘

  En x86:
  Las instrucciones CISC de longitud variable (1-15 bytes) se decodifican
  en micro-ops (μops) RISC de tamaño fijo. Una instrucción x86 puede
  generar 1-4+ μops.

  ADD RAX, RBX         → 1 μop
  ADD RAX, [mem]       → 2 μops (load + add)
  REP MOVSB            → muchas μops (loop en microcode)

  El micro-op cache (DSB) almacena μops ya decodificadas:
  si un loop se repite, las instrucciones se toman del DSB
  sin necesidad de re-decodificar → ahorra energía y ancho de banda.
```

### Register Renaming

```
  Problema: x86-64 tiene solo 16 registros arquitecturales,
  pero el CPU puede tener 200+ instrucciones en vuelo.
  Muchas instrucciones usan los mismos registros, creando
  dependencias falsas (WAW, WAR).

  Ejemplo:
  ADD RAX, RBX       ← escribe RAX
  MOV RCX, RAX       ← lee RAX
  ADD RAX, RDX       ← escribe RAX (WAW con la primera ADD)

  Sin renaming: la tercera instrucción debe esperar a que
  la segunda lea RAX antes de escribir → dependencia falsa.

  Con renaming:
  ADD P47, P12, P23     ← RAX → P47 (registro físico 47)
  MOV P48, P47          ← lee P47 (el RAX original)
  ADD P49, P47, P30     ← RAX → P49 (nuevo registro físico)

  Ahora no hay conflicto: P47 y P49 son registros diferentes.
  La tercera instrucción puede ejecutarse en paralelo con la segunda.

  RAT (Register Alias Table):
  Mapea registros arquitecturales → registros físicos.
  Se actualiza con cada instrucción que escribe un registro.

  Intel Skylake: 180 registros físicos integer, 168 FP/vector.
  Apple M1: ~350+ registros físicos (estimado).
```

### Out-of-Order Execution Engine

```
  El back-end OoO es donde se ejecutan las instrucciones.

  ┌────────────────────────────────────────────────────┐
  │                 BACK-END (OoO)                      │
  │                                                    │
  │  Renamed μops → Reservation Station (RS)           │
  │                  (espera operandos)                 │
  │                       │                            │
  │            ┌──────────┼──────────┐                 │
  │            ▼          ▼          ▼                 │
  │       ┌────────┐ ┌────────┐ ┌────────┐            │
  │       │ Port 0 │ │ Port 1 │ │ Port 2 │ ...        │
  │       │ ALU    │ │ ALU    │ │ Load   │            │
  │       │ FMA    │ │ FMA    │ │        │            │
  │       │ Branch │ │ MUL    │ │        │            │
  │       └────────┘ └────────┘ └────────┘            │
  │            │          │          │                 │
  │            ▼          ▼          ▼                 │
  │       ┌────────────────────────────────┐           │
  │       │      Reorder Buffer (ROB)      │           │
  │       │  (mantiene orden del programa) │           │
  │       └────────────────┬───────────────┘           │
  │                        ▼                           │
  │                   Retirement                       │
  │              (commit en orden)                     │
  └────────────────────────────────────────────────────┘

  Flujo:
  1. Las μops entran al RS con sus operandos (o tags si no están listos)
  2. Cuando TODOS los operandos de una μop están listos → se despacha
     al puerto de ejecución disponible
  3. La ejecución ocurre FUERA DE ORDEN (la primera μop lista se ejecuta primero)
  4. El resultado se escribe en el registro físico y se broadcast
     a las μops que esperaban ese resultado (wakeup)
  5. El ROB mantiene las μops en orden del programa
  6. Cuando la μop más antigua del ROB está completa → se retira (commit)
     y su resultado se hace arquitecturalmente visible


  Intel Skylake numbers:
  - RS entries: 97 (instrucciones esperando operandos)
  - ROB entries: 224 (instrucciones en vuelo)
  - Execution ports: 8
  - Dispatch width: 6 μops/ciclo
  - Retire width: 4 μops/ciclo

  Apple M1 (Firestorm):
  - ROB entries: ~630 (estimado — masivamente más grande)
  - Dispatch width: 8 μops/ciclo
  - El ROB grande permite "ver más lejos" y encontrar más ILP
```

### Reorder Buffer (ROB)

```
  El ROB es la estructura que permite ejecución fuera de orden
  con retiro en orden.

  ¿Por qué retirar en orden?
  - Excepciones precisas: si la instrucción #5 causa una excepción,
    las instrucciones #6, #7 (ejecutadas especulativamente) deben
    descartarse como si nunca hubieran existido.
  - Branch misprediction recovery: si un branch se predijo mal,
    todo lo ejecutado especulativamente después del branch se descarta.
  - Estado consistente: el programa ve los resultados como si
    se ejecutaran secuencialmente, aunque internamente no lo hagan.

  ROB como buffer circular:
  ┌───┬───┬───┬───┬───┬───┬───┬───┐
  │ 0 │ 1 │ 2 │ 3 │ 4 │ 5 │ 6 │ 7 │
  │ADD│MUL│LD │SUB│BRN│ADD│ST │LD │
  │ ✓ │ ✓ │ ✓ │ ✓ │ ? │ ✓ │   │   │
  └───┴───┴───┴───┴───┴───┴───┴───┘
    ↑ retire                    ↑ allocate

  Entry 4 (BRN) aún no resuelta → entries 0-3 pueden retirarse.
  Entry 5 completada pero NO puede retirarse hasta que 4 se resuelva.
  Si 4 mispredicts → entries 5, 6, 7 se descartan (flush).
```

## Modelo mental

La microarquitectura OoO es como un restaurante con múltiples chefs. Los pedidos (instrucciones) llegan en orden del menú (programa). El maître (front-end) los prepara y asigna. Cada chef (unidad de ejecución) trabaja en el plato cuyo ingredientes estén listos, sin importar el orden de los pedidos. Pero el mesero (ROB) entrega los platos a la mesa estrictamente en el orden en que fueron pedidos. Si un plato se cancela (misprediction), todos los platos posteriores se descartan, aunque algunos ya estaban listos.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| ROB full → front-end stall | Instrucción de larga latencia bloquea retirement | Evitar cache misses en hot loops |
| RS full → dispatch stall | Muchas instrucciones esperando operandos | Reducir cadenas de dependencias |
| Port contention | Demasiadas instrucciones compiten por el mismo puerto | Mezclar tipos de instrucciones |
| Front-end bottleneck | DSB miss, decode stall | Código más compacto, menos instrucciones |

## Insights de ingeniería

- **El tamaño del ROB determina cuánto ILP puede explotar el CPU**: un ROB de 224 entries (Skylake) puede "ver" ~224 instrucciones adelante para encontrar trabajo independiente. Un ROB de 630 (Apple M1) ve casi 3× más lejos. Esto explica parte de por qué Apple Silicon tiene IPC tan alto: puede encontrar paralelismo que procesadores con ROBs más pequeños no ven.
- **Register renaming elimina dependencias falsas**: sin renaming, código como `RAX = A; RAX = B; RAX = C` parece serial. Con renaming, cada asignación a RAX va a un registro físico diferente, eliminando las dependencias WAW y permitiendo ejecución paralela. Más registros físicos = más renaming posible = más ILP.
- **Spectre/Meltdown explotan la ejecución especulativa**: la CPU ejecuta instrucciones especulativamente (antes de saber si el branch se toma). Si la especulación es incorrecta, los resultados arquitecturales se descartan, pero los efectos en cache NO se revierten. Un atacante puede medir timing de cache para inferir datos que nunca debería haber visto. Las mitigaciones (retpoline, IBRS) reducen rendimiento 5-30% en ciertos workloads.
