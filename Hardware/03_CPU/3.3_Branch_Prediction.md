# 3.3 Branch Prediction

## Definición

Branch prediction es el mecanismo hardware que predice la dirección de un salto condicional (branch) antes de que la condición se evalúe. En un pipeline profundo, la CPU necesita saber qué instrucciones fetch a continuación ANTES de resolver el branch. Si predice correctamente, el pipeline sigue sin interrupción. Si predice incorrectamente (misprediction), todo el trabajo especulativo se descarta y el pipeline se vacía — una penalización de 15-20 ciclos en CPUs modernos.

## Mecánica interna

### Por qué es necesaria

```
  Pipeline de 14 etapas (Intel Skylake):

  El branch se resuelve en la etapa de ejecución (~etapa 10).
  Pero el fetch de la siguiente instrucción ocurre en la etapa 1.

  Sin predicción: el CPU se detiene 10+ ciclos en cada branch
  esperando la resolución. Dado que ~20% de instrucciones son branches,
  el rendimiento se destruiría.

  Con predicción al 95%:
  - 95% de branches: 0 ciclos de penalización
  - 5% de branches: ~15 ciclos de penalización
  - Penalización promedio: 0.05 × 15 = 0.75 ciclos/branch

  Con predicción al 99% (real en loops):
  - Penalización promedio: 0.01 × 15 = 0.15 ciclos/branch
```

### Predicción estática

```
  Reglas fijas, sin historial:

  1. "Branch backward taken, forward not taken":
     - Branches hacia atrás (loops): predecir TAKEN
       for (i=0; i<1000; i++) ← el branch de vuelta al inicio
       se toma 999 de 1000 veces
     - Branches hacia adelante (if): predecir NOT TAKEN
       if (error) ← la mayoría del código no es error path

  2. Hint bits en la instrucción (x86 tiene prefijos 2E/3E):
     Raramente usados; la predicción dinámica es muy superior.

  Precisión: ~60-70%. Insuficiente para CPUs modernos.
```

### Predicción dinámica

```
  Usa historial de ejecución para predecir.


  1. Branch History Table (BHT) / 2-bit Counter:

  Cada branch tiene un contador de saturación de 2 bits:

  ┌────────────────────────────────────────────┐
  │ 00: Strongly Not Taken                     │
  │ 01: Weakly Not Taken                       │
  │ 10: Weakly Taken                           │
  │ 11: Strongly Taken                         │
  └────────────────────────────────────────────┘

  Si el branch se toma: incrementar (max 11)
  Si no se toma: decrementar (min 00)
  Predecir: bit más significativo (1x = taken, 0x = not taken)

  Ventaja del 2-bit: tolera una anomalía sin cambiar predicción.
  Ejemplo: un loop de 100 iteraciones. La última iteración sale
  del loop (not taken). Con 1-bit, la siguiente entrada al loop
  predeciría mal. Con 2-bit, solo pasa de 11 → 10 (aún predice taken).


  2. Two-Level Predictor (correlating predictor):

  Problema: el resultado de un branch puede depender del historial
  de OTROS branches previos.

  if (x > 0) ...       ← branch A
  if (x > 0 && y > 0)  ← branch B (correlado con A)

  Solución: usar un Global History Register (GHR) que almacena
  los últimos N resultados de branches (taken/not taken).
  El GHR se usa como índice para una tabla de contadores.

  GHR (últimos 12 branches): 110101001011
  Index = hash(PC, GHR) → tabla de contadores de 2 bits


  3. TAGE (TAgged GEometric history length):

  El predictor más avanzado, usado en CPUs modernos (Intel, AMD, ARM).

  Múltiples tablas con diferentes longitudes de historial:
  ┌──────────────────────────────────────────────┐
  │ Tabla 0: historial de 0 branches (base)      │
  │ Tabla 1: historial de 4 branches             │
  │ Tabla 2: historial de 16 branches            │
  │ Tabla 3: historial de 64 branches            │
  │ Tabla 4: historial de 256 branches           │
  └──────────────────────────────────────────────┘

  Cada tabla tiene tags para verificar que el match es correcto.
  Se usa la predicción de la tabla con el historial más largo que tenga match.

  Precisión: 97-99%+ en workloads típicos.


  4. Branch Target Buffer (BTB):

  Predice la DIRECCIÓN de destino del branch (no solo taken/not taken).

  Para branches directos (offset fijo): fácil, calcular PC + offset.
  Para branches indirectos (computed goto, virtual dispatch):
  BTB almacena la última dirección de destino observada.

  Indirect branch misprediction es costosa y frecuente en:
  - Virtual method dispatch (C++, Java)
  - Switch statements compilados como jump tables
  - Interpreter dispatch loops (Python, Ruby bytecode interpreters)


  5. Return Address Stack (RAS):

  Las instrucciones CALL/RET tienen un patrón predecible:
  CALL empuja la dirección de retorno → RET salta a esa dirección.

  El RAS es un stack hardware que:
  - En CALL: push la dirección de retorno
  - En RET: pop y usa como predicción

  Precisión: ~100% (salvo stack overflow del RAS, ~16-32 entries).
```

### Impacto en rendimiento

```
  Branch misprediction penalty en CPUs modernos:

  ┌────────────────────┬────────────────┐
  │ CPU                │ Penalty        │
  ├────────────────────┼────────────────┤
  │ ARM Cortex-A53     │ ~8 ciclos      │
  │ ARM Cortex-A78     │ ~11 ciclos     │
  │ Intel Skylake      │ ~15 ciclos     │
  │ Apple M1 Firestorm │ ~14 ciclos     │
  │ AMD Zen 4          │ ~13 ciclos     │
  └────────────────────┴────────────────┘

  Impacto práctico:
  Un programa con 20% de instrucciones como branches
  y 5% de misprediction rate:

  IPC ideal: 4.0
  Penalización: 0.20 × 0.05 × 15 = 0.15 ciclos/instr promedio
  IPC real: ~4.0 / (1 + 0.15) ≈ 3.5 (12.5% de pérdida)

  Con 10% misprediction (código con patterns impredecibles):
  Penalización: 0.20 × 0.10 × 15 = 0.30 ciclos/instr
  IPC real: ~4.0 / (1 + 0.30) ≈ 3.1 (22.5% de pérdida)


  Código branchless (sin branches):
  Reemplazar branches con instrucciones condicionales:

  // Con branch (puede mispredecir):
  if (a > b) x = a; else x = b;

  // Branchless (CMOV en x86, CSEL en ARM):
  x = (a > b) ? a : b;  // compilador puede generar CMOV

  // Branchless manual con bit tricks:
  int mask = (a - b) >> 31;  // 0 si a>=b, -1 si a<b
  x = a ^ ((a ^ b) & mask);

  Branchless es beneficioso cuando:
  - El branch es impredecible (50/50, random data)
  - Ambas ramas son baratas (simples asignaciones)

  Branchless es perjudicial cuando:
  - El branch es predecible (95%+ hacia un lado)
  - Una rama es mucho más cara que la otra (branch permite evitarla)
```

## Modelo mental

Branch prediction es una apuesta calculada. El CPU no puede esperar a saber el resultado del branch porque perdería 15 ciclos. Así que apuesta basándose en historial: "este branch se tomó las últimas 99 veces → probablemente se tome de nuevo". Acierta 97% de las veces. El 3% que falla, todo el trabajo especulativo se tira a la basura. La clave es que el costo promedio de predecir (0.03 × 15 = 0.45 ciclos) es mucho menor que el costo de no predecir (15 ciclos siempre).

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Alto misprediction rate | Datos aleatorios, patterns impredecibles | Branchless code, sort data first |
| Indirect branch misprediction | Virtual dispatch, interpreter dispatch | Devirtualization, direct dispatch |
| BTB miss (cold code) | Primera ejecución, código nuevo | Warmup, PGO (Profile-Guided Optimization) |
| Performance drop con Spectre mitigations | Retpoline elimina indirect branch prediction | Hardware mitigations (IBRS, eIBRS) |

## Herramientas de diagnóstico

```
  Linux perf:
  perf stat -e branches,branch-misses ./program

  Output:
  1,234,567,890  branches
     12,345,678  branch-misses  (1.00%)  ← excelente

  > 5% branch-misses → investigar hot branches

  perf record -e branch-misses ./program
  perf report → muestra qué funciones/líneas causan más mispredictions
```

## Insights de ingeniería

- **Sorting data mejora branch prediction**: un loop que compara `if (array[i] > threshold)` es impredecible con datos aleatorios (~50% misprediction) pero predecible con datos ordenados (~0% en cada mitad). Este es el famoso "Why is processing a sorted array faster than an unsorted array?" de Stack Overflow. Sorting cambia el pattern de branches de aleatorio a predecible.
- **Los interpreters son víctimas de indirect branch misprediction**: el dispatch loop de un bytecode interpreter (`switch(opcode)`) es un indirect branch con decenas de posibles destinos. Los CPUs modernos mitigan esto con indirect target arrays, pero la misprediction rate sigue siendo 10-20%. Es una razón fundamental por la que los interpreters son más lentos que código compilado, y por qué JIT compilation (que elimina el dispatch loop) da speedups masivos.
- **Profile-Guided Optimization (PGO)** permite al compilador usar datos reales de ejecución para optimizar branches: pone el camino más probable primero (better I-cache layout), alinea hot loops, e informa hints de branch prediction. Google reporta 5-15% de mejora en rendimiento con PGO en sus workloads.
