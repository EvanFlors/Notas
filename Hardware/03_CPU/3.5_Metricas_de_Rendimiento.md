# 3.5 Métricas de Rendimiento

## Definición

Las métricas de rendimiento de CPU cuantifican la velocidad de procesamiento del procesador. Las tres métricas fundamentales son: clock frequency (velocidad del reloj en GHz), IPC (Instructions Per Cycle — instrucciones completadas por ciclo de reloj), y CPI (Cycles Per Instruction — ciclos necesarios por instrucción). El rendimiento real es el producto de frecuencia × IPC, no un factor individual.

## Mecánica interna

### Clock Frequency

```
  La frecuencia de reloj determina cuántos ciclos por segundo ejecuta el CPU.

  f = 4 GHz = 4 × 10⁹ ciclos/segundo
  T = 1/f = 0.25 ns = 250 ps por ciclo

  Base clock vs Turbo:
  ┌────────────────────┬──────────┬──────────┐
  │ Estado             │ Frecuencia│ Condición │
  ├────────────────────┼──────────┼──────────┤
  │ Idle               │ ~0.8 GHz │ DVFS     │
  │ All-core load      │ 3.5 GHz  │ Base     │
  │ Single-core load   │ 5.5 GHz  │ Turbo    │
  │ Thermal throttle   │ ~2.0 GHz │ Overheat │
  └────────────────────┴──────────┴──────────┘

  Turbo Boost:
  Cuando solo 1-2 cores están activos, el power budget total
  se concentra en esos cores → pueden subir de frecuencia.
  Con todos los cores activos, el presupuesto se reparte → base clock.

  Más frecuencia ≠ siempre más rápido:
  - Un core a 5 GHz con IPC=3 ejecuta 15 × 10⁹ instrucciones/s
  - Un core a 3.5 GHz con IPC=6 ejecuta 21 × 10⁹ instrucciones/s
  - El segundo es 40% más rápido a menor frecuencia

  Esto es exactamente Apple M1 vs Intel: menor frecuencia, mayor IPC.
```

### IPC (Instructions Per Cycle)

```
  IPC mide cuántas instrucciones completa el CPU por ciclo de reloj.
  Es la métrica de eficiencia del pipeline y la microarquitectura.

  IPC teórico máximo (dispatch width):
  ┌────────────────────┬──────────┐
  │ CPU                │ Max IPC  │
  ├────────────────────┼──────────┤
  │ Pipeline escalar   │ 1.0      │
  │ Intel Skylake (6w) │ 6.0      │
  │ Apple M1 (8w)      │ 8.0      │
  │ AMD Zen 4 (6w)     │ 6.0      │
  └────────────────────┴──────────┘

  IPC real (medido con workloads):
  ┌────────────────────┬──────────────┐
  │ Workload           │ IPC típico   │
  ├────────────────────┼──────────────┤
  │ Integer compute    │ 3.0 - 5.0    │
  │ FP/SIMD compute   │ 2.0 - 4.0    │
  │ Memory-bound       │ 0.5 - 1.5   │
  │ Branch-heavy       │ 1.5 - 3.0   │
  │ Server workloads   │ 0.8 - 2.0   │
  └────────────────────┴──────────────┘

  IPC real << IPC teórico porque:
  - Data hazards: dependencias entre instrucciones
  - Cache misses: CPU espera 100+ ciclos por dato
  - Branch mispredictions: pipeline flush (~15 ciclos)
  - Front-end stalls: I-cache miss, decode bottleneck


  Medir IPC:
  perf stat ./program

  Output:
  10,000,000,000  instructions
   3,000,000,000  cycles
  IPC = 10B / 3B = 3.33

  IPC < 1.0 → probablemente memory-bound (muchos stalls)
  IPC > 2.0 → compute-bound (buena utilización del pipeline)
```

### CPI (Cycles Per Instruction)

```
  CPI = 1 / IPC
  Es el inverso de IPC: cuántos ciclos tarda en promedio cada instrucción.

  CPI = 1.0 → una instrucción por ciclo (pipeline escalar ideal)
  CPI = 0.33 → tres instrucciones por ciclo (IPC = 3)
  CPI = 5.0 → una instrucción cada 5 ciclos (muy pobre, memory-bound)

  CPI breakdown (descomposición):

  CPI_total = CPI_base + CPI_stalls

  CPI_base: CPI ideal sin stalls (determinado por dependencias de datos)
  CPI_stalls = CPI_cache_miss + CPI_branch_miss + CPI_frontend + ...

  Ejemplo:
  ┌──────────────────────────┬───────┐
  │ Componente               │ CPI   │
  ├──────────────────────────┼───────┤
  │ CPI base (ideal)         │ 0.30  │
  │ L1 D-cache miss stalls   │ 0.05  │
  │ L2 miss stalls           │ 0.10  │
  │ L3 miss / DRAM stalls    │ 0.25  │
  │ Branch misprediction     │ 0.05  │
  │ Front-end stalls         │ 0.05  │
  ├──────────────────────────┼───────┤
  │ CPI total                │ 0.80  │
  │ IPC = 1/0.80             │ 1.25  │
  └──────────────────────────┴───────┘

  En este ejemplo, DRAM stalls contribuyen el 31% del CPI total.
  Optimizar acceso a memoria (mejor locality, prefetching)
  tendría el mayor impacto en rendimiento.
```

### Rendimiento real: la ecuación fundamental

```
  Tiempo de ejecución = Instrucciones × CPI × T_ciclo

  O equivalentemente:

  Rendimiento = Instrucciones/segundo = IPC × Frecuencia

  ┌──────────────────────────────────────────────────────┐
  │ Tres formas de mejorar rendimiento:                  │
  │                                                      │
  │ 1. Reducir instrucciones: mejor algoritmo,           │
  │    mejor compilador, SIMD (1 instrucción = N datos)  │
  │                                                      │
  │ 2. Aumentar IPC: mejor microarquitectura (OoO,       │
  │    branch prediction, más puertos de ejecución)      │
  │    o mejor código (menos dependencias, mejor locality)│
  │                                                      │
  │ 3. Aumentar frecuencia: mejor proceso de fabricación │
  │    (7nm → 5nm → 3nm), más voltaje (más calor)       │
  └──────────────────────────────────────────────────────┘

  Comparación real:
  ┌──────────────────┬───────┬──────┬────────────────────┐
  │ CPU              │ Freq  │ IPC  │ Rendimiento relativo│
  ├──────────────────┼───────┼──────┼────────────────────┤
  │ Intel i9-13900K  │ 5.8GHz│ ~3.5 │ 20.3 (reference)   │
  │ Apple M2 Pro     │ 3.5GHz│ ~6.0 │ 21.0 (+3%)         │
  │ AMD Ryzen 9 7950X│ 5.7GHz│ ~3.8 │ 21.7 (+7%)         │
  └──────────────────┴───────┴──────┴────────────────────┘

  Apple alcanza rendimiento comparable con ~40% menos frecuencia
  porque su IPC es ~70% mayor.
```

### Top-Down Microarchitecture Analysis (TMA)

```
  Intel desarrolló TMA para categorizar dónde se pierden ciclos:

  ┌─────────────────────────────────────────────┐
  │              Pipeline Slots                  │
  │  ┌───────────────┬─────────────────────┐    │
  │  │  Retiring     │  Not Retiring       │    │
  │  │  (trabajo     │  ┌───────────────┐  │    │
  │  │   útil)       │  │ Front-end     │  │    │
  │  │              │  │ Bound         │  │    │
  │  │              │  ├───────────────┤  │    │
  │  │              │  │ Back-end      │  │    │
  │  │              │  │ Bound         │  │    │
  │  │              │  │ ├─ Memory     │  │    │
  │  │              │  │ └─ Core       │  │    │
  │  │              │  ├───────────────┤  │    │
  │  │              │  │ Bad           │  │    │
  │  │              │  │ Speculation   │  │    │
  │  │              │  └───────────────┘  │    │
  │  └───────────────┴─────────────────────┘    │
  └─────────────────────────────────────────────┘

  perf stat -M TopdownL1 ./program

  Resultado ejemplo:
  Retiring:         35%   ← solo 35% de slots hacen trabajo útil
  Bad Speculation:  10%   ← 10% descartado por mispredictions
  Front-end Bound:  15%   ← 15% perdido en fetch/decode
  Back-end Bound:   40%   ← 40% perdido esperando datos
    Memory Bound:   30%
    Core Bound:     10%

  Diagnóstico: este programa es memory-bound (30%).
  Optimización: mejorar data locality, prefetching, reducir cache misses.
```

## Modelo mental

El rendimiento de un CPU es el producto de dos factores: cuántos ciclos por segundo (frecuencia) y cuántas instrucciones por ciclo (IPC). Es como una fábrica: la frecuencia es la velocidad de la línea de montaje, y el IPC es cuántos productos se producen por paso de la línea. Puedes mejorar la producción haciendo la línea más rápida (más GHz) o produciendo más por paso (más IPC). El CPI breakdown te dice exactamente dónde se pierden los ciclos: si es esperando memoria, prediciendo mal branches, o por stalls en el front-end.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| IPC < 1.0 | Memory-bound: cache misses dominan | Data locality, prefetching, NUMA affinity |
| Frecuencia baja bajo carga | Thermal throttling o power limit | Mejorar refrigeración, configurar power limits |
| Rendimiento no escala con cores | Parte serial domina (Amdahl) | Reducir sección serial, mejorar paralelismo |
| Retiring% < 30% | Mayoría de trabajo es especulativo o stalls | TMA para identificar bottleneck específico |

## Herramientas de diagnóstico

```
  perf stat:           IPC, cache misses, branch misses
  perf stat -M:        TMA (Top-Down Microarchitecture Analysis)
  turbostat:           frecuencia, voltaje, potencia por core
  likwid-perfctr:      bandwidth de memoria, FLOPS
  Intel VTune:         análisis microarquitectural detallado
  AMD uProf:           equivalente para AMD
```

## Insights de ingeniería

- **IPC es más difícil de mejorar que frecuencia**: subir frecuencia es "solo" un problema de ingeniería eléctrica (menor voltaje, mejor proceso). Mejorar IPC requiere cambios arquitecturales profundos (ROB más grande, más puertos de ejecución, mejor branch predictor). Por eso cada generación de CPU mejora IPC solo 5-15%, mientras que el salto de frecuencia entre generaciones puede ser mayor con nuevos procesos de fabricación.
- **El rendimiento single-thread sigue importando**: aunque el mundo se mueve a multicore, muchas cargas tienen componentes seriales (garbage collection, query planning, process startup). El rendimiento single-thread determina el floor de latencia de tu sistema. Es la razón por la que Apple Silicon es tan efectivo para desarrollo: IPC extremadamente alto → baja latencia en tareas interactivas.
- **Server workloads tienen IPC notoriamente bajo (~1.0)**: porque son memory-bound (bases de datos, web servers acceden a grandes datasets que no caben en cache). La optimización de servidores se centra más en memory bandwidth, cache efficiency, y NUMA locality que en IPC puro. Por esto los server CPUs (EPYC, Xeon) priorizan cores, cache, y memory channels sobre frecuencia turbo.
