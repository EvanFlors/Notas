# 3.1 Componentes Internos

## Definición

La CPU se compone de unidades funcionales especializadas: la ALU (Arithmetic Logic Unit) ejecuta operaciones aritméticas y lógicas sobre enteros; la FPU (Floating Point Unit) procesa operaciones con punto flotante; la unidad de control orquesta el flujo de datos y genera las señales de control; y los registros almacenan los datos de trabajo con latencia de acceso cercana a cero.

## Mecánica interna

### ALU (Arithmetic Logic Unit)

```
  La ALU es el corazón computacional de la CPU para operaciones con enteros.

  ┌─────────────────────────────────┐
  │            ALU                   │
  │                                 │
  │  Operando A ──▶ ┌─────┐        │
  │                 │     │──▶ Resultado
  │  Operando B ──▶ │     │        │
  │                 └─────┘        │
  │  Opcode    ──▶   Flags ──▶ Status Register
  └─────────────────────────────────┘

  Operaciones aritméticas:
  - ADD, SUB (suma, resta con complemento a 2)
  - MUL (multiplicación — más compleja, puede tomar 3-5 ciclos)
  - DIV (división — la más costosa, 20-90 ciclos según ancho)

  Operaciones lógicas:
  - AND, OR, XOR, NOT (bit a bit)
  - Shift left/right (SHL, SHR, SAR para aritmético)
  - Rotate (ROL, ROR)

  Operaciones de comparación:
  - CMP (resta sin almacenar resultado, solo actualiza flags)
  - TEST (AND sin almacenar resultado)

  Flags (Status Register / EFLAGS en x86):
  ┌──────────┬────────────────────────────────────────┐
  │ Zero (ZF) │ Resultado = 0                         │
  │ Carry (CF)│ Carry/borrow en operación unsigned    │
  │ Sign (SF) │ Bit más significativo del resultado   │
  │ Overflow  │ Overflow en operación signed           │
  │ (OF)      │                                       │
  └──────────┴────────────────────────────────────────┘

  CPUs modernos tienen MÚLTIPLES ALUs (4-6) para ejecución superscalar.
  Un Intel Skylake tiene 4 puertos de ejecución de ALU integer.
```

### FPU (Floating Point Unit)

```
  La FPU es una unidad separada optimizada para aritmética IEEE 754.

  Operaciones:
  - FADD, FSUB, FMUL: 3-5 ciclos de latencia
  - FDIV: 10-20 ciclos (la división es costosa)
  - FSQRT: 10-20 ciclos
  - FMA (Fused Multiply-Add): a × b + c en una sola operación
    → fundamental para ML (dot products, matrix multiply)
    → mayor precisión (un solo redondeo en vez de dos)

  Registros FPU (x86-64):
  - XMM0-XMM15: 128 bits (SSE) — 2× double o 4× float
  - YMM0-YMM15: 256 bits (AVX) — 4× double o 8× float
  - ZMM0-ZMM31: 512 bits (AVX-512) — 8× double o 16× float

  La FPU está pipelined: aunque una multiplicación tarda 5 ciclos,
  puede iniciar una nueva multiplicación cada ciclo.
  Throughput: 1 FMUL/ciclo. Latencia: 5 ciclos.

  Excepciones de punto flotante:
  - Division by zero → ±Infinity
  - Overflow → ±Infinity
  - Underflow → denormalized number o 0
  - Invalid → NaN (e.g., sqrt(-1), 0/0)

  Denormalized numbers (subnormals):
  Números muy cercanos a cero que pierden precisión gradualmente.
  Operar con denormals puede ser 10-100× más lento en algunos CPUs
  (microcode assist). "Flush denormals to zero" es una optimización
  común en audio/video processing.
```

### Unidad de Control

```
  La unidad de control es el "director de orquesta" de la CPU.
  No procesa datos; genera señales de control que dirigen
  el flujo de datos entre los demás componentes.

  Responsabilidades:
  1. Fetch: envía PC al bus de direcciones, activa lectura de I-cache
  2. Decode: interpreta el opcode, genera señales de control
  3. Dispatch: asigna instrucciones a unidades de ejecución
  4. Control de MUXes: selecciona fuentes de operandos (registro, inmediato, forwarded)
  5. Control de escritura: determina qué registro se escribe y cuándo
  6. Control de memoria: activa lectura/escritura en D-cache

  Implementación:

  Hardwired control:
  Circuitos combinacionales generan señales directamente del opcode.
  Más rápido pero más difícil de modificar. Usado en RISC.

  Microprogrammed control:
  Cada instrucción se traduce a una secuencia de micro-operaciones
  almacenadas en una ROM interna (microcode).
  Más flexible (se puede actualizar microcode sin cambiar hardware).
  Usado en x86 CISC para instrucciones complejas.

  Microcode updates:
  Intel y AMD liberan actualizaciones de microcode que corrigen
  bugs de hardware (e.g., Spectre/Meltdown mitigations).
  El OS carga el microcode al boot.
  Linux: /lib/firmware/intel-ucode/ o amd-ucode/
```

### Registros

```
  Los registros son la memoria más rápida de la máquina.
  Están físicamente dentro del core, conectados directamente
  a la ALU y FPU por cables dedicados.

  x86-64 (AMD64) — Registros visibles al programador:
  ┌──────────────────────────────────────────────┐
  │ General Purpose (64-bit):                    │
  │ RAX, RBX, RCX, RDX, RSI, RDI, RBP, RSP     │
  │ R8, R9, R10, R11, R12, R13, R14, R15        │
  │ (16 registros × 64 bits)                     │
  │                                              │
  │ Sub-registros (x86 legacy):                  │
  │ RAX (64) → EAX (32) → AX (16) → AH:AL (8:8)│
  │                                              │
  │ Registros especiales:                        │
  │ RIP: Program Counter (instruction pointer)   │
  │ RSP: Stack Pointer                           │
  │ RFLAGS: Status flags (ZF, CF, SF, OF)        │
  │                                              │
  │ Segment Registers: CS, DS, SS, ES, FS, GS   │
  │ (mayormente legacy, FS/GS usados para TLS)  │
  └──────────────────────────────────────────────┘

  ARM64 (AArch64):
  ┌──────────────────────────────────────────────┐
  │ General Purpose: X0-X30 (31 × 64-bit)       │
  │ W0-W30: vista de 32 bits de X0-X30          │
  │ XZR: Zero register (lectura siempre 0)      │
  │ SP: Stack Pointer                            │
  │ PC: Program Counter (no accesible directo)   │
  │ NZCV: Condition flags                        │
  └──────────────────────────────────────────────┘

  Registros arquitecturales vs físicos:
  Los 16 registros visibles en x86-64 son ARQUITECTURALES.
  Internamente, el CPU tiene 100-200+ registros FÍSICOS.
  Register renaming mapea arquitecturales a físicos para
  eliminar WAW/WAR hazards y permitir más OoO execution.

  Acceso: ~0.3ns (dentro del core, sin bus)
  Compare: L1 cache ~1ns, RAM ~100ns
```

## Modelo mental

La CPU es una fábrica con departamentos especializados. La ALU es el departamento de matemáticas enteras (sumas, restas, lógica). La FPU es el departamento de matemáticas con decimales (operaciones floating-point, vectoriales). La unidad de control es el gerente de planta: no hace trabajo productivo, pero coordina quién hace qué y cuándo. Los registros son los escritorios de trabajo inmediatos: la información que el trabajador tiene a la mano, sin necesidad de ir al archivo (cache) ni al almacén (RAM).

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Throughput bajo en código FP-heavy | FPU saturada o denormals | FMA instructions, flush denormals |
| Division extremadamente lenta | Integer/FP division es multi-ciclo | Reemplazar con multiplicación por recíproco |
| Register spilling (datos van a stack) | Más variables que registros disponibles | Reducir variables vivas simultáneamente |
| Microcode assist (pérdida de rendimiento) | Operaciones que disparan microcode lento | Evitar denormals, misaligned access |

## Insights de ingeniería

- **La división es la operación más cara**: integer DIV toma 20-90 ciclos vs 1 ciclo para ADD. Los compiladores reemplazan divisiones por constantes con multiplicaciones e shifts cuando es posible (`x / 7` → `x * 0x2492... >> 35`). Evita divisiones en loops calientes.
- **Register pressure importa**: si tu función usa más variables simultáneas que registros disponibles, el compilador "derrama" (spills) variables al stack (memoria L1). Un spill es una lectura/escritura a L1 (~4 ciclos) en vez de un acceso a registro (~0.3ns). El código con muchas variables locales activas simultáneamente es más lento.
- **FMA es la instrucción más importante para ML**: `a * b + c` en una sola instrucción con un solo redondeo. Un core moderno puede ejecutar 2 FMA de 256 bits por ciclo = 2 × 8 = 16 float32 multiply-adds por ciclo. A 4GHz = 128 GFLOPS por core. Esto es lo que permite que las CPUs puedan hacer inference de modelos pequeños sin GPU.
