# 1.5 Representación de Datos

## Definición

La representación de datos define cómo se codifican números, texto, y estructuras de datos en secuencias de bits. El sistema binario es la base: todo dato en una computadora es, en última instancia, una secuencia de 0s y 1s. El complemento a 2 permite representar números negativos. Las codificaciones de texto (ASCII, Unicode) mapean caracteres a valores numéricos. Endianness determina el orden en que los bytes se almacenan en memoria.

## Mecánica interna

### Sistema binario

```
  Base 2: cada dígito (bit) tiene valor 0 o 1.
  El valor de un número binario se calcula como suma de potencias de 2.

  Ejemplo: 1101₂
  = 1×2³ + 1×2² + 0×2¹ + 1×2⁰
  = 8 + 4 + 0 + 1
  = 13₁₀

  Unidades:
  ┌────────────────────────────────────────────┐
  │ 1 bit  = 0 o 1                             │
  │ 1 nibble = 4 bits (0-15, un dígito hex)    │
  │ 1 byte = 8 bits (0-255)                    │
  │ 1 word = depende de la arquitectura:       │
  │          32 bits en x86, 64 bits en x86-64 │
  └────────────────────────────────────────────┘

  Hexadecimal (base 16): representación compacta de binario.
  Cada dígito hex = 4 bits (1 nibble).

  0000 = 0    0100 = 4    1000 = 8    1100 = C
  0001 = 1    0101 = 5    1001 = 9    1101 = D
  0010 = 2    0110 = 6    1010 = A    1110 = E
  0011 = 3    0111 = 7    1011 = B    1111 = F

  Ejemplo: 0xDEADBEEF = 1101 1110 1010 1101 1011 1110 1110 1111₂


  Operaciones binarias:
  Suma:      1011        Shift left (×2):    1011 << 1 = 10110
           + 0110        Shift right (÷2):   1011 >> 1 = 0101
           ------
            10001        AND (máscara):  1011 & 0110 = 0010
                         OR  (set bits): 1011 | 0100 = 1111
                         XOR (toggle):   1011 ^ 0110 = 1101

  Bit masking es omnipresente en sistemas:
  - Flags en registros de CPU: each bit = un flag
  - Permisos Unix: rwxr-xr-x = 755₈ = 111 101 101₂
  - IP subnetting: 255.255.255.0 = máscara de red
  - Feature flags en protocolos: cada bit habilita una feature
```

### Complemento a 2 — números con signo

```
  Problema: representar números negativos en binario.

  Complemento a 2 (el estándar universal):
  - Los números positivos se representan normalmente.
  - Para negar un número: invertir todos los bits y sumar 1.
  - El bit más significativo (MSB) indica el signo: 0 = positivo, 1 = negativo.

  Ejemplo con 8 bits:
  +5  = 00000101
  -5  = 11111011  (invertir: 11111010, sumar 1: 11111011)

  Verificación: 00000101 + 11111011 = 100000000 (9 bits, el carry se descarta) = 0 ✓

  Rango para N bits:  [-2^(N-1), 2^(N-1) - 1]
  ┌─────────┬──────────────────────────────┐
  │ 8 bits  │ -128 a +127                  │
  │ 16 bits │ -32,768 a +32,767            │
  │ 32 bits │ -2,147,483,648 a +2,147,483,647 │
  │ 64 bits │ -9.2 × 10^18 a +9.2 × 10^18 │
  └─────────┴──────────────────────────────┘

  Ventaja del complemento a 2:
  La suma y resta usan el MISMO circuito hardware (el sumador).
  A - B = A + (-B) = A + (complemento a 2 de B).
  No se necesita un circuito separado para resta.


  Overflow:
  Si la suma de dos positivos produce un negativo (o viceversa):
  01111111 (+127) + 00000001 (+1) = 10000000 (-128) ← overflow!

  El CPU detecta overflow con el flag de overflow (OF en x86).
  En lenguajes como C, el signed integer overflow es undefined behavior.
  En Rust, el compilador inserta checks en modo debug.


  Punto flotante (IEEE 754):
  Para números con decimales: signo + exponente + mantisa.

  32 bits (float):  1 bit signo + 8 bits exponente + 23 bits mantisa
  64 bits (double): 1 bit signo + 11 bits exponente + 52 bits mantisa

  Ejemplo: -6.25 en float:
  Signo: 1 (negativo)
  6.25 = 110.01₂ = 1.1001 × 2²
  Exponente: 2 + 127 (bias) = 129 = 10000001₂
  Mantisa: 10010000000000000000000

  Resultado: 1 10000001 10010000000000000000000

  Problemas de punto flotante:
  0.1 + 0.2 ≠ 0.3 en punto flotante (error de representación)
  0.1₁₀ es un decimal periódico en binario → no se puede representar exactamente.
  Por eso NUNCA uses float para dinero. Usa enteros (centavos) o tipos decimales.
```

### Codificación de texto

```
  ASCII (American Standard Code for Information Interchange):
  7 bits → 128 caracteres.

  ┌──────────┬──────────────────┐
  │ 0-31     │ Caracteres de control (newline=10, tab=9, null=0) │
  │ 32-47    │ Símbolos (espacio=32, !=33, "=34) │
  │ 48-57    │ Dígitos 0-9 │
  │ 65-90    │ Letras mayúsculas A-Z │
  │ 97-122   │ Letras minúsculas a-z │
  │ 127      │ DEL │
  └──────────┴──────────────────┘

  'A' = 65 = 0x41 = 01000001₂
  'a' = 97 = 0x61 = 01100001₂
  Diferencia: bit 5 (0x20 = 32). Convertir mayúscula a minúscula: OR 0x20.

  ASCII solo cubre inglés. Para el resto del mundo:


  Unicode:
  Estándar que asigna un número (code point) a cada carácter de cada idioma.
  ~150,000 caracteres definidos. Code points: U+0000 a U+10FFFF.

  'A' = U+0041
  'ñ' = U+00F1
  '日' = U+65E5
  '😀' = U+1F600

  Unicode define QUÉ caracteres existen. La codificación define
  CÓMO se almacenan en bytes:


  UTF-8 (el estándar dominante):
  Codificación de longitud variable: 1 a 4 bytes por carácter.

  ┌───────────────┬────────┬──────────────────────────┐
  │ Rango         │ Bytes  │ Patrón                   │
  ├───────────────┼────────┼──────────────────────────┤
  │ U+0000-007F   │ 1      │ 0xxxxxxx                │
  │ U+0080-07FF   │ 2      │ 110xxxxx 10xxxxxx       │
  │ U+0800-FFFF   │ 3      │ 1110xxxx 10xxxxxx 10xx  │
  │ U+10000-10FFFF│ 4      │ 11110xxx 10xx 10xx 10xx │
  └───────────────┴────────┴──────────────────────────┘

  ASCII (0-127) es idéntico en UTF-8 → compatibilidad total.
  Texto en inglés: 1 byte/carácter (eficiente).
  Texto en español (ñ, á): 2 bytes para caracteres acentuados.
  CJK (chino, japonés, coreano): 3 bytes/carácter.
  Emojis: 4 bytes/carácter.

  UTF-8 es auto-sincronizante: si empiezas a leer en medio de un
  carácter multi-byte, puedes detectarlo porque los bytes de
  continuación empiezan con 10xxxxxx.


  UTF-16: 2 o 4 bytes por carácter. Usado internamente en Windows y Java.
  UTF-32: 4 bytes por carácter siempre. Simple pero desperdicia espacio.
```

### Endianness

```
  Endianness define el orden en que los bytes de un valor multi-byte
  se almacenan en memoria.

  Valor: 0x12345678 (32 bits = 4 bytes)

  Big-endian (byte más significativo primero):
  Dirección: 0x00  0x01  0x02  0x03
  Contenido:  12    34    56    78

  Little-endian (byte menos significativo primero):
  Dirección: 0x00  0x01  0x02  0x03
  Contenido:  78    56    34    12


  ¿Quién usa qué?
  ┌──────────────────────────────────────────────────┐
  │ Little-endian: x86, x86-64 (Intel, AMD), ARM    │
  │                en modo default, RISC-V           │
  │                                                  │
  │ Big-endian: Network byte order (TCP/IP),         │
  │             Java bytecode, PowerPC (legacy)      │
  │                                                  │
  │ Bi-endian: ARM (configurable), MIPS, PowerPC     │
  └──────────────────────────────────────────────────┘

  Network byte order = big-endian.
  Cuando envías un número por la red, debes convertirlo:
  htonl() = host-to-network-long (convierte a big-endian)
  ntohl() = network-to-host-long (convierte desde big-endian)

  Si no conviertes, un servidor x86 (little-endian) enviando el puerto
  8080 (0x1F90) a otro servidor se interpreterá como 0x901F = 36895.


  Ventaja de little-endian:
  Leer un valor de 8, 16, o 32 bits desde la misma dirección de memoria
  siempre empieza por el byte menos significativo.
  *(uint8_t*)&x  lee los bits 0-7
  *(uint16_t*)&x lee los bits 0-15
  *(uint32_t*)&x lee los bits 0-31
  No necesitas ajustar la dirección según el tamaño del tipo.
```

## Modelo mental

Todo dato en una computadora es una secuencia de bits. El significado de esos bits depende completamente de la interpretación: los mismos 8 bits 01000001 pueden ser el número 65, la letra 'A', parte de una instrucción de máquina, o un pixel gris. El sistema binario es la base, el complemento a 2 permite restar con el mismo circuito que suma, IEEE 754 permite representar decimales (con errores inherentes), Unicode asigna números a caracteres, UTF-8 los codifica eficientemente, y endianness determina el orden de los bytes en memoria. Todos estos son contratos de interpretación sobre la misma materia prima: bits.

## Modos de fallo

| Síntoma | Causa | Solución |
|---------|-------|----------|
| Integer overflow silencioso | Superar rango de complemento a 2 | Usar tipos más grandes o checked arithmetic |
| 0.1 + 0.2 ≠ 0.3 | Error de representación en IEEE 754 | Comparar con epsilon, usar decimal para dinero |
| Texto corrupto (mojibake) | Interpretar UTF-8 como Latin-1 o viceversa | Especificar encoding explícitamente siempre |
| Valores incorrectos en red | Endianness mismatch sin htonl/ntohl | Siempre convertir a network byte order |

## Insights de ingeniería

- **UTF-8 ganó la web**: >98% de las páginas web usan UTF-8. Es el default en Linux, Git, y la mayoría de APIs modernas. Si tu sistema no asume UTF-8, estás nadando contra corriente. En bases de datos, usa `utf8mb4` en MySQL (no `utf8`, que solo soporta 3 bytes y falla con emojis).
- **Endianness importa en serialización**: si serializas structs directamente a disco o red (sin un formato como JSON o protobuf), endianness te morderá cuando tu sistema corra en una arquitectura diferente. Protobuf, MessagePack, y todos los formatos de serialización modernos definen explícitamente el byte order.
- **Bit manipulation es la base del rendimiento extremo**: operaciones como population count (contar bits en 1), leading zeros (encontrar el bit más significativo), y bit masking son instrucciones nativas del CPU (POPCNT, LZCNT en x86). Las usan bases de datos (bitmap indexes), networking (IP masking), y estructuras de datos (Bloom filters, hash maps con open addressing).
