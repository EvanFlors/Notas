## 1. Introducción

El enfoque **declarativo** es el corazón de Kubernetes en producción: describes el estado deseado en archivos YAML, y Kubernetes se encarga de converger el estado actual hacia ese objetivo.

Este archivo profundiza en:
- La estructura de los manifiestos YAML
- Cómo explorar la API para construir specs
- El mecanismo interno de `kubectl apply` (three-way merge)
- Las reglas de merge por tipo de campo

---

## 2. Objetos en Kubernetes

### ¿Qué es un objeto?

Un objeto de Kubernetes es una **entidad persistente** que representa el estado deseado del clúster. Es un "registro de intención" — al crearlo, le dices a Kubernetes cómo quieres que se vea tu workload.

```
                    ┌──────────────┐
    Tú declaras ──► │   Objeto     │ ◄── Kubernetes reconcilia
    (desired state) │  K8s (etcd)  │     (actual state)
                    └──────────────┘
```

Los objetos describen:
- Qué aplicaciones containerizadas están corriendo (y en qué nodos)
- Los recursos disponibles para esas aplicaciones
- Las políticas de comportamiento (restart, upgrades, tolerancia a fallos)

### Spec vs Status

Todo objeto tiene dos campos principales:

| Campo | Quién lo escribe | Propósito |
|-------|------------------|-----------|
| `spec` | **Tú** (el usuario) | Estado deseado — qué quieres |
| `status` | **Kubernetes** (control plane) | Estado actual — qué existe realmente |

```
  ┌─────────────────────────────────────────────────┐
  │                    Deployment                    │
  │                                                  │
  │   spec:                    status:               │
  │     replicas: 3              replicas: 3         │
  │     image: nginx:1.25        availableReplicas: 3│
  │                              conditions:         │
  │   (lo que TÚ defines)       (lo que K8s reporta)│
  └─────────────────────────────────────────────────┘
                        │
          Control Plane reconcilia continuamente
          spec (deseado) vs status (actual)
```

> 💡 Si un Pod falla, Kubernetes ve que `status.replicas < spec.replicas` y crea uno nuevo. Este **reconciliation loop** es el motor fundamental de Kubernetes.

## 3. Estructura de un manifiesto YAML

Todo manifiesto requiere **cuatro campos raíz**:

```yaml
apiVersion: apps/v1          # ① Versión de la API
kind: Deployment              # ② Tipo de recurso
metadata:                     # ③ Identificación
  name: nginx-deployment
  labels:
    app: nginx
spec:                         # ④ Especificación (estado deseado)
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.25.0
        ports:
        - containerPort: 80
```

### Desglose de cada campo

| Campo | Descripción | Ejemplo |
|-------|-------------|---------|
| `apiVersion` | Grupo y versión de la API | `v1`, `apps/v1`, `batch/v1` |
| `kind` | Tipo de recurso | `Pod`, `Deployment`, `Service` |
| `metadata` | Datos de identificación | `name` (obligatorio), `labels`, `annotations`, `namespace` |
| `spec` | Estado deseado del recurso | Varía completamente según el `kind` |

### Múltiples manifiestos en un archivo

Separa con `---`:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp
spec:
  # ...
---
apiVersion: v1
kind: Service
metadata:
  name: webapp-svc
spec:
  # ...
```

## 4. Construyendo archivos YAML

### Descubrir recursos y versiones

```bash
# ¿Qué tipos de recursos soporta el clúster?
kubectl api-resources
# Columnas: NAME, SHORTNAMES, APIVERSION, NAMESPACED, KIND

# ¿Qué versiones de API están disponibles?
kubectl api-versions
# Ejemplo: apps/v1, batch/v1, v1, networking.k8s.io/v1
```

> ⚠️ Algunos recursos tienen múltiples versiones de API (vieja vs nueva). Siempre usa la versión estable (`v1`) sobre betas.

### Explorar la spec con `kubectl explain`

`kubectl explain` es tu **documentación interactiva** para construir YAML:

```bash
# Ver todos los campos de un recurso (recursivo)
kubectl explain deployment --recursive
# → Muestra TODA la estructura (puede ser muy largo)

# Ver campos de primer nivel
kubectl explain deployment.spec
# → Muestra campos directos de spec con descripciones

# Navegar campo por campo
kubectl explain deployment.spec.replicas
kubectl explain deployment.spec.template.spec.containers
kubectl explain deployment.spec.strategy.rollingUpdate

# Ver specs anidados
kubectl explain deployment.spec.template.spec.volumes.nfs.server
```

### Flujo para construir un YAML desde cero

```
  1. kubectl api-resources         → Encontrar el kind
              │
  2. kubectl api-versions          → Encontrar el apiVersion
              │
  3. kubectl explain KIND          → Ver estructura general
              │
  4. kubectl explain KIND.spec     → Ver campos del spec
              │
  5. Generar base con --dry-run    → Plantilla inicial
              │
  6. Editar y personalizar         → Añadir tu configuración
              │
  7. kubectl diff -f file.yaml    → Verificar cambios
              │
  8. kubectl apply -f file.yaml   → Aplicar
```

### Generar plantilla base

```bash
# Deployment
kubectl create deployment webapp --image=nginx \
  --dry-run=client -o yaml > webapp.yaml

# Pod
kubectl run test-pod --image=busybox \
  --dry-run=client -o yaml > pod.yaml

# Service
kubectl create service clusterip my-svc --tcp=80:80 \
  --dry-run=client -o yaml > svc.yaml
```

### Documentación adicional

- Referencia API: `kubernetes.io/docs/reference/#api-reference`
- Cada campo está documentado con tipo, descripción y si es obligatorio

## 5. kubectl apply en profundidad

### Operaciones básicas

```bash
# Crear o actualizar desde archivo
kubectl apply -f deployment.yaml

# Desde directorio completo
kubectl apply -f manifests/

# Recursivo (subdirectorios)
kubectl apply -R -f manifests/

# Desde URL
kubectl apply -f https://example.com/manifest.yaml

# Con filtro de labels
kubectl apply -f manifests/ -l app=nginx
```

### La anotación `last-applied-configuration`

Cada vez que ejecutas `kubectl apply`, Kubernetes almacena el contenido del archivo YAML en una anotación del objeto:

```yaml
metadata:
  annotations:
    kubectl.kubernetes.io/last-applied-configuration: |
      {"apiVersion":"apps/v1","kind":"Deployment","metadata":{"name":"nginx"},
       "spec":{"replicas":3,"template":{"spec":{"containers":[{"image":"nginx:1.25",
       "name":"nginx"}]}}}}
```

Esta anotación es **esencial** para el three-way merge: permite a `apply` saber qué campos estaban en el archivo anterior y cuáles se eliminaron.

> ⚠️ Por esto **no debes mezclar** `kubectl create` con `kubectl apply`: `create` no escribe esta anotación, y `apply` no puede calcular correctamente los diffs sin ella.

## 6. Dry Runs y Diffs

### Dry run (simulación)

```bash
# Client-side: solo valida sintaxis local, no contacta al servidor
kubectl apply -f app.yaml --dry-run=client

# Server-side: envía al servidor sin persistir (valida permisos, webhooks, etc.)
kubectl apply -f app.yaml --dry-run=server
```

### kubectl diff

```bash
# Ver qué cambiaría al aplicar
kubectl diff -f deployment.yaml
```

Salida en formato diff estándar:

```diff
-  replicas: 3
+  replicas: 5
-  image: nginx:1.24
+  image: nginx:1.25
```

> 💡 `diff` usa internamente server-side dry-run. Requiere permisos de `PATCH`, `CREATE` y `UPDATE`.

### Flujo recomendado

```bash
# 1. Editar el YAML
vim deployment.yaml

# 2. Ver qué va a cambiar
kubectl diff -f deployment.yaml

# 3. Si el diff es correcto, aplicar
kubectl apply -f deployment.yaml

# 4. Verificar
kubectl get deployment nginx -o yaml
```

## 7. Cómo apply calcula el merge

`kubectl apply` envía un **patch** al API server (no un replace). El patch se calcula comparando tres fuentes:

```
  ┌───────────────────────┐
  │  Configuration File   │  ← Lo que tiene tu archivo YAML ahora
  │  (archivo local)      │
  └───────────┬───────────┘
              │
              ▼
  ┌───────────────────────┐     ┌─────────────────────────┐
  │  Last-Applied Config  │     │   Live Configuration    │
  │  (anotación en etcd)  │     │   (estado real en etcd) │
  └───────────┬───────────┘     └────────────┬────────────┘
              │                               │
              └───────────┬───────────────────┘
                          │
                          ▼
              ┌───────────────────────┐
              │   Merge Patch Calc    │
              ├───────────────────────┤
              │ 1. Campos a ELIMINAR: │
              │    en last-applied    │
              │    pero NO en archivo │
              │                       │
              │ 2. Campos a AÑADIR/   │
              │    MODIFICAR:         │
              │    en archivo pero    │
              │    diferentes en live │
              └───────────┬───────────┘
                          │
                          ▼
              ┌───────────────────────┐
              │   PATCH Request       │
              │   → API Server        │
              └───────────────────────┘
```

### Ejemplo paso a paso

**Estado 1**: Aplicamos YAML inicial

```yaml
spec:
  replicas: 3
  minReadySeconds: 10
  template:
    spec:
      containers:
      - name: nginx
        image: nginx:1.14.2
```

**Estado 2**: Alguien ejecuta `kubectl scale --replicas=2`

Resultado en live: `replicas: 2`, pero `last-applied` sigue con `replicas: 3`

**Estado 3**: Actualizamos el archivo (cambiamos imagen, eliminamos minReadySeconds)

```yaml
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: nginx
        image: nginx:1.25.0     # Cambió
  # minReadySeconds eliminado
```

**Cálculo del merge**:

| Campo | En archivo | En last-applied | En live | Acción |
|-------|-----------|-----------------|---------|--------|
| `image` | `nginx:1.25.0` | `nginx:1.14.2` | `nginx:1.14.2` | **SET** → `nginx:1.25.0` |
| `minReadySeconds` | ❌ ausente | `10` | `10` | **CLEAR** → eliminado |
| `replicas` | `3` | `3` | `2` | **SET** → `3` (⚠️ sobrescribe el scale) |

> 💡 Si quieres que `kubectl scale` se preserve, **omite** `replicas` del YAML.

## 8. Merge por tipo de campo

### Campos primitivos (string, integer, boolean)

Regla: **Replace o clear**.

| En archivo | En live | En last-applied | Acción |
|-----------|---------|-----------------|--------|
| ✅ Sí | ✅ Sí | — | Set live al valor del archivo |
| ✅ Sí | ❌ No | — | Set live al valor del archivo |
| ❌ No | — | ✅ Sí | **Clear** del live |
| ❌ No | — | ❌ No | No hacer nada, mantener live |

### Campos map (object)

Regla: **Merge** — se comparan subcampos individualmente.

```yaml
# Archivo YAML
metadata:
  labels:
    app: nginx          # ← Mantener
    # env: prod         # ← Eliminado del archivo

# Live
metadata:
  labels:
    app: nginx
    env: prod
    team: backend       # ← Añadido por otro writer
```

Resultado: `app: nginx` se mantiene, `env: prod` se **elimina** (estaba en last-applied), `team: backend` se **preserva** (no estaba en last-applied).

### Campos list

Tres estrategias dependiendo del campo:

#### 1. Lista de primitivos → **Replace completo**

```yaml
# last-applied:  args: ["a", "b"]
# archivo:       args: ["a", "c"]
# live:          args: ["a", "b", "d"]
# resultado:     args: ["a", "c"]    ← Replace completo con el archivo
```

#### 2. Lista de maps con `patchMergeKey` → **Merge por key**

Los contenedores usan `name` como `patchMergeKey`:

```yaml
# last-applied
containers:
- name: nginx         # Key: "nginx"
  image: nginx:1.14
- name: helper-a      # Key: "helper-a"
  image: busybox

# archivo actual
containers:
- name: nginx
  image: nginx:1.25   # ← Actualizado
- name: helper-c      # ← Nuevo
  image: alpine

# live
containers:
- name: nginx
  image: nginx:1.14
- name: helper-a
  image: busybox
- name: helper-d      # ← Añadido por otro writer
  image: debug:latest

# resultado
containers:
- name: nginx
  image: nginx:1.25   # ← Actualizado
- name: helper-c      # ← Añadido (del archivo)
  image: alpine
- name: helper-d      # ← Preservado (no estaba en last-applied)
  image: debug:latest
# helper-a ELIMINADO  # ← Estaba en last-applied pero no en archivo
```

#### 3. Sin `patchStrategy` definido → **Replace**

Si un campo de tipo lista no tiene `patchStrategy` configurado en el código fuente de Kubernetes, se trata como replace completo.

## 9. Valores por defecto del servidor

El API server asigna **valores por defecto** a campos no especificados al crear un objeto.

### El problema

```yaml
# Tu archivo (sin strategy)
spec:
  replicas: 3
  template: ...

# El servidor añade por defecto
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 25%
      maxUnavailable: 25%
```

Si luego cambias `strategy.type` a `Recreate` **sin limpiar** `rollingUpdate`:

```yaml
spec:
  strategy:
    type: Recreate
    # rollingUpdate sigue presente del default → ERROR
```

> ❌ `strategy.rollingUpdate` no es válido con `strategy.type: Recreate`.

### Recomendación

**Define explícitamente** en tu YAML los campos que el servidor defaultea, especialmente:

- `strategy` en Deployments
- Selectors y labels de PodTemplate
- `restartPolicy` en Pods

```yaml
# Explícito y seguro
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 25%
      maxUnavailable: 25%
```

### Limpiar campos defaulteados

Para forzar re-default de un campo, establécelo a `null`:

```yaml
spec:
  strategy:
    type: Recreate
    rollingUpdate: null    # ← Limpia explícitamente
```

## 10. Ownership de campos

### ¿Quién "posee" un campo?

Dos entidades pueden escribir campos:
1. **Tu archivo YAML** (via `kubectl apply`)
2. **Writers directos** (via `kubectl scale`, HPA, controllers)

### Cambiar ownership

#### De writer directo → archivo YAML

```bash
# 1. Añade el campo al archivo YAML
# 2. Deja de modificarlo directamente
# 3. kubectl apply se encarga desde ahora
```

#### De archivo YAML → writer directo

```bash
# 1. Elimina el campo del archivo YAML
# 2. Elimina el campo de la anotación last-applied
kubectl edit deployment nginx
# → Eliminar el campo de la anotación manualmente
# 3. Ahora el writer directo puede gestionarlo
```

## 11. Eliminación de objetos

### Método recomendado: explícito

```bash
kubectl delete -f deployment.yaml
```

Más explícito, menos riesgo de eliminar algo por error.

### Método alternativo: prune

```bash
# Elimina objetos que ya no están en el directorio
kubectl apply -f manifests/ --prune -l app=myapp \
  --prune-allowlist=apps/v1/Deployment \
  --prune-allowlist=v1/Service
```

> ⚠️ **Cuidado con `--prune`**: sin `--selector` y `--prune-allowlist` puede eliminar recursos inesperadamente. Siempre usa labels para acotar el scope.

### ApplySet-based pruning (v1.27+, alpha)

Método más seguro que usa un Secret server-side para rastrear membresía:

```bash
kubectl apply -f manifests/ --prune --applyset=my-set
```

## 12. Buenas prácticas

| Práctica | Razón |
|----------|-------|
| Siempre `kubectl diff` antes de `apply` | Ver cambios antes de aplicarlos |
| Guardar YAML en Git | Historial, rollback, code review |
| Un commit = un cambio lógico | Trazabilidad clara |
| Definir campos defaulteados explícitamente | Evitar sorpresas con valores del servidor |
| Omitir `replicas` si usas HPA | Permitir que el autoscaler controle el campo |
| Usar `--dry-run=server` para validar | Detecta errores de permisos y webhooks |
| No mezclar `create`/`replace` con `apply` | Rompe el three-way merge |
| Usar `kubectl explain` para construir specs | Documentación integrada y actualizada |

## 13. Errores comunes

| # | Error | Consecuencia | Solución |
|---|-------|-------------|----------|
| 1 | Mezclar `apply` con `create`/`replace` | `last-applied-configuration` corrupta | Usar un solo método por objeto |
| 2 | No entender que `replicas` en YAML sobrescribe `scale` | Perder escalado manual/HPA | Omitir `replicas` del YAML si usas autoescalado |
| 3 | No definir `strategy` explícitamente | Error al cambiar de RollingUpdate a Recreate | Siempre definir `strategy` completo |
| 4 | Usar `--prune` sin labels ni allowlist | Eliminación masiva accidental | Siempre usar `-l` y `--prune-allowlist` |
| 5 | Aplicar desde URLs sin inspeccionar | Ejecutar manifiestos desconocidos | `curl -L URL` primero, luego `apply` |
| 6 | Ignorar la salida de `kubectl diff` | Cambios inesperados en producción | Integrar `diff` en el workflow |
