## 1. Full App Lifecycle con Compose

### 1.1 El sueño: un solo set de archivos

El objetivo final de Docker Compose + Swarm es tener **un conjunto único de archivos de configuración** que sirva para todo el ciclo de vida de tu aplicación:

```
UN SOLO SET DE ARCHIVOS COMPOSE
────────────────────────────────

  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
  │ DESARROLLO  │     │   CI/TEST   │     │ PRODUCCIÓN  │
  │             │     │             │     │             │
  │ docker      │     │ docker      │     │ docker      │
  │ compose up  │     │ compose up  │     │ stack       │
  │             │     │ -f ... -f   │     │ deploy      │
  └──────┬──────┘     └──────┬──────┘     └──────┬──────┘
         │                   │                    │
         └───────────┬───────┘────────────────────┘
                     │
              docker-compose.yml      ← Base (común a todos)
              + override por entorno  ← Específico por entorno
```

### 1.2 Estructura de archivos

```
proyecto/
├── Dockerfile
├── docker-compose.yml              ← Base: servicios, imágenes
├── docker-compose.override.yml     ← Dev: build, bind mounts, ports
├── docker-compose.test.yml         ← CI: build, test data, no volumes
├── docker-compose.prod.yml         ← Prod: deploy, replicas, secrets
└── secrets/
    ├── db_password.txt             ← Solo para dev local
    └── db_root_password.txt
```

| Archivo | Propósito | Quién lo usa |
|---|---|---|
| `docker-compose.yml` | Definición base (servicios, imágenes, redes) | Todos los entornos |
| `docker-compose.override.yml` | Overrides para desarrollo local | `docker compose up` (automático) |
| `docker-compose.test.yml` | Config para CI/testing | `docker compose -f ... -f ... up` |
| `docker-compose.prod.yml` | Config para producción | `docker stack deploy` |

### 1.3 Ejemplo: archivo base

```yaml
# docker-compose.yml — Base común
version: "3.9"

services:
  drupal:
    image: custom-drupal:latest

  postgres:
    image: postgres:15
```

> Este archivo define **solo lo mínimo** que es común a todos los entornos: qué servicios existen y qué imágenes usan.

## 2. Compose overrides y multi-entorno

### 2.1 Override automático (desarrollo)

Cuando ejecutas `docker compose up`, Docker Compose lee **automáticamente** dos archivos en orden:
1. `docker-compose.yml` (base)
2. `docker-compose.override.yml` (si existe)

No necesitas especificar `-f` para el override — es automático.

```yaml
# docker-compose.override.yml — Desarrollo local
services:
  drupal:
    build: .                           # ← Build local (no existe en prod)
    ports:
      - "8080:80"                      # ← Puerto para dev
    volumes:
      - drupal-modules:/var/www/html/modules
      - drupal-profiles:/var/www/html/profiles
      - drupal-sites:/var/www/html/sites
      - ./themes:/var/www/html/themes  # ← Bind mount para hot-reload

  postgres:
    environment:
      - POSTGRES_PASSWORD_FILE=/run/secrets/psql_password
    secrets:
      - psql_password
    volumes:
      - drupal-data:/var/lib/postgresql/data

volumes:
  drupal-data:
  drupal-modules:
  drupal-profiles:
  drupal-sites:

secrets:
  psql_password:
    file: ./secrets/psql_password.txt  # ← File-based (ok para dev)
```

```bash
# Esto lee automáticamente base + override
$ docker compose up -d

# Verificar que aplicó el override (bind mounts presentes)
$ docker inspect drupal | grep Mounts
```

### 2.2 Override manual (CI/Test)

Para CI, especificas **explícitamente** qué archivos combinar:

```yaml
# docker-compose.test.yml — CI environment
services:
  drupal:
    build: .
    ports:
      - "80:80"
    # Sin bind mounts — no necesitamos persistir nada en CI

  postgres:
    environment:
      - POSTGRES_PASSWORD=test_password  # ← Ok en CI, no es producción
    volumes:
      - ./sample-data:/var/lib/postgresql/data  # ← Datos de prueba
```

```bash
# CI: especificar archivos explícitamente
$ docker compose -f docker-compose.yml -f docker-compose.test.yml up -d

# Nota: NO se aplica el override.yml porque especificamos -f
```

### 2.3 Override para producción

```yaml
# docker-compose.prod.yml — Producción
services:
  drupal:
    ports:
      - "80:80"
    volumes:
      - drupal-modules:/var/www/html/modules
      - drupal-profiles:/var/www/html/profiles
      - drupal-sites:/var/www/html/sites
    deploy:                              # ← Solo Swarm lee esto
      replicas: 2
      update_config:
        parallelism: 1
        delay: 10s
      restart_policy:
        condition: on-failure

  postgres:
    environment:
      - POSTGRES_PASSWORD_FILE=/run/secrets/psql_password
    secrets:
      - psql_password
    volumes:
      - drupal-data:/var/lib/postgresql/data
    deploy:
      replicas: 1
      placement:
        constraints: [node.role == manager]

volumes:
  drupal-data:
  drupal-modules:
  drupal-profiles:
  drupal-sites:

secrets:
  psql_password:
    external: true                       # ← Pre-creado en Swarm
```

### 2.4 Generar config combinada para producción

El comando `config` combina múltiples archivos Compose y genera un **YAML unificado**:

```bash
# Combinar base + prod y generar archivo para stack deploy
$ docker compose -f docker-compose.yml -f docker-compose.prod.yml config > output.yml

# Inspeccionar el resultado
$ cat output.yml

# Desplegar con el archivo generado
$ docker stack deploy -c output.yml myapp
```

> **¿Por qué generar `output.yml`?** Porque `docker stack deploy` no soporta múltiples `-f` como `docker compose`. Necesitas un solo archivo.

### 2.5 Diagrama: Flujo de override

```
docker-compose.yml (BASE)
  │
  │  services:
  │    drupal:
  │      image: custom-drupal
  │    postgres:
  │      image: postgres:15
  │
  ├──────────────────┬──────────────────┬──────────────────┐
  │                  │                  │                  │
  ▼                  ▼                  ▼                  ▼
override.yml       test.yml          prod.yml           config
(automático)       (-f ... -f)       (-f ... -f)        (output)
  │                  │                  │                  │
  │  + build: .      │  + build: .     │  + deploy:       │  YAML
  │  + ports: 8080   │  + ports: 80    │  + replicas      │  unificado
  │  + bind mounts   │  + sample-data  │  + external      │  para
  │  + file secrets  │  + test pass    │    secrets       │  stack
  │                  │                  │                  │
  ▼                  ▼                  ▼                  ▼
docker compose up  compose up -f     docker stack       deploy
(local dev)        (CI pipeline)     deploy -c output   (Swarm)
```

## 3. Compose extends

### 3.1 ¿Qué es `extends`?

El atributo `extends` permite **reutilizar la configuración de un servicio** definido en otro archivo o en el mismo archivo. Es una forma más declarativa de compartir configuración común.

```yaml
# common-services.yml — Configuración compartida
services:
  webapp:
    build: .
    ports:
      - "8000:8000"
    volumes:
      - "/data"
```

```yaml
# docker-compose.yml — Extender desde otro archivo
services:
  web:
    extends:
      file: common-services.yml
      service: webapp
    environment:
      - DEBUG=1
```

> Resultado: el servicio `web` hereda `build`, `ports` y `volumes` de `webapp`, y **añade** `environment`.

### 3.2 Extends dentro del mismo archivo

```yaml
services:
  web:
    build: ./alpine
    extends: webapp        # ← Hereda de webapp (mismo archivo)

  webapp:
    environment:
      - DEBUG=1
```

> Ambos servicios (`web` y `webapp`) serán parte de la configuración final.

### 3.3 Ejemplo práctico: compartir config entre servicios

```yaml
# common.yaml — App base
services:
  app:
    build: .
    environment:
      CONFIG_FILE_PATH: /code/config
      API_KEY: xxxyyy
    cpu_shares: 5
```

```yaml
# compose.yaml — Servicios concretos
services:
  webapp:
    extends:
      file: common.yaml
      service: app
    command: /code/run_web_app
    ports:
      - 8080:8080
    depends_on:
      - queue
      - db

  queue_worker:
    extends:
      file: common.yaml
      service: app
    command: /code/run_worker
    depends_on:
      - queue
```

```
RESULTADO:

  webapp                              queue_worker
  ├── build: .           (de common)  ├── build: .           (de common)
  ├── environment:       (de common)  ├── environment:       (de common)
  │   ├── CONFIG_FILE_PATH            │   ├── CONFIG_FILE_PATH
  │   └── API_KEY                     │   └── API_KEY
  ├── cpu_shares: 5      (de common)  ├── cpu_shares: 5      (de common)
  ├── command: run_web_app (propio)   ├── command: run_worker (propio)
  ├── ports: 8080        (propio)     └── depends_on: queue  (propio)
  └── depends_on: queue, db (propio)
```

### 3.4 Rutas relativas con extends

Cuando usas `extends` con `file:` que apunta a otra carpeta, las rutas relativas se resuelven **desde el archivo extendido**, no desde tu archivo:

```yaml
# compose.yaml
services:
  webapp:
    image: example
    extends:
      file: ../commons/compose.yaml
      service: base

# ../commons/compose.yaml
services:
  base:
    env_file: ./container.env    # ← Relativo a commons/

# Resultado (docker compose config):
services:
  webapp:
    image: example
    env_file:
      - ../commons/container.env  # ← Resuelto correctamente
```

> **⚠️ Limitación:** `extends` no funciona actualmente con `docker stack deploy`. Solo funciona con `docker compose`.

## 4. Compose en producción

### 4.1 Ajustes para producción

Al pasar de desarrollo a producción, necesitas modificar tu configuración:

| Cambio | Por qué | Ejemplo |
|---|---|---|
| **Eliminar bind mounts de código** | El código debe estar en la imagen, no montado | Quitar `./src:/app` |
| **Cambiar puertos** | Evitar conflictos con otros servicios | `80:80` en vez de `8080:80` |
| **Variables de entorno** | Reducir logs, apuntar a servicios externos | `LOG_LEVEL=warn` |
| **Restart policy** | Evitar downtime | `restart: always` |
| **Agregar servicios** | Logging, monitoring, reverse proxy | Agregar Prometheus, ELK |
| **Secrets external** | No depender de archivos en el host | `external: true` |

### 4.2 Desplegar cambios en producción

```bash
# Rebuild + redeploy de un servicio específico
$ docker compose build web
$ docker compose up --no-deps -d web

# --no-deps: NO recrea servicios de los que `web` depende
# Solo recrea `web` con la nueva imagen
```

### 4.3 Deploy remoto

```bash
# Configurar conexión al host remoto
$ export DOCKER_HOST=tcp://remote-host:2376
$ export DOCKER_TLS_VERIFY=1
$ export DOCKER_CERT_PATH=~/.docker/certs

# Ahora todos los comandos docker van al host remoto
$ docker compose up -d
$ docker stack deploy -c compose.yml myapp
```

## 5. Service Updates

### 5.1 ¿Qué es un Service Update?

Un service update es el mecanismo de Swarm para **reemplazar progresivamente los tasks (containers) de un servicio** con nuevas versiones. Es la base del zero-downtime deployment.

```
ROLLING UPDATE (default)

  Antes:     [v1.0] [v1.0] [v1.0] [v1.0] [v1.0]    (5 replicas)
                │
  Paso 1:    [v1.1] [v1.0] [v1.0] [v1.0] [v1.0]    ← 1 actualizada
                       │
  Paso 2:    [v1.1] [v1.1] [v1.0] [v1.0] [v1.0]    ← 2 actualizadas
                              │
  Paso 3:    [v1.1] [v1.1] [v1.1] [v1.0] [v1.0]    ← 3 actualizadas
                                     │
  Paso 4:    [v1.1] [v1.1] [v1.1] [v1.1] [v1.0]    ← 4 actualizadas
                                            │
  Paso 5:    [v1.1] [v1.1] [v1.1] [v1.1] [v1.1]    ← Todas actualizadas ✅

  → Por defecto: 1 a la vez, sin delay
  → Configurable: parallelism, delay, failure-action
```

### 5.2 Qué provoca un update

| Cambio | ¿Reemplaza containers? | Ejemplo |
|---|---|---|
| Cambiar imagen | ✅ Sí | `--image nginx:1.25` |
| Agregar env var | ✅ Sí | `--env-add NEW=value` |
| Cambiar puertos | ✅ Sí | `--publish-add 8080:80` |
| Agregar/quitar secrets | ✅ Sí | `--secret-add`, `--secret-rm` |
| Agregar/quitar mounts | ✅ Sí | `--mount-add`, `--mount-rm` |
| Cambiar labels | ❌ No (metadata) | `--label-add env=prod` |
| Cambiar parallelism | ❌ No (config) | `--update-parallelism 2` |
| `--force` | ✅ Sí (forzado) | Rebalancear, rolling restart |

### 5.3 Update vs Rollback: ¿qué orden usan?

```
UPDATE ORDER:

  stop-first (default):
  ──────────────────────
  1. Detener task viejo
  2. Crear task nuevo
  3. Esperar que esté healthy
  4. Continuar al siguiente

  → Menor uso de recursos
  → Breve momento sin ese replica

  start-first:
  ─────────────
  1. Crear task nuevo
  2. Esperar que esté healthy
  3. Detener task viejo
  4. Continuar al siguiente

  → Usa más recursos temporalmente
  → Sin downtime (siempre hay N replicas activas)
```

### 5.4 Comandos de Service Update

#### Actualizar imagen

```bash
# El caso más común: deploy de nueva versión
$ docker service update --image myapp:1.2.1 web
```

#### Agregar/quitar variables de entorno y puertos

```bash
$ docker service update \
    --env-add NODE_ENV=production \
    --publish-rm 8080 \
    --publish-add published=9090,target=80 \
    web
```

#### Escalar servicios

```bash
# Escalar uno
$ docker service scale web=5

# Escalar múltiples a la vez
$ docker service scale web=5 api=3 worker=10
```

#### Rolling restart (sin cambios)

```bash
# Forzar reemplazo de todos los tasks (útil para rebalancear)
$ docker service update --force \
    --update-parallelism 1 \
    --update-delay 30s \
    web
```

> **¿Cuándo usar `--force`?** Cuando añades nodos nuevos y quieres redistribuir tasks. Swarm no rebalancea automáticamente — necesitas forzar un update para que el scheduler coloque tasks en los nodos menos cargados.

#### Rollback

```bash
# Rollback manual a la versión anterior
$ docker service rollback web

# Rollback con subcomando (equivalente)
$ docker service update --rollback web

# Rollback sin delay entre tasks
$ docker service update --rollback --update-delay 0s web
```

#### Rollback automático

```bash
# Configurar rollback automático al crear el servicio
$ docker service create \
    --name web \
    --replicas 5 \
    --update-failure-action rollback \
    --update-max-failure-ratio 0.3 \
    --rollback-parallelism 3 \
    --rollback-delay 5s \
    nginx:1.24

# Si > 30% de los tasks fallan durante un update
# → rollback automático, 3 a la vez, 5s entre cada grupo
```

### 5.5 Parámetros de Update

| Parámetro | Default | Descripción |
|---|---|---|
| `--update-parallelism` | 1 | Tasks a actualizar simultáneamente |
| `--update-delay` | 0s | Espera entre cada grupo de tasks |
| `--update-order` | stop-first | `stop-first` o `start-first` |
| `--update-failure-action` | pause | `pause`, `continue`, o `rollback` |
| `--update-max-failure-ratio` | 0 | % de fallos tolerados (0.0 - 1.0) |
| `--update-monitor` | 5s | Tiempo de monitoreo post-update por task |

### 5.6 Parámetros de Rollback

| Parámetro | Default | Descripción |
|---|---|---|
| `--rollback-parallelism` | 1 | Tasks a rollback simultáneamente |
| `--rollback-delay` | 0s | Espera entre cada grupo |
| `--rollback-failure-action` | pause | Acción si el rollback falla |
| `--rollback-max-failure-ratio` | 0 | % de fallos tolerados en rollback |
| `--rollback-monitor` | 5s | Tiempo de monitoreo post-rollback |
| `--rollback-order` | stop-first | `stop-first` o `start-first` |

### 5.7 Updates con Stack files

No existe un comando separado para "stack update". **Re-desplegar** un stack es el update:

```bash
# Editar docker-compose.yml (cambiar imagen, replicas, etc.)
# Luego re-deploy:
$ docker stack deploy -c docker-compose.yml myapp

# Swarm compara el estado actual con el deseado
# y aplica solo los cambios necesarios como service updates
```

```
STACK DEPLOY = UPDATE INTELIGENTE

  Estado actual                Estado deseado (nuevo YAML)
  ──────────────               ─────────────────────────────
  web: nginx:1.24 (3)    →    web: nginx:1.25 (5)     ← update image + scale
  api: myapp:2.0 (2)     →    api: myapp:2.0 (2)      ← sin cambios
  db: postgres:15 (1)    →    db: postgres:16 (1)      ← update image

  Swarm solo toca web y db, deja api como está.
```

### 5.8 Diagrama: Flujo completo de un Update

```
  docker service update --image nginx:1.25 web
          │
          ▼
  ┌─── Scheduler ─────────────────────────────┐
  │                                            │
  │  1. Seleccionar task a actualizar          │
  │     (según --update-parallelism)           │
  │          │                                 │
  │          ▼                                 │
  │  2. Detener task viejo (stop-first)        │
  │     o crear task nuevo (start-first)       │
  │          │                                 │
  │          ▼                                 │
  │  3. Crear nuevo task con nueva imagen      │
  │          │                                 │
  │          ▼                                 │
  │  4. Esperar healthcheck (si configurado)   │
  │     ┌────┴────┐                            │
  │     │         │                            │
  │   healthy   unhealthy                      │
  │     │         │                            │
  │     │    ┌────┴────┐                       │
  │     │    │ failure-action                   │
  │     │    ├─ pause: detener update          │
  │     │    ├─ continue: seguir adelante      │
  │     │    └─ rollback: revertir todo        │
  │     │                                      │
  │     ▼                                      │
  │  5. Esperar --update-delay                 │
  │          │                                 │
  │          ▼                                 │
  │  6. Siguiente task (loop hasta completar)  │
  │                                            │
  └────────────────────────────────────────────┘
```

## 6. Docker Healthchecks

### 6.1 ¿Qué es un Healthcheck?

Un healthcheck es un comando que Docker ejecuta **dentro del container** periódicamente para verificar que la aplicación funciona correctamente. Antes de healthchecks, Docker solo verificaba que el **proceso principal estuviera corriendo** — pero un proceso puede estar vivo y no funcionar (ej: un web server en loop infinito).

```
SIN HEALTHCHECK:                    CON HEALTHCHECK:
─────────────────                   ──────────────────
Docker pregunta:                    Docker pregunta:
"¿El proceso PID 1                 "¿La app responde
 está corriendo?"                    correctamente?"

  Proceso corriendo                 curl localhost:80
  pero en deadlock  → "healthy"     → 200 OK → healthy
                    (INCORRECTO)    → 500    → unhealthy ← CORRECTO
```

### 6.2 Los tres estados

```
ESTADOS DE UN CONTAINER CON HEALTHCHECK

  Container start
       │
       ▼
  ┌──────────┐     Primera verificación
  │ starting │────────────────────────────┐
  └──────────┘     (espera --interval)    │
                                          ▼
                              ┌─── exit 0? ───┐
                              │               │
                            Sí │             No │
                              ▼               ▼
                        ┌─────────┐    ¿Retries agotados?
                        │ healthy │         │
                        └────┬────┘       Sí │    No
                             │              ▼     │
                        Cada interval   ┌──────────┐
                        re-check        │unhealthy │
                             │          └──────────┘
                             ▼
                        exit 0? → healthy (loop)
                        exit 1? → cuenta retry
```

| Estado | Significado | Acción de Docker |
|---|---|---|
| **starting** | Container recién iniciado, aún no se verificó | Espera `--interval` para primera verificación |
| **healthy** | El healthcheck retornó `exit 0` | Nada (todo bien) |
| **unhealthy** | Falló `--retries` veces consecutivas | `docker run`: solo marca. `service`: reemplaza task |

### 6.3 Opciones del Healthcheck

| Opción | Default | Descripción |
|---|---|---|
| `--interval` | 30s | Frecuencia de ejecución del healthcheck |
| `--timeout` | 30s | Tiempo máximo de espera por respuesta |
| `--start-period` | 0s | Tiempo de gracia al inicio (fallos no cuentan) |
| `--start-interval` | 5s | Frecuencia durante el start-period (Docker 25.0+) |
| `--retries` | 3 | Fallos consecutivos para marcar como unhealthy |

### 6.4 Códigos de salida

| Código | Significado | Acción |
|---|---|---|
| `0` | Success — container healthy | Marcar como healthy |
| `1` | Unhealthy — container no funciona | Contar como fallo |
| `2` | Reserved — no usar | — |

> **⚠️ Importante:** Muchas herramientas (como `curl`) pueden retornar códigos de error distintos a `1`. Siempre usa `|| exit 1` o `|| false` al final del comando para normalizar el código de salida.

```bash
# ❌ Problema: curl puede retornar exit 7, 22, 28, etc.
HEALTHCHECK CMD curl -f http://localhost/

# ✅ Correcto: siempre normalizar a exit 1
HEALTHCHECK CMD curl -f http://localhost/ || exit 1

# ✅ Equivalente:
HEALTHCHECK CMD curl -f http://localhost/ || false
```

### 6.5 Healthcheck en Dockerfile

```dockerfile
# Forma básica (opciones por defecto)
HEALTHCHECK CMD curl -f http://localhost/ || exit 1

# Con opciones personalizadas
HEALTHCHECK --interval=30s --timeout=3s --start-period=15s --retries=3 \
    CMD curl -f http://localhost/ || exit 1

# Deshabilitar healthcheck heredado de imagen base
HEALTHCHECK NONE
```

> Solo puede haber **un HEALTHCHECK** por Dockerfile. Si pones más de uno, solo aplica el último.

### 6.6 Healthcheck en docker run

```bash
# Agregar healthcheck a una imagen que no tiene uno
$ docker run -d \
    --name db \
    --health-cmd="pg_isready -U postgres || exit 1" \
    --health-interval=30s \
    --health-timeout=10s \
    --health-start-period=30s \
    --health-retries=3 \
    postgres:15
```

### 6.7 Healthcheck en Compose/Stack

```yaml
services:
  web:
    image: nginx:alpine
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost/"]
      interval: 30s
      timeout: 10s
      start_period: 5s
      retries: 3

  db:
    image: postgres:15
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres || exit 1"]
      interval: 30s
      timeout: 10s
      start_period: 30s
      retries: 5
```

| Formato de `test` | Sintaxis | Cuándo usarlo |
|---|---|---|
| `CMD` | `["CMD", "curl", "-f", "http://localhost/"]` | Comando sin shell (exec form) |
| `CMD-SHELL` | `["CMD-SHELL", "curl -f http://localhost/ \|\| exit 1"]` | Necesitas pipes, `||`, redirecciones |
| String | `"curl -f http://localhost/ \|\| exit 1"` | Shorthand (se ejecuta en shell) |

#### Nginx / Apache (web server)

## 7. Healthchecks en Swarm Services

### 7.1 La diferencia crítica: run vs service

```
docker run + healthcheck:
─────────────────────────
  unhealthy → Docker MARCA el container como unhealthy
              pero NO toma ninguna acción
              (solo informativo en ls e inspect)

docker service + healthcheck:
─────────────────────────────
  unhealthy → Swarm REEMPLAZA el task
              (detiene el container unhealthy,
               crea uno nuevo en su lugar)
              Esto es self-healing automático.
```

### 7.2 Healthchecks y Service Updates

Los healthchecks se integran directamente con el proceso de update:

```
SERVICE UPDATE CON HEALTHCHECK

  1. Detener task viejo
  2. Crear task nuevo
  3. Esperar que pase de "starting" a "healthy"
     │
     ├── healthy → continuar con el siguiente task
     │
     └── unhealthy → ejecutar --update-failure-action
                     (pause | continue | rollback)

  SIN healthcheck:
  Starting → Running (instantáneo, sin verificación real)

  CON healthcheck:
  Starting → (espera 30s) → healthy → Running
  (el update NO continúa hasta que el nuevo task sea healthy)
```

> Esto significa que si una nueva versión de tu imagen está rota (no pasa healthcheck), el update se pausa o hace rollback automáticamente **antes de afectar a todos los replicas**.

### 7.3 Demo: servicio con y sin healthcheck

```bash
# Sin healthcheck — starting → running instantáneo
$ docker service create --name p1 postgres:15
# Preparing → Starting → Running (inmediato)

# Con healthcheck — starting espera verificación
$ docker service create \
    --name p2 \
    --health-cmd="pg_isready -U postgres || exit 1" \
    postgres:15
# Preparing → Starting → (30 segundos esperando) → Running
# El servicio solo se considera "running" cuando es "healthy"
```

## 8. Buenas prácticas

### 8.1 Service Updates

| Práctica | Por qué | Cómo |
|---|---|---|
| Siempre usar tags de imagen | Evitar `latest` que es impredecible | `nginx:1.25.3` no `nginx:latest` |
| Configurar `--update-delay` | Dar tiempo para detectar problemas | `--update-delay 10s` |
| Usar `start-first` para zero-downtime | Siempre hay N replicas activas | `--update-order start-first` |
| Configurar rollback automático | Self-healing en deploys fallidos | `--update-failure-action rollback` |
| Testear updates en staging | No descubrir problemas en producción | Ambiente de staging con Swarm |
| Usar healthchecks siempre | Updates esperan verificación | `HEALTHCHECK CMD ...` en Dockerfile |

### 8.2 Healthchecks

| Práctica | Por qué | Cómo |
|---|---|---|
| Agregar healthcheck a **toda** imagen de producción | Self-healing, updates inteligentes | `HEALTHCHECK` en Dockerfile |
| Usar `--start-period` para apps lentas | Java, DBs tardan en iniciar | `--start-period=60s` |
| Mantener healthchecks **ligeros** | Se ejecutan cada 30s, no deben ser costosos | `curl localhost` no `run full test suite` |
| Normalizar exit codes con `\|\| exit 1` | Docker solo entiende 0 y 1 | `curl ... \|\| exit 1` |
| No reemplazar monitoring externo | Healthchecks son verificación básica | Healthcheck + Prometheus/Grafana |

### 8.3 Compose multi-entorno

| Práctica | Por qué | Cómo |
|---|---|---|
| Base mínima | Solo lo común a todos los entornos | Servicios + imágenes en `docker-compose.yml` |
| Override automático para dev | Menos comandos que recordar | `docker-compose.override.yml` |
| `external: true` para secrets en prod | No depender de archivos en el server | Secrets pre-creados en Swarm |
| Generar output.yml para stack deploy | Stack no soporta múltiples `-f` | `docker compose -f ... config > output.yml` |
| No usar `extends` en stacks | No soportado actualmente | Usar override files con `-f` |

## 9. Errores comunes

| # | Error | Consecuencia | Solución |
|---|---|---|---|
| 1 | No usar healthchecks en producción | Updates ciegos — no detectan containers rotos | Agregar `HEALTHCHECK` a toda imagen |
| 2 | Usar `latest` como tag de imagen | Impossible saber qué versión corre, rollback impredecible | Siempre tags explícitos: `nginx:1.25.3` |
| 3 | Deploy directo a prod sin staging | Descubrir problemas con usuarios reales | Ambiente staging con Swarm idéntico |
| 4 | Olvidar `\|\| exit 1` en healthcheck | Curl retorna exit 7 o 22, Docker no lo entiende como unhealthy | Siempre normalizar: `cmd \|\| exit 1` |
| 5 | Healthcheck muy pesado | Consume recursos cada 30s, ralentiza la app | Verificación ligera (ping, status endpoint) |
| 6 | No configurar `--start-period` | Apps lentas (Java, DBs) marcadas como unhealthy antes de iniciar | `--start-period=60s` para apps lentas |
| 7 | Asumir que updates = zero downtime | Depende de la app, el protocolo y la configuración | Testear updates, usar `start-first` |
| 8 | No usar `--update-failure-action rollback` | Un deploy roto afecta todos los replicas | Rollback automático con `max-failure-ratio` |
| 9 | Editar containers manualmente en Swarm | Swarm los recrea, se pierden los cambios | Todo declarativo via stack files o service update |
| 10 | Mezclar `docker compose` y `docker stack` en el mismo entorno | Conflictos de naming y gestión | Compose para dev, Stack para prod — nunca ambos |
| 11 | No versionar los archivos Compose | Impossible reproducir un deploy anterior | Git para todos los archivos de configuración |
| 12 | `extends` en stack deploy | No funciona, falla silenciosamente | Usar override files con `-f` y `config` |
