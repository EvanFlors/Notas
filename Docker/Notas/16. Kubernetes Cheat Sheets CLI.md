# ☸️ kubectl — Referencia CLI y Cheat Sheet

## 1. 🎯 Introducción

`kubectl` es la herramienta de línea de comandos para interactuar con el API Server de Kubernetes. Toda operación en K8s pasa por `kubectl` (o directamente por la API REST).

### Configuración

```bash
# kubectl busca configuración en:
~/.kube/config                    # Default
$KUBECONFIG                       # Variable de entorno
--kubeconfig /path/to/config      # Flag explícito
```

### Sintaxis general

```
kubectl [command] [TYPE] [NAME] [flags]

# Ejemplos equivalentes (singular, plural, abreviado):
kubectl get pod my-pod
kubectl get pods my-pod
kubectl get po my-pod
```

## 2. 🔧 Configuración de Contexto

```bash
# Ver configuración actual
kubectl config view
kubectl config view --raw                       # Con certificados

# Contextos
kubectl config get-contexts                     # Listar contextos
kubectl config current-context                  # Contexto actual
kubectl config use-context my-cluster           # Cambiar contexto

# Namespace por defecto
kubectl config set-context --current --namespace=my-ns

# Alias útiles (zsh/bash)
alias k=kubectl
alias kx='f() { [ "$1" ] && kubectl config use-context $1 || kubectl config current-context ; } ; f'
alias kn='f() { [ "$1" ] && kubectl config set-context --current --namespace $1 || kubectl config view --minify | grep namespace | cut -d" " -f6 ; } ; f'
```

### Autocompletado

```bash
# Bash
source <(kubectl completion bash)
echo "source <(kubectl completion bash)" >> ~/.bashrc

# Zsh
source <(kubectl completion zsh)
echo '[[ $commands[kubectl] ]] && source <(kubectl completion zsh)' >> ~/.zshrc

# Con alias
alias k=kubectl
complete -o default -F __start_kubectl k       # Bash
```

## 3. 📋 Operaciones Principales

### 3.1 Crear recursos

```bash
# Desde YAML
kubectl apply -f manifest.yaml                  # Crear o actualizar
kubectl apply -f ./dir/                         # Todos los manifiestos en directorio
kubectl apply -f https://example.com/m.yaml     # Desde URL

# Imperativos (para aprendizaje y testing)
kubectl create deployment nginx --image=nginx
kubectl run my-pod --image nginx
kubectl create job hello --image=busybox:1.28 -- echo "Hello"
kubectl create cronjob hello --image=busybox:1.28 --schedule="*/1 * * * *" -- echo "Hello"

# Generar YAML sin crear
kubectl run nginx --image=nginx --dry-run=client -o yaml > pod.yaml
kubectl create deployment nginx --image=nginx --dry-run=client -o yaml > deploy.yaml
```

### 3.2 Ejecutar un shell interactivo en un Pod

Desde la versión **1.18**, `kubectl run` solo crea **un Pod con un contenedor** desde una imagen. El formato completo es:

```bash
kubectl run NAME --image=image [--env="key=value"] [--port=port] [--dry-run=server|client] [--overrides=inline-json] [--command] -- [COMMAND] [args...] [options]
```

Solo **NAME** e **--image** son obligatorios. Desglose de las opciones más usadas:

| Opción | Descripción |
|--------|-------------|
| `NAME` | Nombre del Pod (obligatorio) |
| `--image` | Imagen del contenedor (obligatorio) |
| `--rm` | Elimina el Pod al salir (como `docker run --rm`) |
| `-i` (`--stdin=true`) | Mantiene la conexión de entrada abierta (permite escribir múltiples comandos sin reiniciar el contenedor) |
| `-t` (`--tty=true`) | Asigna una terminal virtual (donde ejecutamos un shell como bash) |
| `-- [COMMAND]` | Sobreescribe el `CMD` del Dockerfile con un nuevo comando |

> 💡 `-it` es la abreviatura combinada de `-i` y `-t`. Se usan juntas cuando queremos un **shell interactivo** dentro del contenedor (con `run`, `exec`, etc.)

#### El doble guion `--`

El `--` es una convención estándar en shells de Linux que **separa las opciones del CLI de los argumentos posicionales**. En kubectl:

```bash
# Todo antes de -- son opciones de kubectl
# Todo después de -- sobreescribe el CMD del contenedor
kubectl run tmp-shell --rm -it --image bretfisher/netshoot -- bash
```

> ⚠️ A diferencia de `docker run`, kubectl **requiere** el `--` para sobreescribir el comando. Alternativamente se puede usar `--command`, pero `--` es preferible porque es una convención universal en Linux y no requiere entrecomillar el comando completo.

### 3.3 Consultar recursos

```bash
# Listado básico
kubectl get pods                                # Pods del namespace actual
kubectl get pods -A                             # Todos los namespaces (--all-namespaces)
kubectl get pods -o wide                        # Con info de Node e IP
kubectl get all                                 # Recursos comunes
kubectl get deploy,svc,pods                     # Múltiples tipos

# Formato de salida
kubectl get pods -o yaml                        # YAML completo
kubectl get pods -o json                        # JSON
kubectl get pods -o name                        # Solo nombres
kubectl get pods -o custom-columns=NAME:.metadata.name,IMAGE:.spec.containers[*].image

# Filtrado
kubectl get pods -l app=nginx                   # Por label
kubectl get pods --field-selector=status.phase=Running   # Por campo
kubectl get pods --sort-by=.metadata.creationTimestamp    # Ordenado

# Inspección detallada
kubectl describe pod my-pod                     # Detalles + eventos
kubectl describe deploy my-deployment
kubectl describe node my-node
```

**`get` vs `describe`:**

| `kubectl get` | `kubectl describe` |
|---|---|
| Un tipo de recurso a la vez | Combina recursos relacionados |
| Datos tabulares | Detalles formateados |
| Soporta `-o yaml/json` | Incluye Events |

### 3.3 Actualizar recursos

```bash
# Actualizar imagen
kubectl set image deploy/frontend www=image:v2

# Escalar
kubectl scale deploy foo --replicas=3

# Labels y annotations
kubectl label pods my-pod env=prod
kubectl label pods my-pod env-                  # Eliminar label
kubectl annotate pods my-pod note="hello"

# Auto-scale
kubectl autoscale deploy foo --min=2 --max=10 --cpu-percent=80

# Editar en editor
kubectl edit deploy my-deploy
KUBE_EDITOR="nano" kubectl edit deploy my-deploy

# Patch
kubectl patch node k8s-node-1 -p '{"spec":{"unschedulable":true}}'
```

### 3.4 Rollouts

```bash
kubectl rollout status deploy/frontend          # Estado del rollout
kubectl rollout history deploy/frontend         # Historial
kubectl rollout undo deploy/frontend            # Rollback
kubectl rollout undo deploy/frontend --to-revision=2
kubectl rollout pause deploy/frontend           # Pausar
kubectl rollout resume deploy/frontend          # Resumir
kubectl rollout restart deploy/frontend         # Rolling restart
```

### 3.5 Eliminar recursos

```bash
kubectl delete -f pod.yaml                      # Desde archivo
kubectl delete pod my-pod                       # Por nombre
kubectl delete pod my-pod --now                 # Sin grace period
kubectl delete pods,services -l name=myLabel    # Por label
kubectl -n my-ns delete pod,svc --all           # Todo en namespace
```

## 4. 🔍 Debugging e Interacción

### 4.1 Logs

```bash
kubectl logs my-pod                             # Logs del pod
kubectl logs my-pod -c my-container             # Container específico
kubectl logs my-pod --all-containers=true        # Todos los containers
kubectl logs my-pod --previous                  # Instancia anterior
kubectl logs -f my-pod                          # Stream (follow)
kubectl logs -f my-pod --tail 100               # Últimas 100 líneas + follow
kubectl logs -l app=nginx                       # Por label selector
kubectl logs deploy/my-deployment               # Primer pod del deployment
```

### 4.2 Exec y Attach

```bash
# Ejecutar comando
kubectl exec my-pod -- ls /
kubectl exec my-pod -c my-container -- ls /

# Shell interactivo
kubectl exec -it my-pod -- /bin/sh
kubectl exec -it my-pod -- /bin/bash

# Attach (conectar a proceso existente)
kubectl attach my-pod -i
```

### 4.3 Port Forward

```bash
# Pod
kubectl port-forward my-pod 5000:6000           # local:remote

# Service
kubectl port-forward svc/my-service 5000:80

# Deployment
kubectl port-forward deploy/my-deploy 5000:6000
```

### 4.4 Debug

```bash
# Debug en pod existente
kubectl debug my-pod -it --image=busybox:1.28

# Debug en nodo
kubectl debug node/my-node -it --image=busybox:1.28
```

### 4.5 Copiar archivos

```bash
kubectl cp /tmp/foo my-pod:/tmp/bar             # Local → Pod
kubectl cp my-pod:/tmp/foo /tmp/bar             # Pod → Local
kubectl cp /tmp/foo my-ns/my-pod:/tmp/bar       # Con namespace
kubectl cp /tmp/foo my-pod:/tmp/bar -c my-container  # Container específico
```

### 4.6 Métricas

```bash
kubectl top pod                                 # CPU/Mem de pods
kubectl top pod my-pod --containers             # Por container
kubectl top pod --sort-by=cpu                   # Ordenado
kubectl top node                                # Métricas de nodos
```

## 5. 🏗️ Nodos y Cluster

```bash
# Info del cluster
kubectl cluster-info
kubectl cluster-info dump                       # Dump completo

# Nodos
kubectl get nodes
kubectl get nodes -o wide
kubectl describe node my-node

# Mantenimiento
kubectl cordon my-node                          # Marcar como no-schedulable
kubectl drain my-node                           # Evacuar pods
kubectl uncordon my-node                        # Restaurar scheduling

# Taints
kubectl taint nodes foo dedicated=special:NoSchedule

# Ver taints
kubectl get nodes -o='custom-columns=NodeName:.metadata.name,TaintKey:.spec.taints[*].key,TaintValue:.spec.taints[*].value,TaintEffect:.spec.taints[*].effect'
```

## 7. 📤 Formatos de Salida

| Flag | Descripción |
|---|---|
| `-o yaml` | Formato YAML |
| `-o json` | Formato JSON |
| `-o wide` | Texto plano con más columnas |
| `-o name` | Solo `type/name` |
| `-o jsonpath='{.spec}'` | Campos específicos vía JSONPath |
| `-o custom-columns=COL:EXPR` | Columnas personalizadas |
| `-o go-template=TEMPLATE` | Go templates |

```bash
# Ejemplos JSONPath
kubectl get pods -o jsonpath='{.items[*].metadata.name}'
kubectl get nodes -o jsonpath='{.items[*].status.addresses[?(@.type=="ExternalIP")].address}'
kubectl get secret my-secret -o jsonpath='{.data.password}' | base64 -d

# Custom columns
kubectl get pods -o custom-columns='NAME:.metadata.name,IMAGE:.spec.containers[*].image'

# Todas las imágenes del cluster
kubectl get pods -A -o custom-columns='DATA:spec.containers[*].image'
```

## 9. 🐳 kubectl vs docker (equivalencias)

| Docker | kubectl | Nota |
|---|---|---|
| `docker run -d nginx` | `kubectl create deploy nginx --image=nginx` | K8s crea Deployment |
| `docker ps` | `kubectl get pods` | |
| `docker exec -it ID bash` | `kubectl exec -it pod -- bash` | |
| `docker logs -f ID` | `kubectl logs -f pod` | |
| `docker stop/rm ID` | `kubectl delete deploy nginx` | Eliminar el Deployment, no el Pod |
| `docker version` | `kubectl version` | |
| `docker info` | `kubectl cluster-info` | |
| `docker login` | No hay equivalente directo | Ver "Using a Private Registry" |

> **Diferencia clave:** En Docker, eliminas el container directo. En K8s, eliminas el **Deployment** (si eliminas solo el Pod, el Deployment lo recrea).

## 10. ⚠️ Errores Comunes

| # | Error | Solución |
|---|---|---|
| 1 | `kubectl delete pod X` y el pod reaparece | Elimina el Deployment, no el Pod |
| 2 | Olvidar `-n namespace` | Usa `kubectl config set-context --current --namespace=X` |
| 3 | No usar `--dry-run=client -o yaml` | Genera YAML antes de crear para revisión |
| 4 | Confundir `apply` con `create` | `apply` es idempotente; `create` falla si ya existe |
| 5 | No instalar autocompletado | Ahorra tiempo y errores con `source <(kubectl completion zsh)` |
