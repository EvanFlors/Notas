## 1. Introducción

Todo en Kubernetes es un **recurso de la API**. Cada Pod, Deployment, Service, Secret — todos son objetos gestionados a través de la API REST del API Server. Entender qué recursos existen, cómo se organizan y cómo explorarlos es fundamental para trabajar eficazmente con Kubernetes.

```
  kubectl get pods          ──►  GET /api/v1/namespaces/default/pods
  kubectl create deploy     ──►  POST /apis/apps/v1/namespaces/default/deployments
  kubectl delete svc nginx  ──►  DELETE /api/v1/namespaces/default/services/nginx

  Cada comando kubectl es una llamada HTTP al API Server
```

## 2. La API de Kubernetes

### Arquitectura

```
  Cliente (kubectl, SDK, dashboard)
       │
       │  HTTPS (REST)
       ▼
  ┌──────────────────────────────────────────────┐
  │               API Server (kube-apiserver)     │
  │                                               │
  │   /api/v1/...              ← Core group       │
  │   /apis/apps/v1/...        ← Apps group       │
  │   /apis/batch/v1/...       ← Batch group      │
  │   /apis/networking.k8s.io/ ← Networking group  │
  │   /apis/storage.k8s.io/    ← Storage group    │
  │   /apis/<custom>/          ← CRDs             │
  │                                               │
  │   Authentication → Authorization → Admission  │
  └──────────────────┬────────────────────────────┘
                     │
                     ▼
                   etcd (almacenamiento)
```

### ¿Por qué importa?

- **Todo** lo que haces en Kubernetes pasa por la API
- Conocer los recursos te permite saber **qué puedes gestionar**
- Los API groups organizan los recursos por funcionalidad
- Las versiones de API indican **estabilidad** (alpha → beta → stable)

## 3. Anatomía de un API Resource

Cada recurso tiene propiedades clave que determinan cómo y dónde se usa:

```
  kubectl api-resources

  NAME          SHORTNAMES   APIVERSION   NAMESPACED   KIND
  ─────────     ──────────   ──────────   ──────────   ────
  deployments   deploy       apps/v1      true         Deployment
  │             │            │            │            │
  │             │            │            │            └── Tipo en YAML (kind:)
  │             │            │            └── ¿Vive dentro de un namespace?
  │             │            └── Grupo/Versión de API
  │             └── Abreviatura para kubectl
  └── Nombre plural (usado en URLs y comandos)
```

| Propiedad | Descripción | Ejemplo |
|-----------|-------------|---------|
| **NAME** | Nombre plural del recurso (para URLs y CLI) | `deployments`, `pods`, `services` |
| **SHORTNAMES** | Alias cortos para kubectl | `deploy`, `po`, `svc` |
| **APIVERSION** | Grupo + versión de la API | `v1`, `apps/v1`, `batch/v1` |
| **NAMESPACED** | Si el recurso vive dentro de un namespace | `true` / `false` |
| **KIND** | Nombre del tipo (usado en YAML `kind:`) | `Deployment`, `Pod`, `Service` |

## 4. API Groups y Versions

### Grupos de API

Kubernetes organiza sus recursos en **grupos** para facilitar la evolución independiente de cada área:

```
  API Server
  ├── /api/v1                          ← Core group (sin nombre explícito)
  │   ├── pods, services, configmaps, secrets, nodes, namespaces...
  │
  ├── /apis/apps/v1                    ← Apps group
  │   ├── deployments, statefulsets, daemonsets, replicasets
  │
  ├── /apis/batch/v1                   ← Batch group
  │   ├── jobs, cronjobs
  │
  ├── /apis/networking.k8s.io/v1       ← Networking group
  │   ├── ingresses, networkpolicies, ingressclasses
  │
  ├── /apis/storage.k8s.io/v1         ← Storage group
  │   ├── storageclasses, csidrivers, volumeattachments
  │
  ├── /apis/rbac.authorization.k8s.io/v1  ← RBAC group
  │   ├── roles, clusterroles, rolebindings, clusterrolebindings
  │
  ├── /apis/autoscaling/v2             ← Autoscaling group
  │   ├── horizontalpodautoscalers
  │
  └── /apis/apiextensions.k8s.io/v1   ← Extensions group
      ├── customresourcedefinitions
```

### Niveles de estabilidad de versiones

| Nivel | Formato | Significado | Ejemplo |
|-------|---------|-------------|---------|
| **Stable (GA)** | `v1`, `v2` | Producción, sin cambios breaking | `apps/v1` |
| **Beta** | `v1beta1`, `v2beta2` | Funcional, puede cambiar | `flowcontrol.apiserver.k8s.io/v1beta3` |
| **Alpha** | `v1alpha1` | Experimental, puede desaparecer | features con feature gates |

> 💡 En YAML, `apiVersion` combina grupo + versión. Para el core group se usa solo `v1` (sin prefijo de grupo).

### Ver versiones disponibles

```bash
kubectl api-versions
# apps/v1
# autoscaling/v2
# batch/v1
# networking.k8s.io/v1
# rbac.authorization.k8s.io/v1
# storage.k8s.io/v1
# v1                          ← Core group
# ...
```

## 5. Recursos Core (v1)

Los recursos del core group son los más fundamentales — existen desde las primeras versiones de Kubernetes y usan `apiVersion: v1`.

### Recursos principales

| Recurso | Short | Kind | Namespaced | Descripción |
|---------|-------|------|------------|-------------|
| `pods` | `po` | Pod | ✅ | Unidad mínima de ejecución — uno o más contenedores |
| `services` | `svc` | Service | ✅ | Abstracción de red para acceder a un conjunto de Pods |
| `configmaps` | `cm` | ConfigMap | ✅ | Configuración no sensible como key/value |
| `secrets` | — | Secret | ✅ | Datos sensibles (passwords, tokens, certificados) |
| `namespaces` | `ns` | Namespace | ❌ | Aislamiento lógico de recursos dentro del clúster |
| `nodes` | `no` | Node | ❌ | Máquina (física o virtual) del clúster |
| `persistentvolumes` | `pv` | PersistentVolume | ❌ | Almacenamiento provisionado por el admin |
| `persistentvolumeclaims` | `pvc` | PersistentVolumeClaim | ✅ | Solicitud de almacenamiento por un Pod |
| `serviceaccounts` | `sa` | ServiceAccount | ✅ | Identidad para Pods (autenticación con la API) |
| `endpoints` | `ep` | Endpoints | ✅ | Lista de IPs y puertos que respaldan un Service |
| `events` | `ev` | Event | ✅ | Registro de eventos del clúster |
| `resourcequotas` | `quota` | ResourceQuota | ✅ | Límites de recursos por namespace |
| `limitranges` | `limits` | LimitRange | ✅ | Defaults y límites de recursos por contenedor |
| `replicationcontrollers` | `rc` | ReplicationController | ✅ | Precursor de ReplicaSet (legado) |

### Relaciones entre recursos core

```
  Namespace
  ├── Pod ◄──── Service (selecciona por labels)
  │    │              │
  │    │              └── Endpoints / EndpointSlice (IPs de Pods)
  │    │
  │    ├── ConfigMap (montado como volumen o env vars)
  │    ├── Secret (montado como volumen o env vars)
  │    └── PersistentVolumeClaim ──► PersistentVolume
  │
  ├── ServiceAccount (identidad del Pod)
  ├── ResourceQuota (límites del namespace)
  └── LimitRange (defaults por contenedor)
```

## 6. Recursos de Workloads (apps/v1)

Gestionan **cómo** se ejecutan los contenedores y su ciclo de vida:

| Recurso | Short | Kind | Descripción |
|---------|-------|------|-------------|
| `deployments` | `deploy` | Deployment | Gestión declarativa de Pods con rolling updates |
| `replicasets` | `rs` | ReplicaSet | Mantiene N réplicas de un Pod (gestionado por Deployment) |
| `statefulsets` | `sts` | StatefulSet | Para apps con estado: identidad persistente + storage |
| `daemonsets` | `ds` | DaemonSet | Un Pod por nodo (logs, monitoring, networking) |
| `controllerrevisions` | — | ControllerRevision | Historial de revisiones de StatefulSets y DaemonSets |

### Batch (batch/v1)

| Recurso | Short | Kind | Descripción |
|---------|-------|------|-------------|
| `jobs` | — | Job | Ejecución de tareas hasta completar (one-off o paralelas) |
| `cronjobs` | `cj` | CronJob | Jobs programados con schedule tipo cron |

### Jerarquía de workloads

```
  Deployment
  └── ReplicaSet (creado automáticamente)
      └── Pod (réplicas gestionadas)

  StatefulSet
  └── Pod (identidad estable: pod-0, pod-1, pod-2)
      └── PersistentVolumeClaim (uno por Pod)

  DaemonSet
  └── Pod (uno por nodo automáticamente)

  CronJob
  └── Job (creado según schedule)
      └── Pod (ejecuta hasta completar)
```

### ¿Cuándo usar cada uno?

| Workload | Caso de uso | Ejemplo |
|----------|-------------|---------|
| **Deployment** | Apps stateless, APIs, frontends | nginx, API REST, microservicio |
| **StatefulSet** | Apps con estado, orden de inicio | PostgreSQL, Kafka, ZooKeeper |
| **DaemonSet** | Agentes que corren en todos los nodos | Fluentd, Datadog agent, kube-proxy |
| **Job** | Tareas batch puntuales | Migración de DB, procesamiento de datos |
| **CronJob** | Tareas recurrentes | Backups nocturnos, limpieza de logs |

## 7. Recursos de Networking

| Recurso | Short | API Group | Kind | Descripción |
|---------|-------|-----------|------|-------------|
| `services` | `svc` | `v1` | Service | Exposición de Pods (ClusterIP, NodePort, LoadBalancer) |
| `endpoints` | `ep` | `v1` | Endpoints | IPs de Pods detrás de un Service (legado) |
| `endpointslices` | — | `discovery.k8s.io/v1` | EndpointSlice | Reemplazo escalable de Endpoints |
| `ingresses` | `ing` | `networking.k8s.io/v1` | Ingress | Enrutamiento HTTP/HTTPS con reglas por host/path |
| `ingressclasses` | — | `networking.k8s.io/v1` | IngressClass | Define qué Ingress Controller usar |
| `networkpolicies` | `netpol` | `networking.k8s.io/v1` | NetworkPolicy | Firewall a nivel de Pod (reglas de ingress/egress) |

### Flujo de tráfico

```
  Internet
      │
      ▼
  ┌─────────────────┐
  │  LoadBalancer/   │
  │  Ingress         │ ◄── IngressClass define el controller
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │    Service       │ ◄── Selecciona Pods por labels
  └────────┬────────┘
           │
           ▼
  ┌────────────────────────────┐
  │  EndpointSlice             │ ◄── IPs y puertos de los Pods
  │  (172.17.0.3:8080,         │
  │   172.17.0.4:8080, ...)    │
  └────────┬───────────────────┘
           │
    ┌──────┼──────┐
    ▼      ▼      ▼
  Pod-1  Pod-2  Pod-3          ◄── NetworkPolicy filtra tráfico
```

### Flujo de provisioning dinámico

```
  Pod spec:
    volumes:
    - persistentVolumeClaim:
        claimName: my-data
                │
                ▼
  PersistentVolumeClaim (pvc)     ──►  StorageClass
  "Quiero 10Gi de SSD"                "Provisioner: aws-ebs"
                                              │
                                              ▼
                                     PersistentVolume (pv)
                                     "10Gi en aws-ebs, bound"
                                              │
                                              ▼
                                     CSIDriver (aws-ebs-csi)
                                     "Crea el disco real"
```

## 9. Recursos de Configuración y Seguridad

### Configuración

| Recurso | Short | Kind | Descripción |
|---------|-------|------|-------------|
| `configmaps` | `cm` | ConfigMap | Config no sensible (archivos, variables, flags) |
| `secrets` | — | Secret | Datos sensibles (base64 encoded, encriptados en etcd) |

### RBAC (Control de acceso)

| Recurso | Short | Kind | Namespaced | Descripción |
|---------|-------|------|------------|-------------|
| `roles` | — | Role | ✅ | Permisos dentro de un namespace |
| `clusterroles` | — | ClusterRole | ❌ | Permisos a nivel de clúster |
| `rolebindings` | — | RoleBinding | ✅ | Vincula Role → User/ServiceAccount |
| `clusterrolebindings` | — | ClusterRoleBinding | ❌ | Vincula ClusterRole → User/ServiceAccount |
| `serviceaccounts` | `sa` | ServiceAccount | ✅ | Identidad para Pods |

### Flujo de RBAC

```
  ¿Quién?                    ¿Qué puede hacer?         ¿Dónde?
  ─────────                  ──────────────────         ────────
  User / ServiceAccount ◄──  RoleBinding  ──►  Role    (namespace)
  User / ServiceAccount ◄──  ClusterRoleBinding ──► ClusterRole (clúster)
```

### Admission Control y Webhooks

| Recurso | Kind | Descripción |
|---------|------|-------------|
| `mutatingwebhookconfigurations` | MutatingWebhookConfiguration | Modifica objetos antes de persistirlos |
| `validatingwebhookconfigurations` | ValidatingWebhookConfiguration | Valida objetos antes de persistirlos |
| `validatingadmissionpolicies` | ValidatingAdmissionPolicy | Políticas de validación declarativas (CEL) |
| `validatingadmissionpolicybindings` | ValidatingAdmissionPolicyBinding | Vincula políticas a namespaces/recursos |

```
  kubectl apply ──► API Server ──► Authentication ──► Authorization ──► Admission
                                                                           │
                                                          ┌────────────────┤
                                                          ▼                ▼
                                                    Mutating           Validating
                                                    Webhooks           Webhooks
                                                    (modifican)        (aprueban/rechazan)
                                                          │                │
                                                          └────────┬───────┘
                                                                   ▼
                                                                 etcd
```

## 10. Recursos de Scheduling y Autoscaling

| Recurso | Short | API Group | Kind | Descripción |
|---------|-------|-----------|------|-------------|
| `horizontalpodautoscalers` | `hpa` | `autoscaling/v2` | HorizontalPodAutoscaler | Escala réplicas por CPU/memoria/custom metrics |
| `priorityclasses` | `pc` | `scheduling.k8s.io/v1` | PriorityClass | Prioridad relativa de Pods (preemption) |
| `poddisruptionbudgets` | `pdb` | `policy/v1` | PodDisruptionBudget | Mínimo de Pods disponibles durante disruptions |
| `leases` | — | `coordination.k8s.io/v1` | Lease | Leader election y heartbeat de nodos |

### Cómo trabajan juntos

```
  HPA: "Si CPU > 70%, escalar a más réplicas"
       │
       ▼
  Deployment (replicas: 3 → 5)
       │
       ├── PodDisruptionBudget: "Siempre ≥ 2 pods available"
       │   (protege durante upgrades, drain, etc.)
       │
       └── PriorityClass: "Este Pod es crítico"
           (si no hay recursos, puede desalojar Pods de menor prioridad)
```

## 11. Recursos de Extensibilidad

| Recurso | Short | Kind | Descripción |
|---------|-------|------|-------------|
| `customresourcedefinitions` | `crd`, `crds` | CustomResourceDefinition | Define nuevos tipos de recursos en la API |
| `apiservices` | — | APIService | Registra API servers adicionales (aggregation layer) |

### CRDs: extender Kubernetes

Los **Custom Resource Definitions** permiten crear tus propios tipos de recursos:

```bash
# Ver CRDs instalados
kubectl get crds

# Ejemplo: si tienes Cert-Manager instalado
kubectl get certificates                    # ← Recurso custom definido por un CRD
kubectl get issuers
```

```
  CRD (define el tipo)                   CR (instancia del tipo)
  ─────────────────────                  ─────────────────────────
  kind: CustomResourceDefinition         kind: Certificate
  metadata:                              metadata:
    name: certificates.cert-manager.io     name: my-tls-cert
  spec:                                  spec:
    group: cert-manager.io                 secretName: my-tls
    names:                                 issuerRef:
      kind: Certificate                      name: letsencrypt
```

> 💡 Herramientas como **Cert-Manager**, **Istio**, **ArgoCD** y **Prometheus Operator** extienden Kubernetes con sus propios CRDs.

## 12. Namespaced vs Cluster-Scoped

Los recursos se dividen en dos ámbitos:

| Ámbito | Descripción | Ejemplo |
|--------|-------------|---------|
| **Namespaced** | Vive dentro de un namespace, aislado lógicamente | Pods, Services, Deployments, ConfigMaps |
| **Cluster-Scoped** | Existe a nivel de clúster, sin namespace | Nodes, PVs, Namespaces, ClusterRoles, CRDs |

### Listar por ámbito

```bash
# Solo recursos namespaced
kubectl api-resources --namespaced=true

# Solo recursos cluster-scoped
kubectl api-resources --namespaced=false
```

### ¿Por qué importa?

```bash
# Namespaced: DEBES especificar namespace (o usa el default)
kubectl get pods -n production
kubectl get pods --all-namespaces        # -A

# Cluster-scoped: no tienen namespace
kubectl get nodes                        # No acepta -n
kubectl get pv                           # Global
kubectl get clusterroles                 # Global
```

### Resumen visual

```
  Cluster Scope (global)
  ┌────────────────────────────────────────────────────┐
  │  Nodes, PersistentVolumes, Namespaces              │
  │  ClusterRoles, ClusterRoleBindings                 │
  │  StorageClasses, CRDs, IngressClasses              │
  │                                                     │
  │  ┌─── Namespace: default ────┐  ┌── Namespace: prod ──┐
  │  │ Pods, Services            │  │ Pods, Services       │
  │  │ Deployments, ConfigMaps   │  │ Deployments, Secrets │
  │  │ Roles, RoleBindings       │  │ Ingresses, HPA       │
  │  │ PVCs, ServiceAccounts     │  │ Jobs, CronJobs       │
  │  └──────────────────────────┘  └──────────────────────┘
  └────────────────────────────────────────────────────┘
```

## 13. Explorando la API con kubectl

### Comandos esenciales

```bash
# Listar TODOS los recursos disponibles
kubectl api-resources

# Filtrar por API group
kubectl api-resources --api-group=apps
kubectl api-resources --api-group=networking.k8s.io
kubectl api-resources --api-group=''              # Core group

# Filtrar por verbo (qué puedes hacer)
kubectl api-resources --verbs=list,get
kubectl api-resources --verbs=create,delete

# Listar versiones de API
kubectl api-versions

# Explorar la estructura de un recurso
kubectl explain pod
kubectl explain pod.spec.containers
kubectl explain deployment.spec.strategy --recursive

# Ver todos los campos posibles (recursivo)
kubectl explain service --recursive | head -50
```

### Combinar filtros

```bash
# Recursos namespaced del grupo apps que puedo crear
kubectl api-resources --api-group=apps --namespaced=true --verbs=create

# Recursos con shortname
kubectl api-resources --no-headers | awk '$2 != "" {print $1, $2}'
```

### Acceder directamente a la API REST

```bash
# Proxy al API server
kubectl proxy --port=8080 &

# Listar grupos de API
curl http://localhost:8080/apis

# Listar pods en default namespace
curl http://localhost:8080/api/v1/namespaces/default/pods

# Listar deployments
curl http://localhost:8080/apis/apps/v1/namespaces/default/deployments
```

## 15. Buenas prácticas

| Práctica | Razón |
|----------|-------|
| Usar `kubectl explain` antes de escribir YAML | Documentación integrada, siempre actualizada |
| Preferir API versions estables (`v1`, `apps/v1`) | Alpha/beta pueden cambiar o desaparecer |
| Usar shortnames en la CLI, nombres completos en scripts | Legibilidad vs rapidez |
| Revisar `kubectl api-resources` al actualizar el clúster | Nuevos recursos y deprecaciones |
| Conocer qué es namespaced vs cluster-scoped | Evitar errores de permisos y scope |
| Usar `--api-group` para filtrar | Encontrar rápidamente el recurso correcto |

## 16. Errores comunes

| # | Error | Consecuencia | Solución |
|---|-------|-------------|----------|
| 1 | Usar `apiVersion` incorrecto en YAML | `error: unable to recognize` | `kubectl api-resources` para ver la versión correcta |
| 2 | Crear recurso cluster-scoped con `-n namespace` | El namespace se ignora silenciosamente | Verificar con `--namespaced=false` |
| 3 | Usar API version deprecated | Warning o error en versiones nuevas | Migrar a la versión estable |
| 4 | No saber que un CRD existe | Intentar gestionar recursos sin saber cómo | `kubectl get crd` para ver extensiones instaladas |
| 5 | Confundir `kind` con `name` | `kind: Deployment` (tipo) vs `name: nginx` (instancia) | `kind` = clase, `name` = instancia |
| 6 | Usar Endpoints en lugar de EndpointSlices | Endpoints no escala bien (>1000 endpoints) | Preferir EndpointSlices en clústers grandes |
