# Docker Compose

---

## 1. Introducción

### ¿Qué es Docker Compose?

Docker Compose es una herramienta para **definir y ejecutar aplicaciones multi-contenedor**. En lugar de lanzar cada contenedor manualmente con `docker run`, describes toda tu infraestructura local en un archivo YAML y la levantas con un solo comando.

**Analogía:** Si `docker run` es dar instrucciones a un trabajador individual, Docker Compose es entregar un plano completo al equipo de construcción — todos saben qué hacer, dónde conectarse y en qué orden.

### ¿Por qué usarlo?

| Problema sin Compose | Solución con Compose |
|---|---|
| Múltiples `docker run` con flags largos | Un solo archivo YAML declarativo |
| Configurar redes manualmente entre contenedores | Red creada automáticamente en `docker compose up` |
| Cada desarrollador configura diferente | Entorno reproducible: `git clone` + `docker compose up` |
| Difícil recordar puertos, volúmenes, variables | Todo documentado en `compose.yaml` |
| Iniciar/parar servicios uno por uno | `up` / `down` para todo el stack |

### Componentes de Compose

Docker Compose se compone de dos partes:

```
Docker Compose
├── 1. Archivo YAML (compose.yaml)
│   └── Define: servicios, redes, volúmenes, configs, secrets
│
└── 2. CLI (docker compose)
    └── Ejecuta: up, down, build, logs, ps, exec, etc.
```

> **Nota importante:** Compose es ideal para **desarrollo local y testing**. Para producción, se utiliza en conjunto con Swarm Stacks o Kubernetes.

## 2. Fundamentos del Archivo Compose

### Estructura básica

El archivo `compose.yaml` (o `docker-compose.yml`) utiliza formato YAML para describir el estado deseado de tu aplicación:

```yaml
# compose.yaml
name: my-app  # Nombre del proyecto (opcional)

services:
  web:
    image: nginx:alpine
    ports:
      - "8080:80"
    volumes:
      - ./html:/usr/share/nginx/html
    depends_on:
      - api

  api:
    build: ./api
    environment:
      - DATABASE_URL=postgres://db:5432/myapp
    depends_on:
      - db

  db:
    image: postgres:16
    environment:
      POSTGRES_DB: myapp
      POSTGRES_PASSWORD: secret
    volumes:
      - db-data:/var/lib/postgresql/data

volumes:
  db-data:
```

### Anatomía de un servicio

Cada servicio en Compose es equivalente a un `docker run` con sus flags:

```
docker run equivalente:
  docker run -d \
    --name my-app-web-1 \
    -p 8080:80 \
    -v ./html:/usr/share/nginx/html \
    --network my-app_default \
    nginx:alpine

En compose.yaml:
  services:
    web:
      image: nginx:alpine
      ports:
        - "8080:80"
      volumes:
        - ./html:/usr/share/nginx/html
```

### Secciones principales del archivo

| Sección | Propósito | Ejemplo |
|---|---|---|
| `services` | Contenedores a ejecutar | Webapp, API, base de datos |
| `networks` | Redes personalizadas | Frontend/backend separation |
| `volumes` | Volúmenes nombrados persistentes | Data de BD, uploads |
| `configs` | Archivos de configuración | nginx.conf, app config |
| `secrets` | Datos sensibles | Passwords, API keys |

### Nombres de archivo válidos

Compose busca archivos en este orden de prioridad:

1. `compose.yaml` ← **Recomendado**
2. `compose.yml`
3. `docker-compose.yaml`
4. `docker-compose.yml`

Se puede especificar un archivo diferente con `-f`:

```bash
docker compose -f custom-compose.yaml up
```

## 3. Configuración de Servicios en Detalle

### image vs build

```yaml
services:
  # Usar una imagen existente de un registry
  redis:
    image: redis:7-alpine

  # Construir desde un Dockerfile
  api:
    build: ./api                  # Dockerfile en ./api/Dockerfile

  # Build con opciones avanzadas
  frontend:
    build:
      context: ./frontend         # Directorio de contexto
      dockerfile: Dockerfile.dev  # Dockerfile específico
      args:
        NODE_ENV: development     # Build arguments
      target: dev-stage           # Multi-stage target
    image: myapp-frontend:dev     # Tag para la imagen construida
```

> **¿Por qué importa?** El `build context` (`.` o una ruta) indica dónde está el Dockerfile y qué archivos puede acceder durante el build. `"."` significa el mismo directorio que el compose file.

### Ports: Publicar puertos

```yaml
services:
  web:
    ports:
      - "8080:80"       # HOST:CONTAINER
      - "443:443"
      - "9090:9090/udp" # Protocolo específico
      - "3000"          # Puerto aleatorio del host → 3000 del container
```

**`ports` vs `expose`:**

| Key | Efecto | Equivalente docker run |
|---|---|---|
| `ports` | Publica al host (accesible externamente) | `-p 8080:80` |
| `expose` | Solo accesible dentro de la red Docker | Sin flag (comunicación inter-contenedor) |

### Environment: Variables de entorno

```yaml
services:
  api:
    environment:
      # Formato map (recomendado)
      DATABASE_URL: postgres://db:5432/myapp
      DEBUG: "true"
      SECRET_KEY: ${SECRET_KEY}  # Desde variable del host o .env

    # O desde un archivo externo
    env_file:
      - .env
      - .env.local
```

> **Buena práctica:** Nunca hardcodear credenciales. Usar `env_file` con un `.env` que esté en `.gitignore`, o Docker secrets para producción.

### Volumes: Persistencia de datos

```yaml
services:
  db:
    volumes:
      # Named volume (gestionado por Docker, persiste datos)
      - db-data:/var/lib/postgresql/data

      # Bind mount (mapea directorio del host)
      - ./init.sql:/docker-entrypoint-initdb.d/init.sql:ro

      # Tmpfs (solo en memoria, no persiste)
      - type: tmpfs
        target: /tmp

volumes:
  db-data:  # Declaración del named volume
```

### Depends_on: Orden de inicio

```yaml
services:
  web:
    depends_on:
      api:
        condition: service_started  # Default

  api:
    depends_on:
      db:
        condition: service_healthy  # Espera healthcheck

  db:
    image: postgres:16
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      timeout: 5s
      retries: 5
```

> **Error común:** `depends_on` controla el **orden de inicio**, NO garantiza que el servicio esté "listo". Usa `condition: service_healthy` con healthchecks para esperar verdaderamente a que el servicio esté operativo.

### Networks: Redes personalizadas

```yaml
services:
  frontend:
    networks:
      - frontend-net

  api:
    networks:
      - frontend-net  # Comunicación con frontend
      - backend-net   # Comunicación con DB

  db:
    networks:
      - backend-net   # Aislada del frontend

networks:
  frontend-net:
  backend-net:
```

```
Diagrama de red:

  ┌──────────────────── frontend-net ────────────────────┐
  │                                                      │
  │  ┌──────────┐              ┌──────────┐              │
  │  │ frontend │ ◄──────────► │   api    │              │
  │  └──────────┘              └────┬─────┘              │
  │                                 │                    │
  └─────────────────────────────────┼────────────────────┘
                                    │
  ┌─────────────────────────────────┼── backend-net ─────┐
  │                                 │                    │
  │                            ┌────┴─────┐              │
  │                            │   api    │              │
  │                            └────┬─────┘              │
  │                                 │                    │
  │                            ┌────┴─────┐              │
  │                            │    db    │              │
  │                            └──────────┘              │
  └──────────────────────────────────────────────────────┘

  ✗ frontend NO puede comunicarse con db directamente
```

### DNS automático

Compose crea automáticamente resolución DNS para cada servicio basado en su nombre:

```yaml
services:
  api:
    environment:
      # "db" se resuelve automáticamente al IP del contenedor de db
      DATABASE_HOST: db
  db:
    image: postgres:16
```

El nombre DNS de cada servicio es el key definido bajo `services:`. No necesitas conocer IPs — simplemente referencia el nombre del servicio.

## 4. Comandos Esenciales

### Ciclo de vida básico

```bash
# Levantar todos los servicios (en modo detached)
docker compose up -d

# Ver servicios corriendo
docker compose ps

# Ver logs (todos los servicios, o específico)
docker compose logs
docker compose logs -f api        # Follow logs de "api"

# Parar y eliminar contenedores + redes
docker compose down
```

### Referencia de comandos clave

| Comando | Descripción | Uso típico |
|---|---|---|
| `up` | Crear e iniciar contenedores | `docker compose up -d` |
| `down` | Parar y remover contenedores + redes | `docker compose down` |
| `build` | Construir/reconstruir imágenes | `docker compose build` |
| `ps` | Listar contenedores del proyecto | `docker compose ps` |
| `logs` | Ver output de contenedores | `docker compose logs -f` |
| `exec` | Ejecutar comando en contenedor corriendo | `docker compose exec api sh` |
| `run` | Ejecutar comando one-off (nuevo contenedor) | `docker compose run api pytest` |
| `stop` | Parar servicios (sin remover) | `docker compose stop` |
| `start` | Iniciar servicios parados | `docker compose start` |
| `restart` | Reiniciar servicios | `docker compose restart api` |
| `pull` | Descargar imágenes de servicios | `docker compose pull` |
| `config` | Validar y mostrar la configuración resultante | `docker compose config` |
| `top` | Mostrar procesos corriendo | `docker compose top` |
| `watch` | Hot-reload: rebuild/refresh en cambios de archivos | `docker compose watch` |

### docker compose down: Opciones de limpieza

```bash
# Solo parar y remover contenedores + redes (default)
docker compose down

# También remover volúmenes nombrados (⚠️ borra datos persistidos)
docker compose down -v

# También remover imágenes
docker compose down --rmi all    # Todas las imágenes
docker compose down --rmi local  # Solo las construidas localmente

# Remover contenedores huérfanos (de servicios eliminados del compose file)
docker compose down --remove-orphans
```

> **Cuidado:** `docker compose down -v` elimina volúmenes nombrados. Esto **borra datos** de bases de datos, uploads, etc. Úsalo solo cuando realmente quieras empezar desde cero.

### docker compose up: Opciones útiles

```bash
# Reconstruir imágenes antes de levantar
docker compose up -d --build

# Forzar recreación de contenedores
docker compose up -d --force-recreate

# Levantar solo un servicio específico (y sus dependencias)
docker compose up -d api

# Escalar un servicio a N réplicas
docker compose up -d --scale api=3

# Modo dry-run: ver qué haría sin ejecutar
docker compose --dry-run up --build -d
```

## 5. Usando Compose para Build

Compose puede construir imágenes custom desde Dockerfiles, lo cual es ideal para desarrollo local.

### Comportamiento del build

```
docker compose up
    │
    ├── ¿Imagen existe en cache? ──► SÍ → Usar cache
    │
    └── NO → Construir desde Dockerfile → Levantar contenedor

docker compose up --build
    │
    └── SIEMPRE reconstruir → Levantar contenedor

docker compose build
    │
    └── Solo construir/reconstruir (sin levantar)
```

### Ejemplo: App con build custom

```yaml
services:
  app:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "3000:3000"
    volumes:
      - .:/app              # Bind mount para hot-reload en dev
      - /app/node_modules   # Evitar sobreescribir node_modules
    environment:
      NODE_ENV: development
```

> **¿Por qué el volume anónimo `/app/node_modules`?** El bind mount `.:app` mapea tu directorio local al contenedor, pero los `node_modules` instalados en el build se perderían. El volume anónimo preserva los modules del contenedor.

## 6. Gestión de Proyectos

### Project Name

Compose asigna un nombre de proyecto que prefija todos los recursos. Se determina por (en orden de prioridad):

1. Flag `-p` en CLI: `docker compose -p myproject up`
2. Variable `COMPOSE_PROJECT_NAME`
3. Campo `name:` en el compose file
4. Nombre del directorio que contiene el compose file

```bash
# Ejemplo
$ docker compose -p myproject ps
NAME                  SERVICE   STATUS    PORTS
myproject-web-1       web       running   0.0.0.0:8080->80/tcp
myproject-api-1       api       running   0.0.0.0:3000->3000/tcp
```

### Profiles: Servicios opcionales

Los profiles permiten definir servicios que solo se activan cuando se necesitan:

```yaml
services:
  web:
    image: nginx:alpine    # Siempre activo (sin profile)

  api:
    build: ./api           # Siempre activo

  debug-tools:
    image: nicolaka/netshoot
    profiles: ["debug"]    # Solo con --profile debug

  mailhog:
    image: mailhog/mailhog
    profiles: ["email"]    # Solo con --profile email
```

```bash
# Solo levanta web y api
docker compose up -d

# Levanta web, api + debug-tools
docker compose --profile debug up -d

# Levanta todo
docker compose --profile debug --profile email up -d
```

### Multiple Compose Files (Override pattern)

Compose permite combinar múltiples archivos. Los archivos subsecuentes sobreescriben y extienden los anteriores:

```bash
# compose.yaml       → Configuración base
# compose.override.yaml → Override automático (si existe)
# compose.prod.yaml  → Override para producción

# Desarrollo (automáticamente combina compose.yaml + compose.override.yaml)
docker compose up -d

# Producción (combinación explícita)
docker compose -f compose.yaml -f compose.prod.yaml up -d
```

```yaml
# compose.yaml (base)
services:
  api:
    image: myapp-api:latest
    ports:
      - "3000:3000"

# compose.override.yaml (dev — se aplica automáticamente)
services:
  api:
    build: ./api
    volumes:
      - ./api:/app
    environment:
      DEBUG: "true"

# compose.prod.yaml
services:
  api:
    environment:
      DEBUG: "false"
      NODE_ENV: production
    deploy:
      replicas: 3
```

### Fuentes remotas para Compose Files

```bash
# Desde un OCI registry
docker compose -f oci://registry.example.com/my-compose:latest up

# Desde un repositorio Git
docker compose -f https://github.com/user/repo.git up
docker compose -f https://github.com/user/repo.git@v1.0.0 up

# Desde stdin
cat compose.yaml | docker compose -f - up
```

### Variables de entorno de Compose

| Variable | Equivalente CLI | Propósito |
|---|---|---|
| `COMPOSE_FILE` | `-f` | Archivo(s) compose a usar |
| `COMPOSE_PROJECT_NAME` | `-p` | Nombre del proyecto |
| `COMPOSE_PROFILES` | `--profile` | Profiles a activar |
| `COMPOSE_PARALLEL_LIMIT` | `--parallel` | Paralelismo máximo |
| `COMPOSE_IGNORE_ORPHANS` | — | Ignorar contenedores huérfanos |
| `COMPOSE_MENU` | `--menu` | Menú de ayuda en `up` attached |

## 7. Compatibilidad de Versiones de Apps

Cuando defines servicios en Compose, recuerda que las aplicaciones tienen **dependencias de versión** entre sí:

```yaml
# ⚠️ Verifica compatibilidad entre versiones
services:
  wordpress:
    image: wordpress:6.4      # ← ¿Compatible con MySQL 8.4?
    depends_on:
      - db

  db:
    image: mysql:8.4          # ← Verificar docs de WordPress
```

**Checklist de compatibilidad:**

- [ ] Revisar documentación oficial de la app para versiones de DB soportadas
- [ ] Verificar compatibilidad entre versiones de lenguaje (Node, Python) y frameworks
- [ ] Probar la combinación de versiones en tu entorno local antes de commitear
- [ ] Pinear versiones específicas (no usar `latest` en producción)

## 8. Buenas Prácticas

### Estructura de proyecto recomendada

```
my-project/
├── compose.yaml               # Configuración principal
├── compose.override.yaml      # Overrides para desarrollo
├── compose.prod.yaml          # Overrides para producción
├── .env                       # Variables de entorno (en .gitignore)
├── .env.example               # Template de variables (commitear)
├── api/
│   ├── Dockerfile
│   ├── .dockerignore
│   └── src/
├── frontend/
│   ├── Dockerfile
│   └── src/
└── db/
    └── init.sql
```

### Do's and Don'ts

| ✅ Do | ❌ Don't |
|---|---|
| Pinear versiones de imágenes (`nginx:1.25-alpine`) | Usar `latest` en servicios críticos |
| Usar `depends_on` con `condition: service_healthy` | Asumir que `depends_on` espera al servicio listo |
| Separar redes (frontend/backend) | Poner todo en la red default |
| Usar `.env` + `.env.example` | Hardcodear secrets en el compose file |
| Usar named volumes para datos persistentes | Usar bind mounts para datos de producción |
| Documentar el compose file con comentarios | Asumir que el siguiente dev sabe la intención |

### Errores comunes

1. **Perder datos con `down -v`** — Borrar volúmenes accidentalmente.
2. **`ports` vs `expose`** — Exponer puertos de BD al host innecesariamente (seguridad).
3. **No usar healthchecks** — Servicios que dependen de otro fallan al inicio porque la dependencia no está lista.
4. **Rebuild olvidado** — Cambiar Dockerfile sin hacer `--build`, usando imagen vieja del cache.
5. **Bind mount sobre node_modules** — El bind mount del host sobreescribe las dependencias instaladas en el contenedor.

## 9. Resumen

```
┌─────────────────────────────────────────────────────────┐
│                    Docker Compose                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  compose.yaml                                           │
│  ┌─────────────────────────────────────────────────┐    │
│  │ services:     → Contenedores (web, api, db)     │    │
│  │ networks:     → Conectividad entre servicios    │    │
│  │ volumes:      → Persistencia de datos           │    │
│  │ configs:      → Archivos de configuración       │    │
│  │ secrets:      → Datos sensibles                 │    │
│  └─────────────────────────────────────────────────┘    │
│                                                         │
│  CLI                                                    │
│  ┌─────────────────────────────────────────────────┐    │
│  │ up -d         → Levantar todo                   │    │
│  │ down          → Parar y limpiar                 │    │
│  │ logs -f       → Ver logs en tiempo real         │    │
│  │ exec          → Shell en contenedor corriendo   │    │
│  │ build         → Reconstruir imágenes            │    │
│  │ ps            → Estado de servicios             │    │
│  └─────────────────────────────────────────────────┘    │
│                                                         │
│  Flujo de trabajo:                                      │
│  git clone → docker compose up -d → desarrollar → down  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**Puntos clave:**

- Compose **declara** tu infraestructura local como código
- Una red bridge se crea automáticamente — los servicios se comunican por nombre DNS
- `up` construye, crea y levanta; `down` para, elimina y limpia
- Ideal para desarrollo y testing, no es una herramienta de producción por sí sola
- Combina con Swarm Stacks o Kubernetes para despliegues productivos
