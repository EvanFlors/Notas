# 🐳 Docker en Producción — Estrategia y Decisiones

## 1. 🎯 Introducción

Llevar Docker a producción **no es simplemente ejecutar contenedores en un servidor**. Es un proceso de toma de decisiones incrementales sobre infraestructura, orquestación, seguridad y operaciones. Este módulo te guía a través de las decisiones clave y te ayuda a evitar las trampas más comunes.

```
Docker en Producción ≠ docker run en un servidor

Docker en Producción = Gestionar complejidad distribuida
                       con decisiones incrementales y controladas
```

### Principio fundamental

> **Limita la innovación simultánea.** Cada nueva tecnología que introduces añade complejidad. No cambies todo al mismo tiempo.

## 2. ⚖️ Estrategia de Adopción

### 2.1 Lo que NO necesitas al inicio

Muchos equipos intentan implementar todo desde el día 1. Esto es un anti-patrón:

| ❌ NO necesitas al inicio | ✅ Necesitas al inicio |
|---|---|
| CI/CD completamente automatizado | Dockerfile funcional y documentado |
| Auto-escalado dinámico | Proceso de build reproducible |
| Migración total a contenedores | Un servicio migrado exitosamente |
| Persistencia de datos en contenedores | Logs a stdout/stderr |
| Orquestador multi-nodo | Entendimiento de tu aplicación en contenedor |

### 2.2 Compatibilidad con sistemas legacy

```
┌──────────────────────────────────────────────┐
│  Mito: "Necesito microservicios para Docker" │
│                                              │
│  Realidad: Docker funciona perfectamente     │
│  con aplicaciones monolíticas.               │
│  Migración progresiva es la clave.           │
└──────────────────────────────────────────────┘
```

- **No requiere microservicios** — puedes containerizar un monolito
- **12 Factor App** es un objetivo a largo plazo, no un prerequisito
- **Migración progresiva** — empieza con un servicio, aprende, repite

## 3. 📦 Dockerfile como Pieza Central

### 3.1 Modelo de madurez del Dockerfile

Tu Dockerfile debe evolucionar a través de estas etapas:

```
Nivel 1 ──► Make it start        (funciona, arranca)
  │
Nivel 2 ──► Logging correcto     (stdout/stderr)
  │
Nivel 3 ──► Documentado          (ENV, EXPOSE, comentarios)
  │
Nivel 4 ──► Reproducible         (otro dev puede construirlo)
  │
Nivel 5 ──► Optimizado           (multi-stage, capas eficientes)
  │
Nivel 6 ──► Escalable            (healthchecks, señales, graceful shutdown)
```

### 3.2 Anti-patrones en Dockerfile

| Anti-patrón | ❌ Problema | ✅ Solución |
|---|---|---|
| **Datos en contenedor** | Persistencia interna se pierde al recrear | Usar `VOLUME` para datos persistentes |
| **Usar `latest`** | Builds no determinísticos | Versionar explícitamente (`nginx:1.25.3`) |
| **Defaults sin revisar** | Configs de imagen base no óptimas | Personalizar vía `ENV`, `RUN`, `ENTRYPOINT` |
| **Config por entorno en build** | Imagen acoplada a un entorno | Config en runtime (`ENV`, secrets, configs) |

### 3.3 Análisis de imágenes base

```bash
# Antes de usar una imagen base, inspecciona su Dockerfile:
docker image inspect nginx --format '{{.Config.Entrypoint}}'
docker image inspect nginx --format '{{.Config.Cmd}}'
docker image history nginx
```

> **Buena práctica:** Estudia el `ENTRYPOINT` y `CMD` de las imágenes oficiales antes de usarlas. Entiende qué hacen antes de sobreescribirlas.

## 4. 🖥️ Infraestructura Base

### 4.1 VM vs Bare Metal

| Factor | VM | Bare Metal |
|---|---|---|
| **Flexibilidad** | Alta (crear/destruir rápido) | Baja (hardware fijo) |
| **Rendimiento** | Overhead de virtualización | Rendimiento máximo |
| **Aislamiento** | Fuerte entre VMs | Aislamiento por contenedor |
| **Recomendación** | Usar lo que ya conoces | Solo si rendimiento es crítico |

### 4.2 Sistema operativo

```
Docker depende del kernel Linux
         │
         ▼
Storage drivers varían según distro/kernel
         │
         ▼
Elige una distro que conozcas bien
```

| Distribución | Pros | Contras |
|---|---|---|
| **Ubuntu LTS** | Baseline confiable, amplio soporte | Más pesada |
| **Alpine Linux** | Ultra-ligera, popular en contenedores | Menos familiar para ops |
| **RHEL/CentOS** | Soporte enterprise, estable | Licencias en RHEL |
| **Container-optimized OS** | Mínimo, solo para contenedores | Curva de aprendizaje |

### 4.3 Imagen base de contenedores (`FROM`)

```
Inicio: Usa lo que conoces (Ubuntu, Debian, etc.)
  │
  ▼
Optimización posterior: Migrar a Alpine (opcional)
  │
  ▼
No optimices prematuramente por tamaño
```

## 5. ⚙️ Arquitecturas de Orquestación con Swarm

### 5.1 Niveles de arquitectura

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  🟢 1 nodo (Baby Swarm)                                    │
│  • Dev/testing, alternativa a docker-compose                │
│  • Sin HA, pero con las APIs de Swarm                       │
│                                                             │
│  🟡 3 nodos (HA mínimo)                                    │
│  • Tolerancia a fallo de 1 nodo                             │
│  • Todos managers (Raft consensus)                          │
│                                                             │
│  🟠 5 nodos (producción recomendada)                        │
│  • Tolerancia a fallo de 2 nodos                            │
│  • Alta disponibilidad real                                 │
│                                                             │
│  🔵 10+ nodos                                               │
│  • Managers (3-5-7) + Workers                               │
│  • Segmentación por roles (labels/constraints)              │
│                                                             │
│  🔴 100+ nodos                                              │
│  • Redes segmentadas                                        │
│  • Control de placement avanzado                            │
│  • Monitoreo dedicado                                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 Regla de Raft para managers

| Managers | Tolerancia a fallos | Quorum |
|---|---|---|
| 1 | 0 | 1 |
| 3 | 1 | 2 |
| 5 | 2 | 3 |
| 7 | 3 | 4 |

> **Nunca uses un número par de managers.** Raft requiere mayoría, y un número par no mejora la tolerancia a fallos.

## 6. 🐄 Filosofía Operativa — Cattle, not Pets

### 6.1 Infraestructura efímera

```
PET (mascota) 🐕               CATTLE (ganado) 🐄
───────────────────             ───────────────────
Nombre único                    Identificador numérico
Cuidado individual              Reemplazable
Se repara cuando falla          Se destruye y recrea
Irremplazable                   Uno más del rebaño

Ejemplo: "el servidor de            Ejemplo: "nodo worker #47"
producción Bob"
```

- **Nodos son reemplazables** — si uno falla, añade otro
- **Contenedores son recreables** — nunca repares, recrea
- **Estado vive fuera** — datos en volúmenes/servicios externos

## 7. 🌍 Estrategia Multi-Cluster

### 7.1 Cuándo sí y cuándo no

| ❌ Malas razones para multi-cluster | ✅ Buenas razones para multi-cluster |
|---|---|
| Diferencias de hardware | Ambientes separados (dev/staging/prod) |
| Diferentes subnets | Requisitos geográficos (latencia) |
| Diferentes configs de OS | Seguridad y control de acceso |
| "Porque sí" | Regulaciones de cumplimiento |

## 8. 🪟 Consideraciones Windows

```
┌── Swarm Híbrido ──────────────────────────────┐
│                                                │
│  Managers: Linux (SIEMPRE recomendado)         │
│                                                │
│  Workers Linux: nginx, node.js, python, etc.   │
│  Workers Windows: .NET Framework, IIS, SQL Svr │
│                                                │
└────────────────────────────────────────────────┘
```

- Managers **siempre en Linux** (más estable, más rápido)
- Windows workers solo para cargas que lo requieran (.NET Framework)

## 9. 🧩 Externalización de Servicios

### 9.1 Criterio de decisión

```
¿Alta complejidad operativa?  ──► SÍ ──► ¿Existe SaaS maduro? ──► SÍ ──► EXTERNALIZAR
         │                                        │
         NO                                      NO
         │                                        │
         ▼                                        ▼
    Self-hosted                             Self-hosted
```

### 9.2 Servicios candidatos a externalización

| Servicio | Self-hosted | SaaS/Managed |
|---|---|---|
| **Registry** | Docker Registry, Harbor | Docker Hub, ECR, ACR, GCR |
| **Logs** | ELK Stack, Loki | Datadog, Splunk Cloud, CloudWatch |
| **Monitoreo** | Prometheus + Grafana | Datadog, New Relic, Sematext |
| **CI/CD** | Jenkins, GitLab CI | GitHub Actions, CircleCI, Travis |
| **Proxy/Ingress** | Traefik, Nginx | Cloud Load Balancers |

## 11. 🔐 Seguridad en Imágenes

| Práctica | Descripción |
|---|---|
| **Escaneo de vulnerabilidades** | Integrar scanning en CI/CD (Trivy, Snyk, Clair) |
| **RBAC** | Control de acceso basado en roles para registry |
| **Image promotion** | Dev → Staging → Prod con gates de calidad |
| **Content Trust** | Firmar imágenes digitalmente (Docker Content Trust) |
| **Base images mínimas** | Reducir superficie de ataque |
| **No ejecutar como root** | `USER nonroot` en Dockerfile |

## 12. 🤔 ¿Siempre Necesitas un Orquestador?

### Alternativa: 1 contenedor por VM

```
┌─── VM ──────────┐
│  ┌────────────┐ │
│  │ Contenedor │ │    • Baja complejidad
│  │  (1 app)   │ │    • Aislamiento fuerte
│  └────────────┘ │    • Fácil adopción
│  Docker Engine  │    • Para migraciones graduales
└─────────────────┘
```

**Cuándo considerar esta alternativa:**
- Infraestructura VM madura con auto-escalado existente
- Requisitos de aislamiento fuerte (regulaciones)
- Migraciones graduales desde infraestructura legacy
- Equipos pequeños sin experiencia en orquestación

## 13. ⚠️ Errores Comunes

| # | Error | Consecuencia | Solución |
|---|---|---|---|
| 1 | Migrar todo al mismo tiempo | Caos operativo | Migrar un servicio a la vez |
| 2 | Usar `latest` en producción | Builds no reproducibles | Tags explícitos siempre |
| 3 | Datos dentro del contenedor | Pérdida de datos al recrear | Volúmenes externos |
| 4 | Ignorar logs | Imposible debuggear | stdout/stderr desde día 1 |
| 5 | Número par de managers | Sin mejora de HA | Siempre número impar (3, 5, 7) |
| 6 | Tratar servidores como pets | No escala, difícil de mantener | Infraestructura efímera |
| 7 | Optimizar prematuramente | Complejidad innecesaria | Funcionalidad primero, optimizar después |
| 8 | Saltar TLS/seguridad | Vulnerabilidades en producción | Seguridad desde el inicio |

---

## 14. 📋 Checklist de Producción

```
□ Dockerfile documentado y reproducible
□ Logs a stdout/stderr
□ Healthcheck configurado
□ Imagen versionada (no latest)
□ Datos en volúmenes externos
□ TLS habilitado
□ Registry configurado
□ Monitoreo y alertas
□ Estrategia de backup
□ Plan de rollback
□ Documentación de runbooks
□ Pruebas de recuperación ante desastres
```
