# ☸️ Kubernetes — Inspección, Logs y Debugging

## 1. 🎯 Introducción

Inspeccionar y debuggear aplicaciones es una habilidad fundamental en Kubernetes. Este módulo cubre el flujo completo: desde listar recursos hasta analizar logs del cluster.

```
Flujo de diagnóstico típico:

  kubectl get all          (vista general)
       │
       ▼
  kubectl get pods -o wide (detalle de pods)
       │
       ▼
  kubectl describe pod X   (eventos + config)
       │
       ▼
  kubectl logs pod X       (output de la aplicación)
       │
       ▼
  kubectl exec -it pod X   (shell interactivo)
```

## 2. 🔍 Inspección con `get`

### 2.1 Comandos fundamentales

```bash
# Vista general
kubectl get all                                  # Recursos comunes del namespace

# Pods con más detalle
kubectl get pods -o wide                         # Incluye Node, IP

# Output YAML completo de un recurso
kubectl get deploy/my-apache -o yaml             # Todo sobre el deployment

# Filtrar por labels
kubectl get pods -l app=my-apache

# Watch (monitorear cambios en tiempo real)
kubectl get pods -w                              # Se actualiza automáticamente
kubectl get events --watch-only                  # Solo eventos futuros
```

### 2.2 Entendiendo Labels y Selectors

Los labels son la **columna vertebral** de cómo K8s conecta recursos entre sí:

```
Deployment: my-apache
  spec.selector.matchLabels:
    app: my-apache                    ← "busca ReplicaSets con este label"
        │
        ▼
ReplicaSet: my-apache-86c695c9b6
  metadata.labels:
    app: my-apache                    ← coincide con el Deployment
    pod-template-hash: 86c695c9b6     ← identifica esta versión
  spec.selector.matchLabels:
    app: my-apache
    pod-template-hash: 86c695c9b6     ← "busca Pods con estos labels"
        │
        ▼
Pod: my-apache-86c695c9b6-8d75v
  metadata.labels:
    app: my-apache                    ← coincide con el ReplicaSet
    pod-template-hash: 86c695c9b6     ← coincide con el ReplicaSet
```

> **Clave:** El label `pod-template-hash` es generado automáticamente y vincula ReplicaSets con sus Pods. Nunca lo modifiques manualmente.

## 3. 📝 Inspección con `describe`

### 3.1 ¿Cuándo usar `describe` vs `get`?

| `get` | `describe` |
|---|---|
| Datos de **un** tipo de recurso | Combina info de recursos **relacionados** |
| Soporta `-o yaml/json/wide` | Formato legible para humanos |
| Para scripts y automatización | Para **debugging interactivo** |
| No muestra eventos | **Siempre muestra eventos** (crucial para debug) |

### 3.2 Describe de un Deployment

```bash
kubectl describe deploy/my-apache
```

Muestra:
- Resumen del Deployment (nombre, namespace, labels, replicas)
- Pod template (imagen, ports, env, mounts)
- Nombres de Old/New ReplicaSets
- **Events** (escalados, rollouts, errores)

### 3.3 Describe de un Pod

```bash
kubectl describe pod/my-apache-86c695c9b6-8d75v
```

Muestra:
- Estado de cada container (Running, Waiting, Terminated)
- Condiciones (Ready, Initialized, PodScheduled)
- Volúmenes montados
- **Events** (Scheduled, Pulling, Created, Started, o errores)

### 3.4 Describe de un Node

```bash
# Flujo típico: high-level → low-level
kubectl get nodes                                # Lista de nodos
kubectl get node/docker-desktop -o wide          # Más info
kubectl describe node/docker-desktop             # Todo: recursos, pods, events
```

El describe de un Node muestra:
- Capacidad (CPU, memoria, pods máximos)
- Recursos asignados vs disponibles
- Pods corriendo en ese nodo
- Conditions (Ready, MemoryPressure, DiskPressure)
- Events del nodo

## 4. 👁️ Watching Resources

### 4.1 Monitoreo en tiempo real

```bash
# Watch pods (se actualiza en cada cambio)
kubectl get pods -w

# En otra terminal, provocar un cambio:
kubectl delete pod/my-apache-xxxx-yyyy
# → Verás el pod morir y el ReplicaSet crear uno nuevo

# Solo eventos futuros
kubectl get events --watch-only

# Todos los eventos recientes
kubectl get events
kubectl get events --sort-by=.metadata.creationTimestamp
```

### 4.2 Ejemplo de observación de self-healing

```
Terminal 1: kubectl get pods -w

NAME                         READY   STATUS    AGE
my-apache-86c695c9b6-8d75v   1/1     Running   5m
my-apache-86c695c9b6-abc12   1/1     Running   5m

Terminal 2: kubectl delete pod my-apache-86c695c9b6-8d75v

Terminal 1 (actualización automática):
my-apache-86c695c9b6-8d75v   1/1     Terminating   5m
my-apache-86c695c9b6-xyz99   0/1     Pending       0s     ← ¡Nuevo pod!
my-apache-86c695c9b6-xyz99   0/1     ContainerCreating   0s
my-apache-86c695c9b6-xyz99   1/1     Running       2s     ← Recuperado
my-apache-86c695c9b6-8d75v   0/1     Terminating   5m
```

## 5. 📊 Container Logs

### 5.1 Comandos de logs

```bash
# Logs básicos (un pod random del deployment, primer container)
kubectl logs deploy/my-apache

# Follow + últimas N líneas
kubectl logs deploy/my-apache --follow --tail 1

# Container específico en un pod
kubectl logs pod/my-apache-xx-yy -c httpd

# Todos los containers de un pod
kubectl logs pod/my-apache-xx-yy --all-containers=true

# Múltiples pods por label
kubectl logs -l app=my-apache

# Logs de instancia anterior (después de un restart)
kubectl logs my-pod --previous
```

### 5.2 ¿Qué container se selecciona por defecto?

Cuando un Pod tiene múltiples containers, `kubectl logs` usa el **primer container** del spec:

```yaml
spec:
  containers:
  - name: httpd          # ← Este se selecciona por defecto
    image: httpd
  - name: sidecar
    image: busybox
```

Para elegir otro: `kubectl logs pod-name -c sidecar`

### 5.3 Herramientas avanzadas de logs

| Herramienta | Descripción |
|---|---|
| **stern** | Multi-pod logging con colores y filtros ([github.com/stern/stern](https://github.com/stern/stern)) |
| **kubetail** | Tail de múltiples pods simultáneamente |
| **kubectl logs -l** | Label selector nativo para múltiples pods |

## 6. 🏢 Logging a Nivel de Cluster

### 6.1 Arquitectura de logs

```
┌─── Node ─────────────────────────────────────────┐
│                                                    │
│  Pod ──► stdout/stderr ──► Container Runtime       │
│                                    │               │
│                                    ▼               │
│                            /var/log/pods/          │
│                            /var/log/containers/    │
│                                    │               │
│                          kubelet (rotación)        │
│                                    │               │
│                                    ▼               │
│                          kubectl logs              │
└────────────────────────────────────────────────────┘
```

### 6.2 Rotación de logs

El kubelet gestiona la rotación automáticamente:

| Parámetro | Default | Descripción |
|---|---|---|
| `containerLogMaxSize` | 10Mi | Tamaño máximo por archivo |
| `containerLogMaxFiles` | 5 | Archivos por container |

> **Importante:** `kubectl logs` solo muestra el archivo de log **más reciente**. Si un Pod genera 40MB de logs con rotación a 10MB, solo verás los últimos ~10MB.

### 6.3 Soluciones de logging centralizado

```
┌────────────────────────────────────────────────────┐
│           Cluster-level Logging                     │
│                                                     │
│  Opción 1: Node-level agent (DaemonSet)            │
│  ┌──────┐  ┌──────┐  ┌──────┐                     │
│  │Agent │  │Agent │  │Agent │  → Backend           │
│  │Node 1│  │Node 2│  │Node 3│    (ELK, Loki)      │
│  └──────┘  └──────┘  └──────┘                     │
│                                                     │
│  Opción 2: Sidecar container                        │
│  ┌─── Pod ──────────────┐                          │
│  │ App → file           │                          │
│  │ Sidecar → stdout     │  → kubelet → agent       │
│  └──────────────────────┘                          │
│                                                     │
│  Opción 3: Push directo desde app                   │
│  App → Backend (sin pasar por kubelet)              │
└────────────────────────────────────────────────────┘
```

| Solución | Tipo | Fortaleza |
|---|---|---|
| **Grafana Loki** | Open-source | Ligero, integración Grafana |
| **Elastic (ELK)** | Open-source | Búsqueda potente, ecosistema maduro |
| **AWS CloudWatch** | Managed | Integración nativa AWS |
| **Splunk** | Enterprise | Análisis avanzado |
| **Datadog** | SaaS | Logs + metrics + traces unificados |

### 6.4 Sidecar pattern para logs

Cuando tu app no escribe a stdout/stderr:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: counter
spec:
  containers:
  - name: app
    image: busybox:1.28
    args: [/bin/sh, -c, 'i=0; while true; do echo "$i: $(date)" >> /var/log/app.log; i=$((i+1)); sleep 1; done']
    volumeMounts:
    - name: varlog
      mountPath: /var/log
  - name: log-streamer                    # ← Sidecar
    image: busybox:1.28
    args: [/bin/sh, -c, 'tail -n+1 -F /var/log/app.log']
    volumeMounts:
    - name: varlog
      mountPath: /var/log
  volumes:
  - name: varlog
    emptyDir: {}
```

```bash
# Ver logs del sidecar (que a su vez lee los de la app)
kubectl logs counter -c log-streamer
```

## 7. 🔧 System Logs

### 7.1 Dónde están los logs del sistema

| Componente | Ubicación |
|---|---|
| **kubelet** | journald (`journalctl -u kubelet`) o `/var/log/kubelet.log` |
| **Container runtime** | journald o `/var/log/` |
| **API Server** | Pod logs (`/var/log/pods/...`) |
| **Scheduler** | Pod logs |
| **Controller Manager** | Pod logs |
| **etcd** | Pod logs |

### 7.2 Formato de logs (klog)

```
I1025 00:15:15.525108  1 controller.go:116] "Pod status updated" pod="kube-system/coredns" status="ready"
│                       │                    │                     │
│                       │                    │                     └─ Datos estructurados
│                       │                    └─ Mensaje
│                       └─ ID del proceso
└─ I=Info, W=Warning, E=Error, F=Fatal + timestamp
```

### 7.3 Formatos de log

| Formato | Flag | Estado |
|---|---|---|
| **klog nativo** | (default) | Estable |
| **Structured** | Automático | Beta (v1.23+) |
| **JSON** | `--logging-format=json` | Alpha (v1.19+) |

### 7.4 Log query (desde nodos)

```bash
# Obtener logs del kubelet desde un nodo (v1.27+, beta)
kubectl get --raw "/api/v1/nodes/node-1/proxy/logs/?query=kubelet"

# Filtrar por patrón
kubectl get --raw "/api/v1/nodes/node-1/proxy/logs/?query=kubelet&pattern=error"

# Opciones: boot, pattern, query, sinceTime, untilTime, tailLines
```

## 8. ✅ Buenas Prácticas

| Práctica | Razón |
|---|---|
| **Siempre loggea a stdout/stderr** | Es el contrato con K8s; kubelet lo captura |
| **Usa labels consistentes** | Facilita `kubectl logs -l app=X` |
| **Configura logging centralizado** | Los Pods son efímeros; los logs no deberían serlo |
| **Usa `--previous` para crashes** | Ver qué loggeó un container antes de morir |
| **Instala stern** | Multi-pod logging mucho más cómodo |
| **No escribas logs enormes** | Configura rotación (`containerLogMaxSize`) |

## 9. ⚠️ Errores Comunes

| # | Error | Solución |
|---|---|---|
| 1 | `kubectl logs deploy/X` muestra solo 1 pod | Es por diseño; usa `-l app=X` para todos |
| 2 | Logs vacíos después de restart | Usa `--previous` para ver la instancia anterior |
| 3 | `kubectl logs` trunca salida | Solo muestra el log file más reciente (rotación) |
| 4 | App loggea a archivo, no a stdout | Usa sidecar pattern o reconfigura la app |
| 5 | No distinguir `get` de `describe` | `describe` muestra eventos; úsalo para debugging |
| 6 | Ignorar Events | Los Events muestran el "por qué" de los problemas |
| 7 | No usar `-w` (watch) | Esencial para observar cambios en tiempo real |

## 10. 📋 Cheat Sheet de Debugging

```bash
# ═══ FLUJO DE DIAGNÓSTICO ═══
kubectl get all                                  # 1. Vista general
kubectl get pods -o wide                         # 2. Estado de pods
kubectl describe pod/NAME                        # 3. Detalles + eventos
kubectl logs pod/NAME [-c CONTAINER]             # 4. Logs de app
kubectl logs pod/NAME --previous                 # 5. Logs pre-crash
kubectl exec -it pod/NAME -- /bin/sh             # 6. Shell interactivo
kubectl get events --sort-by=.metadata.creationTimestamp  # 7. Eventos

# ═══ MONITORING ═══
kubectl get pods -w                              # Watch pods
kubectl get events --watch-only                  # Watch events
kubectl top pod                                  # CPU/Mem por pod
kubectl top node                                 # CPU/Mem por node

# ═══ LOGS MULTI-POD ═══
kubectl logs -l app=my-app                       # Todos los pods con label
kubectl logs -l app=my-app --all-containers      # + todos los containers
kubectl logs -l app=my-app -f --tail 10          # Follow últimas 10 líneas
```

## 11. 🔑 Resumen

| Concepto | Punto clave |
|---|---|
| **`kubectl get`** | Listado tabular; `-o wide/yaml/json` para más detalle |
| **`kubectl describe`** | Debugging; combina recursos relacionados + Events |
| **Labels** | Vinculan Deployment → ReplicaSet → Pod automáticamente |
| **Watch (`-w`)** | Monitoreo en tiempo real sin herramientas externas |
| **Logs** | stdout/stderr capturado por kubelet; `--previous` para crashes |
| **Rotación** | 10Mi × 5 archivos por default; `kubectl logs` solo muestra el último |
| **Logging centralizado** | DaemonSet agent (Loki, ELK) o sidecar pattern |
| **System logs** | klog format; journald o /var/log según componente |
