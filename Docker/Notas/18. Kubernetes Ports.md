# ☸️ Kubernetes — Services, Networking y DNS

## 1. 🎯 Introducción

En Kubernetes, los **Pods son efímeros** — se crean, mueren y se recrean con nuevas IPs. Los **Services** resuelven este problema proporcionando un **endpoint de red estable** para acceder a un conjunto de Pods.

```
Sin Service:                        Con Service:

Pod (10.244.0.5) ← muere            Service (ClusterIP: 10.96.0.1)
Pod (10.244.0.9) ← nueva IP!               │
                                    ┌──────┼──────┐
El cliente pierde                  Pod    Pod    Pod
la conexión                     (cualquier IP, no importa)
```

### Analogía

Un Service es como un **número de teléfono de oficina** — siempre el mismo número, aunque los empleados (Pods) que atienden cambien constantemente.

## 2. 🧩 Tipos de Services

### 2.1 Visión general

```
                    ExternalName
                    (DNS CNAME)

                    LoadBalancer
                    (IP externa + LB)
                        │
                        │ crea
                        ▼
                     NodePort
                    (puerto en cada nodo)
                        │
                        │ crea
                        ▼
                     ClusterIP
                    (IP interna del cluster)
                        │
                        │ selecciona
                        ▼
                      Pods
```

> **Clave:** Los tipos son **aditivos**. LoadBalancer crea un NodePort, que crea un ClusterIP. Cada nivel incluye los anteriores.

### 2.2 Comparación detallada

| Tipo | Acceso | IP | Puerto | Caso de uso |
|---|---|---|---|---|
| **ClusterIP** | Solo dentro del cluster | Virtual interna | App port | Comunicación entre servicios |
| **NodePort** | Desde fuera vía IP del nodo | IP de cada nodo | 30000-32767 | Acceso externo sin LB |
| **LoadBalancer** | Desde internet vía LB | IP pública del LB | Cualquiera | Producción con cloud provider |
| **ExternalName** | DNS CNAME | N/A | N/A | Apuntar a servicios externos |

## 3. 🔵 ClusterIP (default)

### 3.1 Concepto

ClusterIP asigna una **IP virtual interna** accesible solo desde dentro del cluster:

```
┌──── Cluster ───────────────────────────────────┐
│                                                │
│   Frontend Pod ──► ClusterIP:10.96.0.1:8888    │
│                         │                      │
│                    ┌────┼────┐                 │
│                   Pod  Pod  Pod  (backend)     │
│                                                │
│   EXTERNO ──✗──► ClusterIP  (NO accesible)     │
└────────────────────────────────────────────────┘
```

### 3.2 Ejercicio práctico

```bash
# 1. Crear un deployment
kubectl create deployment httpenv --image=bretfisher/httpenv

# 2. Escalar a 5 réplicas
kubectl scale deployment/httpenv --replicas=5

# 3. Crear el Service ClusterIP (default)
kubectl expose deployment/httpenv --port 8888

# 4. Verificar
kubectl get service
# NAME      TYPE        CLUSTER-IP    EXTERNAL-IP   PORT(S)
# httpenv   ClusterIP   10.96.0.123   <none>        8888/TCP

# 5. Acceder desde dentro del cluster
kubectl run tmp-shell --rm -it --image bretfisher/netshoot -- bash
# Dentro del pod:
curl httpenv:8888      # ← Usa DNS, no IP
```

> **¿Por qué funciona `curl httpenv:8888`?** CoreDNS resuelve el nombre del Service a su ClusterIP automáticamente.

## 4. 🟡 NodePort

### 4.1 Concepto

NodePort abre un **puerto alto (30000-32767)** en **todos los nodos** del cluster:

```
┌──── Internet ────────────────────────────────┐
│                                               │
│   curl <NodeIP>:31234                         │
└───────────────┬───────────────────────────────┘
                │
┌───────────────▼───────────────────────────────┐
│              Cluster                           │
│                                                │
│  Node 1 (:31234) ──┐                          │
│  Node 2 (:31234) ──┤──► ClusterIP ──► Pods   │
│  Node 3 (:31234) ──┘                          │
│                                                │
└────────────────────────────────────────────────┘
```

### 4.2 Ejercicio práctico

```bash
# Crear NodePort (también crea ClusterIP automáticamente)
kubectl expose deployment/httpenv --port 8888 \
  --name httpenv-np --type NodePort

# Verificar
kubectl get svc
# NAME         TYPE       CLUSTER-IP    PORT(S)
# httpenv-np   NodePort   10.96.0.200   8888:31234/TCP
#                                            └── Puerto asignado en cada nodo

# Acceder desde fuera (si estás en Docker Desktop)
curl localhost:31234
```

### 4.3 Elegir puerto específico

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  type: NodePort
  selector:
    app.kubernetes.io/name: MyApp
  ports:
    - port: 80
      targetPort: 80
      nodePort: 30007    # Puerto específico (30000-32767)
```

## 5. 🟢 LoadBalancer

### 5.1 Concepto

LoadBalancer solicita un **balanceador de carga externo** al cloud provider:

```
┌──── Internet ────────────────────────────────┐
│                                               │
│   curl <LoadBalancer-IP>:8888                 │
└───────────────┬───────────────────────────────┘
                │
┌───────────────▼───────────────────────────────┐
│         Load Balancer (cloud)                  │
│              │                                 │
│         ┌────┼────┐                            │
│         ▼    ▼    ▼                            │
│       Node  Node  Node  (:NodePort)           │
│         │    │    │                            │
│         └────┼────┘                            │
│              ▼                                 │
│         ClusterIP ──► Pods                    │
└────────────────────────────────────────────────┘
```

### 5.2 Ejercicio práctico

```bash
# Crear LoadBalancer
kubectl expose deployment/httpenv --port 8888 \
  --name httpenv-lb --type LoadBalancer

# Verificar
kubectl get svc
# NAME         TYPE           CLUSTER-IP    EXTERNAL-IP   PORT(S)
# httpenv-lb   LoadBalancer   10.96.0.300   localhost      8888:32100/TCP

# Docker Desktop: acceder directamente
curl localhost:8888
```

### 5.3 Disponibilidad por plataforma

| Plataforma | LoadBalancer disponible | Alternativa |
|---|---|---|
| **Docker Desktop** | ✅ Built-in (publica en localhost) | — |
| **EKS / AKS / GKE** | ✅ Cloud LB nativo | — |
| **minikube** | ⚠️ `minikube tunnel` | NodePort |
| **kubeadm / bare metal** | ❌ Se queda en "Pending" | MetalLB, NodePort |
| **k3s** | ✅ ServiceLB (Klipper) | — |

## 6. 🟣 ExternalName

### 6.1 Concepto

ExternalName **no crea proxy ni routing**. Solo añade un **registro CNAME** en CoreDNS para apuntar a un servicio externo:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-database
  namespace: prod
spec:
  type: ExternalName
  externalName: db.external-provider.com
```

```bash
# Dentro del cluster:
curl my-database.prod.svc.cluster.local
# → Se resuelve como CNAME → db.external-provider.com
```

**Caso de uso:** Migración gradual — hoy apunta a DB externa, mañana cambias a un Service interno sin que los clientes cambien nada.

> ⚠️ **Limitación:** HTTP/HTTPS pueden fallar porque el header `Host:` no coincidirá con el nombre externo.

## 7. 🌐 DNS en Kubernetes (CoreDNS)

### 7.1 Resolución de nombres

```
Dentro del mismo Namespace:
  curl httpenv:8888                              ← Solo nombre

Desde otro Namespace:
  curl httpenv.production:8888                   ← nombre.namespace

FQDN completo:
  curl httpenv.production.svc.cluster.local:8888 ← FQDN
```

### 7.2 Formato del FQDN

```
<service>.<namespace>.svc.cluster.local

  httpenv.default.svc.cluster.local
     │       │     │      │
     │       │     │      └── Dominio del cluster
     │       │     └── Siempre "svc" para Services
     │       └── Namespace
     └── Nombre del Service
```

### 7.3 DNS SRV Records

Para Services con puertos nombrados:

```bash
# Si el Service tiene un port llamado "http" con protocolo TCP:
dig _http._tcp.my-service.my-ns.svc.cluster.local SRV
# → Devuelve puerto y dirección
```

### 7.4 Service Discovery

K8s ofrece dos mecanismos:

| Mecanismo | Cómo funciona | Limitación |
|---|---|---|
| **DNS** (recomendado) | CoreDNS resuelve nombres de Services | Ninguna significativa |
| **Variables de entorno** | kubelet inyecta `{SVC}_SERVICE_HOST` y `{SVC}_SERVICE_PORT` | El Service debe existir ANTES que el Pod |

```bash
# Variables de entorno inyectadas automáticamente:
REDIS_PRIMARY_SERVICE_HOST=10.0.0.11
REDIS_PRIMARY_SERVICE_PORT=6379
```

## 8. 🔧 Definición de Services en YAML

### 8.1 Service básico con selector

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  selector:
    app.kubernetes.io/name: MyApp     # ← Selecciona pods con este label
  ports:
    - protocol: TCP
      port: 80                         # ← Puerto del Service
      targetPort: 9376                 # ← Puerto del container
```

### 8.2 Service multi-puerto

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-service
spec:
  selector:
    app.kubernetes.io/name: MyApp
  ports:
    - name: http                       # ← Nombre obligatorio con multi-port
      protocol: TCP
      port: 80
      targetPort: 9376
    - name: https
      protocol: TCP
      port: 443
      targetPort: 9377
```

### 8.3 Named ports (desacoplamiento)

```yaml
# Service apunta a nombre de puerto, no número
apiVersion: v1
kind: Service
spec:
  ports:
  - port: 80
    targetPort: http-web-svc           # ← Nombre, no número
---
# Pod define el nombre del puerto
apiVersion: v1
kind: Pod
spec:
  containers:
  - name: nginx
    ports:
    - containerPort: 80
      name: http-web-svc               # ← Coincide con el Service
```

> **Ventaja:** Puedes cambiar el puerto del container sin actualizar el Service.

### 8.4 Service sin selector (endpoints manuales)

Para backends externos al cluster:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-db
spec:
  ports:
    - port: 5432
---
apiVersion: discovery.k8s.io/v1
kind: EndpointSlice
metadata:
  name: external-db-1
  labels:
    kubernetes.io/service-name: external-db
addressType: IPv4
ports:
  - port: 5432
endpoints:
  - addresses: ["10.1.2.3"]
  - addresses: ["10.4.5.6"]
```

## 9. 🎭 Headless Services

Cuando **no necesitas** load balancing ni ClusterIP:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-headless
spec:
  clusterIP: None                      # ← "None" = headless
  selector:
    app: my-app
  ports:
    - port: 80
```

**Comportamiento DNS:**
- Con selector → A/AAAA records apuntan directamente a cada Pod IP
- Sin selector → Busca EndpointSlice o CNAME (ExternalName)

**Caso de uso:** StatefulSets donde cada Pod necesita ser direccionable individualmente (ej: bases de datos con replicación).

## 10. 🔀 Traffic Policies

### 10.1 External Traffic Policy

```yaml
spec:
  externalTrafficPolicy: Local    # Solo envía al nodo que recibe
  # vs
  externalTrafficPolicy: Cluster  # Distribuye a todos los nodos (default)
```

| Policy | Comportamiento | Pro | Contra |
|---|---|---|---|
| **Cluster** (default) | Distribuye tráfico a todos los nodos | Distribución uniforme | Doble hop, pierde IP origen |
| **Local** | Solo pods en el nodo receptor | Preserva IP origen | Distribución desigual |

### 10.2 Traffic Distribution (K8s 1.31+)

```yaml
spec:
  trafficDistribution: PreferSameZone    # Preferir pods en misma zona
  # PreferSameNode                         # Preferir pods en mismo nodo
```

### 10.3 Session Affinity

```yaml
spec:
  sessionAffinity: ClientIP              # Sticky sessions por IP
  sessionAffinityConfig:
    clientIP:
      timeoutSeconds: 10800              # 3 horas
```

## 11. 🌍 EndpointSlices

Los **EndpointSlices** reemplazaron a Endpoints (deprecated v1.33) para mejor escalabilidad:

| Endpoints (legacy) | EndpointSlices (actual) |
|---|---|
| Un objeto por Service | Múltiples slices (100 endpoints c/u) |
| Trunca a 1000 backends | Sin límite práctico |
| Sin soporte dual-stack | Soporte IPv4 + IPv6 |

## 12. ⚠️ Errores Comunes

| # | Error | Causa | Solución |
|---|---|---|---|
| 1 | Service no alcanza Pods | Selector no coincide con labels de Pods | Verificar `kubectl get pods --show-labels` |
| 2 | LoadBalancer en "Pending" | No hay LB provider (bare metal) | Instalar MetalLB o usar NodePort |
| 3 | No se puede resolver nombre del Service | CoreDNS no está corriendo | `kubectl get pods -n kube-system` |
| 4 | `curl service:port` falla desde fuera | ClusterIP solo es interno | Usar NodePort o LoadBalancer |
| 5 | Conflicto de NodePort | Puerto ya asignado a otro Service | Dejar que K8s asigne o verificar con `kubectl get svc` |
| 6 | ExternalName + HTTPS falla | Host header no coincide | Usa Service sin selector + EndpointSlice |
| 7 | Variables de entorno vacías | Service creado después del Pod | Crear Services antes que Pods, o usar DNS |
| 8 | `port-forward` a Service sin selector | API server no permite proxy a endpoints sin pods | Usa `kubectl run` para crear un pod temporal |

## 13. 📋 Cheat Sheet

```bash
# ═══ CREAR SERVICES ═══
kubectl expose deployment/app --port 80                          # ClusterIP (default)
kubectl expose deployment/app --port 80 --type NodePort          # NodePort
kubectl expose deployment/app --port 80 --type LoadBalancer      # LoadBalancer

# ═══ INSPECCIONAR ═══
kubectl get svc                                                  # Listar services
kubectl get svc -o wide                                          # Con selector
kubectl get endpoints                                            # Ver endpoints
kubectl get endpointslices                                       # EndpointSlices
kubectl describe svc my-service                                  # Detalles

# ═══ ACCEDER DESDE DENTRO ═══
kubectl run tmp --rm -it --image bretfisher/netshoot -- bash
curl <service-name>:<port>                                       # Mismo namespace
curl <service-name>.<namespace>:<port>                           # Otro namespace

# ═══ ACCEDER DESDE FUERA ═══
curl <NodeIP>:<NodePort>                                         # NodePort
curl <LoadBalancerIP>:<port>                                     # LoadBalancer
kubectl port-forward svc/my-service 8080:80                      # Port forward

# ═══ DNS ═══
kubectl get namespaces
# FQDN: <svc>.<ns>.svc.cluster.local

# ═══ CLEANUP ═══
kubectl delete svc my-service
kubectl delete deployment httpenv
```

## 14. 🔑 Resumen

| Concepto | Punto clave |
|---|---|
| **Service** | Endpoint estable para Pods efímeros; selecciona por labels |
| **ClusterIP** | Default; solo accesible dentro del cluster |
| **NodePort** | Puerto alto en cada nodo; acceso externo básico |
| **LoadBalancer** | IP pública vía cloud LB; crea NodePort + ClusterIP |
| **ExternalName** | CNAME DNS; para servicios externos |
| **Headless** | `clusterIP: None`; DNS apunta directo a Pod IPs |
| **CoreDNS** | Resolución automática: `<svc>.<ns>.svc.cluster.local` |
| **Tipos aditivos** | LB → NodePort → ClusterIP; cada uno incluye los anteriores |
| **EndpointSlices** | Reemplazo moderno de Endpoints; mejor escalabilidad |
