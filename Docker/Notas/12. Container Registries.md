# 🐳 Container Registries — Almacenamiento y Distribución de Imágenes

## 1. 🎯 Introducción

Un **Container Registry** es un servicio que almacena y distribuye imágenes de contenedores. Es una pieza **no opcional** en cualquier ecosistema de contenedores — sin él, cada máquina tendría que construir o importar manualmente sus imágenes.

```
┌──────────────┐     push      ┌──────────────┐     pull      ┌──────────────┐
│  Developer   │ ────────────► │   Registry   │ ◄──────────── │  Production  │
│  Workstation │               │  (Hub/Priv)  │               │   Servers    │
└──────────────┘               └──────────────┘               └──────────────┘
```

### ¿Por qué necesitas un registry?

| Sin Registry | Con Registry |
|---|---|
| Exportar/importar imágenes como archivos ZIP | Push/pull automático por red |
| Sin versionado centralizado | Tags y digest para cada versión |
| Sin control de acceso | Autenticación y autorización |
| Sin escaneo de seguridad | Integración con scanning tools |

> **Punto clave:** Planifica tu estrategia de registry desde el inicio. Es tan fundamental como tu repositorio de código fuente.

## 2. 📦 Docker Hub — El Registry Público

### 2.1 ¿Qué es Docker Hub?

Docker Hub es el registry público más popular del mundo. Es el **registry por defecto** de Docker — cuando ejecutas `docker pull nginx`, está descargando desde Docker Hub.

Es más que un simple almacén de imágenes:

```
┌─────────────────────────────────────────────┐
│              Docker Hub                      │
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌───────────┐ │
│  │ Registry │  │  Image   │  │  Webhook  │ │
│  │ Storage  │  │ Building │  │  System   │ │
│  └──────────┘  └──────────┘  └───────────┘ │
│                                              │
│  ┌──────────┐  ┌──────────┐  ┌───────────┐ │
│  │  Teams/  │  │  Auto-   │  │  Repo     │ │
│  │  Orgs    │  │  Builds  │  │  Links    │ │
│  └──────────┘  └──────────┘  └───────────┘ │
└─────────────────────────────────────────────┘
```

### 2.2 Características principales

| Característica | Descripción |
|---|---|
| **Repositorios públicos** | Ilimitados y gratuitos |
| **Repositorios privados** | 1 gratis, más requieren plan de pago |
| **Organizaciones** | Gestión de equipos y permisos (gratis) |
| **Automated Builds** | Construcción automática desde GitHub/BitBucket |
| **Webhooks** | Notificaciones POST cuando una imagen es pusheada |
| **Repository Links** | Reconstruir tu imagen cuando su base cambia |
| **Build Triggers** | URLs únicas para disparar builds externamente |

### 2.3 Automated Builds (CI/CD integrado)

Los Automated Builds conectan Docker Hub con tu repositorio de código:

```
┌──────────┐   commit    ┌──────────┐   webhook   ┌───────────┐
│  GitHub  │ ──────────► │  Docker  │ ───────────► │  Jenkins  │
│  /Bitbck │             │   Hub    │              │  /CI Tool │
└──────────┘             │          │              └───────────┘
                         │  Build   │
                         │  Image   │
                         └──────────┘
```

**Configuración paso a paso:**

```bash
# 1. NO crear repositorio manualmente
# 2. NO hacer docker push antes de configurar
# 3. Ir a Create > Create Automated Build
# 4. Seleccionar repositorio de GitHub/BitBucket
# 5. Configurar branches y tags para build
```

### 2.4 Repository Links — Cadena de reconstrucción

Cuando tu imagen depende de otra (vía `FROM`), configura **Repository Links** para que se reconstruya automáticamente cuando la imagen base cambie:

```
Imagen base actualizada (ej: ruby:3.2)
         │
         ▼
Tu imagen se reconstruye automáticamente
         │
         ▼
Webhook notifica a tu pipeline de CI/CD
```

> **Buena práctica:** Siempre configura Repository Links para imágenes que dependen de bases que reciben actualizaciones de seguridad.

### 2.5 Permisos y organizaciones

│                                         │
│  ┌─ Team A ──────┐  ┌─ Team B ──────┐ │

## 3. 🔒 Docker Registry Privado

### 3.1 ¿Qué es?

Docker Registry (Distribution Registry) es una aplicación **open-source** (Apache License) que te permite ejecutar tu propio registry privado. Es el backend que usa Docker Hub, pero sin la interfaz web ni las funciones avanzadas.

```
┌────────────────────────────────────────────┐
│         Docker Registry (v2)                │
│                                             │
│  • Escrito en Go                            │
│  • API HTTP/REST                            │
│  • Sin GUI web                              │
│  • Auth básica (todo o nada, no RBAC)       │
│  • Soporte TLS                              │
│  • Múltiples storage backends               │
│  • OCI Distribution Spec v1.0.1             │
└────────────────────────────────────────────┘
```

### 3.2 Storage Backends soportados

| Backend | Uso típico |
|---|---|
| **Filesystem** (default) | Desarrollo, equipos pequeños |
| **S3** | AWS, producción escalable |
| **Azure Blob** | Infraestructura Azure |
| **Google Cloud** | Infraestructura GCP |
| **Alibaba OSS** | Infraestructura Alibaba |
| **OpenStack Swift** | Infraestructura OpenStack |

### 3.3 Ejercicio práctico: Ejecutar un registry privado

```bash
# 1. Iniciar el registry en el puerto 5000
docker container run -d -p 5000:5000 --name registry registry

# 2. Descargar una imagen de prueba
docker pull hello-world

# 3. Re-etiquetar la imagen para tu registry
docker tag hello-world 127.0.0.1:5000/hello-world

# 4. Push al registry privado
docker push 127.0.0.1:5000/hello-world

# 5. Eliminar la copia local
docker image remove hello-world
docker image remove 127.0.0.1:5000/hello-world

# 6. Pull desde tu registry privado
docker pull 127.0.0.1:5000/hello-world

# 7. Con persistencia (bind mount)
docker container run -d -p 5000:5000 --name registry \
  -v $(pwd)/registry-data:/var/lib/registry registry
```

### 3.4 Registry con TLS (producción)

Docker **rechaza conexiones sin HTTPS** por defecto, excepto `localhost` (127.0.0.0/8):

```
┌─────────┐        HTTPS        ┌──────────┐
│  Docker  │ ◄─────────────────► │ Registry │
│  Client  │   TLS obligatorio   │  (TLS)   │
└─────────┘                      └──────────┘

EXCEPCIÓN: localhost (127.0.0.1) → sin TLS requerido
```

Para registries remotos con certificados auto-firmados:

```json
// /etc/docker/daemon.json
{
  "insecure-registries": ["myregistry.example.com:5000"]
}
```

> ⚠️ **Nunca uses `insecure-registries` en producción.** Configura TLS correctamente.

### 3.5 Registry en Docker Swarm

```
┌─────────────────────────────────────────────┐
│              Docker Swarm                    │
│                                              │
│  Node 1 ──┐                                 │
│  Node 2 ──┤── Routing Mesh ── 127.0.0.1:5000│
│  Node 3 ──┘     (todos ven localhost)        │
│                                              │
│  Registry como Service del Swarm             │
└─────────────────────────────────────────────┘
```

- Funciona igual que en localhost gracias al Routing Mesh
- Todos los nodos acceden vía `127.0.0.1:5000`
- Considera el **volume driver** para almacenamiento compartido
- **Todos los nodos** deben poder acceder a las imágenes

> **ProTip:** Si puedes, usa un registry SaaS hospedado en lugar de gestionar uno propio.

## 4. 🪞 Registry Mirror (Pull-Through Cache)

### 4.1 ¿Por qué un mirror?

Cuando tienes múltiples máquinas con Docker, cada una descarga imágenes independientemente de Docker Hub. Un mirror local cachea las imágenes y reduce el tráfico:

```
SIN MIRROR:                           CON MIRROR:

Server 1 ──► Docker Hub               Server 1 ──┐
Server 2 ──► Docker Hub               Server 2 ──┤──► Mirror ──► Docker Hub
Server 3 ──► Docker Hub               Server 3 ──┘    (cache)
  (3 descargas iguales)                  (1 descarga, 3 servidas local)
```

### 4.2 Cómo funciona

1. **Primera solicitud:** El mirror descarga la imagen de Docker Hub, la cachea y la entrega
2. **Solicitudes posteriores:** Servidas directamente desde el cache local
3. **Contenido actualizado:** Al hacer pull con tag, el mirror verifica contra el remoto
4. **Limpieza automática:** El Registry elimina periódicamente contenido obsoleto

### 4.3 Configuración del mirror

**Configurar el Registry como proxy:**

```yaml
# /etc/docker/registry/config.yml
proxy:
  remoteurl: https://registry-1.docker.io
  username: [username]    # opcional, para imágenes privadas
  password: [password]    # opcional
```

> ⚠️ **Seguridad:** Si configuras credenciales, todas las imágenes privadas del usuario quedan accesibles en el mirror. Implementa autenticación en el mirror.

**Configurar los Docker daemons:**

```json
// /etc/docker/daemon.json
{
  "registry-mirrors": ["https://my-mirror.example.com"]
}
```

### 4.4 Limitaciones

| Limitación | Detalle |
|---|---|
| Solo Docker Hub | No puedes hacer mirror de registries privados |
| Fair use policy | Sujeto a las políticas de Docker |
| Storage backend | Usa `filesystem` driver para mejor rendimiento |
| RAM restrictions | Si tienes RAM habilitado, imágenes de Hub no pasan por mirror |

## 5. 🗑️ Garbage Collection

### 5.1 El problema del espacio en disco

Los registries acumulan capas y manifiestos rápidamente. Cada `docker push` puede añadir capas que ya no se necesitan. Sin limpieza, el almacenamiento crece indefinidamente.

### 5.2 Cómo funciona

El garbage collector opera en **dos fases** (mark and sweep):

```
FASE 1 - MARK                        FASE 2 - SWEEP

Escanear todos los manifiestos        Escanear todos los blobs
         │                                     │
         ▼                                     ▼
Construir conjunto de blobs           ¿Blob está en el
referenciados (mark set)              mark set?
                                       │          │
                                      SÍ         NO
                                       │          │
                                    Mantener   ELIMINAR
```

**Ejemplo visual:**

```
ANTES del GC:                    DESPUÉS del GC:

Manifest A ──► layer a ◄── Manifest B    Manifest A ──► layer a
          └──► layer b         │                    └──► layer b
               layer c ◄──────┘

(Se elimina Manifest B vía API)          layer c → ELIMINADO
```

### 5.3 Ejecutar garbage collection

```bash
# Dry run (ver qué se eliminaría sin eliminar)
bin/registry garbage-collect --dry-run /path/to/config.yml

# Ejecución real
bin/registry garbage-collect /path/to/config.yml

# Eliminar manifiestos sin tag
bin/registry garbage-collect --delete-untagged /path/to/config.yml

# Modo silencioso
bin/registry garbage-collect --quiet /path/to/config.yml
```

> ⚠️ **Importante:** El registry debe estar en **modo read-only o detenido** durante el GC. Si subes una imagen mientras el GC ejecuta, sus capas podrían ser eliminadas accidentalmente (stop-the-world GC).

**Configuración necesaria:**

```yaml
version: 0.1
storage:
  filesystem:
    rootdirectory: /registry/data
  delete:
    enabled: true   # REQUERIDO para que el GC funcione
```

## 6. 🏢 Alternativas de Registry (Terceros)

| Registry | Tipo | Características destacadas |
|---|---|---|
| **Docker Hub** | SaaS público | Auto-builds, scanning, mayor catálogo |
| **Quay.io** (Red Hat) | SaaS público/privado | Security scanning, geo-replicación |
| **GitHub Packages** | SaaS integrado | Integración nativa con GitHub Actions |
| **AWS ECR** | Cloud managed | Integración IAM, scanning, replicación |
| **Azure ACR** | Cloud managed | Geo-replicación, tasks, Helm charts |
| **Google Artifact Registry** | Cloud managed | Multi-formato (Docker, npm, Maven) |
| **Harbor** | Open-source | RBAC, scanning, replicación, audit logs |
| **GitLab Container Registry** | Integrado | Parte del pipeline GitLab CI/CD |
| **JFrog Artifactory** | Enterprise | Multi-formato, alta disponibilidad |

### Criterios de selección

```
¿Necesitas control total?
  │
  ├── SÍ → Docker Registry (self-hosted) o Harbor
  │
  └── NO → ¿Ya usas un cloud provider?
              │
              ├── SÍ → ECR / ACR / Artifact Registry
              │
              └── NO → Docker Hub / GitHub Packages / Quay.io
```

## 7. ⚠️ Errores comunes

| # | Error | Causa | Solución |
|---|---|---|---|
| 1 | `http: server gave HTTP response to HTTPS client` | Registry sin TLS | Configurar TLS o añadir a `insecure-registries` |
| 2 | `denied: requested access to the resource is denied` | Sin autenticación | `docker login` al registry |
| 3 | Imágenes no disponibles en nodos Swarm | Nodos no acceden al registry | Verificar conectividad de red de todos los nodos |
| 4 | Disco lleno en registry | Sin garbage collection | Habilitar `delete: enabled` y ejecutar GC periódicamente |
| 5 | `manifest unknown` al pull | Imagen/tag no existe | Verificar nombre y tag exactos |
| 6 | Build automático no se dispara | Repository Links no configurados | Configurar links en Docker Hub settings |
| 7 | Imágenes desactualizadas en mirror | Cache no actualizado | El mirror verifica tags automáticamente; para `latest` es problemático |
| 8 | Registry mirror no funciona con RAM | Registry Access Management bloquea | Pull manual desde mirror y re-tag |

## 8. 📋 Cheat Sheet

```bash
# === Docker Hub ===
docker login                              # Autenticarse en Docker Hub
docker push user/image:tag                # Subir imagen
docker pull user/image:tag                # Descargar imagen

# === Registry Privado ===
docker run -d -p 5000:5000 --name registry registry  # Iniciar registry
docker tag image 127.0.0.1:5000/image     # Re-etiquetar para registry local
docker push 127.0.0.1:5000/image          # Push a registry local
docker pull 127.0.0.1:5000/image          # Pull desde registry local

# Con persistencia
docker run -d -p 5000:5000 --name registry \
  -v $(pwd)/data:/var/lib/registry registry

# === Inspección ===
curl http://127.0.0.1:5000/v2/_catalog    # Listar repositorios
curl http://127.0.0.1:5000/v2/image/tags/list  # Listar tags

# === Limpieza ===
docker container stop registry && docker container rm -v registry
```

## 9. 🔑 Resumen

| Concepto | Punto clave |
|---|---|
| **Docker Hub** | Registry público por defecto, auto-builds, webhooks, organizaciones |
| **Docker Registry** | Open-source, self-hosted, sin GUI, auth básica |
| **TLS** | Obligatorio excepto localhost; nunca `insecure-registries` en producción |
| **Mirror** | Pull-through cache para reducir tráfico a Docker Hub |
| **Garbage Collection** | Stop-the-world; mark & sweep en dos fases |
| **Swarm + Registry** | Routing Mesh permite `127.0.0.1:5000` desde todos los nodos |
| **Alternativas** | Harbor, ECR, ACR, Quay.io, GitHub Packages según tu infraestructura |
