## 2. рҹӨ” ВҝPor QuГ© Kubernetes?

### 2.1 ВҝNecesitas orquestaciГіn?

No toda aplicaciГіn necesita orquestaciГіn. Usa esta fГіrmula simple:

```
Beneficio de OrquestaciГіn = NГәmero de Servidores Г— Tasa de Cambio

  Pocos servidores + cambios infrecuentes  вҶ’  OrquestaciГіn innecesaria
  Muchos servidores + cambios frecuentes   вҶ’  OrquestaciГіn indispensable
```

**Alternativas a orquestaciГіn:**
- AWS Elastic Beanstalk, Heroku, Railway
- Auto Scaling Groups + Load Balancers
- Plataformas PaaS (Cloud Run, App Service)

### 2.2 ВҝQuГ© problema resuelve?

| Sin orquestaciГіn | Con Kubernetes |
|---|---|
| Deploy manual servidor por servidor | Deploy declarativo a todo el cluster |
| RecuperaciГіn manual ante fallos | Self-healing automГЎtico |
| Escalado manual | Auto-scaling por mГ©tricas |
| Updates con downtime | Rolling updates sin interrupciГіn |
| ConfiguraciГіn inconsistente | Estado deseado garantizado |

## 3. вҡ”пёҸ Kubernetes vs Swarm

### 3.1 ComparaciГіn directa

| Aspecto | Docker Swarm | Kubernetes |
|---|---|---|
| **Complejidad** | Simple de desplegar y gestionar | MГЎs complejo, mГЎs flexible |
| **Curva de aprendizaje** | Baja | Media-Alta |
| **Incluido en Docker** | вң… SГӯ, built-in | вқҢ No, instalaciГіn separada |
| **Regla** | 80/20 (20% features, 80% use cases) | Cubre edge cases |
| **Vendor support** | Docker Inc. | Amplio ecosistema de vendors |
| **Cloud managed** | Limitado | EKS, AKS, GKE, etc. |
| **Comunidad** | PequeГұa | Enorme |
| **Seguridad** | Seguro por defecto (mTLS, etc.) | Requiere configuraciГіn |
| **Troubleshooting** | MГЎs fГЎcil (menos partes) | MГЎs complejo |
| **Plataformas** | Docker Engine (Linux, Windows, ARM, IoT) | Principalmente Linux |

### 3.2 Ventajas especГӯficas de Swarm

- **Viene con Docker** вҖ” sin instalar nada extra
- **Secure by default** вҖ” mTLS, control plane encriptado
- **Mismo ecosistema** вҖ” `docker logs`, `docker events`, mismos comandos
- **Multi-arquitectura** вҖ” Linux, Windows, ARM, 32-bit, IoT
- **Recursos mГӯnimos** вҖ” overhead casi nulo sobre Docker

### 3.3 Ventajas especГӯficas de Kubernetes

- **Cloud managed** вҖ” los providers despliegan y gestionan K8s por ti
- **Distribuciones vendor** вҖ” Rancher, OpenShift, VMware Tanzu, etc.
- **MГЎxima flexibilidad** вҖ” mГЎs tipos de recursos, controladores custom
- **"Kubernetes first"** вҖ” vendors priorizan soporte para K8s
- **Ecosistema** вҖ” Helm, Istio, Argo, OPA, miles de herramientas
- **Carrera profesional** вҖ” skill muy demandado en el mercado

### 4.2 Componentes del Control Plane

| Componente | FunciГіn |
|---|---|
| **kube-apiserver** | Punto de entrada REST; toda comunicaciГіn pasa por aquГӯ |
| **etcd** | Base de datos clave-valor distribuida; almacena TODO el estado del cluster |
| **kube-scheduler** | Asigna Pods a Nodes segГәn recursos disponibles |
| **kube-controller-manager** | Ejecuta controladores (Deployment, ReplicaSet, Node, etc.) |
| **cloud-controller-manager** | Integra con APIs del cloud provider (opcional) |

### 4.3 Componentes de cada Node

| Componente | FunciГіn |
|---|---|
| **kubelet** | Agente que asegura que los Pods estГ©n corriendo |
| **kube-proxy** | Reglas de red para implementar Services (iptables/IPVS) |
| **Container runtime** | Ejecuta contenedores (containerd, CRI-O) |

## 5. рҹ”Ө TerminologГӯa Esencial

| TГ©rmino | DefiniciГіn |
|---|---|
| **Cluster** | Conjunto de nodos gestionados por Kubernetes |
| **Node** | Un servidor (fГӯsico o VM) en el cluster |
| **Pod** | Unidad mГӯnima de despliegue; 1+ contenedores co-localizados |
| **Deployment** | Gestiona ReplicaSets y rolling updates |
| **ReplicaSet** | Mantiene N rГ©plicas de un Pod |
| **Service** | Endpoint de red estable para acceder a Pods |
| **Namespace** | AgrupaciГіn lГіgica de recursos (aislamiento soft) |
| **ConfigMap** | ConfiguraciГіn como datos clave-valor |
| **Secret** | Datos sensibles (contraseГұas, tokens, certs) |
| **Label** | Metadata clave-valor para organizar y seleccionar recursos |
| **Selector** | Query contra labels para encontrar recursos |

### RelaciГіn entre recursos

```
Deployment
```

### 7.2 API de Kubernetes

La API es **declarativa**: describes el estado deseado, K8s se encarga de alcanzarlo.

```yaml
# Declaras: "quiero 3 rГ©plicas de nginx"
spec:
  replicas: 3

# Kubernetes:
# 1. Lee el estado deseado (3 rГ©plicas)
# 2. Compara con estado actual (0 rГ©plicas)
# 3. Crea 3 Pods
# 4. Monitorea continuamente
# 5. Si uno muere, crea otro (self-healing)
```

Cada objeto K8s tiene:
- **`spec`** вҖ” estado deseado (lo que TГҡ defines)
- **`status`** вҖ” estado actual (lo que K8s reporta)
- **`metadata`** вҖ” nombre, namespace, labels, uid
