# Docker Swarm — Orquestación Nativa

---

## 1. Introducción: ¿Por qué Orquestación?

### El problema de los contenedores en producción

Ejecutar un contenedor en tu máquina local es simple. Ejecutar **docenas o cientos de contenedores en múltiples servidores** genera problemas que `docker run` nunca fue diseñado para resolver:

| Problema | Pregunta |
|---|---|
| **Escalabilidad** | ¿Cómo ejecuto 10 réplicas de mi API en 5 servidores? |
| **Alta disponibilidad** | ¿Cómo reemplazo un contenedor que falló sin downtime? |
| **Updates sin downtime** | ¿Cómo hago un rolling update (blue/green deploy)? |
| **Descubrimiento** | ¿Cómo saben mis contenedores dónde están los demás? |
| **Networking** | ¿Cómo se comunican contenedores en diferentes hosts? |
| **Seguridad** | ¿Cómo gestiono secrets (passwords, API keys) de forma segura? |
| **Tracking** | ¿Cómo sé en qué servidor está corriendo cada contenedor? |

**Analogía:** `docker run` es conducir un auto. La orquestación es gestionar una flota de camiones de reparto — necesitas saber qué camión va a dónde, qué hacer si uno se avería, cómo redistribuir la carga y cómo agregar más camiones en temporada alta.

### ¿Qué es Docker Swarm?

Docker Swarm (Swarm Mode) es la **solución de orquestación nativa de Docker**, integrada directamente en el Docker Engine. Permite agrupar múltiples hosts Docker en un **cluster** que se gestiona como una sola unidad.

```
Sin Swarm:                          Con Swarm:
┌──────────┐ ┌──────────┐          ┌─────────────────────────────┐
│ Server 1 │ │ Server 2 │          │         SWARM CLUSTER       │
│ docker   │ │ docker   │          │  ┌─────┐ ┌─────┐ ┌─────┐  │
│ run...   │ │ run...   │    →     │  │ N1  │ │ N2  │ │ N3  │  │
│ run...   │ │ run...   │          │  └─────┘ └─────┘ └─────┘  │
└──────────┘ └──────────┘          │  Gestionados como UNO      │
 Manual en cada servidor            └─────────────────────────────┘
                                    Un solo punto de control
```

**Características principales:**

- Integrado en Docker (no requiere software adicional)
- No habilitado por defecto — se activa con `docker swarm init`
- Modelo declarativo: defines el estado deseado, Swarm lo mantiene
- Seguridad automática: TLS, certificados, encriptación
- Base de datos distribuida (Raft) incluida

### Swarm Mode vs Swarm Classic

| Aspecto | Swarm Classic (pre-1.12) | Swarm Mode (1.12+) |
|---|---|---|
| Arquitectura | Contenedor separado sobre Docker | Integrado en el Docker Engine |
| Configuración | Requería key-value store externo (Consul, etcd) | Raft DB integrada |
| Seguridad | Manual | PKI y TLS automáticos |
| Alcance | Replicar `docker run` a múltiples hosts | Orquestación completa con servicios |
| Estado | **Deprecado** | **Actual** |

## 2. Arquitectura del Swarm

### Componentes: Managers y Workers

Un Swarm se compone de **nodos** (hosts Docker) con dos roles:

```
┌─────────────────────────────── SWARM CLUSTER ────────────────────────────────┐
│                                                                              │
│  ┌─────────────────────── MANAGER NODES ───────────────────────┐             │
│  │                                                             │             │
│  │  ┌───────────┐    ┌───────────┐    ┌───────────┐           │             │
│  │  │ Manager 1 │◄──►│ Manager 2 │◄──►│ Manager 3 │           │             │
│  │  │ (Leader)  │    │(Follower) │    │(Follower) │           │             │
│  │  └─────┬─────┘    └─────┬─────┘    └─────┬─────┘           │             │
│  │        │                │                │                  │             │
│  │  ┌─────┴────────────────┴────────────────┴──────┐           │             │
│  │  │         Raft Consensus Database              │           │             │
│  │  │  (Configuración, estado, secrets, certs)     │           │             │
│  │  └──────────────────────────────────────────────┘           │             │
│  └─────────────────────────────┬───────────────────────────────┘             │
│                                │ Control Plane (encrypted)                   │
│  ┌─────────────────────────────┴───────────────────────────────┐             │
│  │                     WORKER NODES                            │             │
│  │                                                             │             │
│  │  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐   │             │
│  │  │ Worker 1 │  │ Worker 2 │  │ Worker 3 │  │ Worker 4 │   │             │
│  │  │ [C][C]   │  │ [C][C]   │  │ [C]      │  │ [C][C]   │   │             │
│  │  └──────────┘  └──────────┘  └──────────┘  └──────────┘   │             │
│  └─────────────────────────────────────────────────────────────┘             │
│                                                                              │
└──────────────────────────────────────────────────────────────────────────────┘
  C = Contenedor (task)
```

#### Manager Nodes

- **Mantienen el estado** del cluster en la Raft Database
- **Orquestan:** deciden dónde colocar tareas y cómo reconciliar el estado
- **Uno es Leader**, el resto son Followers que replican la base de datos
- Los managers **también pueden ejecutar tareas** (son workers + permisos de gestión)
- Número recomendado: **3 o 5** (siempre impar para quórum)

#### Worker Nodes

- **Ejecutan tareas** (contenedores) asignadas por los managers
- **Reportan estado** constantemente al manager
- No toman decisiones de orquestación
- Pueden ser promovidos a manager y viceversa

> **Regla de oro:** Un manager es un worker con permisos para controlar el swarm. La diferencia es funcional, no de hardware.

### Servicios, Tasks y Contenedores

Swarm introduce una jerarquía de abstracción sobre los contenedores:

```
Service (estado deseado)
  │
  ├── Task 1 ──► Container (nginx.1) en Node 1
  ├── Task 2 ──► Container (nginx.2) en Node 2
  └── Task 3 ──► Container (nginx.3) en Node 3
```

| Concepto | Descripción | Analogía |
|---|---|---|
| **Service** | Definición declarativa de qué ejecutar y cuántas réplicas | La orden de trabajo |
| **Task** | Unidad atómica de scheduling asignada a un nodo | La asignación a un trabajador |
| **Container** | El proceso real corriendo | El trabajo ejecutándose |

**`docker run` vs `docker service create`:**

| `docker run` | `docker service create` |
|---|---|
| Un contenedor, un host | Múltiples réplicas, múltiples hosts |
| Si falla, no se recupera | Si falla, Swarm lo recrea automáticamente |
| Acción directa | Declaración al orquestador |
| Sin rolling updates | Rolling updates nativos |
| Sin load balancing | Routing mesh integrado |

### Flujo interno: ¿Qué ocurre al crear un servicio?

```
$ docker service create --replicas 3 nginx

     ┌──────────── Manager Node ─────────────┐
     │                                       │
     │  1. API                               │
     │     Recibe el comando, crea el        │
     │     objeto "service"                  │
     │            │                          │
     │  2. Orchestrator                      │
     │     Detecta: "Necesito 3 tasks"       │
     │     Crea 3 objetos "task"             │
     │            │                          │
     │  3. Allocator                         │
     │     Asigna IPs a cada task            │
     │            │                          │
     │  4. Scheduler                         │
     │     Decide en qué nodo va cada task   │
     │     (disponibilidad, constraints)     │
     │            │                          │
     │  5. Dispatcher                        │
     │     Envía las tasks a los workers     │
     │                                       │
     └───────────┬───────────────────────────┘
                 │
     ┌───────────┴───────────────────────────┐
     │         Worker Nodes                  │
     │                                       │
     │  6. Worker                            │
     │     Recibe la task asignada           │
     │            │                          │
     │  7. Executor                          │
     │     Lanza el contenedor              │
     │     Reporta estado al manager         │
     └───────────────────────────────────────┘
```

> **Diferencia clave:** Con `docker run` hablas directamente al runtime — "crea este contenedor AHORA". Con `docker service create` hablas al **orquestador** — "quiero que existan 3 réplicas de nginx, encárgate de cómo y dónde".

## 3. Raft: Consenso Distribuido

### ¿Por qué necesitamos consenso?

En un cluster con múltiples managers, todos deben estar de acuerdo sobre:

- Qué servicios existen
- Cuántas réplicas hay
- En qué nodo está cada task
- Qué secrets están almacenados

**Raft** es el protocolo que garantiza que todos los managers tengan la misma información, incluso si algunos fallan.

### Conceptos clave de Raft

| Concepto | Descripción |
|---|---|
| **Leader** | El único manager que acepta escrituras y las replica a los demás |
| **Follower** | Replica las entradas del leader; puede ser elegido leader si el actual falla |
| **Candidate** | Follower que inicia una elección al no recibir heartbeats del leader |
| **Term** | Período durante el cual un leader está en poder |
| **Log** | Registro ordenado de todas las operaciones (append-only) |
| **Quórum** | Mayoría de nodos que deben confirmar una escritura para commitearla |

### Log Replication (Cómo se replican cambios)

```
1. Cliente envía cambio al Leader
2. Leader agrega la entrada a su log (uncommitted)
3. Leader replica la entrada a los Followers
4. Followers escriben la entrada en su log
5. Cuando la MAYORÍA confirma → Leader commitea
6. Leader notifica a Followers que commiteen
7. Cluster alcanza consenso

                    ┌──────────┐
  Cambio ──────►   │  Leader  │  ← Escribe en su log
                    └────┬─────┘
                         │ Replica
              ┌──────────┼──────────┐
              ▼          ▼          ▼
         ┌────────┐ ┌────────┐ ┌────────┐
         │Follower│ │Follower│ │Follower│
         │  ✓ OK  │ │  ✓ OK  │ │  ✗ OFF │
         └────────┘ └────────┘ └────────┘

  Mayoría confirmó (2 de 3) → COMMIT
```

### Leader Election (Elección de líder)

1. Cada follower tiene un **election timeout** aleatorio (150-300ms)
2. Si no recibe heartbeat del leader en ese tiempo, se convierte en **candidate**
3. El candidate vota por sí mismo y solicita votos a los demás nodos
4. Si obtiene la **mayoría**, se convierte en leader
5. El nuevo leader envía heartbeats periódicos para mantener su liderazgo

```
Estado normal:
  Leader ──heartbeat──► Follower 1
  Leader ──heartbeat──► Follower 2

Leader falla:
  Leader ✗ (muerto)
  Follower 1: timeout! → se convierte en Candidate
  Candidate solicita votos → Follower 2 vota ✓
  Candidate obtiene mayoría → nuevo Leader
```

> **¿Por qué el timeout es aleatorio?** Para evitar que dos followers se conviertan en candidates simultáneamente (split vote). Si ocurre, se inicia un nuevo term y se intenta de nuevo.

### Quórum y tolerancia a fallos

| Managers | Quórum | Tolerancia a fallos |
|---|---|---|
| 1 | 1 | 0 (sin HA) |
| 2 | 2 | 0 (⚠️ peor que 1 — ambos necesarios) |
| **3** | **2** | **1** |
| 4 | 3 | 1 (no mejora sobre 3) |
| **5** | **3** | **2** |
| 7 | 4 | 3 |

> **Buena práctica:** Usa **3 managers** para la mayoría de clusters. Solo usa 5 si necesitas tolerar 2 fallos simultáneos. **Nunca uses un número par** — no mejora la tolerancia y sí añade overhead.

## 4. Heart of SwarmKit: Topology Management

### Modelo de comunicación: Pull Model

SwarmKit adoptó el modelo **Pull** donde los workers se conectan a los managers (no al revés):

| Aspecto | Push Model | Pull Model (SwarmKit) |
|---|---|---|
| Dirección | Manager → Worker | Worker → Manager |
| Descubrimiento | Manager necesita descubrir workers | Worker conoce las IPs de managers |
| Seguridad | Workers necesitan puertos abiertos | Workers inician conexión outbound |
| Escalabilidad | Manager maneja todas las conexiones | Conexiones distribuidas entre managers |
| Simplicidad | Más complejo | Menos partes móviles |

### Rate Control: Heartbeats y Workloads

**Heartbeats (presencia):**

- El manager dicta la frecuencia de heartbeats a los workers
- Los managers añaden **jitter** (variación aleatoria) para evitar ráfagas simultáneas de pings
- La frecuencia se acuerda por consenso Raft entre managers

**Workloads (asignación de trabajo):**

- El worker abre un **stream gRPC** (HTTP/2) hacia el manager
- El manager envía trabajo en **batches** (cada 100 cambios o cada 100ms, lo que ocurra primero)
- Esto reduce drásticamente el tráfico vs enviar cada cambio individualmente

### Replicación y distribución de conexiones

```
Cluster: 10,000 workers, 5 managers

  Workers se distribuyen entre managers:

  ┌───────────────────────────────────────────────────┐
  │                                                   │
  │  Manager 1 (Leader) ◄── ~2,000 workers            │
  │       ▲                                           │
  │       │ (single socket per follower)              │
  │  Manager 2 (Follower) ◄── ~2,000 workers          │
  │       │                                           │
  │  Manager 3 (Follower) ◄── ~2,000 workers          │
  │       │                                           │
  │  Manager 4 (Follower) ◄── ~2,000 workers          │
  │       │                                           │
  │  Manager 5 (Follower) ◄── ~2,000 workers          │
  │                                                   │
  └───────────────────────────────────────────────────┘

  Cada Follower multiplexea sus ~2,000 workers al Leader
  usando UN SOLO socket → reduce carga de red del Leader
```

**Tolerancia a fallos de red:**

- Si un manager cae, sus workers se reconectan a otro manager automáticamente
- gRPC maneja: exponential backoff, jitter de reconexión, connection pooling
- Si el Leader cae: nuevo Leader elegido → followers redirigen tráfico → transparente para workers
- El manager envía a cada worker la lista de **todos los managers** → los workers siempre saben a quién reconectarse

### Presence: Estado de Workers

- El Leader registra el estado de cada worker (Up/Down) en Raft
- Los TTLs de heartbeats se mantienen **en memoria del Leader** (no en Raft, sería demasiado costoso)
- Si el Leader falla y se elige uno nuevo:
  - Workers reciben un **grace period** para reconectarse
  - Durante este período su estado es "Unknown"
  - Si reconectan → vuelven a "Up"
  - Si no reconectan → marcados como "Down" → sus tasks se reasignan

## 5. Inicialización y Comandos Básicos

### Crear un Swarm

```bash
# Verificar si Swarm está activo
$ docker info | grep Swarm
Swarm: inactive

# Inicializar Swarm (este nodo se convierte en Manager + Leader)
$ docker swarm init
Swarm initialized: current node (abc123) is now a manager.

To add a worker to this swarm, run the following command:
    docker swarm join --token SWMTKN-1-xxx... 192.168.1.100:2377

To add a manager to this swarm, run 'docker swarm join-token manager'
```

**¿Qué ocurre con `docker swarm init`?**

1. Crea una **Root CA** (Certificate Authority) para el swarm
2. Emite un **certificado** para este primer manager
3. Genera **join tokens** (uno para workers, otro para managers)
4. Crea la **Raft Database** encriptada en disco
5. Habilita la **Swarm API**
6. Configura **mutual TLS** para toda comunicación inter-nodo

> **Diferenciador clave:** Swarm no necesita un key-value store externo (Consul, etcd, ZooKeeper). La Raft Database está integrada y se gestiona automáticamente.

### Gestionar nodos

```bash
# Listar nodos del swarm
$ docker node ls
ID              HOSTNAME   STATUS   AVAILABILITY   MANAGER STATUS   ENGINE VERSION
abc123 *        node1      Ready    Active         Leader           24.0.7
def456          node2      Ready    Active         Reachable        24.0.7
ghi789          node3      Ready    Active                          24.0.7

# Agregar un worker
$ docker swarm join --token SWMTKN-1-worker-token... manager-ip:2377

# Agregar un manager
$ docker swarm join --token SWMTKN-1-manager-token... manager-ip:2377

# Obtener tokens
$ docker swarm join-token worker
$ docker swarm join-token manager

# Promover worker a manager
$ docker node promote node3

# Degradar manager a worker
$ docker node demote node2

# Sacar un nodo del swarm (ejecutar en el nodo)
$ docker swarm leave          # Worker
$ docker swarm leave --force  # Manager
```

### CLI del Swarm: Comandos principales

```bash
# ── Swarm ──
docker swarm init              # Inicializar swarm
docker swarm join              # Unir nodo al swarm
docker swarm join-token        # Gestionar tokens
docker swarm leave             # Salir del swarm
docker swarm unlock            # Desbloquear swarm encriptado
docker swarm unlock-key        # Gestionar claves de desbloqueo
docker swarm update            # Actualizar configuración del swarm

# ── Nodes ──
docker node ls                 # Listar nodos
docker node inspect            # Detalles de un nodo
docker node promote/demote     # Cambiar rol
docker node update             # Actualizar metadatos (labels)
docker node rm                 # Remover nodo

# ── Services ──
docker service create          # Crear servicio
docker service ls              # Listar servicios
docker service ps              # Listar tasks de un servicio
docker service inspect         # Detalles del servicio
docker service update          # Actualizar servicio
docker service scale           # Escalar réplicas
docker service rollback        # Revertir último update
docker service logs            # Ver logs
docker service rm              # Eliminar servicio

# ── Stacks ──
docker stack deploy            # Deploy desde compose file
docker stack ls                # Listar stacks
docker stack services          # Listar servicios de un stack
docker stack rm                # Remover stack

# ── Secrets & Configs ──
docker secret create/ls/rm     # Gestionar secrets
docker config create/ls/rm     # Gestionar configs
```

## 6. Servicios en Profundidad

### Crear y gestionar servicios

```bash
# Crear un servicio básico
$ docker service create --name web nginx

# Crear con réplicas y puertos publicados
$ docker service create \
    --name web \
    --replicas 3 \
    --publish 8080:80 \
    nginx:alpine

# Listar servicios
$ docker service ls
ID           NAME   MODE         REPLICAS   IMAGE          PORTS
abc123       web    replicated   3/3        nginx:alpine   *:8080->80/tcp

# Ver tasks del servicio (en qué nodos corren)
$ docker service ps web
ID        NAME    IMAGE          NODE    DESIRED STATE   CURRENT STATE
x1y2z3    web.1   nginx:alpine   node1   Running         Running 2 min ago
a4b5c6    web.2   nginx:alpine   node2   Running         Running 2 min ago
d7e8f9    web.3   nginx:alpine   node3   Running         Running 2 min ago
```

**Interpretación de REPLICAS `3/3`:**

```
  3/3 → 3 corriendo / 3 deseados
  │       │
  │       └── Cuántos definiste (--replicas)
  └────────── Cuántos están realmente corriendo

  El orquestador siempre intenta que ambos números coincidan.
```

### Escalar servicios

```bash
# Escalar a 5 réplicas
$ docker service scale web=5

# O con update
$ docker service update --replicas 5 web

# Escalar múltiples servicios a la vez
$ docker service scale web=5 api=3 worker=10
```

### Auto-healing: Recuperación automática

Si un contenedor falla o se elimina manualmente, Swarm lo recrea automáticamente:

```bash
# Estado inicial: 3 réplicas corriendo
$ docker service ls
NAME   REPLICAS
web    3/3

# Eliminamos un contenedor "por accidente"
$ docker container rm -f web.1.abc123

# Swarm detecta inmediatamente y reconcilia
$ docker service ls
NAME   REPLICAS
web    2/3        ← detecta fallo

# Segundos después...
$ docker service ls
NAME   REPLICAS
web    3/3        ← nueva réplica lanzada automáticamente

# Historial de tasks
$ docker service ps web
NAME       NODE    CURRENT STATE           ERROR
web.1      node2   Running 5 seconds ago            ← nueva
\_web.1    node1   Failed 10 seconds ago   removed   ← historial
web.2      node2   Running 5 minutes ago
web.3      node3   Running 5 minutes ago
```

> **Concepto clave:** No le dices a Swarm "crea un contenedor". Le dices "quiero 3 réplicas de nginx". Swarm lo interpreta como un **estado deseado** y trabaja continuamente para mantenerlo.

### Replicated vs Global Services

| Modo | Descripción | Caso de uso |
|---|---|---|
| **Replicated** (default) | N réplicas distribuidas en el cluster | APIs, web servers, workers |
| **Global** | Exactamente 1 task por nodo | Agentes de monitoreo, log collectors, antivirus |

```bash
# Replicated (default)
$ docker service create --name api --replicas 3 myapp:latest

# Global (un task por nodo)
$ docker service create --name agent --mode global datadog/agent
```

### Configuración del runtime

```bash
$ docker service create --name api \
    --env MYVAR=myvalue \             # Variables de entorno
    --workdir /app \                  # Working directory
    --user appuser \                  # Usuario
    --mount type=volume,src=data,dst=/data \  # Volúmenes
    --network my-overlay \            # Red overlay
    --secret db_password \            # Secrets
    --config nginx_conf \             # Configs
    --constraint 'node.role==worker' \  # Placement constraint
    --limit-cpu 0.5 \                 # Límite CPU
    --limit-memory 256M \             # Límite memoria
    --reserve-cpu 0.25 \              # CPU reservada
    --reserve-memory 128M \           # Memoria reservada
    myapp:latest
```

## 7. Networking: Routing Mesh y Overlay

### Routing Mesh

El routing mesh permite acceder a un servicio en **cualquier nodo del swarm**, aunque el contenedor solo corra en algunos nodos:

```
Cliente hace request a puerto 8080

  ┌──────────────────── SWARM ─────────────────────┐
  │                                                 │
  │   Node 1 (sin web)    Node 2 (web.1)           │
  │   :8080 ──────────►   :8080 ── container       │
  │                                                 │
  │   Node 3 (web.2)      Node 4 (sin web)         │
  │   :8080 ── container   :8080 ──────────►        │
  │                                                 │
  └─────────────────────────────────────────────────┘

  Puerto 8080 está disponible en TODOS los nodos.
  El routing mesh redirige al contenedor correcto automáticamente.
```

```bash
# Publicar vía routing mesh (default)
$ docker service create --name web \
    --replicas 3 \
    --publish published=8080,target=80 \
    nginx

# Publicar directamente en el nodo (bypass routing mesh)
$ docker service create --name web \
    --mode global \
    --publish mode=host,target=80,published=8080 \
    nginx
```

**Trade-offs del `mode=host`:**

| Routing Mesh (default) | mode=host |
|---|---|
| Accesible en cualquier nodo | Solo en nodos con el task |
| Load balancing automático | Tú gestionas el routing |
| 1 réplica por puerto por servicio | 1 réplica por nodo (limitación) |

### Overlay Networks

Las overlay networks permiten comunicación entre contenedores en **diferentes hosts**:

```bash
# Crear overlay network
$ docker network create --driver overlay my-overlay

# Asignar a un servicio
$ docker service create \
    --name api \
    --network my-overlay \
    myapp:latest

# Agregar red a servicio existente
$ docker service update --network-add my-overlay api

# Remover red de servicio
$ docker service update --network-rm my-overlay api
```

## 8. Updates y Rollbacks

### Rolling Updates

Swarm actualiza servicios **sin downtime** reemplazando tasks uno por uno:

```bash
$ docker service create --name web \
    --replicas 10 \
    --update-delay 10s \             # Esperar 10s entre cada update
    --update-parallelism 2 \         # Actualizar 2 tasks a la vez
    --update-failure-action pause \  # Pausar si falla (o "continue")
    --update-max-failure-ratio 0.1 \ # Tolerar 10% de fallos
    --update-monitor 30s \           # Monitorear 30s por task actualizada
    nginx:1.24

# Actualizar la imagen
$ docker service update --image nginx:1.25 web
```

```
Rolling update de 6 réplicas (parallelism=2, delay=10s):

  Tiempo   Task 1   Task 2   Task 3   Task 4   Task 5   Task 6
  ──────   ──────   ──────   ──────   ──────   ──────   ──────
  t=0s     v1→v2    v1→v2    v1       v1       v1       v1
  t=10s    v2 ✓     v2 ✓     v1→v2    v1→v2    v1       v1
  t=20s    v2 ✓     v2 ✓     v2 ✓     v2 ✓     v1→v2    v1→v2
  t=30s    v2 ✓     v2 ✓     v2 ✓     v2 ✓     v2 ✓     v2 ✓

  En todo momento hay tasks corriendo → zero downtime
```

### Rollbacks

```bash
# Rollback manual al estado anterior
$ docker service update --rollback web

# Rollback automático si el update falla
$ docker service create --name web \
    --rollback-delay 0s \
    --rollback-failure-action pause \
    --rollback-max-failure-ratio 0.2 \
    --rollback-monitor 20s \
    --rollback-parallelism 2 \
    nginx:latest
```

| Flag | Default | Descripción |
|---|---|---|
| `--rollback-delay` | `0s` | Tiempo entre rollback de cada task |
| `--rollback-failure-action` | `pause` | `pause` o `continue` si falla el rollback |
| `--rollback-max-failure-ratio` | `0` | Fracción de fallos tolerados (0.0 a 1.0) |
| `--rollback-monitor` | `5s` | Tiempo de monitoreo post-rollback por task |
| `--rollback-parallelism` | `1` | Tasks a rollback simultáneamente |

## 9. Placement: Control de ubicación

### Constraints (restricciones)

Fuerzan dónde puede o no puede correr un servicio:

```bash
# Solo en workers
$ docker service create --constraint 'node.role==worker' --name api myapp

# Solo en nodos con label específico
$ docker node update --label-add region=us-east node1
$ docker service create \
    --constraint 'node.labels.region==us-east' \
    --name api myapp

# Múltiples constraints (AND lógico)
$ docker service create \
    --constraint 'node.labels.region==us-east' \
    --constraint 'node.labels.type!=devel' \
    --name api myapp
```

### Preferences (preferencias)

Distribuyen tasks de forma equilibrada según un label (best-effort, no forzado):

```bash
# Distribuir equitativamente entre datacenters
$ docker service create \
    --replicas 9 \
    --placement-pref 'spread=node.labels.datacenter' \
    --name redis redis:7

# Si hay 3 nodos us-east y 3 us-west:
#   → ~4-5 tasks en us-east, ~4-5 en us-west
```

### Reserva de recursos

```bash
$ docker service create \
    --reserve-cpu 0.5 \      # Reservar 0.5 CPUs por task
    --reserve-memory 256M \  # Reservar 256MB por task
    --limit-cpu 1 \          # Límite máximo de CPU
    --limit-memory 512M \    # Límite máximo de memoria
    --name api myapp
```

> **Importante:** Si ningún nodo tiene los recursos reservados disponibles, el servicio queda en estado `Pending` hasta que un nodo esté disponible.

## 10. Volumes y Bind Mounts en Swarm

```bash
# Named volume (creado automáticamente si no existe)
$ docker service create \
    --mount type=volume,src=app-data,dst=/data \
    --name api myapp

# Bind mount (la ruta debe existir en TODOS los nodos donde corra el task)
$ docker service create \
    --mount type=bind,src=/host/path,dst=/container/path \
    --name api myapp

# Read-only bind mount
$ docker service create \
    --mount type=bind,src=/host/path,dst=/container/path,readonly \
    --name api myapp
```

> **⚠️ Cuidado con bind mounts en Swarm:** La ruta debe existir en cada nodo donde Swarm pueda programar un task. Si Swarm mueve el task a otro nodo, la ruta podría no existir. Prefiere named volumes o storage drivers distribuidos.

## 11. Secrets y Configs

### Secrets (datos sensibles)

```bash
# Crear secret desde archivo
$ echo "my-super-password" | docker secret create db_password -

# Crear secret desde archivo
$ docker secret create ssl_cert ./server.crt

# Usar en un servicio (montado en /run/secrets/<name>)
$ docker service create \
    --secret db_password \
    --name api myapp

# Dentro del contenedor:
$ cat /run/secrets/db_password
my-super-password
```

### Configs (archivos de configuración)

```bash
# Crear config
$ docker config create nginx_conf ./nginx.conf

# Usar en un servicio
$ docker service create \
    --config source=nginx_conf,target=/etc/nginx/nginx.conf \
    --name web nginx
```

**Secrets vs Configs:**

| Aspecto | Secrets | Configs |
|---|---|---|
| Encriptado en tránsito | ✓ | ✓ |
| Encriptado en disco (Raft) | ✓ | ✗ |
| Montado en tmpfs (memoria) | ✓ | ✗ |
| Caso de uso | Passwords, keys, tokens | nginx.conf, app config |

## 13. Buenas Prácticas

### Arquitectura del Cluster

- **3 managers** para la mayoría de casos (tolerancia a 1 fallo)
- **5 managers** solo si necesitas tolerar 2 fallos simultáneos
- **Nunca número par** de managers
- Managers en diferentes availability zones/racks
- Workers escalables según demanda

### Seguridad

- Rotar join tokens periódicamente: `docker swarm join-token --rotate worker`
- Usar autolock para encriptar la Raft DB: `docker swarm init --autolock`
- Secrets para toda credencial — nunca variables de entorno para passwords
- Overlay networks con encriptación: `--opt encrypted`

### Operaciones

- Pinear versiones de imágenes (no usar `latest`)
- Configurar rolling updates con `--update-delay` y `--update-failure-action`
- Siempre definir healthchecks para detección temprana de fallos
- Usar placement constraints para separar workloads (prod vs staging)
- Drain nodos antes de mantenimiento: `docker node update --availability drain node1`

### Errores Comunes

1. **Número par de managers** — No mejora tolerancia, sí añade overhead de consenso
2. **No usar healthchecks** — Swarm no puede detectar si la app está "sana" internamente
3. **Bind mounts en Swarm** — La ruta debe existir en todos los nodos; prefiere volumes
4. **Olvidar `--with-registry-auth`** — En registries privados, los workers no pueden pullear sin esto
5. **Escalar managers excesivamente** — Más de 7 managers degrada el rendimiento por overhead de Raft

## 14. Resumen

```
┌──────────────────────────────────────────────────────────────────┐
│                       DOCKER SWARM                               │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  Arquitectura                                                    │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  Managers: Orquestan, almacenan estado (Raft), API         │  │
│  │  Workers: Ejecutan tasks, reportan estado                  │  │
│  │  Raft DB: Consenso distribuido, encriptada                 │  │
│  └────────────────────────────────────────────────────────────┘  │
│                                                                  │
│  Modelo declarativo                                              │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  TÚ defines: "Quiero 5 réplicas de nginx en puerto 80"    │  │
│  │  SWARM hace: Crear, colocar, monitorear, recuperar         │  │
│  └────────────────────────────────────────────────────────────┘  │
│                                                                  │
│  Capacidades nativas                                             │
│  ┌────────────────────────────────────────────────────────────┐  │
│  │  ✓ Auto-healing        ✓ Rolling updates                  │  │
│  │  ✓ Load balancing      ✓ Rollbacks                        │  │
│  │  ✓ Overlay networking  ✓ Secrets management               │  │
│  │  ✓ Mutual TLS          ✓ Placement control                │  │
│  └────────────────────────────────────────────────────────────┘  │
│                                                                  │
│  docker run → contenedores individuales (dev)                    │
│  docker service → servicios orquestados (producción)             │
│  docker stack → multi-servicio desde compose file (producción)   │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

**Puntos clave:**

- Swarm transforma Docker de "runtime de contenedores" a **plataforma de orquestación**
- El modelo declarativo (estado deseado) reemplaza la gestión imperativa (`docker run`)
- Raft garantiza consistencia del cluster incluso ante fallos de managers
- Routing mesh + overlay networks = networking transparente entre nodos
- Auto-healing, rolling updates y rollbacks son nativos y automáticos
- `docker service` reemplaza a `docker run` en contexto de cluster
