## 1. Stacks: Compose en producción

### 1.1 ¿Qué es un Stack?

Un Stack es una capa de abstracción en Docker Swarm que permite definir y desplegar aplicaciones multi-servicio completas usando archivos Compose como declaración. Es, esencialmente, **Compose para producción**.

Piensa en un Stack como el "director de orquesta" que toma tu partitura (archivo Compose) y coordina todos los músicos (servicios, redes, volúmenes, secrets) para que toquen juntos.

```
SIN STACKS (manual, imperativo):
─────────────────────────────────
$ docker network create -d overlay myapp_net
$ docker service create --name web --network myapp_net ...
$ docker service create --name db --network myapp_net ...
$ docker secret create db_pass ...
→ Múltiples comandos, fácil olvidar algo, difícil reproducir

CON STACKS (declarativo):
─────────────────────────
$ docker stack deploy -c docker-compose.yml myapp
→ Un comando, un archivo, reproducible, versionable
```

### 1.2 Stack vs Docker Compose vs Service Create

| Aspecto | `docker compose up` | `docker service create` | `docker stack deploy` |
|---|---|---|---|
| **Entorno** | Desarrollo local | Swarm (un servicio) | Swarm (app completa) |
| **Orquestación** | No (single host) | Sí (un servicio) | Sí (multi-servicio) |
| **Archivo** | docker-compose.yml | N/A (CLI flags) | docker-compose.yml |
| **Redes** | Bridge (default) | Overlay manual | Overlay automática por stack |
| **Secrets** | File-based (inseguro) | Swarm secrets | Swarm secrets |
| **Build** | ✅ Soportado | ❌ No aplica | ❌ Ignorado |
| **deploy:** | ❌ Ignorado | N/A | ✅ Soportado |

> **Punto clave:** Compose ignora la clave `deploy:` y Swarm ignora la clave `build:`. Esto permite que el **mismo archivo** funcione en ambos contextos sin conflictos.

### 1.3 Anatomía de un Stack

```yaml
# docker-compose.yml para Stack
version: "3.9"

services:
  web:
    image: nginx:alpine          # ← Stacks requieren image (no build)
    ports:
      - "80:80"
    networks:
      - frontend
    deploy:                       # ← Solo Swarm lee esta sección
      replicas: 3
      update_config:
        parallelism: 1
        delay: 10s
      restart_policy:
        condition: on-failure

  api:
    image: myapp/api:1.2
    networks:
      - frontend
      - backend
    secrets:
      - db_password
    deploy:
      replicas: 2

  db:
    image: postgres:15
    volumes:
      - db_data:/var/lib/postgresql/data
    networks:
      - backend
    secrets:
      - db_password
      - db_root_password

networks:
  frontend:     # ← Overlay network automática
  backend:

volumes:
  db_data:

secrets:
  db_password:
    file: ./secrets/db_password.txt
  db_root_password:
    file: ./secrets/db_root_password.txt
```

### 1.4 Comandos de Stack

```bash
# Desplegar un stack (crea o actualiza)
$ docker stack deploy -c docker-compose.yml myapp

# Listar stacks activos
$ docker stack ls

# Ver servicios de un stack
$ docker stack services myapp

# Ver tareas (contenedores) de un stack
$ docker stack ps myapp

# Eliminar un stack (servicios, redes, secrets del stack)
$ docker stack rm myapp
```

### 1.5 Convención de nombres

Cuando despliegas un stack, Docker prefija todos los objetos con el nombre del stack:

```
Stack name: myapp
─────────────────────────────────────────────
Servicios:    myapp_web, myapp_api, myapp_db
Redes:        myapp_frontend, myapp_backend
Secrets:      myapp_db_password, myapp_db_root_password
Volúmenes:    myapp_db_data
```

> Esto permite correr múltiples stacks en el mismo Swarm sin colisiones de nombres.

### 1.6 Diagrama: Flujo de despliegue

```
                    docker-compose.yml
                          │
                          ▼
              ┌─── docker stack deploy ───┐
              │                           │
              ▼                           ▼
     ┌─── Servicios ──┐          ┌─── Redes ──┐
     │                │          │             │
     │  myapp_web (3) │          │  frontend   │
     │  myapp_api (2) │          │  backend    │
     │  myapp_db  (1) │          └─────────────┘
     └────────────────┘
              │                   ┌── Secrets ──┐
              │                   │  db_password │
              ▼                   │  db_root_pw  │
     ┌── Volúmenes ──┐           └──────────────┘
     │   db_data      │
     └────────────────┘

     Todo gestionado como una unidad.
     Un `docker stack rm myapp` limpia todo.
```

## 2. Docker Secrets: Fundamentos

### 2.1 ¿Qué es un Secret?

Un secret es cualquier dato que **no querrías ver en la portada de un periódico**. Si se filtrara y tuvieras que cambiarlo inmediatamente, es un secret.

| Tipo de dato | Ejemplo | ¿Es un secret? |
|---|---|---|
| Contraseñas de BD | `POSTGRES_PASSWORD=s3cret` | ✅ Sí |
| Certificados TLS | `site.crt`, `site.key` | ✅ Sí |
| Claves SSH | `id_rsa` | ✅ Sí |
| API keys | `STRIPE_SECRET_KEY=sk_live_...` | ✅ Sí |
| Tokens OAuth | `GITHUB_TOKEN=ghp_...` | ✅ Sí |
| Puerto de la app | `PORT=3000` | ❌ No (usar configs o env vars) |
| Nombre de la BD | `DB_NAME=myapp` | ⚠️ Depende del contexto |

**Características técnicas:**
- Soporta strings o binarios hasta **500 KB**
- No requiere reescribir la aplicación
- Solo disponible para **Swarm services** (no standalone containers)
- Encriptado en tránsito y en reposo

### 2.2 El problema que resuelven los Secrets

```
❌ ANTES DE SECRETS (inseguro):

  Opción 1: Variables de entorno
  $ docker service create -e DB_PASS=s3cret myapp
  → Visible en `docker inspect`, logs, /proc/environ

  Opción 2: Hardcodeado en la imagen
  ENV DB_PASS=s3cret    # en Dockerfile
  → Cualquiera con acceso a la imagen ve el password

  Opción 3: Archivo en el host
  $ docker service create -v /host/secrets:/app/secrets myapp
  → El archivo está en texto plano en el host
  → No funciona en Swarm multi-nodo

✅ CON DOCKER SECRETS:

  $ printf "s3cret" | docker secret create db_pass -
  $ docker service create --secret db_pass myapp
  → Encriptado en Raft DB (managers)
  → Transmitido por TLS (control plane)
  → Montado en RAM (nunca en disco dentro del container)
  → Solo accesible por servicios autorizados
  → Invisible en `docker inspect` del contenedor
```

### 2.3 Secrets vs Configs

| Aspecto | Secrets | Configs |
|---|---|---|
| **Uso** | Datos sensibles | Datos no sensibles |
| **Almacenamiento en container** | RAM disk (tmpfs) | Filesystem directo |
| **Encriptación en reposo** | ✅ Sí (Raft DB) | ✅ Sí (Raft DB) |
| **Encriptación en tránsito** | ✅ Sí (TLS) | ✅ Sí (TLS) |
| **Ubicación default** | `/run/secrets/<name>` | `/<name>` (raíz) |
| **Ejemplo** | Passwords, TLS keys | nginx.conf, app config |

## 3. Cómo Docker gestiona los Secrets

### 3.1 Arquitectura de seguridad

```
FLUJO DE UN SECRET EN SWARM

  1. CREACIÓN                    2. ALMACENAMIENTO
  ────────────                   ─────────────────
  $ docker secret create         Swarm Raft DB
  db_pass -                      (encriptado en disco)
       │                              │
       ▼                              │  Replicado a todos
  Manager Node ◄──────────────────────┘  los managers (HA)
       │
       │  3. ASIGNACIÓN
       │  ──────────────
       │  $ docker service create --secret db_pass
       │
       ▼
  4. DISTRIBUCIÓN
  ────────────────
  Manager → Worker (TLS mutuo, control plane encriptado)
       │
       │  Solo al nodo que ejecuta un task del servicio autorizado
       ▼
  5. MONTAJE EN CONTAINER
  ───────────────────────
  /run/secrets/db_pass      ← archivo en RAM (tmpfs)
  (solo lectura, solo dentro del container asignado)
       │
       │  6. CLEANUP
       │  ─────────
       ▼
  Cuando el task se detiene → desmontado de RAM
  Cuando el nodo pierde conexión → mantiene acceso
  pero no recibe actualizaciones
```

### 3.2 Puntos clave de seguridad

| Capa | Protección |
|---|---|
| **En reposo (disk)** | Raft DB encriptada, solo en manager nodes |
| **En tránsito (network)** | TLS mutuo + autenticación PKI |
| **En el container** | RAM filesystem (tmpfs), nunca en disco |
| **Acceso** | Solo servicios explícitamente autorizados |
| **Commit** | `docker commit` NO incluye secrets |
| **Inspect** | `docker inspect` del container NO muestra el contenido |

### 3.3 Cómo accede la aplicación al Secret

Desde la perspectiva de tu app, un secret es simplemente **un archivo** en `/run/secrets/`:

```
Container filesystem:
─────────────────────
/
├── app/
│   └── server.js
├── run/
│   └── secrets/
│       ├── db_password         ← contiene: "s3cret"
│       ├── db_host             ← contiene: "db.internal"
│       └── api_key             ← contiene: "sk_live_abc123"
└── ...
```

```javascript
// Node.js — Leer un secret
const fs = require('fs');
const dbPassword = fs.readFileSync('/run/secrets/db_password', 'utf8').trim();
```

```python
# Python — Leer un secret
with open('/run/secrets/db_password') as f:
    db_password = f.read().strip()
```

```bash
# Shell — Leer un secret
DB_PASS=$(cat /run/secrets/db_password)
```

### 3.4 Patrón `_FILE` en imágenes oficiales

Muchas imágenes oficiales (MySQL, Postgres, WordPress) soportan el patrón `_FILE`: en lugar de pasar el valor directamente como variable de entorno, le indicas **la ruta al archivo** que contiene el valor.

```bash
# ❌ Inseguro: valor directo en variable de entorno
-e MYSQL_ROOT_PASSWORD=s3cret

# ✅ Seguro: referencia al archivo del secret
-e MYSQL_ROOT_PASSWORD_FILE=/run/secrets/mysql_root_password
```

> **¿Por qué no usar variables de entorno directamente?** Porque las env vars pueden filtrarse entre containers (ej: `--link`), son visibles en `docker inspect`, y persisten en la historia del shell.

## 4. Secrets con servicios (CLI)

### 4.1 Crear un Secret

```bash
# Desde stdin (más seguro, no queda en archivos ni en history)
$ printf "mi_password_seguro" | docker secret create db_password -

# Desde un archivo
$ docker secret create site_cert ./certs/site.crt

# Generar password aleatorio y crear secret
$ openssl rand -base64 20 | docker secret create db_password -
```

> **⚠️ Importante:** Una vez creado, un secret **no se puede actualizar**. Solo puedes eliminarlo y recrearlo (y no puedes eliminar un secret que un servicio esté usando).

### 4.2 Gestionar Secrets

```bash
# Listar todos los secrets
$ docker secret ls
ID                          NAME            CREATED          UPDATED
wwwrxza8sxy025bas86593fqs   db_password     2 hours ago      2 hours ago
aya1dh363719pkiuoldpter4b   site_cert       1 hour ago       1 hour ago

# Inspeccionar metadata (NO muestra el contenido)
$ docker secret inspect db_password

# Eliminar un secret (falla si un servicio lo está usando)
$ docker secret rm db_password
```

### 4.3 Asignar Secrets a un servicio

```bash
# Al crear el servicio
$ docker service create \
    --name redis \
    --secret db_password \
    redis:alpine

# El container puede leer: /run/secrets/db_password

# Con target personalizado (ruta diferente dentro del container)
$ docker service create \
    --name nginx \
    --secret source=site_cert,target=/etc/nginx/ssl/cert.pem \
    --secret source=site_key,target=/etc/nginx/ssl/key.pem \
    nginx:latest

# Agregar un secret a un servicio existente
$ docker service update --secret-add new_secret myservice

# Revocar acceso a un secret
$ docker service update --secret-rm old_secret myservice
```

### 4.4 Verificar que los Secrets funcionan

```bash
# Ver el secret dentro del container
$ docker exec $(docker ps --filter name=redis -q) \
    ls -l /run/secrets/
total 4
-r--r--r--  1 root root  19 Apr 16 10:00 db_password

$ docker exec $(docker ps --filter name=redis -q) \
    cat /run/secrets/db_password
mi_password_seguro

# Verificar que un commit NO incluye secrets
$ docker commit $(docker ps --filter name=redis -q) test_image
$ docker run --rm test_image cat /run/secrets/db_password
cat: can't open '/run/secrets/db_password': No such file or directory
# ✅ El secret NO persiste en la imagen
```

### 4.5 Ejemplo: Secrets con Nginx TLS

Un caso de uso real donde almacenas certificados TLS como secrets:

```bash
# 1. Crear secrets para TLS
$ docker secret create site.key site.key
$ docker secret create site.crt site.crt
$ docker secret create site.conf site.conf

# 2. Crear servicio Nginx con target personalizado
$ docker service create \
    --name nginx \
    --secret site.key \
    --secret site.crt \
    --secret source=site.conf,target=/etc/nginx/conf.d/site.conf \
    --publish published=443,target=443 \
    nginx:latest

# El container tiene:
#   /run/secrets/site.key       ← TLS private key
#   /run/secrets/site.crt       ← TLS certificate
#   /etc/nginx/conf.d/site.conf ← Nginx config (custom target)
```

La configuración de Nginx referencia los secrets por su ruta:

```nginx
server {
    listen                443 ssl;
    server_name           localhost;
    ssl_certificate       /run/secrets/site.crt;
    ssl_certificate_key   /run/secrets/site.key;

    location / {
        root   /usr/share/nginx/html;
        index  index.html index.htm;
    }
}
```

## 5. Secrets con Stacks (Compose)

### 5.1 Definir Secrets en Compose

Hay tres formas de definir secrets en un archivo Compose:

#### Desde archivo (file-based)

```yaml
secrets:
  db_password:
    file: ./secrets/db_password.txt    # Ruta relativa al archivo
  db_root_password:
    file: ./secrets/db_root_password.txt
```

#### External (pre-creados en Swarm)

```yaml
secrets:
  db_password:
    external: true    # Ya existe en Swarm via `docker secret create`
```

#### Desde variable de entorno del host

```yaml
secrets:
  api_token:
    environment: "OAUTH_TOKEN"    # Lee del env var del host
```

### 5.2 Asignar Secrets a servicios

```yaml
services:
  db:
    image: mysql:latest
    secrets:
      - db_password               # ← Short form: /run/secrets/db_password
      - db_root_password
    environment:
      MYSQL_PASSWORD_FILE: /run/secrets/db_password
      MYSQL_ROOT_PASSWORD_FILE: /run/secrets/db_root_password

  web:
    image: wordpress:latest
    secrets:
      - source: db_password       # ← Long form: target personalizado
        target: wp_db_password     #   → /run/secrets/wp_db_password
        uid: '1000'
        gid: '1000'
        mode: 0400                 #   → Solo lectura para el user
```

| Forma | Sintaxis | Ubicación en container | Permisos |
|---|---|---|---|
| **Short** | `- db_password` | `/run/secrets/db_password` | root:root, 0444 |
| **Long** | `source:`, `target:`, `uid:`, `gid:`, `mode:` | Personalizable | Personalizable |

### 5.3 Ejemplo completo con Stack

```yaml
# docker-compose.yml
version: "3.9"

services:
  db:
    image: mysql:latest
    volumes:
      - db_data:/var/lib/mysql
    environment:
      MYSQL_ROOT_PASSWORD_FILE: /run/secrets/db_root_password
      MYSQL_DATABASE: wordpress
      MYSQL_USER: wordpress
      MYSQL_PASSWORD_FILE: /run/secrets/db_password
    secrets:
      - db_root_password
      - db_password
    networks:
      - backend

  wordpress:
    depends_on:
      - db
    image: wordpress:latest
    ports:
      - "8000:80"
    environment:
      WORDPRESS_DB_HOST: db:3306
      WORDPRESS_DB_USER: wordpress
      WORDPRESS_DB_PASSWORD_FILE: /run/secrets/db_password
    secrets:
      - db_password
    networks:
      - backend

secrets:
  db_password:
    file: ./db_password.txt
  db_root_password:
    file: ./db_root_password.txt

networks:
  backend:

volumes:
  db_data:
```

```bash
# Desplegar
$ docker stack deploy -c docker-compose.yml myapp

# Verificar
$ docker stack services myapp
$ docker secret ls
# Resultado: myapp_db_password, myapp_db_root_password

# Eliminar (limpia secrets del stack también)
$ docker stack rm myapp
```

> **Nota:** Al eliminar el stack con `docker stack rm`, los secrets definidos dentro del stack se eliminan automáticamente. Los secrets `external` NO se eliminan.

## 6. Ejemplo práctico: WordPress + MySQL

### 6.1 Setup paso a paso

```bash
# 1. Crear secrets desde la CLI
$ openssl rand -base64 20 | docker secret create mysql_password -
$ openssl rand -base64 20 | docker secret create mysql_root_password -

# 2. Crear red overlay privada
$ docker network create -d overlay mysql_private

# 3. Crear servicio MySQL
$ docker service create \
    --name mysql \
    --replicas 1 \
    --network mysql_private \
    --mount type=volume,source=mydata,destination=/var/lib/mysql \
    --secret source=mysql_root_password,target=mysql_root_password \
    --secret source=mysql_password,target=mysql_password \
    -e MYSQL_ROOT_PASSWORD_FILE="/run/secrets/mysql_root_password" \
    -e MYSQL_PASSWORD_FILE="/run/secrets/mysql_password" \
    -e MYSQL_USER="wordpress" \
    -e MYSQL_DATABASE="wordpress" \
    mysql:latest

# 4. Crear servicio WordPress
$ docker service create \
    --name wordpress \
    --replicas 1 \
    --network mysql_private \
    --publish published=30000,target=80 \
    --mount type=volume,source=wpdata,destination=/var/www/html \
    --secret source=mysql_password,target=wp_db_password \
    -e WORDPRESS_DB_USER="wordpress" \
    -e WORDPRESS_DB_PASSWORD_FILE="/run/secrets/wp_db_password" \
    -e WORDPRESS_DB_HOST="mysql:3306" \
    -e WORDPRESS_DB_NAME="wordpress" \
    wordpress:latest

# 5. Verificar
$ docker service ls
```

### 6.2 Diagrama de la arquitectura

```
                    Puerto 30000
                        │
                        ▼
              ┌─────────────────┐
              │    WordPress    │
              │  ┌───────────┐  │
              │  │wp_db_pass │  │  ← /run/secrets/wp_db_password
              │  └───────────┘  │     (alias de mysql_password)
              └────────┬────────┘
                       │
                mysql_private
                (overlay network)
                       │
              ┌────────┴────────┐
              │      MySQL      │
              │  ┌───────────┐  │
              │  │mysql_pass │  │  ← /run/secrets/mysql_password
              │  │mysql_root │  │  ← /run/secrets/mysql_root_password
              │  └───────────┘  │
              │  ┌───────────┐  │
              │  │  mydata   │  │  ← Volume persistente
              │  └───────────┘  │
              └─────────────────┘
```

### 6.3 Cleanup

```bash
$ docker service rm wordpress mysql
$ docker volume rm mydata wpdata
$ docker secret rm mysql_password mysql_root_password
$ docker network rm mysql_private
```

## 7. Rotación de Secrets

### 7.1 ¿Por qué rotar?

Los secrets no se pueden actualizar in-place. Para cambiar un password debes:
1. Crear un nuevo secret con el nuevo valor
2. Actualizar los servicios para usar el nuevo secret
3. Ejecutar los cambios necesarios en la aplicación (ej: cambiar password en MySQL)
4. Revocar y eliminar el secret antiguo

> **Tip:** Agrega un número de versión al nombre del secret (ej: `mysql_password_v2`) para facilitar la rotación.

### 7.2 Proceso de rotación paso a paso

```bash
# 1. Crear nuevo secret
$ openssl rand -base64 20 | docker secret create mysql_password_v2 -

# 2. Dar acceso a MySQL a ambos secrets (viejo y nuevo)
$ docker service update \
    --secret-rm mysql_password mysql

$ docker service update \
    --secret-add source=mysql_password,target=old_mysql_password \
    --secret-add source=mysql_password_v2,target=mysql_password \
    mysql

# 3. Cambiar el password en MySQL
$ docker exec $(docker ps --filter name=mysql -q) \
    bash -c 'mysqladmin --user=wordpress \
    --password="$(< /run/secrets/old_mysql_password)" \
    password "$(< /run/secrets/mysql_password)"'

# 4. Actualizar WordPress para usar el nuevo secret
$ docker service update \
    --secret-rm mysql_password \
    --secret-add source=mysql_password_v2,target=wp_db_password \
    wordpress

# 5. Revocar el secret viejo de MySQL y eliminarlo
$ docker service update --secret-rm old_mysql_password mysql
$ docker secret rm mysql_password
```

### 7.3 Diagrama de rotación

```
ESTADO INICIAL:
  mysql_password ──→ MySQL (/run/secrets/mysql_password)
                 ──→ WordPress (/run/secrets/wp_db_password)

PASO 2 (ambos secrets en MySQL):
  mysql_password    ──→ MySQL (/run/secrets/old_mysql_password)  ← viejo
  mysql_password_v2 ──→ MySQL (/run/secrets/mysql_password)      ← nuevo

PASO 3 (cambiar password en BD):
  MySQL ahora usa el nuevo password internamente

PASO 4 (WordPress usa el nuevo):
  mysql_password_v2 ──→ WordPress (/run/secrets/wp_db_password) ← nuevo

PASO 5 (cleanup):
  mysql_password eliminado de Swarm
  mysql_password_v2 ──→ MySQL + WordPress                       ← único
```

## 8. Buenas prácticas

### 8.1 Seguridad

| Práctica | Por qué | Cómo |
|---|---|---|
| **Nunca** guardar secrets en el Dockerfile | Se incluyen en la imagen, visible para todos | Usar `docker secret create` en runtime |
| **Nunca** pasar secrets como env vars directas | Visibles en `inspect`, logs, `/proc` | Usar patrón `_FILE` o leer desde `/run/secrets/` |
| **Nunca** dejar archivos de secrets en el host | Anula el propósito del secret | Eliminar archivos después de crear el secret |
| Limpiar bash history | Los comandos con secrets quedan en el historial | Usar `printf` + pipe con `-`, o `history -c` |
| Usar `external: true` en prod | Los secrets ya deben existir en Swarm | No depender de archivos en el servidor |
| Rotar secrets periódicamente | Limitar ventana de exposición | Versionado: `_v1`, `_v2`, etc. |
| Usar long form con `mode: 0400` | Restringir permisos de lectura | Solo el user de la app puede leer |

### 8.2 Organización

```bash
# Convención de nombres recomendada
<stack>_<servicio>_<tipo>_<versión>

# Ejemplos:
myapp_db_password_v1
myapp_api_jwt_secret_v2
myapp_nginx_tls_cert_v1
```

### 8.3 Desarrollo local vs Producción

```
DESARROLLO LOCAL (docker compose up):
──────────────────────────────────────
secrets:
  db_password:
    file: ./dev_secrets/db_password.txt     ← Archivo local (no seguro)
                                               Pero funcional para dev

PRODUCCIÓN (docker stack deploy):
─────────────────────────────────
secrets:
  db_password:
    external: true                          ← Pre-creado en Swarm
                                               Encriptado en Raft DB
```

> **Recuerda:** `docker compose` en local "finge" la seguridad montando el archivo como bind mount en texto plano. Es funcional para desarrollo pero **nunca** seguro para producción.

### 8.4 Soporte de Secrets en tus imágenes

Para que tu propia imagen soporte secrets, implementa el patrón `_FILE`:

```bash
#!/bin/bash
# entrypoint.sh — Leer secrets desde archivos

# Si existe la variable _FILE, leer el contenido del archivo
if [ -n "$DB_PASSWORD_FILE" ]; then
    export DB_PASSWORD=$(cat "$DB_PASSWORD_FILE")
fi

if [ -n "$API_KEY_FILE" ]; then
    export API_KEY=$(cat "$API_KEY_FILE")
fi

# Ejecutar el comando principal
exec "$@"
```

## 9. Errores comunes

| # | Error | Consecuencia | Solución |
|---|---|---|---|
| 1 | Pasar secrets como `-e DB_PASS=valor` | Visible en inspect, logs, process table | Usar `--secret` + patrón `_FILE` |
| 2 | Hardcodear secrets en Dockerfile | Cualquiera con la imagen ve el secret | Montar en runtime con secrets |
| 3 | Dejar archivos `.txt` de secrets en el servidor | Accesibles para cualquiera con acceso al host | Eliminar después de `docker secret create` |
| 4 | Intentar actualizar un secret existente | Docker no lo permite | Crear `_v2`, actualizar servicio, eliminar `_v1` |
| 5 | Eliminar un secret usado por un servicio | Docker rechaza la operación | Revocar acceso primero con `--secret-rm` |
| 6 | Usar secrets con standalone containers | No funciona (solo Swarm services) | Adaptar el container a servicio con `replicas: 1` |
| 7 | Olvidar `external: true` en producción | Stack intenta crear el secret desde archivo (que no existe en el server) | Definir como external |
| 8 | No limpiar el bash history | El password queda en `~/.bash_history` | Usar `printf` + pipe, o `history -c` |
| 9 | Montar secrets con permisos abiertos | Otros procesos pueden leer el secret | Usar long form con `mode: 0400` |
| 10 | No versionar secrets | Imposible rotar sin downtime | Naming: `secret_name_v1`, `_v2` |

### Nota: Windows Support

Docker soporta secrets en Windows containers con diferencias importantes:

| Aspecto | Linux | Windows |
|---|---|---|
| **Almacenamiento en container** | RAM disk (tmpfs) | Disco del container (texto plano) |
| **Ubicación default** | `/run/secrets/<name>` | `C:\ProgramData\Docker\secrets` |
| **Bind mount** | Directo al archivo | Symlinks desde `internal\secrets` |
| **UID/GID/mode** | ✅ Soportado | ❌ No soportado |
| **Limpieza** | Automática al detener task | Automática al detener task |
| **Recomendación** | — | Habilitar BitLocker en el volumen Docker |

> Aunque Windows persiste secrets en disco dentro del container, se eliminan explícitamente cuando el container se detiene, y `docker commit` no los captura.
