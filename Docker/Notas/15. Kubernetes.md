# ☸️ Kubernetes — Pods, Deployments y Escalado

## 1. 🎯 Introducción

Este módulo cubre los recursos fundamentales de Kubernetes: **Pods**, **Deployments** y **ReplicaSets**. Entender la relación entre estos tres es esencial para cualquier trabajo con K8s.

```
                     Deployment
                         │
                    crea/gestiona
                         │
                    ReplicaSet
                         │
                   crea/gestiona
                         │
                  ┌──────┼──────┐
                  │      │      │
                 Pod    Pod    Pod
                  │      │      │
              Container Container Container
```

---

## 2. 📦 Pods — La Unidad Mínima

### 2.1 ¿Qué es un Pod?

Un Pod es la **unidad mínima de despliegue** en Kubernetes. No puedes crear un container directamente — siempre creas un Pod que contiene uno o más containers.

```
┌─── Pod ──────────────────────────┐
│                                   │
│  ┌───────────┐  ┌───────────┐   │
│  │Container 1│  │Container 2│   │  ← Comparten IP,
│  │  (app)    │  │ (sidecar) │   │    volúmenes y
│  └───────────┘  └───────────┘   │    localhost
│                                   │
│  IP: 10.244.0.5                  │
│  Volumes: shared-data            │
└──────────────────────────────────┘
```

### ¿Por qué Pods y no containers?

| Aspecto | Docker Container | Kubernetes Pod |
|---|---|---|
| **Creación** | `docker run` crea container directo | `kubectl run` crea Pod → kubelet → container |
| **IP** | Una por container | Una por Pod (containers comparten) |
| **Agrupación** | No nativa | Containers co-localizados por diseño |
| **Abstracción** | Runtime-specific | Runtime-agnostic (containerd, CRI-O, etc.) |

### 2.2 Flujo de creación de un Pod

```
kubectl run my-nginx --image nginx
         │
         ▼
    API Server (recibe request)
         │
         ▼
    etcd (almacena registro del Pod)
         │
         ▼
    Scheduler (asigna Pod a un Node)
         │
         ▼
    kubelet en el Node asignado
         │
         ▼
    Container Runtime (crea container nginx)
         │
         ▼
    Pod Running ✓
```

### 2.3 Comandos básicos de Pods

```bash
# Crear un Pod
kubectl run my-nginx --image nginx

# Listar Pods
kubectl get pods

# Ver todos los recursos comunes
kubectl get all

# Ver detalles de un Pod
kubectl describe pod my-nginx

# Eliminar un Pod
kubectl delete pod my-nginx
```

> **Importante:** `kubectl get all` no muestra TODOS los recursos (~80 tipos). Muestra solo los comunes: Pods, Deployments, ReplicaSets, Services.

## 3. 🚀 Deployments — El Recurso Principal

### 3.1 ¿Qué es un Deployment?

Un **Deployment** es el recurso más utilizado en Kubernetes. Es el equivalente aproximado de un `docker service` en Swarm. Gestiona:

- Creación de ReplicaSets (que a su vez crean Pods)
- Rolling updates (nuevas versiones sin downtime)
- Rollbacks (volver a versión anterior)
- Escalado (más o menos réplicas)

### 3.2 Deployment vs Pod directo

| Pod directo (`kubectl run`) | Deployment (`kubectl create deployment`) |
|---|---|
| Un solo Pod | Gestiona N Pods vía ReplicaSet |
| Sin self-healing | Si un Pod muere, se recrea automáticamente |
| Sin rolling updates | Actualización sin downtime |
| Sin rollback | Historial de revisiones |
| Útil para testing/debug | **Lo que usarás en producción** |

### 3.3 Crear un Deployment

```bash
# Crear un Deployment de nginx
kubectl create deployment my-nginx --image nginx

# Ver qué se creó
kubectl get all
```

**Output:**

```
NAME                            READY   STATUS    AGE
pod/my-nginx-5b56ccd65f-abc12   1/1     Running   10s

NAME                       READY   UP-TO-DATE   AVAILABLE   AGE
deployment.apps/my-nginx   1/1     1            1           10s

NAME                                  DESIRED   CURRENT   READY   AGE
replicaset.apps/my-nginx-5b56ccd65f   1         1         1       10s
```

### 3.4 La cadena de recursos

```
Deployment: my-nginx
     │
     └── ReplicaSet: my-nginx-5b56ccd65f     (hash del pod spec)
              │
              └── Pod: my-nginx-5b56ccd65f-abc12  (hash + ID único)
```

**Nomenclatura de nombres:**
```
my-nginx          ← nombre del Deployment
    │
my-nginx-5b56ccd65f    ← Deployment + hash del ReplicaSet
    │
my-nginx-5b56ccd65f-abc12  ← ReplicaSet + ID del Pod
```

### 3.5 Flujo interno al crear un Deployment

```
kubectl create deployment my-nginx --image nginx
         │
         ▼
    API Server → etcd
         │
         ▼
    Controller Manager detecta nuevo Deployment
         │
         ▼
    Deployment Controller crea ReplicaSet
         │
         ▼
    ReplicaSet Controller crea Pod(s)
         │
         ▼
    Scheduler asigna Pod a Node
         │
         ▼
    kubelet → Container Runtime → Container
```

> **¿Por qué existe ReplicaSet como capa intermedia?** Porque durante un rolling update, coexisten el ReplicaSet viejo (con pods de la versión anterior) y el nuevo (con pods de la nueva versión). Esto permite actualizaciones sin downtime.

## 4. 📈 Escalado de ReplicaSets

### 4.1 Escalar un Deployment

```bash
# Crear un Deployment
kubectl create deployment my-apache --image httpd

# Verificar recursos
kubectl get all

# Escalar a 2 réplicas
kubectl scale deploy/my-apache --replicas 2

# Equivalente (con espacio en lugar de /)
kubectl scale deploy my-apache --replicas 2

# Verificar
kubectl get all
```

### 4.2 Abreviaciones de kubectl

```
deploy  = deployment  = deployments     (todos funcionan)
svc     = service     = services
po      = pod         = pods
rs      = replicaset  = replicasets
ns      = namespace   = namespaces

# Separador: / o espacio
kubectl scale deploy/my-apache --replicas 2
kubectl scale deploy my-apache --replicas 2
```

### 4.3 ¿Qué sucede al escalar?

```
kubectl scale deploy/my-apache --replicas 2
         │
         ▼
    API Server actualiza Deployment spec (replicas: 2)
         │
         ▼
    Controller Manager ve que replica count cambió
    (NO se crea nuevo ReplicaSet — solo cambió el conteo)
         │
         ▼
    ReplicaSet se actualiza: desired = 2
         │
         ▼
    Scheduler asigna el nuevo Pod a un Node
         │
         ▼
    kubelet crea el container
         │
         ▼
    2 Pods running ✓
```

**Puntos clave sobre escalado:**

| Concepto | Detalle |
|---|---|
| **`--replicas` es declarativo** | Defines el estado final, no "añade N más" |
| **Solo cambia replica count** | No crea nuevo ReplicaSet (eso ocurre al cambiar imagen/spec) |
| **Mismo Pod spec** | Las nuevas réplicas son idénticas a las existentes |
| **Scheduling** | Cada Pod nuevo pasa por el Scheduler para asignación de Node |

## 5. 🔄 Rolling Updates y Rollbacks

### 5.1 Concepto de Rolling Update

Cuando cambias la imagen o el pod spec de un Deployment, K8s crea un **nuevo ReplicaSet** y gradualmente migra Pods:

```
ANTES del update:

ReplicaSet-v1 (3 réplicas)
  Pod-1 [nginx:1.14] ✓
  Pod-2 [nginx:1.14] ✓
  Pod-3 [nginx:1.14] ✓

DURANTE el update:

ReplicaSet-v1 (2 réplicas, reduciendo)
  Pod-1 [nginx:1.14] ✓
  Pod-2 [nginx:1.14] ✓

ReplicaSet-v2 (1 réplica, creciendo)
  Pod-4 [nginx:1.16] ✓

DESPUÉS del update:

ReplicaSet-v1 (0 réplicas)
  (vacío)

ReplicaSet-v2 (3 réplicas)
  Pod-4 [nginx:1.16] ✓
  Pod-5 [nginx:1.16] ✓
  Pod-6 [nginx:1.16] ✓
```

### 5.2 Comandos de Rolling Update

```bash
# Actualizar imagen
kubectl set image deployment/my-nginx nginx=nginx:1.16.1

# Ver estado del rollout
kubectl rollout status deployment/my-nginx

# Ver historial de revisiones
kubectl rollout history deployment/my-nginx

# Ver detalle de una revisión
kubectl rollout history deployment/my-nginx --revision=2

# Rollback a la versión anterior
kubectl rollout undo deployment/my-nginx

# Rollback a una revisión específica
kubectl rollout undo deployment/my-nginx --to-revision=2
```

### 5.3 Parámetros del Rolling Update

```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxUnavailable: 25%   # Máx pods no disponibles durante update
      maxSurge: 25%          # Máx pods adicionales durante update
```

| Parámetro | Default | Efecto |
|---|---|---|
| **maxUnavailable** | 25% | Cuántos pods pueden estar caídos simultáneamente |
| **maxSurge** | 25% | Cuántos pods extra se crean durante la transición |

**Con 4 réplicas y defaults (25%):**
- Mínimo pods disponibles: 3 (4 - 25% = 3)
- Máximo pods totales: 5 (4 + 25% = 5)

### 5.4 Pausar y Resumir Rollouts

```bash
# Pausar (permite acumular múltiples cambios)
kubectl rollout pause deployment/my-nginx

# Hacer cambios sin disparar rollout
kubectl set image deployment/my-nginx nginx=nginx:1.16.1
kubectl set resources deployment/my-nginx -c=nginx --limits=cpu=200m

# Resumir (aplica todos los cambios juntos)
kubectl rollout resume deployment/my-nginx
```

### 5.5 Estrategias de Deployment

| Estrategia | Comportamiento | Uso |
|---|---|---|
| **RollingUpdate** (default) | Gradual, sin downtime | La mayoría de casos |
| **Recreate** | Mata todo, crea todo nuevo | Cuando no puede haber 2 versiones simultáneas |

### 5.6 Historial y Clean Up

```bash
# El historial se almacena como ReplicaSets antiguos
# Default: se guardan los últimos 10 (revisionHistoryLimit)

# Ver ReplicaSets (incluye antiguos con 0 réplicas)
kubectl get rs
```

```yaml
spec:
  revisionHistoryLimit: 10   # Default. 0 = sin historial (no puedes rollback)
```

## 6. 📊 Deployment Status y Condiciones

### 6.1 Estados del Deployment

```
Progressing ──► Complete
     │
     └──► Failed (si excede progressDeadlineSeconds)
```

### 6.2 Causas comunes de fallo

| Causa | Diagnóstico |
|---|---|
| Quota insuficiente | `kubectl describe deployment` → ReplicaFailure |
| Imagen no encontrada | Pod en estado `ImagePullBackOff` |
| Readiness probe falla | Pod nunca pasa a Ready |
| Permisos insuficientes | Verificar RBAC |
| Límites de recursos | `kubectl describe pod` → OOMKilled |

```bash
# Verificar estado
kubectl rollout status deployment/my-nginx

# Timeout configurable
kubectl patch deployment/my-nginx \
  -p '{"spec":{"progressDeadlineSeconds":600}}'
```

## 7. 🔁 ReplicaSets

### 7.1 Propósito

Un ReplicaSet **mantiene N réplicas de un Pod corriendo en todo momento**. Si un Pod muere, el ReplicaSet crea otro.

> **Regla:** Nunca crees ReplicaSets directamente. Usa Deployments, que los gestionan por ti.

### 7.2 ReplicaSet manifest

```yaml
apiVersion: apps/v1
kind: ReplicaSet
metadata:
  name: frontend
  labels:
    app: guestbook
spec:
  replicas: 3
  selector:
    matchLabels:
      tier: frontend
  template:
    metadata:
      labels:
        tier: frontend
    spec:
      containers:
      - name: php-redis
        image: us-docker.pkg.dev/google-samples/containers/gke/gb-frontend:v5
```

### 7.3 Scaling y Pod Deletion Cost

Al reducir réplicas, K8s prioriza eliminar en este orden:

1. Pods pendientes (unschedulable)
2. Pods con menor `controller.kubernetes.io/pod-deletion-cost`
3. Pods en nodos con más réplicas
4. Pods más recientes
5. Aleatorio si todo es igual

## 8. ✅ Buenas Prácticas

| Práctica | Razón |
|---|---|
| **Usa Deployments, no Pods directos** | Self-healing, rolling updates, rollbacks |
| **Nunca gestiones ReplicaSets directamente** | El Deployment los controla |
| **Versionado de imágenes explícito** | `nginx:1.25.3`, nunca `nginx:latest` |
| **Configura `revisionHistoryLimit`** | Equilibra historial vs recursos |
| **Usa `progressDeadlineSeconds`** | Detecta rollouts atascados |
| **Labels consistentes** | Facilita selección y organización |
| **No solapar selectors** | Dos Deployments con mismo selector = conflicto |

---

## 9. ⚠️ Errores Comunes

| # | Error | Consecuencia | Solución |
|---|---|---|---|
| 1 | Crear Pods directamente | Sin self-healing ni rolling updates | Usa Deployments |
| 2 | Usar `latest` como tag | Updates impredecibles, sin rollback real | Versionado explícito |
| 3 | Olvidar que `--replicas` es declarativo | Esperar que "añada" en vez de "establezca" | `--replicas=N` es el estado final |
| 4 | Solapar selectors entre Deployments | Pods adoptados incorrectamente | Labels únicos por Deployment |
| 5 | No verificar `kubectl rollout status` | No detectar rollouts fallidos | Siempre verificar después de update |
| 6 | Imagen incorrecta en update | Pod en `ImagePullBackOff` eternamente | K8s detiene el rollout automáticamente; `kubectl rollout undo` |
| 7 | `revisionHistoryLimit: 0` | Imposible hacer rollback | Mínimo 2-3 para tener opciones |

## 10. 📋 Cheat Sheet

```bash
# === Pods ===
kubectl run my-pod --image nginx           # Crear Pod
kubectl get pods                           # Listar Pods
kubectl get pods -o wide                   # Con info de Node
kubectl describe pod my-pod                # Detalles + eventos
kubectl delete pod my-pod                  # Eliminar

# === Deployments ===
kubectl create deployment my-app --image nginx  # Crear
kubectl get deploy                              # Listar
kubectl describe deploy my-app                  # Detalles
kubectl delete deployment my-app                # Eliminar

# === Scaling ===
kubectl scale deploy my-app --replicas 3        # Escalar

# === Updates ===
kubectl set image deploy/my-app nginx=nginx:1.16.1  # Update imagen
kubectl rollout status deploy/my-app                  # Ver estado
kubectl rollout history deploy/my-app                 # Historial
kubectl rollout undo deploy/my-app                    # Rollback
kubectl rollout undo deploy/my-app --to-revision=2    # A revisión X
kubectl rollout pause deploy/my-app                   # Pausar
kubectl rollout resume deploy/my-app                  # Resumir

# === Otros ===
kubectl get all                             # Recursos comunes
kubectl get rs                              # ReplicaSets
kubectl edit deploy/my-app                  # Editar en editor
```

## 11. 🔑 Resumen

| Concepto | Punto clave |
|---|---|
| **Pod** | Unidad mínima; wrapper de containers; no se crea directo en producción |
| **Deployment** | Recurso principal; gestiona ReplicaSets, updates y rollbacks |
| **ReplicaSet** | Mantiene N réplicas; gestionado por Deployment, nunca manualmente |
| **Escalado** | `--replicas` es declarativo; no crea nuevo RS si solo cambia conteo |
| **Rolling Update** | Nuevo RS crece, viejo RS reduce; maxUnavailable/maxSurge controlan velocidad |
| **Rollback** | `kubectl rollout undo`; historial en RS antiguos |
| **Nombres** | `deployment-name` → `deployment-RS_hash` → `deployment-RS_hash-Pod_ID` |
