## 1. Introducción

Los **labels** son el mecanismo fundamental de organización en Kubernetes. No son simples etiquetas decorativas — son la base sobre la que operan Services, Deployments, scheduling y prácticamente todo recurso que necesite "encontrar" a otros recursos.

```
  Labels: el "pegamento" de Kubernetes
  ─────────────────────────────────────

  Deployment                    Service
  selector:                     selector:
    app: nginx  ──────────────►   app: nginx
    tier: frontend                tier: frontend
         │                             │
         └──────────┬──────────────────┘
                    │
                    ▼
              ┌───────────┐  ┌───────────┐  ┌───────────┐
              │  Pod      │  │  Pod      │  │  Pod      │
              │  app:nginx│  │  app:nginx│  │  app:nginx│
              │  tier:fe  │  │  tier:fe  │  │  tier:fe  │
              └───────────┘  └───────────┘  └───────────┘
```

## 2. Labels: fundamentos

### ¿Qué son?

Labels son pares **key/value** adjuntos a objetos de Kubernetes. Están diseñados para contener **información identificatoria** que permite seleccionar, agrupar y filtrar recursos.

```yaml
metadata:
  labels:
    app: nginx
    tier: frontend
    env: production
    version: v2.1.0
```

### Sintaxis y restricciones

#### Keys

Un label key tiene dos partes: `prefijo/nombre` (el prefijo es opcional).

```
  [prefijo/]nombre

  prefijo:  DNS subdomain, max 253 chars (opcional)
  nombre:   max 63 chars, alfanumérico al inicio y final
            permite: a-z, 0-9, A-Z, -, _, .
```

| Componente | Regla | Ejemplo |
|------------|-------|---------|
| **Sin prefijo** | Privado al usuario | `app`, `tier`, `env` |
| **Con prefijo** | Componentes automatizados | `app.kubernetes.io/name` |
| **Prefijos reservados** | Solo K8s core | `kubernetes.io/`, `k8s.io/` |

#### Values

- Máximo 63 caracteres (puede estar vacío)
- Si no está vacío: empieza y termina con alfanumérico
- Permite: `a-z`, `0-9`, `A-Z`, `-`, `_`, `.`

### Ejemplos comunes

```yaml
# Por entorno
"environment": "dev"      # "qa", "staging", "production"

# Por aplicación
"app": "api-gateway"
"tier": "frontend"        # "backend", "cache", "database"

# Por release
"release": "stable"       # "canary", "beta"

# Por equipo
"team": "payments"
"partition": "customerA"

# Por tracking
"track": "daily"          # "weekly"
```

## 3. Annotations vs Labels

Ambos son key/value en `metadata`, pero tienen propósitos muy diferentes:

| Aspecto | Labels | Annotations |
|---------|--------|-------------|
| **Propósito** | Identificar y seleccionar | Almacenar metadatos no identificatorios |
| **Usados por selectores** | ✅ Sí | ❌ No |
| **Restricciones de valor** | Max 63 chars, alfanuméricos | Sin restricción de tamaño ni formato |
| **Optimizados para queries** | ✅ Sí (indexados) | ❌ No |
| **Uso típico** | `app: nginx`, `env: prod` | Timestamps, commit SHAs, descripciones |

```yaml
metadata:
  labels:                              # Para seleccionar
    app: nginx
    env: production
  annotations:                         # Para informar
    description: "API gateway principal"
    git-commit: "a1b2c3d4e5f6"
    deploy-timestamp: "2026-04-25T10:30:00Z"
    contact: "team-platform@company.com"
```

> 💡 **Regla**: si Kubernetes o tus selectores necesitan encontrar el recurso por ese dato → **label**. Si es información para humanos o herramientas externas → **annotation**.

## 4. Label Selectors

### ¿Qué son?

Los selectores permiten **identificar un conjunto de objetos** basándose en sus labels. Son el **mecanismo de agrupación fundamental** en Kubernetes.

### Dos tipos de selectores

#### 1. Equality-based (igualdad)

Operadores: `=`, `==` (sinónimos), `!=`

```bash
# Pods en producción
kubectl get pods -l environment=production

# Pods que NO son frontend
kubectl get pods -l tier!=frontend

# Combinación (AND implícito con coma)
kubectl get pods -l environment=production,tier!=frontend
```

#### 2. Set-based (conjuntos)

Operadores: `in`, `notin`, `exists`, `!exists`

```bash
# Pods en producción O qa
kubectl get pods -l 'environment in (production, qa)'

# Pods que NO son frontend ni backend
kubectl get pods -l 'tier notin (frontend, backend)'

# Pods que TIENEN el label "partition" (cualquier valor)
kubectl get pods -l partition

# Pods que NO tienen el label "partition"
kubectl get pods -l '!partition'

# Combinación
kubectl get pods -l 'partition in (customerA, customerB),environment!=qa'
```

> ⚠️ **No existe operador OR** entre condiciones separadas por coma. La coma siempre es **AND**.

### Selectores en YAML

#### Recursos antiguos (Service, ReplicationController): solo equality

```yaml
# Service
spec:
  selector:
    component: redis          # Equality-based implícito
```

#### Recursos nuevos (Deployment, ReplicaSet, Job, DaemonSet): set-based

```yaml
spec:
  selector:
    matchLabels:               # Equality-based
      app: webapp
      tier: frontend
    matchExpressions:          # Set-based
    - key: environment
      operator: In
      values: ["production", "staging"]
    - key: track
      operator: NotIn
      values: ["canary"]
    - key: partition
      operator: Exists
```

> 💡 `matchLabels` y `matchExpressions` se combinan con **AND** — todas las condiciones deben cumplirse.

### En la API REST (query parameters)

```
# Equality-based
?labelSelector=environment%3Dproduction,tier%3Dfrontend

# Set-based
?labelSelector=environment+in+%28production%2Cqa%29%2Ctier+in+%28frontend%29
```

## 5. Labels como pegamento entre recursos

Los labels conectan Deployments → ReplicaSets → Pods → Services:

```yaml
# ═══════════════════════════════════════════
# Deployment: define qué Pods crear
# ═══════════════════════════════════════════
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp
  labels:
    app: webapp                    # Label del Deployment mismo
spec:
  replicas: 3
  selector:
    matchLabels:
      app: webapp                  # ◄── "Busca Pods con app=webapp"
      tier: frontend
  template:
    metadata:
      labels:
        app: webapp                # ◄── Labels de los Pods creados
        tier: frontend             #     (DEBEN coincidir con selector)
    spec:
      containers:
      - name: webapp
        image: webapp:2.0

---
# ═══════════════════════════════════════════
# Service: enruta tráfico a los Pods
# ═══════════════════════════════════════════
apiVersion: v1
kind: Service
metadata:
  name: webapp-svc
spec:
  selector:
    app: webapp                    # ◄── "Enruta a Pods con app=webapp"
    tier: frontend                 #     (DEBE coincidir con template labels)
  ports:
  - port: 80
    targetPort: 8080
```

```
  Deployment (selector: app=webapp, tier=frontend)
       │
       │ crea
       ▼
  ReplicaSet (selector heredado)
       │
       │ crea
       ▼
  Pod (labels: app=webapp, tier=frontend)
       ▲
       │ selecciona
       │
  Service (selector: app=webapp, tier=frontend)
```

> ⚠️ **Los selectores en Deployments son inmutables** — no puedes cambiarlos después de la creación. Planifica tus labels antes de crear el recurso.

## 6. Recommended Labels

Kubernetes define labels estándar con el prefijo `app.kubernetes.io/` que facilitan la interoperabilidad entre herramientas (Helm, ArgoCD, dashboards, etc.):

| Key | Descripción | Ejemplo |
|-----|-------------|---------|
| `app.kubernetes.io/name` | Nombre de la aplicación | `mysql` |
| `app.kubernetes.io/instance` | Instancia única de la app | `mysql-abcxyz` |
| `app.kubernetes.io/version` | Versión actual | `5.7.21` |
| `app.kubernetes.io/component` | Componente en la arquitectura | `database` |
| `app.kubernetes.io/part-of` | Aplicación de nivel superior | `wordpress` |
| `app.kubernetes.io/managed-by` | Herramienta de gestión | `Helm` |

### Ejemplo: WordPress con MySQL

```yaml
# WordPress Deployment
metadata:
  labels:
    app.kubernetes.io/name: wordpress
    app.kubernetes.io/instance: wordpress-prod
    app.kubernetes.io/version: "6.4"
    app.kubernetes.io/component: frontend
    app.kubernetes.io/part-of: wordpress
    app.kubernetes.io/managed-by: Helm

---
# MySQL StatefulSet
metadata:
  labels:
    app.kubernetes.io/name: mysql
    app.kubernetes.io/instance: mysql-wordpress-prod
    app.kubernetes.io/version: "8.0"
    app.kubernetes.io/component: database
    app.kubernetes.io/part-of: wordpress           # ◄── Parte de WordPress
    app.kubernetes.io/managed-by: Helm
```

> 💡 Estos labels son **recomendados, no obligatorios**. Pero usarlos permite que herramientas como Helm, ArgoCD y dashboards entiendan y visualicen tu aplicación automáticamente.

## 7. Gestión de labels con kubectl

### Añadir y modificar labels

```bash
# Añadir label a un pod
kubectl label pod my-pod tier=frontend

# Añadir label a TODOS los pods con app=nginx
kubectl label pods -l app=nginx tier=frontend

# Sobrescribir label existente (requiere --overwrite)
kubectl label pod my-pod tier=backend --overwrite

# Eliminar label (sintaxis key-)
kubectl label pod my-pod tier-
```

### Consultar con labels

```bash
# Filtrar por label
kubectl get pods -l app=nginx

# Mostrar labels como columnas
kubectl get pods -L app -L tier -L role
# NAME                    READY  STATUS   APP        TIER       ROLE
# guestbook-fe-4nlpb      1/1    Running  guestbook  frontend   <none>
# guestbook-redis-master  1/1    Running  guestbook  backend    master
# guestbook-redis-replica 1/1    Running  guestbook  backend    replica

# Mostrar todos los labels
kubectl get pods --show-labels

# Aplicar YAML solo a recursos con cierto label
kubectl apply -f manifests/ -l app=nginx
```

### Slicing multidimensional

```bash
# Solo frontend
kubectl get pods -l tier=frontend

# Solo replicas de redis
kubectl get pods -l app=guestbook,role=replica

# Todo excepto frontend
kubectl get pods -l 'tier notin (frontend)'
```

## 8. Scheduling: control de ubicación de Pods

Los labels también controlan **en qué nodos** se ejecutan los Pods.

### Node labels

Kubernetes asigna labels estándar a los nodos:

```bash
kubectl get nodes --show-labels
# kubernetes.io/hostname=node-1
# kubernetes.io/os=linux
# kubernetes.io/arch=amd64
# topology.kubernetes.io/zone=us-east-1a
# topology.kubernetes.io/region=us-east-1
```

```bash
# Añadir label custom a un nodo
kubectl label node node-1 disktype=ssd
kubectl label node node-2 gpu=nvidia-tesla-p100
```

#### Node isolation/restriction (seguridad)

Si usas labels para aislar nodos (ej: nodos PCI-DSS, FIPS), elige keys que el **kubelet no pueda modificar**. Esto previene que un nodo comprometido se asigne labels a sí mismo para atraer workloads.

El admission plugin **NodeRestriction** impide que el kubelet modifique labels con el prefijo `node-restriction.kubernetes.io/`:

```bash
# Labels seguros para aislamiento de nodos
kubectl label node secure-node-1 example.com.node-restriction.kubernetes.io/fips=true
kubectl label node secure-node-2 example.com.node-restriction.kubernetes.io/pci-dss=true
```

> 💡 Requiere usar el **Node authorizer** y tener habilitado el **NodeRestriction admission plugin**.

### Métodos de asignación de Pods a Nodes

Kubernetes ofrece varios mecanismos, de menor a mayor expresividad:

| Método | Descripción | Uso recomendado |
|--------|-------------|-----------------|
| `nodeName` | Asignación directa (bypassa el scheduler) | Casos avanzados, schedulers custom |
| `nodeSelector` | Match exacto de labels | Caso simple y común |
| `nodeAffinity` | Reglas required/preferred con operadores | Reglas complejas con fallbacks |
| `podAffinity/Anti-Affinity` | Basado en labels de otros Pods | Co-localización / distribución |
| `Taints/Tolerations` | Nodo repele pods sin toleration | Nodos dedicados, hardware especial |
| Topology Spread Constraints | Distribución uniforme por dominio | HA entre zonas/regiones |

### nodeSelector (más simple)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: gpu-pod
spec:
  nodeSelector:
    gpu: nvidia-tesla-p100       # Solo nodos con este label
  containers:
  - name: cuda
    image: nvidia/cuda:12.0
```

### Node Affinity (más expresivo)

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: with-node-affinity
spec:
  affinity:
    nodeAffinity:
      # OBLIGATORIO: debe cumplirse para schedular
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: topology.kubernetes.io/zone
            operator: In
            values:
            - us-east-1a
            - us-east-1b
      # PREFERIDO: intenta pero no bloquea si no se cumple
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 80                # Peso 1-100
        preference:
          matchExpressions:
          - key: disktype
            operator: In
            values:
            - ssd
  containers:
  - name: nginx
    image: nginx
```

#### Operadores disponibles

| Operador | Descripción | Ejemplo |
|----------|-------------|---------|
| `In` | Valor está en el conjunto | `zone In [us-east-1a, us-east-1b]` |
| `NotIn` | Valor NO está en el conjunto | `env NotIn [dev, staging]` |
| `Exists` | El key existe (cualquier valor) | `gpu Exists` |
| `DoesNotExist` | El key no existe | `temporary DoesNotExist` |
| `Gt` | Mayor que (numérico) | `cores Gt 4` |
| `Lt` | Menor que (numérico) | `cores Lt 16` |

#### Reglas de combinación

```
nodeSelectorTerms: [term1, term2]    → OR  (cualquiera satisface)
matchExpressions:  [expr1, expr2]    → AND (todas deben satisfacerse)
nodeSelector + nodeAffinity          → AND (ambos deben satisfacerse)
```

#### `IgnoredDuringExecution`

El nombre completo `requiredDuringScheduling**IgnoredDuringExecution**` significa:

- **DuringScheduling**: la regla se evalúa cuando el Pod se está schedulando
- **IgnoredDuringExecution**: si los labels del nodo **cambian después** de que el Pod fue schedulado, el Pod **sigue ejecutándose** — no se expulsa

> 💡 Actualmente no existe `RequiredDuringExecution` — es un feature planificado que expulsaría Pods si las condiciones dejan de cumplirse.

#### Node Affinity Weight (scoring)

Para `preferred` rules puedes asignar **pesos entre 1 y 100**. El scheduler suma los pesos de todas las reglas satisfied y elige el nodo con mayor puntuación total:

```yaml
preferredDuringSchedulingIgnoredDuringExecution:
- weight: 1                         # Baja prioridad
  preference:
    matchExpressions:
    - key: label-1
      operator: In
      values: ["key-1"]
- weight: 50                        # Alta prioridad (50x más importante)
  preference:
    matchExpressions:
    - key: label-2
      operator: In
      values: ["key-2"]
```

Si hay dos nodos candidatos: uno con `label-1` (score +1) y otro con `label-2` (score +50), el scheduler elige el segundo.

### Pod Affinity y Anti-Affinity

Controla la ubicación de Pods basándote en **labels de otros Pods**, no de nodos:

```yaml
spec:
  affinity:
    # Co-localizar con pods que tienen app=cache
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
          - key: app
            operator: In
            values:
            - cache
        topologyKey: topology.kubernetes.io/zone

    # NO co-localizar con pods que tienen app=webapp
    podAntiAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        podAffinityTerm:
          labelSelector:
            matchExpressions:
            - key: app
              operator: In
              values:
              - webapp
          topologyKey: kubernetes.io/hostname
```

**Casos de uso**:

| Tipo | Caso de uso |
|------|-------------|
| **Pod Affinity** | Co-localizar API + cache en la misma zona para baja latencia |
| **Pod Anti-Affinity** | Distribuir réplicas en diferentes nodos para alta disponibilidad |

> ⚠️ Pod affinity/anti-affinity requiere **procesamiento intensivo** — no recomendado en clústers con más de varios cientos de nodos.

> ⚠️ Pod anti-affinity requiere que **todos los nodos** tengan el label correspondiente al `topologyKey`. Si faltan labels, puede producir comportamiento inesperado.

#### Ejemplo práctico: Redis cache + Web server

Un caso real donde affinity y anti-affinity trabajan juntos. Objetivo: co-localizar cada web server con un cache Redis en el **mismo nodo**, pero distribuir las réplicas en **nodos diferentes**:

```yaml
# ═══ Redis Cache: una réplica por nodo ═══
apiVersion: apps/v1
kind: Deployment
metadata:
  name: redis-cache
spec:
  replicas: 3
  selector:
    matchLabels:
      app: store
  template:
    metadata:
      labels:
        app: store
    spec:
      affinity:
        podAntiAffinity:
          # No poner dos caches en el mismo nodo
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values: ["store"]
            topologyKey: "kubernetes.io/hostname"
      containers:
      - name: redis-server
        image: redis:3.2-alpine
---
# ═══ Web Server: junto al cache, distribuido entre nodos ═══
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-server
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-store
  template:
    metadata:
      labels:
        app: web-store
    spec:
      affinity:
        # Co-localizar con Redis (mismo nodo)
        podAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values: ["store"]
            topologyKey: "kubernetes.io/hostname"
        # No poner dos web servers en el mismo nodo
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
          - labelSelector:
              matchExpressions:
              - key: app
                operator: In
                values: ["web-store"]
            topologyKey: "kubernetes.io/hostname"
      containers:
      - name: web-app
        image: nginx:1.16-alpine
```

Resultado en un clúster de 3 nodos:

```
  node-1              node-2              node-3
  ┌───────────────┐   ┌───────────────┐   ┌───────────────┐
  │ web-server-1  │   │ web-server-2  │   │ web-server-3  │
  │ redis-cache-1 │   │ redis-cache-2 │   │ redis-cache-3 │
  └───────────────┘   └───────────────┘   └───────────────┘

  Cada web server accede a su cache local → mínima latencia
```

### nodeName (asignación directa)

`nodeName` es la forma más directa: asigna el Pod a un nodo específico **sin pasar por el scheduler**:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  nodeName: kube-01              # Bypassa el scheduler completamente
  containers:
  - name: nginx
    image: nginx
```

#### Limitaciones importantes

| Limitación | Detalle |
|------------|---------|
| Nodo no existe | El Pod no se ejecuta y puede ser eliminado automáticamente |
| Sin recursos suficientes | El Pod falla con `OutOfMemory` o `OutOfCpu` |
| Nombres inestables | En cloud, los nombres de nodo no son predecibles |
| Sin scheduler | No hay balanceo de carga, puede sobrecargar el nodo |

> ⚠️ `nodeName` está pensado para **schedulers custom** o casos muy avanzados. Para casos normales, usa `nodeSelector` o `nodeAffinity`.

### Pod Topology Spread Constraints

Controla cómo se **distribuyen los Pods** entre dominios de fallo (regiones, zonas, nodos):

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: webapp
spec:
  replicas: 6
  template:
    spec:
      topologySpreadConstraints:
      - maxSkew: 1                               # Diferencia máxima entre dominios
        topologyKey: topology.kubernetes.io/zone  # Distribuir entre zonas
        whenUnsatisfiable: DoNotSchedule          # Hard constraint
        labelSelector:
          matchLabels:
            app: webapp
      containers:
      - name: webapp
        image: webapp:2.0
```

| Campo | Descripción |
|-------|-------------|
| `maxSkew` | Diferencia máxima permitida de pods entre dominios |
| `topologyKey` | Label del nodo que define el dominio (`zone`, `hostname`, etc.) |
| `whenUnsatisfiable` | `DoNotSchedule` (hard) o `ScheduleAnyway` (soft) |

Resultado con `maxSkew: 1` entre 3 zonas y 6 réplicas:

```
  zone-a: 2 pods    zone-b: 2 pods    zone-c: 2 pods
  (diferencia máxima entre zonas = 0, cumple maxSkew=1)
```

> 💡 Los Pods heredan automáticamente los labels `topology.kubernetes.io/zone` y `topology.kubernetes.io/region` de su nodo asignado.

### Taints y Tolerations

Mientras que **node affinity atrae** Pods hacia ciertos nodos, los **taints repelen** Pods de los nodos. Son mecanismos complementarios:

```
  Node Affinity                    Taints & Tolerations
  ─────────────                    ─────────────────────

  Pod ──────────► Node             Node ──────────✕ Pod
  "Quiero ir a                     "No acepto pods
   nodos con SSD"                   sin toleration"

                                   Pod + Toleration ──► Node ✓
                                   "Tengo permiso
                                    para entrar"
```

#### Concepto

- **Taint**: se aplica a un **nodo** — marca que el nodo rechaza pods que no toleren el taint
- **Toleration**: se aplica a un **pod** — permite que el pod se schedule en nodos con el taint correspondiente

> ⚠️ Una toleration **permite** el scheduling pero **no lo garantiza** — el scheduler evalúa otros parámetros también.

#### Gestión de taints

```bash
# Añadir taint a un nodo
kubectl taint nodes node1 key1=value1:NoSchedule

# Eliminar taint (nota el - al final)
kubectl taint nodes node1 key1=value1:NoSchedule-

# Ver taints de un nodo
kubectl describe node node1 | grep -i taint
```

#### Efectos de taint

| Efecto | Comportamiento |
|--------|----------------|
| `NoSchedule` | **No** se schedulan nuevos pods sin toleration. Pods existentes **no se afectan** |
| `PreferNoSchedule` | Versión **soft** — el scheduler **intenta** evitar el nodo, pero no está garantizado |
| `NoExecute` | Pods sin toleration son **expulsados inmediatamente**. Nuevos pods tampoco se schedulan |

#### Tolerations en el Pod spec

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx
  tolerations:
  # Operador "Equal": key + value + effect deben coincidir
  - key: "key1"
    operator: "Equal"
    value: "value1"
    effect: "NoSchedule"

  # Operador "Exists": solo key + effect (ignora value)
  - key: "key1"
    operator: "Exists"
    effect: "NoSchedule"
```

> 💡 El operador por defecto es `Equal`. Con `Exists` no se especifica `value`.

#### Reglas de matching

Una toleration "coincide" con un taint si:
- Los **keys** son iguales
- Los **effects** son iguales
- El operador es `Exists` (cualquier valor), o `Equal` con el mismo value

**Casos especiales**:
- Key vacío + `Exists` → coincide con **todos** los taints
- Effect vacío → coincide con **todos los effects** de ese key

#### Procesamiento de múltiples taints

Kubernetes procesa los taints como un **filtro**:

```
  Nodo con 3 taints          Pod con 2 tolerations
  ─────────────────          ─────────────────────
  key1=v1:NoSchedule    ◄──  toleration key1=v1:NoSchedule    ✓ Match
  key1=v1:NoExecute     ◄──  toleration key1=v1:NoExecute     ✓ Match
  key2=v2:NoSchedule         (sin toleration)                 ✗ No match
                                    │
                                    ▼
                         Taint restante: key2=v2:NoSchedule
                         Efecto: Pod NO se schedula en este nodo
```

#### NoExecute con `tolerationSeconds`

Permite que un pod **permanezca temporalmente** en un nodo tainted:

```yaml
tolerations:
- key: "key1"
  operator: "Equal"
  value: "value1"
  effect: "NoExecute"
  tolerationSeconds: 3600    # Permanece 1 hora, luego es expulsado
```

Si el taint se elimina antes de que expire el tiempo, el pod **no** es expulsado.

#### Taints automáticos del sistema (Taint-based Evictions)

El node controller añade taints automáticamente cuando detecta problemas:

| Taint automático | Condición |
|-----------------|-----------|
| `node.kubernetes.io/not-ready` | Node no está Ready (`Ready=False`) |
| `node.kubernetes.io/unreachable` | Node inalcanzable (`Ready=Unknown`) |
| `node.kubernetes.io/memory-pressure` | Presión de memoria |
| `node.kubernetes.io/disk-pressure` | Presión de disco |
| `node.kubernetes.io/pid-pressure` | Presión de PIDs |
| `node.kubernetes.io/network-unavailable` | Red no disponible |
| `node.kubernetes.io/unschedulable` | Nodo marcado como unschedulable |

> 💡 Kubernetes añade automáticamente tolerations para `not-ready` y `unreachable` con `tolerationSeconds=300` (5 minutos) a todos los pods. Los **DaemonSet pods** reciben tolerations **sin** `tolerationSeconds` para que nunca sean expulsados.

#### Casos de uso típicos

| Caso de uso | Implementación |
|-------------|----------------|
| **Nodos dedicados** | `kubectl taint nodes node1 dedicated=team-gpu:NoSchedule` + toleration en pods del equipo + nodeSelector para garantizar exclusividad |
| **Hardware especial (GPUs)** | Taint en nodos GPU + toleration automática via admission controller (`ExtendedResourceToleration`) |
| **Mantenimiento de nodo** | `kubectl taint nodes node1 maintenance=true:NoExecute` → pods migran automáticamente |
| **Evicción controlada** | `tolerationSeconds` en pods con estado local para dar tiempo de migración |

#### Ejemplo completo: nodos GPU dedicados

```bash
# 1. Taint en nodos GPU
kubectl taint nodes gpu-node-1 nvidia.com/gpu=true:NoSchedule
kubectl label nodes gpu-node-1 hardware=gpu
```

```yaml
# 2. Pod que necesita GPU
apiVersion: v1
kind: Pod
metadata:
  name: ml-training
spec:
  nodeSelector:
    hardware: gpu                           # Ir a nodos GPU
  tolerations:
  - key: "nvidia.com/gpu"                   # Tolerar el taint
    operator: "Exists"
    effect: "NoSchedule"
  containers:
  - name: trainer
    image: tensorflow/tensorflow:latest-gpu
    resources:
      limits:
        nvidia.com/gpu: 1
```

### Comparación de mecanismos de scheduling

```
  Simplicidad ──────────────────────────────────────────► Expresividad

  nodeSelector     nodeAffinity        Taints/           podAffinity/
  (match exacto)   (In, NotIn,         Tolerations       Anti-Affinity
                    Exists, Gt, Lt)     (repeler pods)    (topologyKey)
  ┌──────────┐    ┌──────────────┐    ┌──────────────┐  ┌──────────────┐
  │ disktype │    │ required +   │    │ Nodo repele  │  │ Basado en    │
  │ = ssd    │    │ preferred    │    │ pods sin     │  │ labels de    │
  │          │    │ + weights    │    │ toleration   │  │ OTROS pods   │
  └──────────┘    └──────────────┘    └──────────────┘  └──────────────┘
    ATRAE pods      ATRAE pods         REPELE pods       ATRAE/REPELE
    a nodos         a nodos            de nodos          entre pods
```

## 9. Buenas prácticas

| Práctica | Razón |
|----------|-------|
| Usar `app.kubernetes.io/*` labels | Interoperabilidad con Helm, dashboards, etc. |
| Definir labels de selector como **inmutables** | No pueden cambiarse después en Deployments |
| Un label de PodTemplate **solo** para el selector | Sin semántica adicional, solo binding |
| Labels para identificar, annotations para informar | Labels indexados y optimizados; annotations libres |
| Planificar esquema de labels antes de crear recursos | Difícil cambiar selectores después |
| Usar labels multidimensionales | `app`, `tier`, `env`, `version` para filtrado flexible |
| Usar anti-affinity para distribuir réplicas | Alta disponibilidad en diferentes nodos/zonas |

## 10. Errores comunes

| # | Error | Consecuencia | Solución |
|---|-------|-------------|----------|
| 1 | Labels del template no coinciden con selector | El Deployment no puede gestionar sus Pods | Asegurar que `selector.matchLabels` ⊆ `template.metadata.labels` |
| 2 | Selectores de dos ReplicaSets se solapan | Controller confundido sobre cuántas réplicas mantener | Labels de selector únicos por controlador |
| 3 | Usar labels para datos no identificatorios | Limitados a 63 chars, sin formato libre | Usar annotations para ese tipo de datos |
| 4 | Intentar modificar selector de un Deployment | Error de la API — son inmutables | Crear nuevo Deployment, migrar tráfico, eliminar el viejo |
| 5 | No especificar `topologyKey` en pod affinity | Error de validación — es obligatorio | Definir siempre `topologyKey` (`kubernetes.io/hostname`, `topology.kubernetes.io/zone`) |
| 6 | Usar pod affinity en clústers grandes | Degradación de rendimiento del scheduler | Preferir nodeSelector/nodeAffinity cuando sea suficiente |
