## 1. Introducción

Kubernetes ofrece **tres formas distintas** de gestionar objetos (Pods, Deployments, Services, etc.). Elegir la correcta depende del contexto: aprendizaje, desarrollo local, producción, o automatización a escala.

```
Complejidad ──────────────────────────────────────►

  Imperative         Imperative          Declarative
  Commands           Objects             Objects
  ┌──────────┐      ┌──────────┐        ┌──────────┐
  │ kubectl  │      │ kubectl  │        │ kubectl  │
  │ run/     │      │ create   │        │ apply    │
  │ create/  │  ──► │ replace  │  ────► │ -f dir/  │
  │ expose   │      │ -f file  │        │          │
  └──────────┘      └──────────┘        └──────────┘
  Dev/Learning      Staging/Small Prod   Production
```

> ⚠️ **Regla fundamental**: Nunca mezcles los tres enfoques para gestionar el **mismo objeto**. Kubernetes almacena metadatos internos que difieren según el método usado, y mezclarlos produce comportamiento indefinido.

## 2. Los tres enfoques de gestión

### Vista rápida

| Aspecto | Imperative Commands | Imperative Objects | Declarative Objects |
|---------|--------------------|--------------------|---------------------|
| **Opera sobre** | Objetos live | Archivos individuales | Directorios de archivos |
| **Entorno recomendado** | Desarrollo | Proyectos pequeños/staging | Producción |
| **Writers soportados** | 1+ | 1 | 1+ |
| **Curva de aprendizaje** | Más baja | Moderada | Más alta |
| **Historial de cambios** | ❌ No | ✅ En archivos | ✅ En archivos + anotaciones |
| **Automatizable** | ❌ Difícil | ⚠️ Parcial | ✅ Ideal |

### Analogía: preparar café ☕

| Enfoque | Equivalente |
|---------|-------------|
| **Imperativo** | "Hiervo agua, muelo 42g de café, vierto 700g de agua..." — yo controlo cada paso |
| **Declarativo** | "Barista, quiero un café" — el barista (Kubernetes) decide los pasos, reintenta si falla, y solo termina cuando tengo mi café |

La diferencia fundamental:
- **Imperativo**: te enfocas en **cómo** se hace
- **Declarativo**: te enfocas en **qué** quieres que exista

## 3. Comandos imperativos

Operas **directamente sobre objetos live** en el clúster, sin archivos intermedios.

### Comandos principales

```bash
# ═══ CREAR ═══
kubectl run nginx --image=nginx                        # Pod
kubectl create deployment nginx --image=nginx          # Deployment
kubectl expose deployment nginx --port=80              # Service
kubectl create service nodeport my-svc --tcp=80:80     # Service NodePort
kubectl autoscale deployment nginx --min=2 --max=5     # HPA

# ═══ ACTUALIZAR ═══
kubectl scale deployment nginx --replicas=3            # Escalar
kubectl set image deploy/nginx nginx=nginx:1.25        # Cambiar imagen
kubectl label pods my-pod tier=frontend                # Añadir label
kubectl annotate pods my-pod desc="API server"         # Añadir anotación
kubectl edit deployment nginx                          # Editar en editor
kubectl patch deploy nginx -p '{"spec":{"replicas":5}}'# Patch directo

# ═══ ELIMINAR ═══
kubectl delete deployment nginx
kubectl delete pod my-pod --grace-period=0 --force     # Forzar eliminación
```

### Ventajas

- Un solo comando = una acción completa
- Sin necesidad de escribir YAML
- Ideal para exploración y aprendizaje
- Rápido para tareas puntuales (one-off tasks)

### Desventajas

- Sin integración con revisión de cambios (code review)
- Sin audit trail — no queda registro de qué se hizo
- Sin plantilla para recrear el mismo objeto
- Imposible reproducir el estado exacto del clúster

## 4. Configuración imperativa de objetos

Operas sobre **archivos individuales** con comandos explícitos (`create`, `replace`, `delete`).

### Comandos principales

```bash
# Crear desde archivo
kubectl create -f nginx.yaml

# Actualizar (reemplaza TODO el spec)
kubectl replace -f nginx.yaml

# Eliminar
kubectl delete -f nginx.yaml -f redis.yaml

# Ver configuración actual
kubectl get -f nginx.yaml -o yaml
```

### ⚠️ Peligro de `replace`

`replace` **reemplaza el spec completo** del objeto con lo que contiene el archivo. Cualquier campo que no esté en el archivo se **elimina**:

```
                    Archivo YAML          Objeto Live
                    ┌────────────┐        ┌────────────┐
                    │ replicas: 3│        │ replicas: 3│
                    │ image: v2  │        │ image: v1  │
                    │            │        │ HPA: min=2 │ ◄── Este campo
                    └────────────┘        └────────────┘     se PIERDE
                                   │
                          kubectl replace
                                   │
                                   ▼
                             ┌────────────┐
                             │ replicas: 3│
                             │ image: v2  │
                             │            │ ◄── HPA desapareció
                             └────────────┘
```

> ❌ **No uses** `replace` con recursos cuyos specs se actualizan independientemente (ej: Services tipo LoadBalancer con `externalIPs` gestionado por el cloud provider).

### Ventajas vs. comandos imperativos

- Los archivos se almacenan en Git → historial + code review
- Sirven como plantilla para crear nuevos objetos
- Más explícito sobre la intención

### Desventajas vs. declarativo

- Funciona mejor con archivos individuales, no directorios
- Los cambios live que no se reflejan en el archivo se **pierden** en el siguiente `replace`
- Requiere conocer qué comando usar (`create` vs `replace`)

## 5. Configuración declarativa de objetos

Operas sobre **archivos o directorios**, y `kubectl apply` detecta automáticamente si debe crear, actualizar o eliminar.

### Comandos principales

```bash
# Crear o actualizar desde archivo
kubectl apply -f nginx.yaml

# Crear o actualizar todo un directorio
kubectl apply -f manifests/

# Recursivo
kubectl apply -R -f manifests/

# Desde URL (inspeccionar primero)
curl -L https://bret.run/pod                    # Ver antes de aplicar
kubectl apply -f https://bret.run/pod.yml

# Ver diferencias antes de aplicar
kubectl diff -f manifests/

# Eliminar (recomendado: explícito)
kubectl delete -f nginx.yaml

# Eliminar (alternativa: prune)
kubectl apply -f manifests/ --prune -l app=myapp
```

### Cómo funciona internamente

`kubectl apply` usa **three-way merge** con tres fuentes de información:

```
    ┌─────────────────────┐
    │   Archivo YAML      │ ◄── Lo que TÚ declaras
    │   (desired state)   │
    └────────┬────────────┘
             │
             ▼
    ┌─────────────────────┐     ┌──────────────────────────────┐
    │   last-applied-     │     │     Live Configuration       │
    │   configuration     │     │     (estado actual en etcd)  │
    │   (anotación)       │     │                              │
    └────────┬────────────┘     └──────────────┬───────────────┘
             │                                  │
             └──────────┬───────────────────────┘
                        │
                        ▼
               ┌─────────────────┐
               │   Merge Patch   │
               │   Calculation   │
               └────────┬────────┘
                        │
                        ▼
               ┌─────────────────┐
               │   API Server    │
               │   (PATCH req)   │
               └─────────────────┘
```

La anotación `kubectl.kubernetes.io/last-applied-configuration` almacena el contenido del YAML de la última vez que se usó `apply`. Esto permite:

1. **Detectar campos eliminados**: si un campo está en `last-applied` pero no en el archivo actual → se elimina del live
2. **Detectar campos añadidos/modificados**: si un campo está en el archivo pero difiere del live → se actualiza
3. **Preservar campos de otros writers**: si un campo fue modificado por `kubectl scale` y no está en el archivo → se conserva

### Ejemplo práctico del three-way merge

```yaml
# 1. Estado inicial — aplicamos este YAML
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 3
  minReadySeconds: 10
  template:
    spec:
      containers:
      - name: nginx
        image: nginx:1.14.2
```

```bash
# 2. Alguien escala manualmente
kubectl scale deployment/nginx --replicas=5
# replicas=5 en live, pero last-applied sigue con replicas=3
```

```yaml
# 3. Actualizamos el YAML: cambiamos imagen y quitamos minReadySeconds
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 3
  template:
    spec:
      containers:
      - name: nginx
        image: nginx:1.25.0    # ◄── Cambió
  # minReadySeconds eliminado   ◄── Eliminado
```

```bash
kubectl apply -f nginx.yaml
```

Resultado:

| Campo | Acción | Razón |
|-------|--------|-------|
| `image` | `nginx:1.14.2` → `nginx:1.25.0` | Presente en archivo con valor diferente al live |
| `minReadySeconds` | **Eliminado** del live | Estaba en `last-applied` pero no en el archivo actual |
| `replicas` | **Se mantiene en 5** | El archivo dice 3, pero como `last-applied` también dice 3, y el live dice 5, se preserva el cambio externo* |

> *\*Nota: en la práctica, `replicas` se sobrescribirá a 3 porque está presente en el archivo. Para preservar el valor del `scale`, debes **omitir** `replicas` del archivo YAML.*

### Ventajas

- Preserva cambios de otros writers (kubectl scale, HPA, etc.)
- Soporte nativo para directorios completos
- Detección automática de operaciones (crear/actualizar)
- Ideal para automatización y CI/CD

### Desventajas

- Más difícil debuggear cuando el resultado es inesperado
- Merge de patches puede ser complejo
- Requiere entender el modelo de three-way merge

## 6. Comparación detallada

### Flujo de trabajo por enfoque

```
IMPERATIVE COMMANDS
───────────────────
  kubectl create deployment nginx --image=nginx
  kubectl scale deployment nginx --replicas=3
  kubectl set image deploy/nginx nginx=nginx:1.25
  → Cada cambio es un comando diferente
  → No queda registro

IMPERATIVE OBJECTS
──────────────────
  vim nginx.yaml                    # Editar archivo
  kubectl replace -f nginx.yaml     # Aplicar
  → replace sobreescribe TODO
  → Cambios externos se pierden

DECLARATIVE OBJECTS
───────────────────
  vim nginx.yaml                    # Editar archivo
  kubectl diff -f nginx.yaml        # Revisar cambios
  kubectl apply -f nginx.yaml       # Aplicar
  → Solo cambia lo necesario
  → Preserva cambios externos
  → Mismo comando siempre
```

### ¿Cuándo usar cada uno?

| Escenario | Enfoque recomendado |
|-----------|-------------------|
| Aprendiendo Kubernetes | Imperative commands |
| Debugging rápido | Imperative commands |
| Proyecto personal / POC | Imperative commands |
| Pequeño equipo, pocos recursos | Imperative objects |
| Producción | Declarative objects |
| CI/CD pipelines | Declarative objects |
| GitOps | Declarative objects |
| Migración entre enfoques | Ver sección 9 |

## 7. Resource Generators

Los comandos imperativos de creación usan internamente **generators**: plantillas que producen la especificación YAML del recurso.

### Ver el YAML que se generaría

```bash
# Deployment
kubectl create deployment test --image=nginx --dry-run=client -o yaml

# Job
kubectl create job test --image=nginx --dry-run=client -o yaml

# Service (requiere que el deployment exista)
kubectl expose deployment test --port=80 --dry-run=client -o yaml

# CronJob
kubectl create cronjob test --image=busybox --schedule="*/5 * * * *" \
  --dry-run=client -o yaml -- echo "hello"
```

### Flujo recomendado: generar → editar → aplicar

```bash
# 1. Generar plantilla base
kubectl create deployment webapp --image=nginx \
  --dry-run=client -o yaml > webapp.yaml

# 2. Editar para añadir configuración específica
vim webapp.yaml    # Añadir resources, probes, volumes, etc.

# 3. Aplicar de forma declarativa
kubectl apply -f webapp.yaml
```

> 💡 Los generators producen **defaults opinados** — un punto de partida razonable que luego personalizas.

### Combinar generators con `set` (pipeline)

```bash
# Crear Service sin selector → inyectar selector → crear
kubectl create service clusterip my-svc --clusterip="None" \
  -o yaml --dry-run=client | \
  kubectl set selector --local -f - 'environment=qa' -o yaml | \
  kubectl create -f -
```

## 8. GitOps: el destino natural

GitOps es la **evolución natural** del enfoque declarativo: cada cambio en la infraestructura pasa por Git.

```
  Developer          Git Repo           GitOps Agent         Cluster
  ─────────          ────────           ────────────         ───────
      │                  │                   │                  │
      │  git push        │                   │                  │
      ├─────────────────►│                   │                  │
      │                  │   detect change   │                  │
      │                  ├──────────────────►│                  │
      │                  │                   │  kubectl apply   │
      │                  │                   ├─────────────────►│
      │                  │                   │                  │
      │                  │                   │  reconcile loop  │
      │                  │                   │◄────────────────►│
      │                  │                   │                  │
```

### Principios de GitOps

| Principio | Descripción |
|-----------|-------------|
| **Git como fuente de verdad** | El estado deseado del clúster está en Git |
| **Declarativo** | Todo se describe como YAML/configuración |
| **Automatizado** | Los cambios en Git se aplican automáticamente al clúster |
| **Reconciliación continua** | El agente detecta y corrige drift (diferencias) |
| **Auditable** | Cada cambio tiene un commit con autor, fecha y mensaje |

### Herramientas populares

| Herramienta | Modelo | Descripción |
|-------------|--------|-------------|
| **ArgoCD** | Pull | Monitorea Git y sincroniza con el clúster |
| **Flux** | Pull | Nativo de CNCF, integración profunda con K8s |
| **Jenkins X** | Push | CI/CD con GitOps integrado |

### Flujo de trabajo GitOps

```bash
# 1. Editar manifiestos localmente
vim manifests/deployment.yaml

# 2. Revisar cambios
kubectl diff -f manifests/

# 3. Commit a Git
git add manifests/
git commit -m "feat: update nginx to 1.25"
git push

# 4. El agente GitOps detecta el cambio y lo aplica
#    (automático — no necesitas ejecutar kubectl apply)
```

## 9. Ruta de aprendizaje recomendada

### Progresión natural

```
  Fase 1: Aprender              Fase 2: Madurar           Fase 3: Producción
  ───────────────               ──────────────             ─────────────────
  kubectl run/create     ──►    kubectl apply -f file  ──► kubectl apply -f dir/
  kubectl expose                git commit cada cambio     GitOps (ArgoCD/Flux)
  kubectl scale                 kubectl diff               CI/CD pipelines
```

### Migración entre enfoques

#### De imperativo a declarativo

```bash
# 1. Exportar objeto live a YAML
kubectl get deployment/nginx -o yaml > deployment_nginx.yaml

# 2. Limpiar campos auto-generados
#    Eliminar: status, metadata.resourceVersion, metadata.uid,
#    metadata.creationTimestamp, metadata.generation

# 3. Establecer la anotación last-applied-configuration
kubectl replace --save-config -f deployment_nginx.yaml

# 4. A partir de ahora, usar solo apply
kubectl apply -f deployment_nginx.yaml
```

#### De imperativo con objetos a declarativo

```bash
# Solo necesitas establecer la anotación
kubectl replace --save-config -f deployment_nginx.yaml

# Y cambiar a apply
kubectl apply -f deployment_nginx.yaml
```

## 10. Errores comunes

| # | Error | Problema | Solución |
|---|-------|----------|----------|
| 1 | Mezclar `create` y `apply` en el mismo objeto | `create` no escribe `last-applied-configuration` → `apply` no puede calcular diffs | Elige un enfoque y mantente |
| 2 | Usar `replace` con campos gestionados externamente | Se pierden campos como `externalIPs`, `replicas` de HPA | Usa `apply` que hace merge inteligente |
| 3 | No hacer `git commit` antes de `apply` | Sin historial de cambios, no puedes hacer rollback del YAML | Commit → apply → verify |
| 4 | Aplicar sin `kubectl diff` primero | Cambios inesperados en producción | Siempre `diff` antes de `apply` |
| 5 | Usar `--prune` sin `--selector` | Puede eliminar recursos no deseados en todo el namespace | Siempre usar `-l` con `--prune` |
| 6 | No entender three-way merge | Confusión sobre por qué un campo se mantiene o se elimina | Estudiar `last-applied-configuration` |

```
│         GESTIÓN DE OBJETOS EN KUBERNETES             │

```
